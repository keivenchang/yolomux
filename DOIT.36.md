## C1 Details — Complete terminal URLs can false-join with an indented next line
ROOT CAUSE (verified): DOIT.27 added hard-wrapped URL stitching in `terminalWrappedLineGroup` (`static_src/js/yolomux/10_core_utils.js`). The helper `terminalTailIsUnterminatedUrl(text)` returns true when the last URL token ends exactly at the end of the row, and the forward pass then appends the next indented row while that remains true. That catches the intended case (`.../runs/26` + indented `919558600/job`), but it also treats a normal complete URL at end-of-line as "unterminated." A normal transcript like `See https://example.com` followed by `    next step` can become one bogus link `https://example.comnext step`.

FIX: Add a regression for "complete URL at EOL + indented non-URL next row" and make continuation depend on terminal/right-edge evidence, not only string end. Options: require the preceding row to be filled to the terminal width, carry row cell width into `terminalWrappedLineGroup`, or otherwise prove the URL was clipped/hard-wrapped before accepting a hanging continuation. Keep the true soft-wrap path unchanged.

VALIDATE: `terminalWrappedLineLinks` still stitches the known hard-wrap URL across both rows; a standalone complete URL at row end links only that URL; an indented URL on the next row remains a separate link at its own real columns; a non-URL indented continuation is not merged.

## C2 Details — Login auth tests can depend on the developer's real saved locale/config

Prefer setting `YOLOMUX_CONFIG_DIR` and `YOLOMUX_STATE_DIR` at the top of `tests/test_login_auth.py` before any project import, or monkeypatch the already-imported `settings.SETTINGS_PATH`/`web.current_language_pref` path explicitly in `start_server`. Then either assert the localized key via `server_string("en", "login.showPassword")` with a forced English config, or add a separate test proving a `yolomux_locale=zh-Hant` cookie localizes the label.

VALIDATE: `python3 -m pytest tests/test_login_auth.py` passes from a clean shell regardless of the real `~/.config/yolomux/settings.yaml` language; the same file passes when the developer's saved language is `zh-Hant`; the full suite remains green.

## C11 Details — Indexed paths/search state is split across UI settings, localStorage, process memory, and disk
REQUEST (user): Research why indexed paths keep toggling between indexed and not showing, why Search takes a while when a past index should exist, how often the index runs, whether it uses Unix `find`, where it is kept, whether two servers share it today, and how to make multi-server behavior sane.

WHAT EXISTS (verified): The backend indexer is `yolomux_lib/file_index.py`. It is not Unix `find`; it uses Python `os.walk(root, topdown=True, followlinks=False)` in `walk_root`, skips `SEARCH_SKIP_DIRS`, records only regular files, caps at `MAX_INDEX_FILES = 400_000`, and stores entries as `(path, name, relative_path, size, mtime)`. Search fallback also uses Python `os.walk` in `filesystem._search_full_tree`, capped by `MAX_SEARCH_DIRS`/`MAX_SEARCH_FILES`.

WHERE IT IS KEPT (verified): On disk it is JSON under `file_index.INDEX_DIR`, which defaults to `STATE_DIR / "search_index"`; `STATE_DIR` defaults to `~/.local/state/yolomux`, so the normal path is `~/.local/state/yolomux/search_index/<sha16>.json`. The file name is the first 16 hex chars of `sha256(str(root))`. Each JSON payload stores `root`, `built_at`, `truncated`, and `entries`. Runtime state is also kept in a per-process in-memory `_REGISTRY: dict[str, RootIndex]`.

HOW OFTEN IT RUNS (verified): There is no cron and no Unix `find`. Indexing runs on access through `file_index.ensure_index(root, SEARCH_SKIP_DIRS)`. `ensure_index` loads the disk JSON if present, marks it ready immediately, and starts a background rebuild when the in-memory index is missing or older than `INDEX_TTL_SECONDS = 300.0` seconds. The frontend triggers that access when a directory is allowed for indexing (`setFileExplorerDirectoryIndexed` -> `/api/fs/index-status`), when exact indexed directory rows render (`updateFileExplorerIndexedDirectoryRows` -> `ensureFileIndexStatus` -> `/api/fs/index-status`), and when quick-open/search hits `/api/fs/search` for a recursive root. Also, any search rooted at a git repo sets `full_tree = true` on the backend, so `search_files` can call `ensure_index` even when the Finder UI does not show that directory as user-indexed.

WHY SEARCH CAN STILL TAKE A WHILE WITH A PAST INDEX (verified): `filesystem.search_files` only serves from the index when all three are true: `full_tree` is true, the query has non-empty `tokens`, and `index.ready` is true. If the query is empty, it explicitly skips `file_index.search_index(...)` and falls through to `_search_full_tree`, so opening quick-open with an empty query can still walk large roots even if disk JSON exists. If an indexed root is not ready yet in this process, the typed query also falls back to the live tree walk. The frontend makes this heavier because `openFileQuickOpen()` immediately schedules a search with the empty query, and `refreshFileQuickOpenCandidates()` sends parallel `/api/fs/search` requests for the active root plus every configured indexed root; for indexed roots it adds `recursive=1`, which makes the empty-query live walk recursive. Browser abort cancels the fetch state, but the server-side walk is still synchronous Python work for that request.

WHY THE BADGE/STATE CAN LOOK INCONSISTENT (verified): The product currently has at least four sources of truth. (1) Browser localStorage key `yolomux.fileExplorer.indexedDirs.v1` means "the user wants these directories indexed" and is origin-scoped, so `:7777` and `:7778` do not share it. (2) Shared settings `file_explorer.indexed_dirs` in `~/.config/yolomux/settings.yaml` is also a desired-root list and is polled by every browser every 1-10 seconds (`refreshSettings` interval is clamped from `eventLogRefreshMs`). (3) `fileExplorerIndexStatus` is a per-page JavaScript cache of `building`/`ready`. (4) Backend `_REGISTRY` is per Python process, while the JSON files are shared on disk. The Finder badge uses `fileExplorerDirectoryIsIndexed(path)` first, which checks only the desired-root set, then overlays cached process readiness with `fileExplorerIndexBadgeText(path)`. A row can therefore disappear from "indexed" when a settings poll reconciles a different desired list into localStorage, even though disk JSON still exists; the inverse can happen too, where the UI says an indexed root is desired but the current server process is still building or has no loaded registry entry yet.

MULTI-SERVER STATUS TODAY (verified): If both servers use the default `YOLOMUX_CONFIG_DIR` and `YOLOMUX_STATE_DIR`, they share `~/.config/yolomux/settings.yaml` and `~/.local/state/yolomux/search_index/*.json`. They do not share browser localStorage because different ports are different origins. They do not share `_REGISTRY`, build timers, `building` flags, or frontend `fileExplorerIndexStatus`. There is also no cross-process build lock around `walk_root` or `_persist`; writes use a temp file plus atomic replace, so the final JSON is not half-written, but two processes can duplicate the same expensive walk and the last writer wins. `unindex(root)` deletes the JSON and clears only the receiving process's registry; another server process can still have a ready in-memory copy until its own registry is cleared/stales/rebuilds, and git-root search can still recreate an index even if the UI desired-root list no longer contains that path.

ROOT CAUSE (verified): The UI conflates "this path is configured for indexing" with "this server has a ready index", and the desired-root list is updated by whole-list replacement from multiple places. `setFileExplorerDirectoryIndexed` writes localStorage and then saves the whole `file_explorer.indexed_dirs` list. `reconcileIndexedDirsFromSetting` later applies the whole settings list back into localStorage and calls `/api/fs/unindex` for removals. With two browser origins or two server ports, stale whole-list saves can remove a path another page just added, and settings polling makes the row flip later. The backend is similarly split: disk JSON is shared, but readiness/building state is process-local and there is no shared manifest/lock to make two servers report the same status at the same time.

FIX: Separate desired roots from actual index status. Make `file_explorer.indexed_dirs` the only durable desired-root source after one migration from localStorage; stop treating localStorage as a peer source of truth. Replace whole-list save paths with operation endpoints or merge-on-save semantics (`add indexed dir`, `remove indexed dir`) so two pages do not overwrite each other's list. Change the Finder badge model to show desired/covered/ready distinctly: exact desired root can show `indexing` or `indexed` based on backend status, descendants can show `covered`, and absent desired roots should not disappear because one process is rebuilding. Add a backend index manifest/status endpoint that reports `root`, `ready`, `building`, `count`, `built_at`, `age`, `truncated`, and `source: memory|disk|building` so the UI can show what is actually happening.

FIX: Make the index coordinator cross-process. Add a per-root file lock under `STATE_DIR/search_index` (for example `<digest>.lock`) around build and unindex so `:7777` and `:7778` do not duplicate huge walks or delete while another build is persisting. Have `ensure_index` reload disk metadata when the manifest/disk mtime changes, and include an index format/version plus skip-dir signature in the JSON so stale incompatible indexes rebuild for a clear reason. When unindexing, mark the manifest removed/tombstoned so another process with an in-memory ready copy does not keep serving it indefinitely.

FIX: Stop cold recursive walks in quick-open. Do not issue recursive `/api/fs/search` on an empty query for indexed roots; show recent/open files and maybe a cheap shallow listing until the user types. Alternatively, allow empty-query indexed search to return a capped list from the existing index, sorted by mtime or recent path score, instead of falling through to `_search_full_tree`. For not-ready indexed roots, avoid the expensive fallback walk for large/user-indexed roots unless the user explicitly asks for live search; show `index warming` and use stale disk entries when available. Keep live fallback for small non-indexed roots and shallow path queries.

VALIDATE: Start both `:7777` and `:7778` with the same default config/state dirs. Remove it in one port and confirm both ports clear it exactly once and no process keeps serving stale in-memory results. - Server head-of-line blocking — the server is `ThreadingHTTPServer` (`server.py:1233`), so the ensure POST is not queued behind the page-load metadata/PR/git poll burst.

LEADING HYPOTHESIS: the "several seconds, first-only" is the combination of the cold first-drag synchronous reflow/paint at `dragstart` (item 2) and the first cold network round-trip in the move (item 1), with the browser's own first-HTML5-drag warmup compounding both. NOTE (ground-truth discipline): this is a code-grounded hypothesis, not a measured finding — the code has more than one plausible first-cold cost, so the exact split MUST be confirmed with one measurement before/while fixing. Do NOT assert a single cause in any user-facing note until measured.

FIX (instrument first, then the two high-value cuts):
