This file documents AI-agent behavior and repo-specific lessons for working in YOLOmux. Keep it focused on how agents should work in this repo; put user-facing instructions and command references in the docs below.

## Development Workflow

- Use [`docs/DEVELOPMENT.md`](docs/DEVELOPMENT.md) as the source of truth for local checks. The canonical local gate is `python3 tools/check.py`, which runs independent lanes in parallel by default.
- When `AGENTS.md` and `docs/DEVELOPMENT.md` disagree on a build/test/development command, update `docs/DEVELOPMENT.md` and keep only a reference here.

## Recent Agent Learnings

- Start with the local task files (`DOIT*.md`, `docs/TODO.md`, `README.md`) and re-read them after each batch. The user updates these files while work is in progress, and stale assumptions caused repeated misses.
- Separate feature completion from cleanup. Commit/push/sync the completed behavior first when asked, then do refactor work as a distinct pass so regressions and review diffs stay easier to reason about.
- All visual UX behavior belongs in [`docs/specs/GUI.md`](docs/specs/GUI.md). When a visual rule is discovered from a screenshot, user report, browser behavior, or design decision, codify it there before or in the same change as the implementation and tests.
- For Claude/Codex TUI and mock parity, reproduce against the real client first: launch real `claude`/`codex` in tmux with the same common inputs and states, capture the pane, then launch `tools/claude.py --mock` / `tools/codex.py --mock` with the matching inputs and compare the panes. The mocks live in the real client entry points plus `tools/mock_agent_common.py`; do not recreate a top-level `mock/` package. Update the mock renderer, prompt fixtures, tests, `tools/CLIENTS.md`, and `docs/specs/GUI.md` from the real capture until the mock matches the real TUI behavior and layout.
- Treat visual bugs as layout-state bugs, not one-off CSS tweaks. Reproduce with the exact URL/layout/tabs query string or screenshot state, then add a layout/browser assertion that checks the failing geometry or DOM contract.
- For pulse/glow/color animation bugs, follow the visual verification contract in [`docs/specs/GUI.md`](docs/specs/GUI.md). Prove the exact reported element changes over time, compare against a known-good element when one exists, and verify the served bundle or restarted dev server before saying a live port is fixed.
- Do not hard-code menu widths, tab capacity, pane sizes, or dropdown behavior for the current browser. Earlier fixed-size guesses broke on different viewport/font/pane combinations. Measure the container/content and use shared CSS variables, flex/grid, percentages, and viewport clamps.
- Keep all pane/tab/window controls on shared helpers. Inconsistent active borders, Mac/PC close/minimize glyphs, pane focus rings, and file-editor tab buttons came from duplicated local button code. If two controls look or behave the same, move the state sync into a helper before adding another branch.
- Preserve Finder as a pane with special width behavior, not as an overlay. Left/right splits should keep Finder from auto-expanding horizontally when its sibling tab closes; top/bottom siblings can expand when the last normal tab is gone. Empty placeholder panes are part of preserving user-chosen split sizes.
- Menu hover/focus behavior is global UI state. Auto-focus is not just terminal panes; it includes menus, popovers, Finder, Preferences, editors, and pane focus rings. If auto-focus is off, hover should not auto-open menus until the user has explicitly opened a menu.
- Popovers need shared timing and ownership. Tab, image, menu, and file-preview popovers flickered because separate timers competed. Prefer one controller per popover family, with explicit show delay, follow delay, hide delay, and close-on-topbar/pane transitions.
- Generated assets are not source. Edit `static_src/js/yolomux/*.js` and `static_src/css/yolomux/*.css`, then run `python3 tools/static_build.py`; use `python3 tools/static_build.py --check` to catch stale `static/yolomux.js` or `static/yolomux.css`.
- Validate browser UI with both code-level and browser-level tests. `node tests/layout_url.test.js` catches URL/layout logic and generated-source contracts; Selenium tests catch real geometry, clipping, alignment, and toolbar regressions. Full pytest needs local socket access, so sandbox failures with `PermissionError: Operation not permitted` should be rerun outside the sandbox before treating them as product failures.
- Live-runtime Selenium fixtures can expose the page to WebDriver before every bundle global is ready under xdist/browser load. Before an async browser script calls app globals, wait for the specific functions and DOM anchors it needs, for example `typeof fileEditorItemFor === 'function'`, `typeof applyLayoutSlots === 'function'`, and `#grid`; otherwise the failure looks like product code (`ReferenceError`) when it is really fixture readiness.
- YO!share replay tests must include the failure sequence, not only the happy path. A late/stale delta after a newer keyframe must be dropped and counted as `staleFrames`, not treated as `viewer-behind`; a true non-contiguous future delta must still request a keyframe. The previous Selenium test asserted the old wrong behavior (`stale epoch -> keyframe request`), so it did not catch the keyframe storm that caused cursor jitter and blanking.
- Replay DOM is serialized DOM, not the original live app DOM. Normal tab/menu/popover event listeners are gone after a keyframe or child-list rebuild, so any viewer-local read-only affordance must be rebound by the replay shell after keyframe and successful delta apply, or must be host-published as replayed DOM. Add a browser or Node regression that proves the replayed element has the listener/visible state after rebuild.
- Host-owned popover replay geometry must include the full measured app-space rectangle, not just visible text or `left/top`. A tab popover whose width comes from `vw` can replay at the right position but wrap wider/taller on a client browser because viewport units resolve against the viewer viewport; serialize the host-computed inline width and compare `left/top/width/height` in Selenium.
- Treat opt-in YO!share debug/profiling uploads as test inputs. If logs say only `reason: gap`, add instrumentation before fixing so the next report includes epoch, sequence, baseSequence, currentEpoch, lastSequence, dropped/stale frames, keyframe requests, suppressed requests, terminal placeholder health, and browser version. Do not stop at a generic `viewer behind` counter when the user reports jitter, blanking, or Safari-only behavior.
- Real generated-link YO!share tests must wait for the host terminal container and host terminal socket before creating/publishing the share if they claim terminal coverage. A passing mirror-status assertion without a host-visible terminal proves only app-shell replay; terminal regressions need assertions that each mirrored placeholder session has a live `/ws/share-view` socket, no session has duplicate live terminal sockets, and terminal placeholder health stays connected through real Finder toggle loops.
- Browser/live tests may start throwaway local servers, but they must also isolate tmux and state: fixture-owned config/state directories, ephemeral HTTP ports, a private tmux socket, and fixture-created session names such as `yt-<pid>-<uuid>-1`. Automated tests must not exercise live dev/prod ports like `8001` or `7777`; those ports are only for explicit smoke checks after a requested restart or sync.
- Safari-cadence regressions need an explicit timer-shaped test even when CI only runs Chrome. Force `geometry-digest` publishes at the Safari-like 2000 ms cadence between real host Finder toggles and assert the viewer does not enter `viewer behind`, does not increase dropped-frame/keyframe-request counters, and keeps terminal placeholders connected.
- Keep screenshots and local debug artifacts out of commits. `cps` must stage explicit files only; untracked `20260*.png` files in `~/yolomux.dev` or `~/yolomux` are local evidence and should remain unstaged.
- For file drag/drop into terminals, browsers may expose dragged files as rich data that Codex/Claude renders as `[Image #...]`, while the shell still receives usable text. Prefer `text/uri-list` plus plain text path data for drag payloads, and test both browser rendering and terminal input behavior.
- File editor/viewer behavior should be capability-based. Markdown/HTML get edit/preview/split; source files get editor/search/diff actions only; large or binary files should show a clear too-large/binary state instead of trying to load everything.
- Context menus and toolbar buttons should carry accessibility state from shared builders: `title`, `aria-label`, `aria-pressed`, `aria-checked`, disabled state, checked state, and keep-open behavior. Missing shared handling caused repeated "same-looking controls behave differently" bugs.
- Prefer small, named state helpers over repeated expressions. Examples: active/pressed button syncing, upload filename defaults, generated upload detection, repo-path matching, and target-session selection. Repeated inline logic was where subtle precedence and drift bugs appeared.
- Keep tests tied to the bug that happened. When a user reports "this exact URL collapses a pane" or "this toolbar shifts after Finder click", add that exact URL/state to a regression test rather than only testing the helper in isolation.
- YO!share replay bugs must be tested in the default replay shell, not only legacy semantic share mode. Regressions like per-second `viewer behind`, cursor target jitter, blank clients, and duplicated Finder/Differ panes can come from DOM replay frames interleaved with semantic frames; tests must cover independent DOM replay sequencing, stale-frame tracking by frame family, five-second keyframe backoff/coalescing, and fixture `/api/share` refreshes that keep the host share active during delayed repair.
- Finder, Differ, and Modified-files must share tree-row, date/sort, status-chip, icon-column, and toolbar helpers. If one pane gains or changes a control (`None`/`Date`/`Ago`, `A-Z`/`Z-A`/`new`/`old`, git status badges, Codex/File icons), apply it through the shared path in the same change; local one-pane fixes caused repeated visual drift.
- Treat toolbar/icon alignment as a geometry contract. For visual bugs like misaligned blame/diff circles or Finder/Differ header icons, add a browser assertion that compares actual bounding boxes across the affected panes and themes; screenshots alone did not catch the recurrence.
- CodeMirror live toggles must exercise the real UI event path. Wrap, line-number, blame, diff, and theme changes should reconfigure compartments or dispatch state effects without rebuilding the editor, and a browser regression should click the button and assert the document text remains visible. Source-grep tests missed a runtime `scheme is not defined` blank-editor failure.
- Git editor actions are capability-gated as a pair. Blame and diff buttons should appear together only for files inside a git repo with meaningful committed history; untracked, no-history, or outside-repo files should hide both rather than showing disabled/mismatched controls.
- Large Markdown cleanups still need surgical edits. Do not generate a replacement file under `/tmp` and overwrite the tracked Markdown with `\cp`; use `apply_patch` or a reviewed script-generated patch so concurrent user edits and unintended deletions are visible in the diff before the file changes.
- In the YOLOmux repo, `cps` means the `yolo-cps` skill, not the Dynamo-utils `dyn-cps` flow. It has two modes: LOCAL (default — "cps"/"yolo-cps" with no qualifier: rebase the dev branch onto local `main` and land it there, NO version bump, NO push) and ORIGIN ("cps origin"/"cps remote": rebase onto `origin/main`, bump the version, land on local `main`, then push). Both run the full local check set with explicit staging. Restart is NOT part of either mode. Full sequence and rules live in [`docs/DEVELOPMENT.md` -> Production Sync](docs/DEVELOPMENT.md).
- The checked-in docs are authoritative over stale skill snippets for YOLOmux CPS. If a skill still mentions `~/yolomux.dev`/`7778`, adapt it to the active worktree and port from `docs/DEVELOPMENT.md` (for example `~/yolomux.dev8003`/`8003`) and restart that active dev server, not an unrelated old dev checkout.
- For YOLOmux CPS checks, `docs/DEVELOPMENT.md` is authoritative: run `python3 tools/check.py`, not a fixed worker count or an old manual command list. The default runner starts the full gate in independent parallel lanes; the `pytest` lane excludes the `node_bridge` marker (the node suite runs once via the `node-layout` lane — see DEVELOPMENT.md). Use `--serial` only when debugging order or machine load.
- After every YOLOmux change in a dev worktree, restart the active dev server before reporting the change ready. Restart prod only when explicitly asked — it is NOT part of `cps`.
- `cps` landing footgun (this is why a just-made commit can appear to "vanish" from `origin/main`): land on local `main` with `git -C ~/yolomux merge --ff-only <branch>` as its OWN exit-checked command. Do NOT pipe it through `tail`/`grep` and do NOT `&&`-chain it straight into the push — a pipe's exit status is the last stage's, so a DIVERGED ff-merge fails silently and the chained `push origin main` then publishes whatever local `main` already points at (often another worktree's concurrent commit), leaving your commit stranded on the dev branch. This is a shared multi-worktree, so local `main` can advance under you mid-`cps`; recovery is `git fetch origin`, `git rebase origin/main` your dev branch (disjoint files rebase clean), re-run the gate, then ff-merge + push. No force-push, nothing lost.
- Keep YOLOmux checks proportional while iterating. Use focused tests or repeated `python3 tools/check.py --lane <name>` quick cycles, reserve the full gate for pre-commit/pre-push validation, and if a full gate already passed then a rebase only adds docs or other non-runtime changes, verify the new diff and run a focused smoke instead of repeating the entire browser suite.
- Good pattern to keep: after each substantial UI change, run the full local check list in [`docs/DEVELOPMENT.md`](docs/DEVELOPMENT.md) (the single source of truth for those commands — do not re-inline them here, per the precedence rule above). Report any sandbox-only failures separately and rerun them with the right permissions.
- Implementer reflections (DOIT.7 retro, 2026-06-01): the rules below are what would have saved time on the build side of the DOIT.5/6 batches.
- Pinned literals live in MULTIPLE test layers — grep before you change one. Every exact CSS token, label, glyph, or constant you edit is usually asserted in more than one place: `tests/layout_url.test.js` (node source-grep), `tests/test_browser_layout.py` (Selenium computed `rgb(...)`), and pytest. This session a single token change (`--pane-tab-unfocused-active-bg` `#285a2f`->`#4f9e3a`) broke both a node source-guard and a Selenium computed-color assertion, and one label (`Branch Info`->`YO!info`) broke three separate node pins. Before changing a pinned value, `grep -rn '<old-literal>' tests/` and update every pin in the same change.
- Reproduce in the harness BEFORE theorizing about whether code is "correct". The vm-built `api` calls bundle internals directly; `tabStrip`/`tabElement` mocks (with a `.rect`) exercise drag-placement math; `prompt_detector` runs against hand-built pane fixtures. For the left->right drag bug I argued from the source that the insert index was already correct for too long — building a 2-tab mock and printing the returned index showed it was a directional-threshold problem (neighbor CENTER vs FAR edge), not the off-by-one the item guessed. Build a failing repro first; it is faster than reading.
- For async root/path/state bugs, reproduce the actual stale transaction. A same-value short-circuit (`root already matches`, `plan already applied`) can be wrong if a different target is still in flight and will apply later; add a deterministic delayed-fetch regression that holds the old operation, switches the explicit target, releases the old operation, and proves stale work cannot clobber the new state.
- When a DOIT item carries "STUDIED FINDINGS" / "naive fixes that are wrong", implement the prescribed durable fix, not your first instinct — the diagnosing agent has usually already tried and rejected the naive path. The Ctrl-T auto-approve fix is the case in point: enumerating todo glyphs (my first move) was exactly the fragile approach the item warned against; the durable fix was a bounded-overlay `break` at the `^\d+\s+tasks?\s+\(` header. And before adopting a "defense-in-depth" suggestion, confirm it does not contradict an existing guarded invariant — the suggested `codeMirrorConfigSignature` scheme key is explicitly forbidden by a test (it forces full editor rebuilds); the compartment swap (`refreshOpenEditorThemePanels`) is the right mechanism, so I used that and skipped the signature change.
- Cross-realm test equality: values returned from the vm-context bundle have a different prototype than the test realm, so `assert.deepStrictEqual` fails with "same structure but not reference-equal". Spread them first (`[...api.fn()]`) or compare primitives.
- FALSE-GREEN / hung-await in the trailing suite IIFE (verified 2026-06-16): the behavioral blocks at the bottom of `tests/layout_url.test.js` run inside one `(async () => {...})().catch().finally()` IIFE, and the pass/fail summary + non-zero exit live in that `.finally`. If ANY block `await`s a promise that never settles, the IIFE never settles, `.finally` never runs, the event loop drains, and node exits 0 with NO `layout suite: N passed, N failed` line — silently masking EVERY later block (including violations of the cross-realm rule above) and passing the cps gate. The classic culprit was: the harness stubbed `setTimeout` as a no-op (so the 8 ms `setTimeout(flushFileExplorerFsBatch,…)` auto-flush never fired), and a block did `await api.applyShareUiStateForTest({finder})` / `await api.fetchDirectoryForTest(...)` (which internally `await`s a batched `/api/fs/batch` directory listing) without driving the flush — the listing promise hung forever. DURABLE FIX (landed 2026-06-16): the harness `setTimeout` shim now fires ONLY the 8 ms fs-batch flush (8 ms is unique to that timer — verified no other bundle timer uses it) via real `setImmediate`; every other timer stays a no-op. So awaited directory listings settle naturally — no per-block flush dance needed; just stub `/api/fs/batch` (echo ok `responses`) if the listing's CONTENTS matter to the assertion. Synchronous back-to-back enqueues still batch into one request (no await between them), so coalescing tests are unaffected. Two guards landed alongside: (1) a `SUITE_WATCHDOG_MS` real-node timer that fails loud (`SUITE DID NOT SETTLE`, exit 1) if the IIFE ever hangs again, instead of exiting 0; (2) ALWAYS confirm a run prints the `layout suite: N passed, N failed` summary — no summary means the tail never ran. Related: these bare `{}` IIFE blocks are NOT isolated like `test()` (first throw/hang aborts all the rest), so the gate surfaces only ONE masked failure per run — prefer `test()` wrappers for new behavioral blocks, and remember cross-realm `deepStrictEqual` needs `canonical(...)` (see entry above).
- The DOIT/TODO checkbox is the shared source of truth, not your internal task tracker. Flip `[ ]`->`[x]` with a DONE note in the DOIT file as part of finishing each item (I once marked internal tasks done but left the DOIT boxes unchecked, so a re-read showed stale "open" items). Re-read the exact line immediately before editing — these files change under you mid-batch, and concurrent edits cause "file modified since read" churn.
- When the user asks to perform a `DOIT*.md` file, finish the queued behavior, validate it, archive the completed work in `docs/DONE.md`, update `README.md` for user-visible behavior, and remove the DOIT file once every item is complete. Keep the raw DOIT file only if unfinished work remains.
- Push the root cause back to the diagnosis. When the diagnosed cause is wrong, state the ACTUAL cause in the DONE note (drag-reorder was a threshold asymmetry, not an index off-by-one; the "PR pillbox" was a `ready-review` state badge, not a PR chip). The `file:symbol + FIX + Validate` contract only gets better if the implementer corrects it instead of silently coding around it.
- Build + test after EACH item, not per batch. Every item this session ended with `tools/static_build.py` + `node tests/layout_url.test.js` (and pytest for backend) before moving on; that caught pinned-value regressions at the item that caused them instead of in a confusing end-of-batch pile.

### Recurring pitfalls (2026-06-01 retro)

Distilled from the DOIT.5/DOIT.6 batches (see `DOIT.7.md` for the concrete failures behind each). These are the bug CLASSES that recurred — check them by default.

- Setting/theme propagation: a global setting change must re-apply to EVERY live surface, not just the obvious one. When you touch `appearance.*` or any global mode, walk the full list — app chrome, OPEN editors (CodeMirror), terminals, tab badges, YO markers, popovers — and confirm each re-applies. New themed components need BOTH a `body.theme-light` and dark rule from the start. (Caused: illegible light-mode badges, View->light not re-theming open editors, terminal theme left behind, stale YO markers.)
- Cache/signature completeness: any memoization or short-circuit key (e.g. `codeMirrorConfigSignature`) must include every input the output depends on, or a real change will no-op. Prefer a cheap live reconfigure (CodeMirror compartment swap) over a full re-render that can short-circuit.
- Detector chrome vs content: text-scraping detectors (`prompt_detector.py`) must treat persistent CLI chrome (Ctrl-T task overlay, input box, mode hints, footers) as NON-activity; recognize bounded overlay blocks (stop at the header) instead of blanket "anything after the footer" rules; and add a fixture for every new Claude/Codex UI variant.
- No silent duplicate code paths / divergent copies: equivalent user actions (double-click vs drag, two open routes) must funnel through ONE shared parent — and not just functions. Before adding ANY new attribute, helper, CSS token, class, or per-session state map, grep for an existing one doing the same job and route both through it (extend it, or pass the difference as data) rather than copy-pasting a parallel one, even when copy-paste is easier; divergent copies are the recurring root cause of inconsistent/half-finished refactors here (light-mode hexes hardcoded per-component instead of via the shared theme tokens; bespoke per-pane builders instead of the shared render pipeline; a git-status→class map written two different ways across the Finder and Differ). Watch too for a function defined in two concatenated `static_src/js/yolomux/*.js` partials — the last one silently wins. Grep for duplicate definitions, and when you unify copies onto a parent, land the parity test in the SAME change.
- Toggles toggle; don't double-show: a control with a checked/visible indicator should open AND hide. Never render the same fact twice (drop the native `title` when a custom popover exists; remove a redundant state label when a dedicated badge exists).
- Right default view mode on open: the file-open path must choose the mode per context (markdown -> preview, untracked/all-added -> edit), not fall through to one default.
- Honest affordances: a control's label/aria/tooltip must match what it actually does, or make it do what it says.
- Stale bundle is not a fix: a frontend change is unverified until `tools/static_build.py` rebuilds and the page is hard-reloaded. Never call a frontend fix done from source inspection alone.
- Stale BACKEND is also not a fix — and it has NO build step to remind you. A Python change (`prompt_detector.py`, `app.py`, `auto_approve_worker.py`, `metadata.py`, `settings.py`, `web.py`, anything under `yolomux_lib/`) only takes effect when the running `yolomux.py` SERVER process is RESTARTED — Python does not hot-reload, and `tools/static_build.py` does not touch it. Passing pytest + a DONE note prove the SOURCE is right, not that the LIVE server runs it. Symptom seen this session: the Ctrl-T auto-approve fix was correct and tested, but the live server still hung because it was started BEFORE the fix landed (and two servers were running — the older one stale). Before reporting a backend behavior fix as live, restart the server (and make sure only ONE instance owns auto-approve — multiple `yolomux.py` instances on the same tmux sessions contend on the auto-approve lock, so a stale older instance can keep serving the old code). Reproduce a backend fix against the actual running process, or restart it, before calling it done.
- Pinned-literal and reproduce-before-theorize rules: covered once in the learnings above ("Pinned literals live in MULTIPLE test layers", "Reproduce in the harness BEFORE theorizing") — they recurred in this retro too. Tooling since added: `static_build.py --check` now carries the undefined-CSS-var + duplicate-declaration lints, and `tools/check.py` runs the whole gate.

### 2026-06-17 retro

- Terminal wheel routing depends on the pane's tmux screen mode, and a "fix" can be half-right. Alt-screen TUIs (claude, codex, vim, less) keep NO tmux scrollback (`history_size` stays 0) and own the mouse, so routing their wheel into tmux `copy-mode` scrolls nothing — the wheel has to reach the app. But just deferring to xterm is too weak: xterm's wheel-to-mouse path emits only ONE mouse report per accumulated cell and discards the line magnitude, so it reads as several physical scrolls per line. The working fix forwards the wheel yourself — re-dispatch N single-line `wheel` events at xterm's screen element (N = the same `terminalWheelSignedLines` count the copy-mode path uses, plus a per-session fractional remainder for touchpads). Verify scroll behavior in a live browser, not just the unit gate; the `sessionPaneIsAlternateScreen` routing gate is unit-tested but the live DOM forwarding is still only on the GUI.md Audit Backlog.
- The gate's wall-clock is bounded by its single longest test, and lanes can silently duplicate work. `test_node_suite.py` (marker `node_bridge`) shells out to the exact `node tests/layout_url.test.js` the `node-layout` lane already runs, so `check.py` was running that ~20s suite twice at once and the two node processes thrashed the cores the 32 xdist browser workers needed. Excluding `node_bridge` from the default `pytest` lane took the gate from ~47s to ~33s. Tuning `-n` did not help (auto=43s vs 8=48s) — both were pinned by the longest single test (the node suite, and a 26s share-link matrix test). To go lower, split the mega-tests, don't change worker count.
