# YO!share Test Inventory

This document lists the current YO!share test coverage, what each test is supposed to prove, and how to reproduce the user-visible behavior by hand. Keep this inventory updated when adding, deleting, renaming, or materially changing a YO!share test.

## Harness Types

Fixture Selenium tests use `load_live_runtime_boot_fixture(...)` in `tests/test_browser_layout.py`. They load the real generated YOLOmux browser bundle into a temporary page, but they stub server pieces such as `fetch`, `WebSocket`, `EventSource`, and xterm. These tests are best for deterministic browser-state assertions: DOM geometry, replay status, replay epoch/sequence counters, stale/dropped frame counts, redaction, popup capture, Finder topology, and host-owned UI state. Manual repro usually means opening a share viewer with the same query mode, then forcing the visible UI state through normal actions.

Live-server Selenium tests use `start_browser_share_server(...)` in `tests/test_browser_layout.py`. They start a real `TmuxWebtermHTTPServer` on a random localhost port, create or open a share URL, then open a viewer browser over the real HTTP and WebSocket routes. Tests that need a host admin only to reach the app use `auth_bypass=True`, which sets `YOLOMUX_TEST_AUTH_BYPASS=1`; tests that validate auth, login redirects, role boundaries, or token scoping keep real auth enabled. These tests are slower, but they catch failures in token auth, share creation, WebSocket relay, replay keyframe cache, queue backpressure, delayed repair, and server/browser timing.

Node source-contract tests in `tests/layout_url.test.js` run the bundle in a VM and inspect exported helpers or source text. They are good for protocol invariants and cheap regression guards, but they do not prove real browser layout or server relay behavior.

Server protocol tests in `tests/test_server_query.py` call Python server helpers directly. They prove keyframe cache, delta ring, WebSocket allowlist, queue overflow, and repair behavior without launching a browser.

## Manual Debug Checklist

Use this checklist when a user reports YO!share drift, blank viewers, duplicate panes, cursor target jitter, or frequent `viewer behind`.

1. Reproduce with the default share path first. Do not add `shareReplay=0` unless testing the legacy semantic escape hatch.
2. Create the share with debug/profiling enabled, or append `?shareDebug=1` to the viewer URL before the `#t=` fragment.
3. Open host and viewer in the reported browsers. For Safari-specific reports, keep Chrome open as a comparison only after the Safari path is reproduced.
4. Watch the viewer banner and debug health. Important fields are `status`, `epoch`, `sequence`, `droppedFrames`, `staleFrames`, `keyframeRequests`, `keyframeRequestsSuppressed`, `keyframeRequestBackoffMs`, `keyframeRequestInFlight`, `keyframeBytes`, `deltaBytes`, and terminal placeholder health.
5. If the problem is duplicate Finder/Differ panes, check whether the viewer is in default replay mode or legacy semantic mode. Default replay should ignore semantic Finder topology and follow DOM keyframes/deltas.
6. If the problem is per-second `viewer behind`, verify whether DOM replay keyframes/deltas have their own contiguous epoch/sequence stream and whether repair keyframes are arriving after backpressure.
7. If the problem is blank viewer content, check whether the latest keyframe reached the viewer, whether it was sanitized to an empty root, and whether the server replaced older queued repair keyframes.
8. If the problem is cursor target jitter, separate pointer-frame jitter from replay repair churn. Pointer jitter with stable replay health is a transform/pointer issue; pointer jitter that coincides with `viewer behind` is usually replay gap/backpressure churn.
9. Preserve raw logs in `/tmp`, not in docs. Summarize only the relevant counters and frame sequence in durable docs or issue notes.

## Maintenance Rules

- Add a fixture Selenium test when the bug can be expressed by injecting frames or host UI state into the browser and reading DOM/JS values.
- Add a live-server Selenium test when the bug depends on auth, share creation, `/ws/share-host`, `/ws/share-ui`, `/ws/share-view`, server fanout, queue pressure, or delayed repair.
- For live-server Selenium tests, use `auth_bypass=True` when the test is about share creation or mirror behavior and not about login/auth itself. Do not use it for tests whose expected result is a login redirect, 401, 403, cookie behavior, readonly/admin distinction, or share-token scoping.
- Add a server unit test when the bug is in keyframe cache, delta ring, allowlist, queue overflow, or repair request behavior.
- Add a Node source-contract test only for cheap protocol/source invariants; do not rely on it for browser geometry or live WebSocket behavior.
- When a test uses `shareReplay=0`, name it or describe it as legacy semantic mode. Default user-facing YO!share behavior is DOM replay.
