# YOLOmux GUI Behavior Spec

This document is the working GUI contract for pane, tab, Finder/Differ, editor, preview, Preferences, keyboard shortcuts, drag/drop, and YO!share live-mirror behavior. It is meant to drive future implementation and tests. User direction during implementation overrides older wording in this spec; when that happens, update the spec and the tests in the same change.

Section order follows the reading path: capabilities and terms, layout and tab mechanics, agent status indicators, pane metadata, chrome/resize behavior, special panes (Finder/Differ and Tabber), editor/preview behavior, search/menus, YO!share, Preferences, keyboard shortcuts, test coverage, and the audit backlog.

## Popular IDE Parity Checklist

- [x] Activity/workbench panes can be arranged as tabs inside split panes.
- [x] Tabs can be reordered within the same pane by pointer drag.
- [x] Tabs can be moved from one pane to another pane's tab strip.
- [x] Tabs can be dropped on pane edges to split the target pane.
- [x] Tabs can be dropped on root edges to create full-span panes beside the current layout.
- [x] Tab insertion preview appears between tabs, not over the target tab.
- [x] Active tabs have rounded top corners and a distinct active background.
- [x] Inactive tabs inherit the tab-container background instead of using a disconnected color.
- [x] Pinned tabs stay at the front of a pane, show a pin icon, are protected from LRU tab eviction, and can be toggled from the tab context menu or the `Mod+K` then `Enter` shortcut.
- [x] Tab hover shows a detail popover with session/file state.
- [x] Tab context menu exposes tab-level actions before destructive session actions.
- [x] Open editors support edit, preview, split, and diff-capable modes based on file capability.
- [x] Markdown task checkboxes in Preview are interactive and update the source document.
- [x] Editor back/forward buttons navigate the visited-file stack.
- [x] Search results can render image/file references as `[Image #N] '/absolute/path'`.
- [x] File quick-open ranks basename matches ahead of path-only matches.
- [x] Typed Quick Search results keep the first visible screen mixed across relevant result types. Cmd-P can still rank files first and Shift-Cmd-P can still rank panes/commands first, but a broad query such as `agent` must not fill the whole first screen with only indexed file hits when matching tabs, commands, or settings exist.
- [x] Newly opened tabs (files, Preferences, YO!info/YO!agent, and other non-tmux tab types) open in the pane that owns the currently active tab; Finder/Differ stays reserved and tmux sessions keep their placement rules. Completed 2026-06-11: generalized from the former quick-open-only targeting rule.
- [x] Finder/Differ rows support selection, multi-select, rename/delete, context menu, date/sort controls, and image preview.
- [x] Preferences exposes searchable, collapsible sections with reset controls and live-applied settings.
- [x] Keyboard shortcut labels use the concrete platform modifier: Mac uses `Cmd`; PC/Linux uses `Ctrl`.
- [x] Global app shortcuts are captured before focused panes swallow them when the shortcut is platform-owned.
- [x] Code editor shortcuts include save, find, replace, go-to-line, comment toggle, indent/outdent, undo/redo, and editor back/forward.
- [x] Inline git blame can annotate the current caret line, with an option to annotate all lines.
- [x] Editor themes include dark/light mode switching, separate dark/light scheme defaults, caret style, and caret color.
- [ ] Partial: pinned-tab behavior is per browser profile through local storage, not yet a shareable URL or server-side preference.
- [ ] Partial: file tabs and tmux tabs share pinning, movement, and pane placement; Finder/Differ remains a reserved special pane.
- [ ] Partial: preview/edit interactions are implemented for Markdown task checkboxes, not for every possible rendered widget.
- [ ] Keyboard-only tab reordering and pane splitting are not implemented.
- [ ] Multi-root project workspaces are represented through Finder roots and companion directories, not a full workspace model.
- [ ] Extension/plugin APIs for editor behavior are not implemented.
- [ ] Integrated source-control side panels are partial: Differ and inline blame exist, but a full branch/commit workbench is not implemented.
- [ ] Problems/debug/test runner panels are not implemented as first-class workbench panels.
- [ ] Terminal/profile management is tmux-session based, not a full shell-profile workbench.

## Terms

- Pane: a visible browser layout region that contains one tab strip and one active tab body.
- Tab: a layout item inside a pane. Current tab types include tmux sessions, Finder/Differ, file editors and image viewers, Preferences, and YO!info/YO!agent.
- Pane pop-outs use one shared same-origin popout host and shared pane-header control. Phase 1 enables detached snapshots only for preview-capable file editors, YO!info, and YO!stats; terminal/transcript panes stay disabled until a live PTY/WebSocket design exists, and Preferences, Finder/Tabber, Search & Runs, and YO!agent stay disabled until interactive command bridging exists. Disabled tab types must have an explicit logged reason instead of silently omitting support.
- Finder/Differ: one special file pane. Finder mode shows the file tree. Differ mode shows changed files and diff controls. Tabber mode shows a live tree of open tabs, tmux sub-windows, and touched repo roots, sorted by recent activity. Legacy `changes` / `__changes__` URLs resolve to Finder in Differ mode.
- Root edge: the outside edge of the pane layout. Dropping there creates a full-span pane beside the existing layout.
- Pane edge: the edge of one target pane. Dropping there splits only that target pane.
- Cross-gutter: the separator between two sibling panes. Dropping there creates a full-span split at that sibling boundary.

## Layout Model

- Pane layout is percentage-based and reload-safe. `layout` and `tabs` URL parameters are the shareable source of truth.
- The URL `state` parameter carries reload-survival state that belongs to the current layout rather than only local browser preferences: editor modes and per-editor view state (`line`, `top`, `left`, `anchor`, `head`), Finder/Differ/Tabber mode/root/session/sort/date/ref/collapse/expanded state, Preferences search/collapse state, and visible scroll offsets for panes that have scrollable surfaces. A full browser reload must reopen editor tabs at the same viewing line using the shared editor line-target path; raw pixel scroll is only a fallback because CodeMirror scroll snapshots are not serializable through the URL.
- A pane split must not advertise a preview if the target rect cannot fit both resulting panes. Left/right splits require enough width for the current target pane plus the incoming tab. Top/bottom splits require enough height for both panes.
- A middle drop must not advertise a preview if the target pane cannot display the incoming tab at its minimum width and height.
- Pane minimum sizes come from shared CSS/JS sizing tokens and tab-type minimums. Do not hard-code a one-window pixel capacity in a local drag handler.
- A local pane-edge drop splits only the target pane. It must not flatten same-axis siblings into equal thirds. Example: dragging into the right half of `1/2 | 1/2` should be able to produce `1/2 | 1/4 | 1/4`.
- A root-edge drop creates a full-span pane beside the entire current content area. If panes are stacked top-to-bottom, dragging to the left or right root edge creates a full-height leftmost or rightmost pane.
- Root top/bottom drops with a docked Finder/Differ column split only the non-Finder content area, so the preview and resulting pane do not cover the reserved Finder/Differ column.
- Root drops over the Finder/Differ column are reserved and must not show a preview.
- Cross-gutter drops preserve the sibling boundary and create a full-span pane at that boundary.
- Closing or moving tabs must preserve user-chosen split percentages where possible. Empty placeholder panes are allowed when they preserve a meaningful user split.
- The app layout menu exposes Single pane, Split, and Grid. The separate `tmux_wall.py` companion is not an app layout mode.
- Resizing the Finder/Differ sash changes the root Finder/Differ percentage and preserves the nested content split percentages. Example: in `Finder | Pane1 | Pane2`, dragging the Finder sash changes the available content width while Pane1 and Pane2 keep their relative ratio.
- The pane-header Info Bar toggle controls the pane metadata strip only. It must be labeled as `show Info Bar` or `hide Info Bar` based on state and must not reuse the YO!info pane label.

## Tab Strip Behavior

- Clicking a tab activates it in its current pane.
- Dragging a tab within the same tab strip reorders it. The preview must appear between tabs, not on top of the target tab.
- A same-strip reorder must commit on a slow, deliberate drag, not only on a quick flick. Drag-and-drop is pointer-based (`dndStrategy: 'pointer'`), so the dragged tab smooth-reorders under the cursor; the drop-target tab must be resolved from the pointer x-coordinate with the dragged tab excluded (nearest remaining tab when the pointer sits over the dragged tab's own gap), never by hit-testing whatever element is under the pointer. Hit-testing the element under the pointer resolves to the dragged tab itself, makes the drop look like a `source == target` no-op, and silently swallows the reorder — only a flick that outran the smooth-reorder transform would land, which reads to the user as "drag sometimes does nothing".
- In a two-tab strip the dragged tab fully covers the drop point. The reorder must still commit, and a secondary pinned pointer-reorder fallback must stand down once the primary drop pipeline has handled the gesture, so it cannot re-run and mirror-swap the order back (which makes the completed drag look undone).
- Dragging a tab to another tab strip moves it to that pane at the indicated insertion position.
- Re-opening an existing file or diff tab activates it in the pane where the user last put that tab. A normal Finder, Differ, Quick Search, or context-menu open must not silently move an existing file editor back to the active, first, largest, or leftmost pane; only an explicit drag/drop or split action may rehome an existing tab.
- Pinned tabs stay in their current pane. A pinned tab can reorder within the pinned partition of its own pane, but it cannot be moved to another pane or split into a new pane.
- Invalid pinned-tab affordances must not take ownership of drag state. They must be derived from the current Dockview preview/target geometry and must clear immediately when the target changes or the drag ends; do not cache an invalid target and replay or repair the drop after release. Invalid pinned/non-pinned partition targets, including pinned tab-strip drops into another pane, show the normal between-tabs dashed preview in the danger color.
- Dragging a tab to another pane edge splits that target pane only.
- Dragging a tab to a root edge creates a full-span pane beside the current layout.
- The tab insertion preview is a visible dashed box between tabs and uses the same configurable color as the pane separator.
- The active tab keeps rounded top corners. Inactive tabs use the same background as the tab container.
- The active pane's tab container is slightly brighter than inactive panes. Inactive tabs inside the active pane use that same active-pane tab-container background.
- `Cmd+Left` / `Cmd+Right` on Mac and `Ctrl+Left` / `Ctrl+Right` on PC/Linux move through the visible pane-tab traversal list: expand `layoutSlots` in visual pane order as `Pane1(Tab1, Tab2, ...) -> Pane2(Tab1, Tab2, ...) -> ...`; when Dockview has a rendered tab order that differs from serialized state, use that rendered order as the same traversal list, not a separate shortcut model. Left and Right must be exact inverses for every adjacent pair, stop at the first/last tab rather than wrap, include file tabs and virtual tabs such as YO!info/Preferences, and always start from `visualActivePaneItem()` so stale terminal capture handlers cannot compute navigation from their own terminal session.
- Tab hover details must show the session/file information popover. Dockview tab popovers must not be clipped by the Dockview tab scroller. Tabber rows must show only one hover surface: YOLOmux's custom Tabber/session popover. Row tooltip text may be kept in `data-tabber-title` for rendering/tests, but Tabber row DOM and visible Tabber session/window chrome must not carry native `title` attributes that create a second browser tooltip, and Tabber activity refreshes must not rebuild a hovered session tab in a way that closes/reopens the custom popover.
- Header actions such as back/forward, minimize, close, and add must stay on the first header row when there is room.
- Many Dockview tabs must keep the configured Preferences tab width when the pane has room for one tab at that width. When there are too many tabs for the pane width, the tab strip wraps to additional rows, the Dockview header grows to contain every row, and content/toolbars start below the complete header. Only the first row reserves horizontal space for the right-side action cluster; wrapped rows use the full pane width, including the space to the right of the first-row actions. Tabs must not be squeezed by tab count, scroll horizontally under hidden scrollbars, paint under the right-side action cluster on row 1, or overlap the editor toolbar/details row.

## Agent Status Indicators

- A Claude/Codex status indicator has two visible parts: a static agent symbol and a separate status ball beside it. The layout is `[symbol][ball]`; the symbol and ball must not alternate, replace each other, or pulse the symbol for working, attention, or cooldown.
- Session-tab balls are true filled circles with a thin black border, fixed equal width and height, and no shadow. Working is green with no yellow/lime tone, attention is red, and cooldown is vibrant yellow from `--agent-status-cooldown`, not the generic `--accent-gold` used for DRAFT/git/warning UI.
- Status balls and sub-window play/stop/pause glyphs change opacity only; they do not glow, resize, filter, or animate a box shadow. Green play is ongoing activity and always pulses through opacity, including when broad status animation or reduced-motion suppression is otherwise active. Red stop and yellow pause pulse only for `performance.workflow_transition_glow_seconds` after a transition from green or idle, then remain static with their full vibrant fill; `0` makes them static immediately. The pulse period is `performance.agent_status_pulse_period_ms`, default `2550`; all affected surfaces use the shared clock phase. A parent circle and its child glyph animate the same `.agent-window-status-dot` opacity property, keyframe, duration, and delay; the child container remains transparent and its fixed-shape pseudo-elements keep `translate(-50%, -50%)`. Opacity moves between `0.16` and `1`.
- Ball size has one parent: `--agent-status-ball-size` on `.agent-window-activity`, consumed as the `.agent-window-status-dot` glyph `font-size`. Aggregate tab/session surfaces use the base ball size. Tmux sub-window buttons in the Info Bar, YO!info, and Tabber use a flat 80% of the base size for visible play/stop/pause glyphs. Surfaces must not define ad hoc local dot sizes or derive ball size from `--agent-window-icon-size`; that token belongs to the static Claude/Codex symbol only.
- The working ball must remain visible through green color and opacity pulse, but it must not pulse by changing geometry, use condensed tab text rendering, use yellow/lime peak colors, or use a separate size path from other status balls or other green balls.
- The `--active` state may use `agent-symbol-glow-cadence` on the symbol itself only when status pulse is enabled; active/current is not a status ball. When status pulse is disabled, active symbols stay static and must not keep running animation or `will-change` work alive.
- A session tab has one filled, thin-black-bordered circle and omits the Claude/Codex symbol to reduce tab clutter. With one visible unacknowledged child tone it uses that red stop, yellow pause, or green play color. With multiple tones it uses two equal, crisp segments for the two most urgent distinct tones in order red, yellow, green: red+green for mixed ASK+working, red+yellow for mixed ASK+done, and yellow+green for mixed done+working; if all three occur, the parent uses red+yellow. Its opacity state follows the children: if any visible child is pulsing, the parent circle pulses in the same global phase; otherwise it is static. Thus a selected red/yellow child that has completed its transition is static unless another child is still pulsing, while any green child keeps the parent pulsing. Sub-window buttons, popovers, and rows can still show the static agent symbol beside their status ball. Any sub-window activity refresh that can add or remove a ball must redraw the parent pane tab strip in the same UI update.
- A session tab that has YOLO enabled locally or on another YOLOmux server must always render its `YO` control. Auto-approval state refreshes, reconnects, and server restarts must rebuild the same shared pane-tab renderer, not only refresh hidden popovers or Info Bar content.
- Red attention and finished yellow balls are acknowledgement notifications, not idle states. The lifecycle is `green working (always opacity-pulsing) -> red ASK or yellow done (opacity-pulsing for performance.workflow_transition_glow_seconds) -> same red/yellow steady -> gray fading over performance.agent_status_pulse_period_ms after user acknowledgement -> absent`; the red/yellow route may go directly from the pulsing phase to acknowledgement before its configured duration elapses. Every observed working-to-idle completion creates a fresh yellow pause timestamp, even if the last terminal activity belongs to an earlier run; the pause pulses for `performance.workflow_transition_glow_seconds` and then stays visible in its full vibrant color until the user clicks/selects or types into the matching tab/sub-window. Acknowledgement is server-owned and keyed to the exact prompt/window transition so all connected clients acknowledge the same red/yellow signal; an approval uses its prompt hash plus a per-window generation that changes only when the immediately previous hash differs, so `A -> A -> A` is one prompt while `A -> B -> A` produces three separately acknowledgeable prompts. Multiple `YOLOmux.py` servers sharing the same state dir must also converge through the shared acknowledgement ledger. The acknowledgement ledger expires keys after seven days and bounds it to 4096 entries; obsolete old-format keys cannot match the generated new keys and age out normally. The acknowledging client turns the red/yellow ball or play/stop/pause glyph gray immediately, fades that gray marker from full opacity to transparent over exactly one configured Status ball pulse period, then removes it. A Claude/Codex sub-window keeps its static identity icon; status-only parent Tab circles disappear entirely after the gray fade. The acknowledgement refresh redraws the clicked window bar and parent pane-tab strip before the click's normal window switch, so the gray marker survives the switch for the full fade. Clicking a red/yellow tmux sub-window button or ball still performs the normal sub-window switch in that same first click, without requiring a second click. The transient marker stays absent until a new attention key or working/stopped timestamp arrives; a fresh backend `attention_acknowledged: false` or `cooldown_acknowledged: false` overrides a browser-local ACK even when the prompt text/key happens to be identical. Parent tab aggregates, YO!stats Agent status, and other summary counts must exclude acknowledged red/yellow states; unacknowledged sticky yellow counts as `Transition`.
- Agent attention acknowledgements, YO!stats owner `yolomux.py` CPU, system CPU, Agent status, Agent tokens/min history, and transcript-token sampler state are global durable state across YOLOmux servers that share `YOLOMUX_STATE_DIR`. They live under `STATE_DIR/tmux-AI-status.json` and use the background-owner `stats-sampler` role for one-writer behavior: the elected owner samples and writes, followers read the recent durable value, and a follower becomes the writer only after owner election promotes it. Browser latency, API/SSE, bandwidth/sec, disconnected gaps, hover guide, zoom range, and graph-range controls remain per-client and must not be written into the shared global stats section.
- The red terminal question highlight must mark the actual user-facing question sentence only. Footer/help chrome such as `? for shortcuts · ← for agents` is not a question and must not receive the red question overlay even when the pane itself is in an attention state.
- Animation acceptance criteria: verification must target the actual `.agent-window-status-dot` rendered on each affected surface, not just the wrapper or stylesheet. Verify green plays continue to animate and red/yellow only animate for their configured transition duration. Compare parent and child `animationName`, duration, delay, timing, play state, opacity, color, filter, and box-shadow; then sample at least three rendered frames or equivalent pixel/computed-style values across one animation period.
- The Preferences Status ball pulse period example uses the live shared status renderer and shows three groups in this order: colored aggregate Tab balls, colored sub-window play/pause/stop glyphs, and gray acknowledged play/pause/stop glyphs. All nine markers use the configured Status ball pulse period and the same global animation phase; the colored markers share the opacity-pulse keyframe while all three gray markers share the one-way acknowledgement-fade keyframe.
- A single screenshot, source grep, or “animation is present” assertion is not enough. If one attention indicator visibly pulses but a ball does not, first compare the exact ball element's computed animation cascade against the working indicator, including `prefers-reduced-motion`; do not change base ball size, swap symbol/ball roles, or add per-surface overrides until that comparison explains the difference.

- The working green ball renders on the same condition a session counts as working: `sessionAgentWindowStatusModel` resolves any per-window `state === 'working'` and the screen-state proxy `screen.key === 'working'` once; `sessionYoloIsWorking` consumes that model.
- The same shared model owns window-bar, Tabber, parent-tab, and YO!info rows. When a session is working only via the screen proxy and has no per-window `working` row, it promotes the current non-attention agent row (or the first non-attention row) to working before every consumer renders it. No surface may add its own screen-proxy fallback.
- The YO marker no longer rotates to signal working; the glowing green ball does that. The YO marker is static on tabs, in the About modal, on the sign-in screen, and in Differ loading states. YO!agent thinking spinners can still spin because they indicate transient model work, not session-working. Differ loading uses inline text plus moving ellipsis dots, with no circular YO marker.
- Tmux session tabs offer the YO button only when YO is already enabled locally, enabled/locked by another YOLOmux server, or the tab currently has an unacknowledged prompt/question for the user. Plain idle or working sessions with YO off still show agent status balls when present, but they do not show an inactive YO button.

## Pane Metadata And Tmux Sub-Windows

- The pane Info Bar reads `path · dirty-count | tmux sub-window buttons | ×` with NO separate agent badge, and its path/repo metadata is scoped to the selected tmux sub-window. If `0:codex` has transcript-touched `~/yolomux.dev8001` but `1:bash` is selected in a different cwd, the Info Bar must show `1:bash`'s cwd and must not inherit `0:codex`'s touched repo. Terminal agent identity is carried by the tmux sub-window buttons (an agent sub-window's name falls back to its process label, so it reads `N:claude` / `N:codex`). Each Claude/Codex tmux sub-window also has its own path/branch/git/HEAD/worktree metadata for popovers and Tabber; a session can have 0, 1, or many AI tmux sub-windows, and non-AI tmux sub-windows carry cwd/path only without a fabricated agent/state. The source of truth for Claude/Codex per-sub-window state, pid, active, touched repo roots, and git metadata is the backend-owned `agent_windows` record served from cached activity/auto-approve payloads; popover, Tabber, Info Bar, and tmux sub-window bar UI must read that one record instead of reconstructing path/git/active from pane cwd, `window_metadata[]`, selected pane, or a frontend `/api/session-files` scan. The agent marker must never appear in the first tab-title row.
- Info Bar metadata must keep full branch/path/description text in the DOM and must not pre-truncate with `...`. When the metadata overflows the available width, the fixed repo selector such as `< 2/3 >` stays outside the scrolling viewport, the full start of the metadata remains visible for 2 seconds, then the metadata text scrolls through the hidden overflow. The tmux sub-window buttons and close button stay fixed while metadata scrolls.
- Tmux session Info Bars show a tmux sub-window bar with one button per tmux sub-window, immediately before the Info Bar close control. Buttons use compact `index:name` labels, keep duplicate-name suffixes when needed, never show process IDs in the visible button text, switch directly to that tmux sub-window on click, and hide with collapsed details. Info Bar tmux sub-window buttons must keep intrinsic width and must not shrink or minimize because metadata text is long; other non-Info-Bar tmux sub-window bars may still use number-only labels when a name bar would be too crowded. The ACTIVE tmux sub-window's button reads as a pressed toggle styled through the shared `--active-control-*` tokens (so it follows the user's Active color preference and flips with the theme); inactive buttons keep the plain button look. Tmux sub-window buttons must not carry per-agent tint attributes or tint CSS, and generic `.tab.active` fallbacks must not repaint active tmux sub-window buttons in light mode. After an explicit tmux sub-window click or `Ctrl-b <number>`, the clicked target remains the UI active target until authoritative read-back confirms or fails; stale transcript or `/api/tmux-signals` payloads must not briefly repaint the old active tmux sub-window, Tabber row, Info Bar path, or Finder sync source in the middle.
- Tmux tab popovers list Claude/Codex tmux sub-windows near the top, then render path/branch/repo/worktree/HEAD/git status under the matching AI tmux sub-window when sub-windows differ. If all AI tmux sub-windows share the same path/repo/branch/worktree, the popover shows that metadata once as shared. A popover must not show a flat session-level `agent`, `state`, `path`, or `branch` row that contradicts the per-sub-window rows.

## Pane Chrome And Resize

- Pane separators are skinny at rest and use the shared separator token. Hover and active hit targets may be wider, but the visible rest line stays skinny.
- Separator hover thickness is at least 5px.
- Dashed tab/file/root previews use the same configurable separator color.
- Pane spacing and active pane rings must not cover terminal/editor content. Content surfaces resize inside the ring/spacing.
- Dockview sash hit targets are transparent at rest so only the shared separator line is visible.
- Active/typing-ready panes feed the same active-ring color into Dockview and legacy pane chrome.
- The active ring is a state indicator, not layout capacity. Changing pane spacing must not shift terminal text under the ring.
- Terminal panes must fit against the full pane content box and then stay stable. The fallback pre-render cell probe uses the same bundled mono font token as xterm (`YOLOmux Mono` via the shared terminal font family), bundled-font readiness performs one shared post-font refit, duplicate ResizeObserver echoes with unchanged pane size and cell metrics must not call `term.resize()` again, and Dockview host resize observer work is coalesced to one layout per animation frame. A fresh terminal WebSocket attach must request a tmux client refresh after attach so the xterm receives the current PTY screen without requiring manual input, resize, or user-run `tmux refresh-client`.
- Browser tabs that are hidden/backgrounded must not publish terminal resize frames or assert host tmux dimensions. When a YOLOmux browser window becomes active, it scans the currently visible pane tabs and sends foreground resize claims for each visible xterm; those claims make only that browser's visible tmux clients count for window sizing. A terminal WebSocket attach with no explicit foreground resize reuses the last known host dimensions for that session instead of attaching a small default PTY that can shrink the foreground pane.
- Terminal wheel scrolling routes by the pane's tmux screen mode. A normal pane with scrollback scrolls tmux copy-mode (`scroll-up`/`scroll-down`). An alternate-screen pane (`alternate_on` — claude, codex, vim, less) carries no tmux scrollback and owns the mouse, so the wheel must reach the app instead of copy-mode (which would scroll nothing): the browser forwards it as the app's negotiated mouse-wheel reports at the same line count the copy-mode path computes, with a per-session fractional remainder so small touchpad deltas accumulate into whole lines. It must not let xterm's default behavior stand for these panes — xterm emits only one mouse report per accumulated cell regardless of delta size, which reads as several physical scrolls per line. Read-only or socket-less panes fall back to local xterm scrollback. The tmux-protocol mechanics live in `docs/specs/TMUX.md` ("Wheel routing by screen mode").
- Local `tools/claude.py` and `tools/codex.py` render assistant-visible Markdown in TTY answer text before writing to the transcript region. Common spans such as `**bold**`, `*italic*`, and inline code use ANSI styling, split streaming chunks are buffered until the line is complete, and non-TTY/redirected output remains plain text. Thinking/reasoning and tool output are not Markdown-rendered; they keep the explicit `thinking|`, `reasoning|`, or `tool|` diagnostic prefixes.
- Local `tools/claude.py` and `tools/codex.py` TTY mode own the composer/status chrome as an immutable bottom footer. The implementation must render footer rows with absolute cursor addressing, keep the terminal scroll region bounded to `1..footer_top-1`, and leave the cursor in that bounded transcript region before any answer, slash-command, thinking/tool, approval, or result text is printed. Submitted prompts are transcript content, but model/status rows are footer content and must not be copied into scrollback. Working, queued, approval, and question states may replace the idle footer with a taller fixed bottom block, but they must also move the scroll-region bottom to the row above that block. Ctrl-C, Esc exits from waiting states, EOF, and normal process exit must reset the terminal scroll region, clear the owned footer/prompt rows, and park any exit footer so the user's next shell prompt appears on the bottom row rather than inheriting the TUI cursor position near the top or middle of the pane.
- Claude TUI composer geometry in terminal panes and mocks follows the real Claude Code 2.1.185 capture from 2026-06-22: the compact startup header is three lines with `Claude Code v2.1.185`, model/billing/org, and cwd; in a TTY it renders as transcript chrome immediately above the fixed footer instead of clearing to row 1, so there is no full-screen home/clear jump or large blank gap between the header and prompt; when launched from an existing interactive shell, that startup header is emitted as normal terminal output and old shell rows scroll instead of being repainted over; when the model row is too narrow, shorten the model/effort text with `…` but keep the right-side `API Usage Billing · NVIDIA Corporation - Power Users` text visible after the ellipsis; the empty prompt uses `❯ Try "fix typecheck errors"` above the status row; the mock footer is separator, prompt, separator, status, and must not render the tmux focus-events advisory as mock UI. The detector still treats `tmux focus-events off ...` as terminal chrome if it appears in a captured pane, so that advisory cannot suppress a current AskUserQuestion. Submitting a typed question commits the submitted `❯ text` line into the scrollable transcript area above the footer, then redraws the empty separator/prompt/separator/status footer immediately and keeps later stdout scrolling above it; pressing Enter on an empty prompt is a no-op and must not append a blank transcript row or scroll the footer. Approval/cancel follow-up output does the same before printing `User approved...`, thinking, Bash output, or done markers. The submitted text and footer must not disappear or be scrolled off by output, and the mock must not add a second dark `> text` header. Interactive Claude mock question fixtures also reserve those footer rows: the question/options render only above the separator/prompt/separator/status footer, and Up/Down selector movement rewrites the original option rows in place rather than appending a second option block or covering the footer. Pressing `?` replaces the normal status with the shortcut block, including the `\⏎ for newline` row. ## Finder/Differ Pane Rules

- Finder/Differ is a real pane in the layout, not an overlay.
- Finder/Differ/Tabber are three modes of the same reserved pane. Dockview adoption, Tabber refresh, Differ refresh, and Finder refresh must preserve that pane and its current tree position unless the user explicitly hides it with the close button, File -> Finder toggle, or the Finder shortcut.
- `Cmd+B` on Mac and `Ctrl+B` on PC/Linux are YOLOmux's Cursor-style sidebar on/off shortcut for the entire reserved Finder/Differ/Tabber pane. The shortcut hides or restores that pane regardless of whether it is currently in Finder, Differ, or Tabber mode. Mac `Cmd+B` must work while a terminal has focus because it does not collide with tmux `Ctrl+B`; PC/Linux `Ctrl+B` must not be stolen from a focused terminal.
- Finder/Differ must self-heal when a URL parse, Dockview adoption, reconnect, wake, hidden-tab measurement, or resize path produces a layout that lost the reserved pane without an explicit user close. Explicit close intent is per browser tab and suppresses self-heal until the user restores Finder/Differ.
- Dockview must not adopt a layout snapshot while its host is hidden or measured at zero area. A visible small host still lays out at its real size; only serialized snapshots use functional fallback dimensions so a sleep/hidden-tab measurement cannot become the persisted layout.
- Finder/Differ is reserved as a target. Nothing can be dropped into its center, left edge, right edge, or top edge.
- The only allowed drop onto Finder/Differ is the bottom edge, and only when Finder/Differ is large enough for the existing Finder/Differ pane and the incoming tab after the split.
- Finder/Differ itself is not draggable as a layout tab. Dragging Finder/Differ must not advertise a pane split, root split, or gutter split.

- Directory drags into terminal content keep the terminal path-insertion affordance. They may show a normal-pane split preview when the target is not Finder/Differ and the pane has enough room.

Agent panes may insert the path first and then show a transient non-modal suggestion overlay on `1` through `9` (top-row or numpad); plain shell panes show explicit read-only shell commands (`file`, `wc`, `tail -F`, `jq`, `column -t`, directory size scans) instead of inserting a stray bare path; server actions can show bounded file info, preview, log scan, data-stats/chart, or OCR results in a result panel. External/upload drops and pasted image uploads use the same action registry after the upload path is known; pasted image uploads must still insert the `[Image #N] '/path'` reference immediately. Escape, click-away, normal typing, or timeout dismisses the overlay without running anything.

Image paste/drop ordering is controlled by `uploads.image_action_order`; that Preference is authoritative, stores prompt actions as `Popup label: ; prompt text inserted after the image path`, accepts non-AI rows as the popup label or a short special action such as `info`, and is capped at 9 rows because the popup assigns shortcut keys `1` through `n`. The pasted image path/reference is inserted before the popup, so `Insert path` is not part of the image action order. The default image order is agent OCR, error diagnosis, image description, then `info`; stale image-order rows for server OCR and file type are dropped. Custom action lines use `Label | prompt text or shell:command | optional comma-separated categories` and support `{path}`, `{qpath}`, `{paths}`, `{qpaths}`, `{name}`, `{count}`, and `{category}` placeholders. Read-only shell autorun is allowed only when the `uploads.suggestion_autorun` Preference is on; agent prompts and server result-panel actions never auto-enter terminal text.

## Reconnect And Wake Rules

- Client-events reconnect must backfill live auto-approve status so YO markers reflect the current working/idle state immediately after the stream is ready again. Page visibility returning to visible and browser `online` events must schedule the shared refresh path, and a narrow auto-status fallback poll may run only while the client-events stream is disconnected.
- The YO browser favicon badge counts active work only: sessions whose live auto-status screen state is `working`, plus explicit active states such as tests-running or YOLO approval. It must not count tmux's selected pane/window flags, open terminal websockets, idle Claude/Codex processes, or attention-only states as active work; every tmux sub-window has a selected pane even when idle.
- For sessions with multiple detected Claude/Codex panes, the backend keeps per-agent-window status rows keyed by tmux sub-window/pane target. Any scalar session summary follows the selected/preferred agent pane first, then a deterministic agent-pane fallback, but YO markers, working counts, Tabber, popovers, and Recent Agents must consume the per-window rows so they do not get stuck on an older idle agent when a later window in the same tmux session is working.
- A session's YO marker is GREEN/active ("on here") when THIS server is auto-approving the session, YELLOW/locked ("on elsewhere") when ANOTHER YOLOmux server on the same machine is auto-approving it, and inactive otherwise. With multiple servers sharing one tmux, clicking YO green on one server must make the same session read yellow on every other server. Cross-server detection: a server with no local worker for the session reports `enabled_elsewhere`/`locked` by probing the auto-approve lock for that session's agent-pane targets (`auto_approve_agent_targets`) — i.e. the exact lock a YO worker holds — falling back to the bare session-name lock for the no-agent case. Probing only the session-name lock silently dropped the yellow marker for every agent-backed session, because YO workers lock per agent pane (e.g. `%1`), not the bare session.
- Session tabs reserve their fixed-width numeric prefix only when the shared leading status ball is present. A tab with no ball compacts that prefix so its branch and PR metadata does not begin after an empty status column.

## Tabber Pane Rules

- The Tabber is the third mode of the one shared file-pane mode control (`Finder / Differ / Tabber`); like Differ it takes over the pane (the file tree is hidden, the changes panel fills the pane). Legacy `files`/`diff` behavior is unchanged. A newly restored/default file pane opens in Finder mode; Tabber is an explicit mode selection, not the startup default for the left file pane.
- Tabber shows a live tree of everything open and defaults to fully expanded. Level 0 = tmux sessions (sorted first), then the other open tabs (Preferences, YO!info/YO!agent, file editors) as leaf rows at the end. Session rows are collapsible like other directory rows: clicking only the disclosure arrow toggles collapse/expand, while clicking anywhere else on the green `N <description>` session line opens/activates that tmux tab. The Tabber session label must render through `tmuxPaneTabTokenHtml` and use the shared `.tmux-pane-tab-token` shell for visible tab styling, including inactive border, hover, active colors, and top-only rounded corners; Tabber CSS may only add row layout/truncation overrides and the selected-row focus ring. Level 1 = the session's tmux sub-windows (`index:process (pid=N)`, e.g. `0:claude (pid=12345)`) rendered through the same compact `.tmux-window-button` shell used by YO!info tmux sub-window values and the pane Info Bar. Claude/Codex tmux sub-windows show the shared agent icon and smaller sub-window status ball inside that button shell. The session's current tmux sub-window is shown by row emphasis such as bold text plus `aria-current`, not by adding another circle/dot beside the agent activity icons. Under a `claude`/`codex` tmux sub-window, Level 2 = absolute touched repo roots attributed to that same tmux agent sub-window from the backend `agent_windows.path_entries` record in `/api/activity`, with a folder icon and the git branch on the matching repo row. A repo touched by `0:codex` must not render under `1:codex` or `1:bash` just because it belongs to the same tmux session. The backend builds those path entries from the cached `/api/session-files` attribution and git metadata; the browser must not refetch or reparse `/api/session-files` to decide per-sub-window path/git/active state. These touched-path rows are file-kind leaves and use normal text color; Tabber must not list every individual file, nested repo subdirectory, `/tmp` artifact, or `.claude/projects/...` transcript path under them. Non-agent tmux sub-windows are leaf rows.
- Startup behavior: immediately after launch transcript metadata is available, Tabber should already be able to render the Level 0/Level 1 session, process, PID, active-window, and Claude/Codex icon rows. PID text is visible inline inside compact Tabber sub-window labels. The browser starts one `/api/activity?hours=<lookback>` request for the server-cached activity snapshot; when that snapshot contains `agent_windows`, path rows render from it. This first useful paint must not wait for a browser-side `/api/session-files` fetch; touched repo/path folding is backend-owned and may update on the normal cached activity cadence.
- Toolbar behavior: Tabber keeps only the shared mode switcher and pane frame controls. Finder/Differ session controls, Finder-only controls (`.*`, Sync/path controls, create file, create folder, Finder sort/date/reload controls), and Differ-only controls (Differ expand/collapse) are hidden because Tabber is an inventory/navigation view, not a filesystem editing or changed-files view.

CONDITION — a file outside every searched root will NOT appear even on an exact-name query; this is a scope limit, not a ranking bug. The fix is to mark that file's directory (or an ancestor) indexed in the Finder, not to change ranking.

So if an exact-name file is absent from results, it is outside the searched roots (see scope above), not mis-ranked. Dominance order within results: (1) per-surface domain prior, (2) anchored prefix on the primary name (typing `6` puts file `6.md` first in `Mod+P`, pane `6 …` first in `Shift+Mod+P`), (3) contiguity (`123` ranks `hello 123` above `1 a 2 b 3 c`), (4) recency (file index mtime; pane recency uses latest meaningful agent transcript event timestamp from the session payload, then summary timestamps as fallback), (5) focused-session repo affinity. Match quality dominates recency; recency dominates repo affinity. For typed blended results, the first visible screen keeps a bounded mix of file and pane matches so async file-index results cannot crowd a strong pane hit offscreen. The scoring and first-screen mix weights are ONE exported data table guarded by table-driven ranking tests, so behavior is tuned by editing weights + expected-order rows, not by rewriting ranking code.

## Session Project And Repo Detection

How a tmux-session tab header and YO!info decide which git repo(s) a session is working in, and when they show `no git checkout detected`. Implemented by `candidate_session_cwds` -> `session_repo_summaries` / `session_git_inventory` in `yolomux_lib/metadata.py`, with the transcript edited-dir signal from `session_touched_dirs` in `yolomux_lib/session_files.py`.

- The repo is derived from MULTIPLE signals, not just the live pane cwd. Candidate directories, strongest first: (1) the directories the agents have actually EDITED files in, read from each agent transcript's edit-tool scan (Claude `Edit`/`Write`/`MultiEdit`/`NotebookEdit`, Codex apply-patch — edits only, NOT files merely read), (2) the focused pane's current path, (3) each agent's launch cwd, (4) other panes' current paths, (5) the session-number default workdir, including a matching `~/yolomux.dev800N` checkout for session `800N` when present, (6) configured companion dirs. The edited-dir signal (1) is strongest because it is real work evidence, not a guess.
- Each candidate directory is resolved to its git top-level (`git rev-parse --show-toplevel`) and deduped by repo root. The PRIMARY repo shown in the tab header is selected by signal strength first, then by the most recent git activity inside the same signal tier (the later of dirty-file mtime and last-commit time). The live pane cwd, agent launch cwd, and other pane cwd signals share one tier; transcript edited dirs sit above them. YO!info lists every distinct touched repo.
- CONDITION — agent launched from a non-repo cwd: a Claude/Codex started in `$HOME` (or any directory with no git checkout) that is editing files inside a real repo MUST still surface that repo and branch via signal (1). The header must not show `no git checkout detected` while the transcript shows edits in a real repo. (This was the "8003 shows no checkout while really working in `~/yolomux.dev8003`" bug.)
- REQUIREMENT — `$HOME` is never a project repo. `focus_root_for_session` explicitly excludes `Path.home()`, so a home-directory checkout (even a valid one) is never treated as the session's project; home-based sessions surface their repo only through the other signals.
- REQUIREMENT — a directory counts as a "checkout" only if `git rev-parse --show-toplevel` succeeds. A `.git` directory existing on disk is NOT sufficient: a stub/incomplete `.git` that lacks `HEAD`/`objects`/`refs` (for example a placeholder with empty `config`/`hooks`) makes git return rc=128, and the directory must be treated as repo-less. This is why `$HOME` with a placeholder `.git` reads as `no git checkout detected` rather than as a repo.
- CONDITION — no signal yields a checkout: when none of the candidate directories resolve to a git repo (fresh shell in a non-repo directory, no edits yet), the tab subtitle is `no git checkout detected` (`t('git.noCheckout')`). That is the correct terminal state, not a failure.
- PERFORMANCE REQUIREMENT — detection runs on the hot metadata-refresh path (every session, every refresh) and now scans agent transcripts for edited dirs. The Claude transcript scan MUST be memoized by `(path, mtime, size)` (matching the Codex scanner's cache) so an unchanged transcript reads from disk once; the cache key MUST include mtime+size so an appended transcript re-scans and a newly-touched repo appears without a stale-cache miss.

## YO!share (Live Session Sharing)

YO!share lets the host share the live YOLOmux layout to other browsers via a short `/share/<id>#t=<secret>` URL. Read-only viewers observe; write viewers may also type through validated input intents. The guiding rule: the client mirrors the host's visible state and never re-derives geometry or invents transient UI state. The detailed DOM replay architecture spec lives in [`SHARE_MIRRORING.md`](SHARE_MIRRORING.md). See also the File menu entry in Search, Menus, And Popovers and the `share.*` Preferences.

Architecture: read-only and write viewers use host-rendered DOM replay by default. The host is the only renderer for mirrored app state, streams sanitized `#appRoot` keyframes/deltas, and terminal panes stay on the host-sized byte stream because xterm internals do not serialize through DOM mutation. Viewer DOM is inert. Write clients send allowed input intents to the host/server and receive host replay frames for visual feedback. Normal viewer chrome shows replay health (`mirrored`, `resyncing`, `host disconnected`, `viewer behind`) rather than semantic bucket names.

Replay repair is keyframe-based. Any host topology or global-UI change that cannot be trusted as a small delta may publish or trigger a complete epoch-keyed DOM keyframe. Clients apply frames only in sequence for the current epoch; gaps, digest mismatch, slow-viewer backpressure, and manual debug requests ask the host/server for a new keyframe. Digestless mutations that target an unknown replay node are treated as stale deltas from a replaced subtree: the viewer advances the replay cursor without entering `viewer behind` and waits for the next scheduled keyframe. Terminal resize remains separate: `terminal-host-resize` resets viewer xterms to host rows/cols before fresh tmux repaint bytes arrive.

### Sharing UI, mirror frame, and viewer rules

- File -> Open File is the first command under File, then Finder, then Search & Runs, then YO!info/YO!agent, and File -> YO!share sits immediately after those commands and before Preferences. YO!share opens the shared modal chrome, not a separate dialog, and shares the entire current YOLOmux layout: pane tree, split percentages, active/background tabs, Finder/Differ/Tabber state, editor tabs and view mode/theme/preview/split state, Preferences state, app-menu state, and every tmux session present in pane tabs. When the share action is available or sharing is already active, the File menu detail says only `sharing`. The create view field order is Max time to share, Max viewers, Read-only, Protocol; Max time to share is a typable minutes number field, not a dropdown. Defaults come from Preferences `share.ttl_seconds`, `share.max_viewers`, `share.read_only`, and `share.scheme`; `share.max_viewers` defaults to 2, read-only defaults on, protocol defaults to http, unchecking Read-only forces https and disables http with a short hint, and a non-TLS page forces Read-only and http. The manage view lists every active share with a readonly URL input whose copy icon sits immediately to the right of the URL text, then expiry, viewer count, mode, protocol, and per-share Stop control, while keeping the create form available for another share inside a bounded, scrollable modal body. Viewer caps count distinct viewer browsers, not the number of terminal websockets opened for that browser's shared pane layout. While shares are active, the host topbar shows a compact single-share pill (`YO!share: N viewers · M:SS left`) or an aggregate multi-share pill (`YO!share: S shares · N viewers · M:SS left`) that reopens the manage view; read shares use the green/good mode color and write shares use the red/bad mode color. A share-view browser shows a slim banner with host, mode, expiry, pushed status, replay health, and a cover/contain fit toggle; this banner lives outside the mirrored app root and must never shift host layout geometry. Share viewers render sanitized host DOM inside a host-sized `#appRoot` replay frame whose viewport, appearance settings, bundled UI/mono fonts, and terminal placeholders come from the host, then one root transform maps that frame into the viewer window. The default fit is `cover` from `share.view_fit`, cropping the long axis symmetrically; `contain` letterboxes instead. The viewer must not reflow the app against its own viewport, and raw `window.innerWidth`/`window.innerHeight` reads outside the shared viewport owner are forbidden. Host terminal cols/rows stay pinned and viewer resize only changes the mirror transform, never the shared tmux size. Read-only viewers are passive. Write viewers may send validated input intents over HTTPS, but they do not publish semantic layout, Finder, editor, Preferences, popup, or scroll state; host replay frames provide visual feedback.
- YO!share URLs must stay short (`/share/<id>#t=<secret>`); sessions, layout, tabs, and UI state live server-side behind the id. Host-owned popups, menus, context menus, language pickers, modals, and transient messages mirror as normal sanitized DOM inside `#appRoot` through replay. Share-modal URLs and token-bearing fields are marked as share secrets and redacted before DOM keyframes, deltas, debug copy, or server UI frames reach clients. Mirrored modals use the host DOM and host content sizing, with no client-only reflow, forced child height, or extra scrollbar. Active share rows are compact: `New share` and `Active share URLs` sections, max time/viewers/read-only/protocol grouped, URL plus copy in one row, and Stop inline with mode/protocol. The active URL block is the primary share affordance in the modal; it must be visually stronger than metadata rows while keeping exactly one copy target beside the URL input. Only the host can stop or extend active shares; viewers only receive refreshed countdown and viewer-count status. Read-only viewers must preserve browser text selection, copy, context-copy, and harmless navigation defaults while blocking activation and mutation. Editor/diff parity comes from host replay plus share-scoped file reads for every host-visible file editor identity (`file:`, `filediff:`, `filecopy:`, and `image:` tabs). Saves still publish file-version invalidations for open shared files. Pre-auth login/setup language pickers use in-page buttons and hidden inputs rather than native selects so OS popups cannot leak as broken share-view boxes.
- When a share viewer's host-provided expiry reaches zero, the viewer must navigate to the login screen immediately through the shared login redirect path; it must not wait for a websocket close, terminal reconnect loop, or the low-frequency `/api/share` backup poll. The login `next` target includes only the pathname and query, never the `#t=` fragment token.
- The YO!share manage view must show a `Users` section inside each active share row. The section lists one row per distinct connected browser, with connected duration, IP address, and summarized browser type. It updates from pushed share status on join/leave without requiring the host to close and reopen the modal, and it must not expose full user-agent strings or share secrets in the mirrored popup layer. The Users section is an inline subsection of the active share row: row separators are allowed, but it must not draw a second card border, framed table box, or card background inside the already-bordered share row.
- YO!share clients mirror the host's visible UI state, not the client's inferred state. When the host opens, closes, hovers, unhovers, scrolls, changes language, changes theme, changes layout, changes active panes/tabs, toggles tab metadata, switches YO!info/YO!agent, or expands/collapses editor diff context, the client must converge to that exact host state and clear any stale mirrored surface immediately. This includes the YO!share modal, all app menus, submenu hover state, language popovers, tab hover popovers, context menus, transient messages, editor scroll, Finder/Differ scroll, pane split geometry, active/focused pane chrome, tab metadata visibility, YO!info/YO!agent active sub-tab, and popup font sizing. Client-side hover or focus must not be required to clear stale host UI, and read-only replay clients must not synthesize local tab popovers from their own pointer position. Host tab popovers must carry their computed app-space `left`, `top`, measured inline `width`, and measured `height` on the replayed popup node, not only in document-level CSS variables or an `inset` shorthand, so Safari/Chrome viewers under a mirror transform render the same host position and wrapping. Client-side scroll must not persist or snap mirrored panes back to a local default; read-only client scroll input is blocked for mirrored app surfaces except where browser text selection requires a harmless default. Host menu and popover mirroring is an open/close contract as well as an HTML snapshot contract: a closed host popup means no mirrored popup remains on any client. Language changes must update host and clients atomically enough that no pane blanks or loses its current content while theme/locale text is re-rendered. Theme changes must update replay-shell viewers through live `appearance` frames as well as DOM replay, preserving the host `theme`, `resolvedTheme`, and `terminalTheme` values so mirrored terminals repaint across light, dark, and system. User-facing language choices show human-readable language names/endonyms only; raw locale codes such as `en`, `es`, `ja`, or `zh-Hans` are not visible UI labels except in explicit developer/debug output.
- YO!share replay keyframes must include the host's current scroll targets, compact YO/auto-approval state, theme/language/font readiness, popup/menu/modal DOM, terminal placeholders, and share-visible file/editor identities, not only the pane tree. This is required for late-joining viewers, keyframe repair, Preferences scroll, tmux/Tabs badge parity, and host menu rows that depend on YO state. Live editor/file refresh for share viewers is push-driven through share UI/file-version frames; `/api/share` polling on read-only viewers is only a low-frequency status backup for banner metadata, not a high-frequency editor refresh mechanism. Same-epoch incremental DOM replay updates should keep applying even when a viewer detects a sequence gap, while viewer repair keyframe requests and host repair keyframes are throttled to a five-second floor. Topology and join keyframes keep host mutation deltas quiet for the keyframe floor plus slack so Finder/Differ/minimize loops do not interleave node-id deltas from an old DOM base between full replay frames. Semantic geometry resync stays on a ten-second floor.
- YO!share visual parity is a replay-health contract. Host and viewer must load the bundled `YOLOmux UI` and `YOLOmux Mono` fonts before the first mirrored layout so serialized DOM dimensions stay stable, but viewers do not compute their own wrapped-text or pane geometry to decide whether they are synced. Read-only replay health asserts `domDigest`, epoch/sequence continuity, dropped-frame/keyframe-request counts, redaction policy version, node count, and terminal placeholder health. Legacy semantic geometry/text-wrap digest buckets such as `slots`, `tabStrips`, `editors`, and `textWraps` may remain only for the explicit semantic escape hatch or targeted diagnostics; normal read-only replay must not show bucket names or repair by applying another semantic widget snapshot.
- YO!share debug diagnostics are opt-in with `?shareDebug=1` on a share-view URL or `localStorage.yolomux.shareDebug=1`. Replay debug records one sanitized `share-replay-health` report on `window.yolomuxShareDebug` with DOM digest, epoch, sequence, dropped-frame count, keyframe request count, redaction policy version, node count, terminal placeholder health, fit mode, viewport, `visualViewport`, DPR, user agent, root/stage rects, and mirror transform. The debug banner copy control appears only in debug mode and copies the latest report as JSON. Share tokens and `#t=...` fragments must be redacted before console logging, window exposure, or clipboard copy. Legacy semantic drift reports are allowed only when the semantic escape hatch is active.
- YO!info is part of the same host-owned share state as Finder, Differ, editors, and Preferences. Branch-table row content, sort column/direction, resizable Branch and desc column widths, vertical scroll, and horizontal scroll must publish from the host and apply on read-only clients. A viewer must not rebuild YO!info rows from its own scoped transcript metadata after a host snapshot arrives; it renders the host's rows until the host sends a newer row snapshot. Column widths are part of the mirrored UI state, not browser-local storage on the viewer; applying them on a client must not persist them or echo-publish a client-authored width. A read-only client wheel, scrollbar drag, hover, focus, or other local interaction over YO!info must not publish or persist a client-authored position; it restores to the last host-authored `#info-content` scroll frame.
- Finder/Differ/Tabber share state is host-owned at the model layer, not only at the DOM event layer. The host snapshot owns mode, root mode, root path, target session, expanded paths, selected paths, tree date/sort settings, changed-file collapsed sets, Tabber collapsed set, and per-repo Differ refs. A read-only client's local transcript refresh, `/api/session-files` result, hover/focus state, active-tab sync, or manual open attempt must not retarget the root, collapse/expand the tree, or switch Finder/Differ context. Repeated host frames with the same root must refresh in place when needed and must not reopen the root in a way that briefly clears expansion; only a changed host root should reopen the tree.
- HTTP read-only share parity covers the same visible surfaces as HTTPS share parity. A viewer loaded from an HTTP share URL must render the host's Finder rows without a red path field, pane tab metadata without `no path` or `finding branch...` fallbacks, file editor contents without `File could not be opened` / `TypeError: Load failed`, and Differ/Tabber data without local fetch-error placeholders. These checks must run in Selenium against a real `http://.../share/<id>#t=...` page so token propagation, plaintext redirect exemptions, and browser fetch/WebSocket behavior are exercised together.
- YO!share popups, menus, context menus, language pickers, modals, and transient UI mirror as ordinary sanitized DOM inside `#appRoot`. Popup roots must mount under the app overlay owner so the replay serializer sees them; stale popup resurrection is prevented by replay epoch/sequence handling and keyframe replacement, not a separate popup-layer shell.
- Share-view terminal grids are host-PTY authoritative. A viewer terminal must size from the host's rows/cols only; it must not fall back to `estimateTerminalSize()` from the viewer pane, native viewport, local font metrics, or transient container visibility. A host resize, `terminal-host-resize` frame, or terminal placeholder update that changes rows/cols must resize the viewer xterm, call `reset()` to discard any smeared/refowed buffer rows, and rely on the server-side `tmux refresh-client` repaint to refill a clean grid at the host size. The server must deliver `terminal-host-resize` before requesting the tmux repaint so clients reset before fresh terminal bytes arrive. A viewer that reattaches to an already-running shared upstream must request a bounded `tmux refresh-client` repaint; otherwise layout rebuilds can create a blank xterm that does not receive a full curses screen until manual refresh. If host dimensions are unavailable, the viewer skips resizing rather than inventing local dimensions.
- Dockview panel show/hide is part of the terminal lifecycle. When a share-view host frame removes or minimizes Finder and Dockview remounts tmux panels or activates tmux tabs that were hidden during the topology change, the viewer must ensure the xterm is bound to the currently connected terminal container, reconnect/restart stale bindings, and keep using the host rows/cols.
- Read-only share viewers are visually host-identical inside the mirrored app. They must not add client-only disabled styling, `readonly access` labels, native scrollbars, draggable scrollbar tracks/thumbs, or other read-only chrome inside the mirrored Preferences/Finder/Differ/editor/terminal surfaces. Read-only protection is enforced by the inert replay shell and host-authored scroll frames; host scroll is applied programmatically and retried after keyframe/delta apply so a late DOM rebuild cannot snap a viewer back to a local default. Viewer-only controls such as the share banner, fit toggle, and replay health stay outside the mirrored app root.

### Transport and server architecture

- HTTP read-only share works even when YOLOmux is launched with HTTPS, on the SAME port (no second listener). The server muxes protocols by peeking the connection's first byte (`TLS_FIRST_BYTES`); the peek timeout must stay generous (`tls_peek_timeout_seconds`, currently 2.0s) and must NOT run so short that a remote client's first byte (one RTT away) is misread as a TLS preconnect — a 50ms peek silently broke the plaintext->https redirect for remote browsers. Plaintext requests on a TLS host are SERVED only for share-scoped paths (`/share/<id>`, `/`, `/static/*`, `/ws/share-view`, `/ws/share-ui`, `/api/ping`, `/api/share`, `/api/share-stream`, readonly share data APIs such as `/api/session-metadata`, `/api/transcripts` compatibility metadata, `/api/activity`, `/api/session-files`, `/api/fs/list`, `/api/fs/info`, `/api/fs/index-status`, scoped file APIs such as `/api/fs/read`, `/api/fs/diff`, `/api/fs/raw`, and readonly batch `/api/fs/batch`) AND only when the matching share record has `http_allowed` (`plaintext_share_scope_allowed`); every other plaintext request gets a 308 redirect to https (`redirect_plaintext_to_https_if_needed`). Plaintext share connections are read-only regardless of token mode, and no cookies are set on them. Write mode requires https.
- Terminal mirroring fans out from ONE upstream pty+`tmux attach` per shared session (`share_terminal_upstream`), never one attach per viewer (viewers x panes attaches would melt tmux). The upstream pty is pinned to the host's rows/cols; viewers never resize it. Each output chunk is serialized into one frame once and enqueued to every viewer (`broadcast`). Backpressure is per-viewer: a bounded outbound queue (`SHARE_VIEWER_QUEUE_HIGH_WATER_BYTES`) — on overflow the viewer's backlog is dropped and ONE rate-limited `tmux refresh-client` (`SHARE_REFRESH_CLIENT_MIN_SECONDS`) repaints all lagging viewers from a single screenful; a viewer that overflows repeatedly (`SHARE_VIEWER_OVERFLOW_LIMIT` within `SHARE_VIEWER_OVERFLOW_WINDOW_SECONDS`) is disconnected `too-slow`. One slow consumer must never buffer the server into OOM or stall the fan-out loop. Upstreams with no viewers are reaped (`close_inactive_share_upstreams`).
- Host resize propagation is a terminal repaint contract, not only a layout contract. `record_host_pty_dimensions` updates each matching share upstream, broadcasts `terminal-host-resize`, and only then requests `tmux refresh-client` after `set_pty_size`/SIGWINCH so terminal bytes following the resize redraw a full clean screen after the client has reset to the host rows/cols. Layout commits publish replay keyframes or deltas so terminal placeholders, tab metadata, YO!agent sub-tab state, Finder/Differ state, editor diff expansion, and Preferences state converge after pane rebuilds; semantic `layout`/`ui-state` snapshots are legacy escape-hatch behavior, not the normal share path.
- Viewer accounting is per distinct browser (`viewer_ids` on the record), not per websocket; status (viewer count, expiry) is PUSHED to host pill/manage-modal and viewer banner via `broadcast_share_status` on join/leave/extend, not only polled. The read-only viewer `/api/share` status poll is a backup and must stay low-frequency because normal UI, editor, file-version, scroll, popup, and viewer-count changes arrive through the share WebSocket. The viewer cap refuses the WS upgrade past `max_viewers`. Records are in-memory: a server restart drops all shares (URLs become invalid -> login), which is acceptable and intended. The data model is keyed by token to allow N concurrent shares later (mixed modes/TTLs); v1 may gate to one active share at mint time.
- Host-local surfaces must NOT be mirrored to viewers: the latency meter is the host browser's own RTT and is meaningless on a client; viewer-only chrome (the share banner, fit toggle, replay health status) belongs to the viewer, not the host; the real OS cursor is replaced by the ghost cursor; share secrets/URLs stay redacted. Decision rule for any new mirrored surface: host-owned visible state inside `#appRoot` mirrors through sanitized DOM replay with the client inert; client-local or host-private state is redacted or excluded. When unsure whether a surface belongs in the mirror, register it with the replay/redaction policy and add a replay health or sanitizer test rather than adding another semantic snapshot/apply pair.

## Preferences Contract

- Preferences is a normal tab type and may be moved, activated, minimized, and closed like other non-Finder tabs.
- Preferences must expose a search field, collapsible sections, per-row reset controls, a global reset flow, the settings file path, and a settings age/status indicator.
- Preference changes save through the shared settings API and live-apply when the setting affects visible UI. Appearance, terminal/editor, Finder/Differ, popover timing, notification, and YOLO changes must not require a page reload unless the setting explicitly says so.
- General settings must include language, auto-focus, and startup tips. The legacy default-sessions setting is accepted from old config files but is hidden because running servers default to all discovered tmux sessions unless launched with `--sessions`.
- Appearance settings must include global theme, default layout, UI and Finder font sizes, tab width, max tabs per pane, pane spacing, pane ring opacity, inactive pane opacity, active color, separator color, editor cursor color, and 12-hour/24-hour clock mode. Notification settings include the green/red/yellow workflow transition glow duration. Changing Active color must live-apply the matching tmux status/window/pane styles to every existing tmux session/window, and newly created tmux sessions must receive that saved Active color immediately after creation.
- Terminal / Editor settings must include terminal theme, terminal/editor/preview font sizes, terminal scrollback, editor dark/light schemes, editor cursor style, word wrap, line numbers, autosave, autosave delay, and all-lines blame.

- YO!agent is the central command surface for YOLOmux. It must reason over all visible tmux sessions, not only the focused pane, and explicit send requests run without an extra confirmation unless the user asks for preview or confirmation. Explicit target-session sends return results by default: the first YO!agent response must confirm the send immediately, state that the target session was verified as accepting an AI prompt, show the exact prompt YO!agent is sending after perspective rewriting/redaction, and say it is awaiting the response; a background watcher then reads the target agent transcript delta, falls back to visible pane text on timeout, and appends a result message through the persisted conversation plus `yoagent_conversation_changed` client event. Target-session prompts preserve perspectives: YO!agent keeps the user's routing perspective local, then writes the target prompt from the target agent's point of view instead of blindly forwarding the user's wording. `ask agent 1 to <do ...>` sends `<do ...>` to agent `1`, not `ask agent 1 to <do ...>`, and `ask session 1 what it has done today` sends `what have you done today?` to session `1`, while YO!agent's local explanation may still say it is asking tmux session `1`. Generated handoff prompts should be concise clean tasks/questions, not multi-line routing/context wrappers, unless the user explicitly provided raw quoted multi-line text. YO!agent may pass original, modified, or summarized information to another target session, but it should do that centrally: capture the source output, bound or validate it, transform it when requested, and send a clean target prompt. When a current target input draft exists and the user explicitly asks YO!agent to send, YO!agent clears that draft first, verifies the composer is empty, then pastes and submits the new prompt; if the draft cannot be cleared, it refuses and does not paste. Old submitted `❯ ...` prompt lines visible above a current blank composer must not be treated as current unsent text. Firing notify jobs use the existing toast/browser notification path. Wait-then-send jobs do not paste into a busy, non-agent, approval-prompt, missing, or stale target; they re-preview and revalidate the exact pane/agent before sending. The job state tracks the per-session states and blockers so a future job list can show which session is still preventing the watch from firing. Batch fanout actions should remain per-target confirm/cancel instead of one opaque batch button.

Existing user-owned visible Claude/Codex panes use `tmux-legacy`: server-resolved visible pane paste plus Return, prompt-state preflight, stale-target checks, and post-send composer-cleared verification. `tmux-legacy` is not a native agent API and must never be described as more reliable than terminal automation. The planned order is Claude SDK for YOLOmux-owned Claude sessions, Claude Channels for opt-in visible Claude panes, Codex SDK before direct Codex app-server calls for YOLOmux-owned Codex sessions, Codex MCP server for central orchestration, `codex-exec` for one-shot jobs, Claude stream-json as a CLI fallback, and `tmux-legacy` last. If a future visible TUI is attached to a managed session, the UI must label both identities: the visible tmux pane and the structured provider/thread that YO!agent controls.
- YO!agent multi-session handoffs are always orchestrated by YO!agent, not delegated from one target session to another. In a normal request, session `1` and session `2` do not know about each other; YO!agent coordinates and passes clean messages between them unless the user explicitly asks for disclosure. Direct relay/chaining between agents is rare and allowed only when the user explicitly requests relay/chaining; when allowed, YO!agent must pass concrete instructions that say how to relay, and must not leave the agent to infer the route. For a request such as "ask session 1..., get info from session 1, pass that to session 2", YO!agent asks tmux session `1`, waits for and records session `1`'s response, derives a bounded handoff prompt itself, verifies tmux session `2` is accepting an AI prompt, sends the handoff to session `2`, waits for session `2`, and appends session `2`'s result back into the YO!agent conversation. Multiple simultaneous waits must render as multiple rows and clear independently as each wait finishes or times out.

## Keyboard Shortcuts Contract

- The app modifier is platform-specific: Mac uses `Cmd`; PC/Linux uses `Ctrl`. Shortcut labels must say the concrete platform modifier and must not display combined `Ctrl/Cmd` text.
- Mac app shortcuts use `Cmd` so they can be captured even when a terminal is focused; Mac `Ctrl` stays available for terminal/tmux input. PC/Linux app shortcuts use `Ctrl`.
- Global shortcuts are captured before focused controls swallow them, but text inputs, rename fields, editors, and terminal input must keep their normal typing shortcuts unless a shortcut is explicitly platform-owned by the app.
- Implemented app shortcuts: `Mod+P` opens unified search with file-first ranking, `Shift+Mod+P` opens the same unified search with pane/command-first ranking, `Mod+K` opens YO!share, `Mod+B` toggles Finder/Differ, `Mod+Alt+B` opens YO!agent in a right pane, `Mod+,` opens Preferences, `?` opens Keyboard Shortcuts and Legends, `Esc` closes shortcut/menu overlays, and `Mod+W` closes the active closable tab.
- Help -> Keyboard Shortcuts and Legends must document implemented global/context shortcuts plus the color, status, icon, and YO button meanings used by tabs, Info Bars, YO!info, and menus. On wide viewports it should use compact three- or four-column packing with balanced section stacks so users can scan it without scrolling. YO buttons are square rounded-control badges, not circular pills.
- Close-tab fallback: outside text-editing contexts, `Mod+Backspace` / `Mod+Delete` may close the active closable tab. This fallback must not fire inside inputs, rename fields, editors, or terminal text entry.
- Editor shortcuts: `Mod+S` saves, `Mod+F` finds, `Mod+H` replaces, `Mod+G` goes to line, `Mod+/` toggles comments, `Tab` / `Shift+Tab` indent and outdent, `Mod+Z` / `Shift+Mod+Z` undo and redo, and `Mod+Alt+[` / `Mod+Alt+]` move editor navigation back and forward.
- Diff editor chunk shortcuts reuse editor undo/redo: `Mod+Z` undoes the selected chunk action and `Shift+Mod+Z` redoes it when the diff editor owns the focus.
- Finder/Differ keyboard behavior follows Finder-style list navigation: Arrow Up/Down moves selection, Shift+Arrow extends selection, Home/End jump, Shift+Home/End extend to edge, Arrow Right expands or steps into children, Arrow Left collapses or steps to parent, Space toggles quick preview, type-ahead selects by prefix, `Mod+A` selects all, `Mod+O` and `Mod+Down` open, `Mod+Up` opens the enclosing folder, and Return starts rename.
- Finder/Differ delete is platform-specific: Mac uses `Cmd+Delete` or `Cmd+Backspace`; PC/Linux uses plain `Delete`. It must be scoped to Finder/Differ and must not run from text inputs or rename controls.
- Terminal copy handling must preserve normal terminal input: `Cmd+C` on Mac or `Ctrl+C` on PC/Linux copies xterm selection when present; without a browser/xterm selection, the tmux copy bridge may request the tmux copy-mode selection. Claude and other TUIs can own the visible highlighted block and expose it only through OSC 52; terminal right-click `Copy` must prefer the selection captured before xterm clears it, treat an explicitly captured empty selection as authoritative, and then fall back to the recent OSC 52 app clipboard payload instead of re-reading a tiny under-cursor browser/xterm fragment. After a terminal menu action consumes selected text, stale browser/xterm selection must be cleared. Explicit `Copy tmux selection` must not leave tmux copy-mode rows visibly selected after the clipboard payload is captured.
- Terminal URL links must stitch hard-wrapped TUI URLs into one clickable link even when the TUI emits each visual row as a separate non-`isWrapped` line. Hanging-indent continuations and zero-indent width-1 continuations are allowed only when the previous row ends in an unterminated URL token at the terminal edge; a following fresh `https://`, `file://`, or `www.` row stays a separate link.
- Terminal file path references that resolve to an existing file through the same context as the right-click/open path action show a subtle persistent underline in the terminal. Missing/unresolved path-looking text stays unmarked. Hovering a resolved file reference keeps the normal xterm link handling but uses only a slight underline contrast increase so the hovered target remains visible without a bright bar.
- Tab drag, pane split, and root-edge layout changes are currently pointer-driven. Keyboard equivalents for moving tabs/panes are not implemented and must not be advertised as available until they exist and are tested.
