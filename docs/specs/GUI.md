# YOLOmux GUI Behavior Spec

This document is the working GUI contract for pane, tab, Finder/Differ, editor, preview, Preferences, keyboard shortcuts, drag/drop, and YO!share live-mirror behavior. It is meant to drive future implementation and tests. User direction during implementation overrides older wording in this spec; when that happens, update the spec and the tests in the same change.

Section order follows the reading path: capabilities and terms, layout and tab mechanics, agent status indicators, pane metadata, chrome/resize behavior, special panes (Finder/Differ and Tabber), editor/preview behavior, search/menus, YO!share, Preferences, keyboard shortcuts, test coverage, and the audit backlog.

## Popular IDE Parity Checklist

- [x] Activity/workbench panes can be arranged as tabs inside split panes.
- [x] Tabs can be reordered within the same pane by pointer drag.
- [x] Tabs can be moved from one pane to another pane's tab strip.
- [x] Tabs can be dropped on pane edges to split the target pane.
- [x] Tabs can be dropped on root edges to create full-span panes beside the current layout.
- [x] Tab insertion preview appears between tabs, not over the target tab.
- [x] Active tabs have rounded top corners and a distinct active background.
- [x] Inactive tabs inherit the tab-container background instead of using a disconnected color.
- [x] Pinned tabs stay at the front of a pane, show a pin icon, cannot be user-dragged to another pane or LRU-evicted, and can be toggled from the tab context menu or the `Mod+K` then `Enter` shortcut. Pane minimize/close/expand is a pane operation rather than a tab drag: it remains available with pinned residents and moves the complete minimized pane normally. Toggling pin state while its hover detail is open updates the icon immediately without closing or rebuilding that detail.
- [x] Tab hover shows a detail popover with session/file state when a cursor-hover-capable pointer is available.
- [x] Tab context menu exposes tab-level actions before destructive session actions.
- [x] Open editors support edit, preview, split, and diff-capable modes based on file capability.
- [x] Markdown task checkboxes in Preview are interactive and update the source document.
- [x] Editor back/forward buttons navigate the visited-file stack.
- [x] Search results can render image/file references as `[Image #N] '/absolute/path'`.
- [x] File quick-open ranks basename matches ahead of path-only matches.
- [x] Cmd-P is a file-only Quick Open surface: exact filename matches rank before fuzzy path matches and command/action rows never displace them. Shift-Cmd-P keeps the mixed pane/command/settings search surface, but it never injects one contextual terminal-action row per verb for the active file; file actions stay in the file context menu. A capped full-tree walk is never presented as a complete Quick Open answer; it waits for the shared persistent index instead of showing misleading fuzzy rows. An empty `index_state: warming` response is still pending work: the palette shows Indexing and retries the same query rather than falsely rendering No matches.
- [x] Newly opened tabs (files, Preferences, YO!info/YO!agent, and other non-tmux tab types) open in the pane that owns the currently active tab; Finder, Differ, and Tabber use their independent-triplet placement rules and tmux sessions keep their placement rules.
- [x] Finder/Differ rows support selection, multi-select, rename/delete, context menu, date/sort controls, and image preview.
- [x] Preferences exposes searchable, collapsible sections with reset controls and live-applied settings.
- [x] Keyboard shortcut labels use the concrete platform modifier: Mac uses `Cmd`; PC/Linux uses `Ctrl`.
- [x] Global app shortcuts are captured before focused panes swallow them when the shortcut is platform-owned.
- [x] Code editor shortcuts include save, find, replace, go-to-line, comment toggle, indent/outdent, undo/redo, and editor back/forward.
- [x] Inline git blame can annotate the current caret line, with an option to annotate all lines.
- [x] Editor themes include dark/light mode switching, separate dark/light scheme defaults, caret style, and caret color.
- [ ] Partial: pinned-tab behavior is per browser profile through local storage, not yet a shareable URL or server-side preference.
- [x] Generic and Vertical Side Pane roles are explicit, and tab placement is capability-based. Finder/Differ/Tabber remain Vertical-Side-only, ordinary tabs remain Generic-only, and YO!info/YO!stats/YO!agent/YO!chat/YO!cost may move or be adopted across the role boundary; edge position alone never changes a pane's role.
- [x] Identity-only changes, including tmux-session and file-path renames, preserve the complete pane topology: split tree, pane role, tab membership, active tab, and pane sizes may not change except for the renamed item ID.
- [ ] Partial: preview/edit interactions are implemented for Markdown task checkboxes, not for every possible rendered widget.
- [ ] Keyboard-only tab reordering and pane splitting are not implemented.
- [ ] Multi-root project workspaces are represented through Finder roots and companion directories, not a full workspace model.
- [ ] Extension/plugin APIs for editor behavior are not implemented.
- [ ] Integrated source-control side panels are partial: Differ and inline blame exist, but a full branch/commit workbench is not implemented.
- [ ] Problems/debug/test runner panels are not implemented as first-class workbench panels.
- [ ] Terminal/profile management is tmux-session based, not a full shell-profile workbench.

## Terms

- Pane: a visible browser layout region that contains one tab strip and one active tab body.
- Tab: a layout item inside a pane. Current tab types include tmux sessions, Finder, Differ, Tabber, file editors and image viewers, Preferences, and YO!info/YO!agent.
- Generic Pane: the ordinary pane specialization. It may appear at any edge, may contain terminals, files, Preferences, Search & Runs, and the dual-role YO!info/YO!stats/YO!agent/YO!chat/YO!cost tabs, and uses the standard pane/tab sizing and controls. Moving a generic-only tab to the far left or right creates another Generic Pane; position alone never creates a Vertical Side Pane.
- Vertical Side Pane: the narrow outermost pane specialization with an explicit `side:left|right`. “Vertical” is part of the formal name: it is a fixed-width edge column whose leaves may stack only top/bottom. At most one Vertical Side Pane column exists per edge. It accepts Finder/Differ/Tabber plus the dual-role YO!info/YO!stats/YO!agent/YO!chat/YO!cost tabs; it rejects terminals and every generic-only tab. Its header exposes only pane minimize `_`, its tabs use intrinsic widths, its sash width is preserved across Generic Pane compaction, and its width is capped at one third of the browser. Finder/Differ/Tabber remain Vertical-Side-only, generic-only tabs remain Generic-only, and the YO!* tabs may drag or be adopted in either direction between the two pane roles. A YO!info tree in this narrow role keeps the same data and nesting, but stacks each record label above its value and uses compact tree indentation; branch/path values must retain a usable line width and may not collapse into one-character vertical wrapping.
- YO!cost is a first-class virtual tab with the established compact in-tab summary and charts. It uses the normal pane lifecycle, URL/layout restore, sharing, localization, retained YO!stats visual system, and internal scroll ownership; its explicit More Info button opens the shared scrollable detail view described below.
- Pane pop-outs use one shared same-origin popout host, detached-document writer, and pane-header control. File Preview supplies only its body/style payload and post-write interactive control binding to that document writer; it must not recreate the doctype/head/viewport/title/stylesheet/open/close shell. Phase 1 enables detached snapshots only for preview-capable file editors, YO!info, and YO!stats; terminal/transcript panes stay disabled until a live PTY/WebSocket design exists, and Preferences, Finder/Tabber, Search & Runs, and YO!agent stay disabled until interactive command bridging exists. Disabled tab types must have an explicit logged reason instead of silently omitting support.
- Finder, Differ, and Tabber: three independent file-surface tabs. Finder shows the file tree; Differ shows changed files and diff controls; Tabber shows open tabs, tmux sub-windows, and touched repo roots by recent activity. Legacy `files` / `__files__` plus `finder=<mode>` expands to the triplet on wide layouts with the corresponding item active; legacy `changes` / `__changes__` resolves to Differ.
- Root edge: the outside edge of the pane layout. Dropping there creates a full-span pane beside the existing layout.
- Pane edge: the edge of one target pane. Dropping there splits only that target pane.
- Cross-gutter: the separator between two sibling panes. Dropping there creates a full-span split at that sibling boundary.

## Layout Model

- Pane layout is percentage-based and reload-safe. `layout` and `tabs` URL parameters are the shareable source of truth.
- The URL `state` parameter carries reload-survival state that belongs to the current layout rather than only local browser preferences: editor modes and per-editor view state (`line`, `top`, `left`, `anchor`, `head`), Finder root/sort/expanded state, Differ session/ref/collapse state, Tabber collapse/lookback state, Preferences search/collapse state, and visible scroll offsets for panes that have scrollable surfaces. A full browser reload must reopen editor tabs at the same viewing line using the shared editor line-target path; raw pixel scroll is only a fallback because CodeMirror scroll snapshots are not serializable through the URL.
- A pane split must not advertise a preview if the target rect cannot fit both resulting panes. Left/right splits require enough width for the current target pane plus the incoming tab. Top/bottom splits require enough height for both panes.
- A middle drop must not advertise a preview if the target pane cannot display the incoming tab at its minimum width and height.
- Pane minimum sizes come from shared CSS/JS sizing tokens and tab-type minimums. Do not hard-code a one-window pixel capacity in a local drag handler.
- The app root always publishes the current layout viewport height instead of trusting CSS `100vh`, which Safari can retain as its larger browser-chrome viewport. If unzoomed `visualViewport.height` is larger than Safari's stale `innerHeight`, that larger current visible height repairs the root before the layout is rendered; a smaller unzoomed visual height replaces it only after the shared 140px keyboard reduction threshold. Responsive width remains tied to the layout viewport. The owner coalesces keyboard/rotation resize work through one animation frame, refits Dockview and visible terminals from that one geometry update, and ignores pinch zoom as keyboard geometry. `debug=1` exposes the raw layout/document/visual/root geometry and the keyboard decision.
- Vertical Side Pane availability is a viewport-capability rule, not a user-agent rule: it derives from the shared 300px minimum pane width and the one-third Vertical Side Pane cap, currently enabling Vertical Side Panes at 900px and above on desktop or touch devices. That 900px threshold controls only the special Side Pane role: below it, existing Side Pane leaves become Generic Panes without flattening ordinary left/right or top/bottom splits. The independent shared minimum-width rule controls true one-column mode; a portrait iPad such as 834px therefore retains and can create Generic Pane splits even though it cannot reserve the capped Finder Side Pane. Only true one-column layouts select one Finder/Differ/Tabber surface and replace the other two without removing ordinary tabs. A fresh phone layout still seeds up to three recent tmux sessions and no file surface. Constrained-to-wide creates `side:left` and restores Finder/Differ/Tabber there; YO!* tabs remain Generic rather than restoring a stale wide snapshot.
- A one-column pane frame has no sibling pane to minimize into. When it contains more than one tab, its minus and X act on the selected tab through the normal hide/close route; terminal panes therefore show X in that state. When only one tab remains, both controls are absent rather than removing the last pane and producing a blank layout. Multi-pane controls keep their normal pane-level minimize and close behavior.
- The compact minus/X glyphs on pane tabs retain their small visual size but use a 36px touch target. Closing a tab from the initial phone set changes the concrete one-column layout; the recent-session seed is only for a fresh view and must not add the hidden tab back during later normalization.
- The topbar packs from actual rendered geometry, never a viewport-width breakpoint. It starts full and, only while the row overflows or two visible controls collide, downgrades in this exact order: version, full wordmark to the existing localized boxed YO marker, background-owner status (`IDX|STATS|SESS`), Search to its short form, activity to nonzero green/red/yellow count balls left of Ping and every action button, optional buttons (latency, Exit, Bell and `#` together as one equal-priority tier, language, and editor navigation; Refresh remains), Search to an icon-only button, then the File/View/tmux/Tabs/Help roots to one localized `Menus` root. After the first fit, it walks that same sequence backward one step at a time and keeps each recovered control only when it still fits. Thus the active reductions always form one prefix of the stated order: spare space is used, but an earlier-priority control is never restored while a later-priority one remains hidden. Any dynamic topbar content change, including the background-owner label changing from loading to `IDX|STATS|SESS: follower`, schedules that same packer before the next paint. Shrinking and expanding through the same rendered width must produce the same reduction prefix even when a topbar control is focused. Compact activity keeps intrinsic width for every nonzero green/red/yellow ball; responsive fixed-icon sizing must never collapse the group to one button slot, and the packer treats overlap between the balls themselves as a collision. `Menus` contains File/View/tmux/Tabs/Help as its second layer and is an accordion: tapping its open File/View/tmux/Tabs/Help item collapses it, while tapping another item closes the previous category and opens the next. Tapping `Menus` again immediately collapses the current menu layer. Executing any navigation/action command, including a checked `File -> Finder`, closes the menu; only explicitly marked repeated settings toggles such as View notification/theme/sort/layout controls remain open. This topbar-only packing never changes desktop panes or split behavior. The editor arrows and Cmd-P launcher sit immediately after the five menu roots, and Search is directly between Help and the right-side controls; its shared center group owns the entire free region, centers that nav/search pair within it, and lets Search grow to a bounded useful width. The right-side controls retain their intrinsic width while that center region yields unused width, so Language or action buttons cannot overflow beside a visibly empty Search box. They remain normal flex siblings so a centered overlay can never cover Help. Topbar masked action icons, including the bell, use the shared centered icon grid and must not inherit a text-baseline offset. The generic operation-status element remains an assistive `role=status` live region rather than visible text beside Exit. These groups must never overlap or paint beyond their topbar grid cell.
- Opening Tabs renders the last accepted session metadata immediately, then starts one coalesced forced metadata refresh that performs the current tmux session listing. When the live result arrives, it updates the already-open Tabs rows in place without closing the menu or replacing its topbar target.
- Pane separators remain a skinny visible line. Desktop/trackpad users can drag its full length; touch users resize only from the centered 44px yellow/blue grab handle, so pane-edge taps keep their normal focus behavior.
- The active-pane surrounding ring uses the shared pane-tab radius token on all four outer corners. Dockview's ring pseudo-element inherits that radius, so the visible active outline and the selected tab have the same corner roundness rather than a rounded tab inside a square pane frame. A phone-only one-pane layout uses the same ring path at 1px because no competing pane needs a strong focus boundary; tablets and multi-pane layouts keep the configured pane-spacing width.
- On coarse-pointer browsers, actual editable inputs, textareas, selects, and contenteditable controls use at least a 16px computed font to prevent Safari focus zoom. xterm's hidden focus textarea follows that rule without changing the user-selected terminal canvas font size.
- A coarse-pointer tmux terminal keeps its full height until the user presses its floating keyboard launcher, initially in the lower-right corner of that pane. The launcher uses the shared inactive-pane-tab foreground/background/border tokens in both themes, suppresses text/emoji selection and the touch callout while retaining normal button focus, and is hidden while its palette is open; it never morphs into the close control. The launcher can be moved anywhere inside its owning `.tab-pane`; its stored coordinates are the single position model for both the launcher and palette. Drag-release never opens the palette by itself, and if that drag produces no synthetic click the next intentional tap must still open the palette with one tap. On first open the palette sits directly above the lower-right launcher position. After the launcher moves, palette placement is derived from launcher geometry: near the pane top it opens below, against a moved side edge it opens on the interior side, and otherwise it chooses the side with the most room while staying clamped inside the pane. Stored launcher coordinates also re-clamp against the current pane during render, pane resize, and window resize; hidden or zero-size transient panes preserve the last good offset instead of collapsing the launcher to `0,0`. The palette derives its width from the fixed key tracks plus the two `1em` navigation gutters and derives its height from the four key rows; at the default UI font it measures 252x171, leaves only a 1px border-safe inset on the right/bottom, and clamps responsively in smaller panes. Switching pages never changes its measured size or creates an internal scrolling full-keyboard surface. A dedicated X stays flush to the key area's top-right border-safe edge on both pages, immediately beside More and below the 21px drag strip. Backspace, More, and X are consecutive on the primary page. The full-width drag strip is the one palette drag owner and uses a contrasting surface, divider, and centered grip pill so the drag affordance remains visible in both themes. Dragging that grabber or other palette background/padding, excluding every key button and the X/More rail, moves the same remembered launcher position so the palette stays adjacent to where the launcher will reappear and remains fully visible inside the pane. The primary page keeps Esc directly above Tab in the left utility column, followed by Shift and Ctrl; Ctrl-B stays compact in the top row; Copy stays directly above Paste; Up/Down/Left/Right and PgUp/PgDn form one navigation block with about `1em` of horizontal space on each side and no added top spacing; PgUp is immediately right of Up and PgDn is immediately right of Down; each arrow button owns one continuous five-sided house silhouette whose square body has no border across the overlapping triangular roof join, while the four roof boundaries meet as a clean center X; Ctrl, Option/Alt, then the platform Meta key remain adjacent across the bottom-left row beneath the left gutter; Return is a narrow two-row key after the navigation block's right gutter; and one shared Ctrl-C (`^C`, `\\x03`) definition sits directly below Return with its complete border visible. Mac renders Option/Alt as `⌥`, Meta as `⌘`, and app actions as `⌘C`, `⌘V`, and `⌘P`; PC renders Alt as `Alt`, Meta as `⊞`, and the actual app shortcuts as `Ctrl+C`, `Ctrl+V`, and `Ctrl+P`. The `platform=mac` and `platform=pc` query parameters simulate either label set independently of `mobile=1`. More replaces the primary page with a bounded three-row grid whose top-right pair is More then X; the remaining actions use the available tracks and end with Ctrl-Z/Ctrl-L/Ctrl-R/Ctrl-C. All visibility changes settle before measuring palette size. While open, primary/overflow content changes preserve the chosen side and launcher-adjacent edge, grow away from the remembered launcher position, and clamp once without oscillating; only a launcher/palette drag or new open may reselect the side. Holding the launcher opens the palette directly in that overflow grid. Paste reads text or images under the tap's browser user activation; image payloads reuse the normal paste-upload path and text reuses the normal terminal transport. A long press starts the same clipboard availability probe while touch activation is still live; the shared terminal context menu shows Paste only when that probe found a text or image payload, then invokes that same paste path. Ctrl, Option/Alt, Shift, and the platform Meta key arm the next OS-keyboard character and then clear; a double-tap locks that modifier until tapped again, and tapping a locked modifier unlocks it. Typing, paste/beforeinput, outside pointer actions, and opening another terminal palette leave every open palette visible, so the accessory can stay beside the iOS keyboard; only its dedicated X closes it and clears its modifiers. Every rendered copy for one session follows the same XOR state: while open all launchers are hidden and all active palette surfaces are shown; after close every launcher reappears at its remembered coordinates and every palette is hidden. The palette keeps xterm focused while tapped, respects application cursor-key mode, and sends bytes through the same terminal transport as physical input, including writable YO!share and acknowledgement handling. It is absent from desktop-pointer terminals and disabled for read-only terminals.
- A late-mounted terminal accessory surface, including a stale closed-state Dockview/replay copy, immediately joins the existing per-session synchronizer: an open palette hides every launcher and shows every palette copy without waiting for another tap or resize. Palette geometry replaces the hidden launcher at the same pane-facing corner and clamps inside the local pane; a duplicate surface never reuses another pane's pixel anchor. In the primary grid, Copy and Paste share one column with Copy directly above Paste, Ctrl/Option-or-Alt/Meta are adjacent left-to-right on the bottom row, and each arrow's visible silhouette is a five-sided house: one square plus its inward triangular roof with no line across their join. The top arrow is an upside-down house, Down is upright, Left and Right rotate the same shape, and the four roof boundaries meet as the center X.
- A single-finger vertical pan over a terminal claims a non-passive touch path only after vertical movement clears the shared slop; the claimed gesture never focuses xterm's hidden textarea or opens the software keyboard. Normal-screen panes convert rendered row distance, including fractional accumulation, into the existing bounded tmux-scroll transport with finger-down revealing older content. Alternate-screen panes send cursor-mode-aware ArrowUp/ArrowDown bytes, while a fast flick sends one PgUp/PgDn sequence. Horizontal or multi-touch movement cancels without input, a plain tap keeps normal terminal focus, and touch end/cancel stops every pending batch with no inertia. Because iPadOS WebKit can synthesize mouse events after touchend even when touchmove was prevented, a claimed pan suppresses only the following 350 ms mousedown/mouseup/click chain; an unclaimed tap never arms that latch and retains normal focus.
- iPad acceptance capture records `innerWidth`, `innerHeight`, document client height, visual viewport width/height/offset/scale, app-root and Dockview-host rectangles, focused element, and the computed font size of the focused text control. Capture both before and after opening the virtual keyboard.
- Standard control, card, tab, and pill corners come from the shared radius tokens (`--radius-sm`, `--radius-control`, `--radius-md`, `--radius-lg`, and `--radius-pill`). Compound and logical corners reuse those same tokens; component CSS must not restate their pixel values.
- Standard scalar and compound component gaps from 1–12 px consume the shared `--space-1` through `--space-12` scale. `gap`, `row-gap`, and `column-gap` declarations may keep a genuine one-off or relative value, but must not restate a standard spacing literal or hide one in a custom-property fallback.
- Repeated fixed paint values use one shared `--paint-*` palette token even when their consumers have different semantic roles; component CSS must not restate an equivalent hex, `rgb()`, or `rgba()` shade. A semantic component token remains preferred when one exists, while the fixed paint palette owns otherwise-identical numeric channel/alpha combinations without changing their theme behavior.
- Persistent active/selected controls use one grouped base paint owner for `--active-control-text`, `--active-control-bg`, and `--active-control-border`. Topbar toggles, app-menu icons, status badges, Preferences search, tmux sub-windows, YO!info presets, Finder controls, preview zoom, and CodeMirror search toggles may keep local geometry or hover effects, but must not restate that three-property paint set; dark, light, and editor-light selectors remain in the same owner so cascade specificity cannot make one control neutral.
- Clickable topbar owner/activity status surfaces share one button-scoped compact shell and one pointer/keyboard interaction rule; variants retain only their gap/padding and the attention variant's red paint. The `IDX|STATS|SESS` owner-status hover title defines each abbreviation (`IDX` search/Quick Open index owner, `STATS` statsd/YO!stats sampler, `SESS` session-file owner), explains leader/follower meaning, names the connected and owner servers from the live payload, and states that right-click takes over as leader. The noninteractive latency meter shares the base surface paint but remains an inline grid with the normal cursor and must not inherit button behavior; its title explains the latency metric without adding click behavior. Their rest paint comes from the same theme-token owner in both dark and light mode; light-theme selectors must not restate the tokenized base background/border declarations. Static computed-style coverage owns shell and paint parity, while the live owner-status fixture covers only payload-derived role text, explanatory title content, order, and geometry.
- Pane/editor pressed controls use one grouped `--pane-ctl-pressed-*` paint owner across pane tabs, attention identities, editor modes, blame, diff, gutter, wrap, and find controls. YO!info path actions share one text-button reset, Finder/Differ/session agent identities share one inline cluster layout, and editor plus preview-find controls share one hover paint owner; consumers keep only their distinct geometry or typography.
- MAIN branch indicators consume one tokenized paint rule in every theme and pane state; light mode changes only the indicator's three custom-property values and excludes it from generic CI-chip paint. The server-update Reload button likewise keeps one theme-safe danger rest/hover owner with no light-only copy. Transcript/summary shells, nested YO!stats views, YO!info/Search & Runs input focus rings, framed panel surfaces, and centered editor pseudo-icons each use one grouped layout or state owner.
- When a long-lived page detects a newer YOLOmux server, it shows the reload banner because the page bundle and backend may no longer agree. The banner keeps Reload and Keep in one shared action row; at <=760px it stacks the content-sized message above that row so neither action can overlap or merge into its neighbor, and no row-oriented flex basis may create empty vertical notification space. In dev mode, a changed bundle still auto-reloads through the dev-reload stream.
- Approval/input state badges, failing CI chips, and changes-requested review chips consume one danger paint owner; failing CI retains only its distinct no-shadow behavior. Light-mode neutral badge paint excludes every specialized danger/disconnected/test/review/YO state so it cannot erase the state color through higher specificity. Differ and tmux sub-window agent SVGs share one compact 14 px geometry owner, while light Changes/Finder surfaces share one text/panel/border owner. A general light-theme surface default must use a zero-specificity `:where(body.theme-light)` gate when an embedded component variant needs to override it; Finder comparison and Reload chrome must win through their normal component selectors, not light-only repair copies.
- Single-line truncated labels use one grouped `min-width: 0` / hidden overflow / ellipsis / no-wrap owner across topbar/share, YO!stats, run history, YO!agent, YO!info, and share-management surfaces. A consumer may keep local color, font, or flex data, but must not copy only part of the shrink-and-ellipsis contract.
- Preformatted terminal snapshots, transcript text, plain YO!agent messages/details, modal output, drop-action results, and conflict comparisons share one `pre-wrap` plus `overflow-wrap: anywhere` owner. Each surface keeps its own overflow mode, size, font, and paint; a wrapping change must propagate through the grouped owner rather than local copies.
- Transient debug, repository, and terminal-drop surfaces combine readable content-relative `ch` capacity with `--popover-max-inline-size`, so they can grow when room exists but never exceed the viewport clamp. The topbar search uses flex/container-relative capacity instead of a fixed pixel maximum; narrow layouts may collapse or hide its label through the existing responsive classes.
- App dialogs choose from the shared compact, standard, and wide content-relative capacity tokens and combine that token with the viewport clamp. About and command dialogs use their assigned capacity in both the live host and replay shell; Keyboard Shortcuts uses the wide capacity and must remain contained at narrow widths in dark and light themes.
- Terminal drop-action labels use that complete single-line contract, including `min-width: 0`, so the label flex item shrinks beside its fixed keycap and never widens the popup past its edge.
- Disabled context-menu and editor/preview controls use one shared `opacity: 0.42` plus `cursor: default` owner; component rules retain geometry and normal/hover paint only.
- Inactive YO/agent markers use one grouped `--agent-inactive-marker-*` paint owner across normal/active pane and dark/light selectors. Vanilla preview inline/fenced code uses one grouped light-text/panel/border paint owner across pane, popout, and editor-light contexts; nested `pre code` transparency remains a separate semantic rule.
- Every CSS or inline-template stacking declaration consumes a named `--z-*` token. Numeric z-index values and layer arithmetic live only in `00_tokens_base.css`; component selectors, detached popout styles, sticky tree layers, editor overlays, and share-viewer overlays must not define parallel stacking numbers or calculations.
- A local pane-edge drop splits only the target pane. It must not flatten same-axis siblings into equal thirds. Example: dragging into the right half of `1/2 | 1/2` should be able to produce `1/2 | 1/4 | 1/4`. On touch browsers, a visible valid root-edge preview is the retained drop intent when the final release event omits usable pointer coordinates; Right and Bottom must commit the pane shown by that preview instead of silently doing nothing.
- A root-edge drop creates a full-span pane beside the entire current content area. If panes are stacked top-to-bottom, dragging to the left or right root edge creates a full-height leftmost or rightmost pane.
- The full root-edge hit band is active along the complete edge; a left/right root drop must not require the pointer to also sit near a top/bottom pane boundary. Inside that outer band, root placement wins before same-strip no-op, tab-header insertion, or tab-strip-end insertion, so dragging C from either `A C / B` or `A / B C` to the right edge produces `(A / B) | C`. Native Dockview drop handling and the pointer fallback use the same root decision and cannot move C back into A or B afterward. A Generic source creates a Generic outer pane even at the far-right edge. A Vertical Side Pane source may create or target the opposite-edge Vertical Side Pane. Existing Vertical Side Panes remain outermost and keep their measured widths while the generic content subtree receives the normal balanced share.
- A tab header inside an outermost pane is not itself a root-edge drop gesture. If a drag begins inside a root hit band, it must first leave that band and then return before it can create a new outer pane on that same edge. This keeps a horizontal move between top-row tab strips from being misread as `Full top`, while preserving a deliberate top-root drop.
- Root drops use the normal full-span geometry inside their source role. Generic-only content cannot target a Vertical Side Pane, and Vertical-Side-required content cannot target a Generic Pane. Dual-role YO!info/YO!stats/YO!agent/YO!chat may cross that boundary: a center drop joins the target pane, while a top/bottom drop from either role creates a real stacked leaf inside the target Vertical Side Pane at the same fixed width. A Vertical Side Pane may subdivide only top/bottom; left/right subdivision remains unavailable.
- Cross-gutter drops preserve the sibling boundary and create a full-span pane at that boundary.
- Closing or moving tabs must preserve user-chosen split percentages where possible. Empty placeholder panes are allowed when they preserve a meaningful user split. Every non-root empty placeholder shows exactly one top-right X, supplied by the shared pane-frame close owner, that removes that specific layout leaf through the shared pane-close layout transaction; a final root placeholder remains available as the single drop target. ## Tab Strip Behavior

- Clicking a tab activates it in its current pane.
- Tab switch: highlight first, content follows. Every tab-switch input that routes through `activatePaneTab()` — pane-tab click, keyboard cycling (Cmd/Mod+Left/Right), Tabs-menu activation, and adjacent-tab selection — must flip the new tab's active chrome (the highlighted tab and the swapped pane body) synchronously in the same frame as the input event; the heavy terminal follow-up (xterm focus/fit) runs in a later frame (phase 2). The highlight must never wait for content: the new tab is highlighted first, the previous tab's content must not linger under the new highlight (Dockview's cheap pre-rendered activate path swaps the shown pane body, showing the target content, its placeholder, or blank), and it is acceptable for the pane to blank or show the shared placeholder while the target renders. Rapid cycling steps the highlight instantly on every press but the phase-2 terminal engagement is generation-guarded and settles on the final target only, so intermediate tabs waste no xterm work and flash no stale content. A deliberately slowed phase 2 (`window.__yolomuxTabSwitchPhaseTwoDelayMs`) is the test seam that locks this paint order.
- Dragging a tab within the same tab strip reorders it. The preview must appear between tabs, not on top of the target tab.
- A same-strip reorder must commit on a slow, deliberate drag, not only on a quick flick. Drag-and-drop is pointer-based (`dndStrategy: 'pointer'`), so the dragged tab smooth-reorders under the cursor; the drop-target tab must be resolved from the pointer x-coordinate with the dragged tab excluded (nearest remaining tab when the pointer sits over the dragged tab's own gap), never by hit-testing whatever element is under the pointer. Hit-testing the element under the pointer resolves to the dragged tab itself, makes the drop look like a `source == target` no-op, and silently swallows the reorder — only a flick that outran the smooth-reorder transform would land, which reads to the user as "drag sometimes does nothing".
- In a two-tab strip the dragged tab fully covers the drop point. The reorder must still commit, and a secondary pinned pointer-reorder fallback must stand down once the primary drop pipeline has handled the gesture, so it cannot re-run and mirror-swap the order back (which makes the completed drag look undone).
- Dragging a tab to another tab strip moves it to that pane at the indicated insertion position.
- Every tab or pane pointer drag ends on release or cancellation, including when the pointer is released over an interactive Finder, Differ, or Tabber row. Tree rows may suppress the synthetic activation caused by that release, but they must not stop the release from reaching Dockview. After either an accepted or rejected drop, no tab/pane drag record, native drag ghost, Dockview target, app drop preview, or invalid-drop class may remain visible.
- Re-opening an existing file or diff tab activates it in the pane where the user last put that tab. A normal Finder, Differ, Quick Search, or context-menu open must not silently move an existing file editor back to the active, first, largest, or leftmost pane; only an explicit drag/drop or split action may rehome an existing tab.
- Pinned tabs stay in their current pane. A pinned tab can reorder within the pinned partition of its own pane, but it cannot be moved to another pane or split into a new pane.
- Invalid pinned-tab affordances must not take ownership of drag state. They must be derived from the current Dockview preview/target geometry and must clear immediately when the target changes or the drag ends; do not cache an invalid target and replay or repair the drop after release. Invalid pinned/non-pinned partition targets, including pinned tab-strip drops into another pane, show the normal between-tabs dashed preview in the danger color.
- The per-pane tab cap (`appearance.max_tabs_per_pane`) is enforced through one capacity owner for every action that inserts, moves, restores, minimizes into, expands into, or otherwise appends tabs to an existing pane. The cap may evict only eligible unpinned, clean, non-Finder, non-kept tabs by deterministic LRU order; pinned tabs, dirty editors, Finder, and the incoming/kept active tab are unevictable. If a target pane is already at cap and there is no eligible tab to evict, the action is refused, the layout remains unchanged, and the existing layout status surface shows a danger-styled localized error that names the pane/cap condition. A pane whose pinned count reaches the cap shows an advisory status because it cannot accept additional tabs until a tab is unpinned, closed, or the cap is raised. Lowering the cap below the current pinned count never force-closes pinned tabs or dirty editors; the pane is grandfathered over cap, load/restore/share replay applies faithfully without error spam, and future incoming tabs are refused or shrink only eligible unpinned overflow.
- Pinning constrains tab-level user movement and eviction only; it never hides or refuses pane minimize. Generic-pane minimize moves every source tab, including pinned tabs, through the shared capacity owner. One-column tab-level minimize removes the selected pinned tab exactly as it removes an unpinned tab, and Vertical Side Pane minimize removes its pinned dual-role resident like any other pane minimize. Minimize into a full target is all-or-nothing: it may evict target-side eligible LRU tabs to fit every moved source tab, but if the target still cannot fit, the source pane remains unchanged and the ordinary capacity error is shown.
- Expand never drags pinned residents out of their panes. Expanding one pane may gather only unpinned tabs from other panes through the same capacity owner; panes left holding pinned tabs survive. If the expand target itself cannot accept the gathered unpinned set, the expand is refused or shrinks only eligible overflow according to the shared capacity result; pinned tabs are never moved or evicted to satisfy expand.
- Dragging a tab to another pane edge splits that target pane only.
- Dragging a tab to a root edge creates a full-span pane beside the current layout.
- The tab insertion preview is a visible dashed box between tabs and uses the same configurable color as the pane separator.
- The active tab keeps rounded top corners. Inactive tabs use the same background as the tab container.

## Agent Status Indicators

- Auto-approve, Tabber, attention indicators, and YO!stats Agent Status use one shared status generation across web ports. A stale complete generation remains visibly stale; otherwise unavailable status uses the existing error state and never fabricates a local roster.

- A Claude/Codex status indicator has two visible parts: a static agent symbol and a separate status ball beside it. The layout is `[symbol][ball]`; the symbol and ball must not alternate, replace each other, or pulse the symbol for working, attention, or cooldown.
- Session-tab balls are true filled circles with a thin black border, fixed equal width and height, and no shadow. Working is green with no yellow/lime tone, attention is red, and cooldown is vibrant yellow from `--agent-status-cooldown`, not the generic `--accent-gold` used for DRAFT/git/warning UI.

Replacing or rebuilding status-ball DOM stamps the current wall-clock phase before first paint, so metadata, activity, pane-tab, Info, or Tabber refreshes cannot restart the pulse from an old page-load phase even when the broad-animation mutation observer is disabled. Aggregate tab/session surfaces use the base ball size. Tmux sub-window buttons in the Info Bar, YO!info, and Tabber use a flat 80% of the base size for visible play/stop/pause glyphs. - The `--active` state may use `agent-symbol-glow-cadence` on the symbol itself only when status pulse is enabled; active/current is not a status ball. When status pulse is disabled, active symbols stay static and must not keep running animation or `will-change` work alive.

With one visible unacknowledged child tone it uses that red stop, yellow pause, or green play color. With multiple tones it uses two equal, crisp segments for the two most urgent distinct tones in order red, yellow, green: red+green for mixed ASK+working, red+yellow for mixed ASK+done, and yellow+green for mixed done+working; if all three occur, the parent uses red+yellow. ## YOPane Metadata And TmuxWindows

### Tmux Runtime And YOLOmux Layout Terminology

- `YOPane` is a physical, resizable browser layout region. A `YOTab` is movable content hosted by a `YOPane`; moving it changes only browser layout state.
- `TmuxSession` is one tmux session. A `TmuxWindow` is a tmux window inside that session (older localized UI text may say “tmux sub-window”), and `Ctrl-B n`/`Ctrl-B p` switch `TmuxWindow`s. A `TmuxPane` is a PTY-backed split inside a `TmuxWindow`.
- Backend runtime records use `TmuxPaneInfo`, never the unqualified `PaneInfo`, to prevent confusion with `YOPane`. A temporary compatibility alias exists only for external Python importers during the migration.
- A `RuntimeActor` (shell, Claude, Codex, or another foreground process) belongs to a `TmuxPane`. A session may have multiple `TmuxWindow`s, a window may have multiple `TmuxPane`s, and a browser `YOPane` must not be used to infer any of those runtime relationships.
- YO!info, tab descriptions, tab popovers, Tabber, Finder/Differ session roots, command-palette metadata, and Info Bar metadata consume the same normalized `work_graph` projection. The graph is authoritative when present, including when it is valid but empty; UI code must not revive stale `project` or `window_metadata[].git` values in that case. The focused chain is `TmuxPane -> RuntimeActor -> PathObservation -> GitWorktree -> LocalGitRepository -> LocalBranch -> PullRequest`. The full LocalGitRepository branch inventory remains one canonical set even when several linked GitWorktrees or RuntimeActors reference it.
- YO!info's Order-by selectors expose each public grouping label at most once. Internal relationship dimensions such as `TmuxSession`, `TmuxPane`, `GitWorktree`, `LocalGitRepository`, and hosted-repository ownership remain available to canonical presets and stored groupings, but they are not parallel manual options beside their public `Tab`, `tmux sub-window`, or `Path` projections. If a preset selects an internal dimension, that selected option remains representable while any same-label public option is suppressed for that selector level.
- Each YO!info path/worktree branch list preserves checkout state through every relationship and YO!share projection. The `LocalBranch` whose id equals that path's `GitWorktree.current_branch_id` renders the localized current/active badge and current-state paint; other repository branches remain `available`, historically observed branches may be `worked`, and missing refs remain `missing`. A branch must never fall back to `available` merely because the tree is grouped through Tab, Path, or tmux sub-window.
- YO!info responds to measured usable width rather than only pane orientation. When its toolbar is too narrow for one stable Order-by line, the four selectors become an even two-column grid and redundant `>` separators disappear instead of dangling on wrapped lines. When it resolves more than one PR, tab-level chrome renders a localized aggregate such as `2 PRs`; individual PR rows remain available in YO!info. ## Internationalization

- The backend locale registry is the single owner for shipped locale identifiers, endonyms, normalization, browser-language resolution, and text direction. Login/setup, first paint, the runtime language picker, auxiliary shells, and catalog asset lookup must consume that registry rather than keeping parallel locale lists.
- Login is rate-limited on both the browser form and HTTP Basic before any password hash (see `docs/DEVELOPMENT.md` → Login rate limiting). Repeated failures eventually return a generic `429`: the login page re-renders with a localized `login.error.rateLimited{Minutes,Hours}` message and the API/Basic path returns a JSON `auth.error.rateLimited{Minutes,Hours}` error. The visible message is fact-free — it states only "wait a few minutes" or "wait a few hours" and never reveals which limit fired, whether the username exists, attempts remaining, or a precise reset time (no scheduler-friendly `Retry-After`); the ordinary `login.error.invalid` (401) message is byte-identical for a known and an unknown username. Correct credentials for an account that is not itself locked still succeed. Setup mode and `YOLOMUX_TEST_AUTH_BYPASS` are never throttled.
- Every shipped source catalog must have the exact English key set, nonblank translations wherever English is nonblank, identical placeholder/protected-token multisets, and no unintended English prose fallback. Simplified and Traditional Chinese use `优` and `優` branding respectively and must not expose ASCII `YO` in localized values; other locales preserve the `YO` product mark.
- A runtime language change relocalizes global chrome and every mounted pane through the shared surface and `TAB_TYPES[].relocalize` registries. It must preserve pane content, editor drafts/selections/scroll, terminal state, Finder expansion/scroll, open overlays, and right-to-left direction while updating visible text, titles, placeholders, and accessibility labels on the same interaction.
- Server-generated user-visible messages must carry a stable locale key plus parameters or be rendered for the normalized request locale. Raw diagnostics may remain in debug/detail fields, but API errors, activity summaries, notifications, and deterministic YO!agent answers must not expose English prose as the localized visible message.

## Finder, Differ, And Tabber Rules

- Finder, Differ, and Tabber are real independent tabs in the layout, never overlays or global modes. Each has one identity, fixed view, panel DOM identity, and shared domain-state controller; they must not be rebuilt or switched as a group.
- Fresh wide layouts seed Finder, Differ, and Tabber in a 22%-width `side:left` tab stack with Finder active; no right Vertical Side Pane is created. File activates an existing item in its current Vertical Side Pane, current left/right edge, and current vertical leaf. The combined wide-screen File command adds only missing Finder/Differ/Tabber items to the default left Vertical Side Pane; it never re-homes an existing item from the right edge or another leaf. Finder/Differ/Tabber require the Vertical Side Pane role on wide layouts and cannot enter a Generic Pane.
- A Vertical Side Pane is one outer left or right fixed-width column containing one or more vertically stacked leaves. Finder/Differ/Tabber and YO!info/YO!stats/YO!agent/YO!chat may reorder or center-stack within a leaf; any of the four YO!* tabs may also split or move into a top/bottom leaf and may drag back to a Generic Pane. Right-click/long-press omits the generic `More desc` row and reuses the normal directional grid: Move Up/Down and unambiguous Swap Up/Down operate inside the column, while the existing Move Left/Right cell toward the opposite browser edge moves the tab across. Creating a missing opposite edge copies the source Vertical Side Pane width exactly. A Move with no vertical neighbor creates another leaf. Horizontal subdivision and left/right resizing are unavailable. Generic-only tabs are rejected. Both edge specializations carry the same native Dockview maximum-width constraint, dynamically capped at one-third of the current host; right-edge resize and Generic-neighbor removal cannot leave a visually oversized Vertical Side Pane while serialized state is already clamped.
- Closing one triplet item removes only that item. Closing the final Vertical Side Pane tab removes its Vertical Side Pane while retaining the generic content or filler. Pane `_` hides the whole Vertical Side Pane without moving its tabs into generic content. Normal URL parsing, Dockview adoption, reconnect, wake, refresh, relocalization, and resize preserve explicit role and edge and repair malformed saved layouts to at most one outer Vertical Side Pane per edge.
- `Cmd+B` on Mac and `Ctrl+B` on PC/Linux are YOLOmux's triplet visibility shortcut. If any triplet item is open, it hides all three. If none is open, it restores Finder, Differ, and Tabber to a fresh 22% `side:left` with Finder active on wide layouts, or the selected single file surface in the constrained Generic Pane. Mac `Cmd+B` works while a terminal has focus; PC/Linux `Ctrl+B` is not stolen from a focused terminal.
- A Differ refresh delegated from a follower YOLOmux server to the shared background owner must start the existing session-files cache worker with the requesting session, refs, and validated exact cache key, then fan completion back to the follower. A rooted `refreshing_elsewhere` placeholder is pending data, not a clean result: with no real files, warnings, or errors it shows the loading state and must never say `0 files changed` / `No Differ results for this session` while Git is dirty.
- A runtime language change must relocalize Finder through the existing Dockview panel renderer. Replacing the cached Finder node must remount that replacement into the current Dockview group without emptying the panel content container or losing its toolbar, tree, expanded state, or scroll state.
- Dockview must not adopt a layout snapshot while its host is hidden or measured at zero area. A visible small host still lays out at its real size; only serialized snapshots use functional fallback dimensions so a sleep/hidden-tab measurement cannot become the persisted layout.
- Pending layout-URL state, its applied marker, and the scheduled URL refresh live in one record. A deep-link snapshot applies once, a replacement snapshot resets the marker with the payload, and duplicate refresh scheduling cannot leave an old timer suppressing the current URL write.
- Pane-role policy is the single placement gate. `TAB_TYPES[].panePlacement` classifies Finder/Differ/Tabber as Vertical-Side-required, terminals/editors/Preferences and other ordinary tabs as Generic-only, and YO!info/YO!stats/YO!agent/YO!chat as dual-role Side-allowed. Only the dual-role classification may cross Generic/Vertical-Side boundaries; Finder/Differ/Tabber never advertise a Generic target on wide layouts, and generic-only tabs never advertise a Vertical Side Pane target.
- A file dragged from Finder or Differ into a normal pane opens in that pane. Dropping near a normal pane edge opens it in a new split and shows the same dashed preview used for tab drops.
- A file dragged from Finder or Differ onto a Vertical Side Pane is rejected; file editors may be dropped into any Generic Pane subject to the shared size gate.
- Directory drags into terminal content keep the terminal path-insertion affordance. They may show a normal-pane split preview when the target is not Finder/Differ and the pane has enough room.
- Tab, pane, Finder, and Differ drags share one operation record for source item/slots, file payload, preview nodes/offset, and tab geometry. Starting one drag kind clears incompatible state from the prior kind; drop, dragend, pointer/mouse release, blur, and visibility cancellation clear the complete operation so a later drag cannot inherit a stale file payload, pane source, preview, or geometry cache.
- File or directory drops into terminal content route through one shared `DROP_ACTIONS` registry. Agent panes may insert the path first and then show a transient non-modal suggestion overlay on `1` through `9` (top-row or numpad); plain shell panes show explicit read-only shell commands (`file`, `wc`, `tail -F`, `jq`, `column -t`, directory size scans) instead of inserting a stray bare path; server actions can show bounded file info, preview, log scan, data-stats/chart, or OCR results in a result panel. External/upload drops and pasted image uploads use the same action registry after the upload path is known; pasted image uploads must still insert the `[Image #N] '/path'` reference immediately. Escape, click-away, normal typing, or timeout dismisses the overlay without running anything.
- File-drop actions support multi-file prompt actions, remember the last chosen action for categories without an explicit order, and include user-defined custom actions from Upload Preferences. Image paste/drop ordering is controlled by `uploads.image_action_order`; that Preference is authoritative, stores prompt actions as `Popup label: ; prompt text inserted after the image path`, accepts non-AI rows as the popup label or a short special action such as `info`, and is capped at 9 rows because the popup assigns shortcut keys `1` through `n`. Image popups must only show configured image-order entries that apply to the current pane, and their visible text must match the configured Preference row verbatim except `info`, which resolves to the runnable `Server: file info` label. The pasted image path/reference is inserted before the popup, so `Insert path` is not part of the image action order. The default image order is agent OCR, error diagnosis, image description, then `info`; stale image-order rows for server OCR and file type are dropped. Custom action lines use `Label | prompt text or shell:command | optional comma-separated categories` and support `{path}`, `{qpath}`, `{paths}`, `{qpaths}`, `{name}`, `{count}`, and `{category}` placeholders. Read-only shell autorun is allowed only when the `uploads.suggestion_autorun` Preference is on; agent prompts and server result-panel actions never auto-enter terminal text.
- Finder, Differ, and Tabber rows share the base tree-row hover paint. Selected rows and the Finder-synced current-file row share one grouped selection-paint owner in both themes; a light-theme rule must not restate token-driven base selection paint.
- Finder, Differ, and Tabber use one shared tree toolbar/row renderer, but each fixed surface owns its own visible tree choices. `Sort by` and the Date/Ago/None toggle are independent for Finder, Tabber, and Differ: changing one surface must neither change nor rerender the other two, and row date/age text must resolve against the row's owning surface rather than falling back to another surface's mode. - Finder and Differ rows share selection, multi-select, context menu, date columns, status columns, image preview, rename/delete, and keyboard behavior through shared helpers.

- A restored Finder directory that is already pointed down but whose child listing has not rendered yet is also loading and must show that same moving spinner until its children become available. Indexed and index-excluded status phrases use the selected row's foreground token when selected, preserving readable contrast in dark, light, and editor-light color schemes.
- Finder and Differ changed directory/path rows render metadata in one shared order: path/name, inherited agent marker, diff counts (`+N -N`), bare changed-file count (`N` with no `file changed` / `files changed` label), git status, then date. The diff column is the first right-side metadata column when it exists; the changed-file count follows it and must not appear before the diff counts. On a file row and repo header, `+N/-N` keeps its existing added/removed-lines meaning.

Normal row opens reuse the Differ preview editor tab and start that editor item with collapsed unchanged context without changing the global/default diff expansion setting, including consecutive clicks across different Differ files. The row context menu lists Open actions first in this exact order: `Open in a Differ`, `Open in a new Differ`, `Open in a new Editor`. When the same physical file is already open, those actions focus that editor and switch modes instead of creating a duplicate editable tab.
- Background file reloads must preserve each same-file editor tab's CodeMirror cursor and scroll. If the server push/watch path replaces shared file content for `/path`, every visible editor item for `/path` must capture its own selection/scroll before the shared state swap and restore it after the render so duplicate editor tabs do not jump to the top. Background reload must also defer replacement while any editor for that path has focus, or immediately after the file was saved/returned to clean, and surface the normal changed-on-disk status instead of replacing the live document under the cursor. Ordinary autosave, watch/SSE, metadata, and chrome rerenders must treat a connected focused CodeMirror view as the newest cursor/selection/scroll authority: they update the cached view state from the live editor and never replay an older cache entry over Shift+Arrow selection or cursor-follow scrolling. Explicit disk replacement may restore the pre-reload snapshot it captured for that operation.

The filesystem package normalizes raw OS failures for single and batch routes: permission failures return a localizable 403 payload, other unclassified OS failures return 500, and neither emits a request-thread traceback. A valid inspectable local file must not be marked `file state unknown`; if the UI shows that state for an existing file, the next diagnosis starts with the actual `/api/fs/list` or `/api/fs/info` error shape, allowed-root validation, and stale absolute-path/worktree state.
- Editor scroll and selection must survive a tab switch away and back (for example `2026.md` -> Preferences -> `2026.md`); switching back must not snap the editor to the top of file. The CodeMirror scroller captures its own view state on scroll through one shared scheduler, so "scroll, then switch away" persists state regardless of tab-event order. A manual Dockview tab activation captures the outgoing pane's editor view state on `pointerdown`/`mousedown` before Dockview switches panels, because manual Dockview tab clicks do not route through `activatePaneTab()`. A pooled, disconnected, or zero-height editor pane must never overwrite an existing saved scroll with `scrollTop: 0`, and restore must reapply the saved scroll after CodeMirror focus and across layout/animation frames. This tab-switch contract is distinct from the background-reload contract above.
- Uploaded files in Differ stay under their actual repo section and repo-relative path, usually as a `.uploads` directory. Differ must not pull them into a synthetic top-level `Uploaded files` group. `.uploads` directories are collapsed by default so screenshots and pasted images do not dominate the first view.

Every selected path must be contained by both the displayed and resolved Finder roots; outside-root, unresolved, symlink-escaping, or mixed selections disable the action rather than copying an ambiguous value. Differ/Modified Files retains its separate Git-relative copy contract. Browser coverage exercises Finder root/session/reload/path/hidden controls; create, sort/date, expansion, selection, keyboard, rename/delete, drag, and readonly states; the file context actions; and Differ session/ref/reload/sort/date, repo/folder/global collapse, and directory-to-Finder action paths.
- Image file rows in Finder, Explorer, Differ, and Tabber path contexts expose `Copy image` through the shared file context menu. When browser image clipboard writes are available, it copies the image bytes; when unavailable, it falls back to copying the absolute path with a clear status message.
- Folder rows in Finder expose `Zip & download` through the shared file context menu for single admin-owned folder selection only. The server zips the folder under a `<folder>.<YYYYMMDD-HHMMSS>.zip` Pacific-time attachment name, includes the top folder in the archive, skips symlinks and special files, rejects folders over the Preferences `File transfer size cap` with a visible JSON-backed status error telling the user to zip it manually, and blocks the action for readonly users just like other filesystem downloads.

The indexed root itself retains the remove-index action, unrelated directories retain the add-index action, unrelated files cannot become roots, and a descendant cannot pretend to override an excluded ancestor. Exact indexed and excluded rows use the readable labels `indexed` and `index excluded` immediately before the Date/Ago column; their status slot grows to fit the words instead of clipping them to the one-character git-badge width or relying on the indexed-directory dot alone.
- Finder, Differ, and Tabber retain their own panel toolbar and fixed view; there is no in-panel `Finder / Differ / Tabber` mode switcher. Switching uses the shared tab strip, so the items remain draggable/reorderable tabs and each panel owns only controls relevant to its view. Vertical Side Pane tabs use intrinsic content width, stay on one line until the available width is exhausted, and ignore Preferences tab width; Generic tabs retain the preference width. Each triplet tab keeps its own `_` to hide that tab, while the one shared pane-frame `_` at the far right hides the Vertical Side Pane. Vertical Side Pane headers never show `+`, actions, details, popout, close, or pane-drag controls, and Dockview never renders a second inner pane-control row.
- Finder, Differ, and Tabber share one visible-surface attach lifecycle. When a move, close, minimize, restore, or layout adoption implicitly exposes one of them as the active remaining tab, that surface must immediately resume its normal work: Finder refreshes its tree, Differ renders cached data and fetches the selected session when needed, and Tabber refreshes activity. Visibility must never leave a newly exposed Differ at `not loaded` until the user clicks Reload or switches tabs.
- The Finder/Differ/Tabber top row must not include a duplicate Differ collapse/expand-all button next to the close button; tree expand/collapse controls live in the normal Finder/Differ toolbar row below.
- Finder's path row is exactly `Sync button | absolute path input | copy-path button`. The path may be synced or manually typed, but the Sync control must not be duplicated on another line. Copy-path copies the normalized value visibly in the input, including a pending manually typed root; it must not silently copy the prior committed root.
- Finder's file-action row starts with create-file/create-folder controls, then puts the `.*` hidden-file toggle before the `A-Z` sort selector. The `~`, `/*`, and `/tmp` quick-root buttons are not visible in the Finder pane chrome because they consume too much space.
- Finder mode and Differ mode must keep the same toolbar alignment rules, date-mode vocabulary (`None`, `Date`, `Ago`), sort vocabulary (`A-Z`, `Z-A`, `new`, `old`), and readable row metadata. Hidden, Sync/header actions, sort, and any quick-path button consume one compact 20px muted control shell for border, radius, paint, and typography in both themes; individual width/padding/layout remains control-specific. Finder starts directly below the pane tab header with `Session: <selector>` and Reload right-aligned on the same row, then `Sync <path>`, then its remaining actions; the trailing tree controls are ordered `Date | Expand all | Collapse all`. Expand all/Collapse all use compact skinny in/out arrow toolbar icons rather than square text glyphs.
- When Finder or Differ auto-switches to a session whose changed-file payload is being fetched and no cached payload is available, it must show an inline loading state with subtle movement, not `not loaded`. The current standard is loading text plus the session label and moving ellipsis dots, with no circular YO marker. If a follower-refresh payload already carries visible loaded Differ data, such as a clean repo section with zero files, the panel must render that result instead of pinning the header to loading. In Finder Sync mode, an explicit session change must immediately plan the root from the cached changed-file payload or the latest tmux transcript metadata, then refresh changed-files in the background; it must not leave the path on the prior session while waiting for `/api/session-files`. If a loaded cached payload exists for that session and comparison, Finder/Differ must apply it immediately, use it for Sync root planning, and refresh in the background instead of replacing it with the loading state.
- Finder/Differ session-files payload, render signature, loading state, and request generation live in one record. Accepted cache/direct/push replacements invalidate older HTTP generations before applying, so a late request cannot restore stale files, errors, refs, loading, or tab counts after a newer server push.
- Finder `Copy relative path(s)` means relative to the root currently displayed in Finder, including a nested root below a Git repository. The action verifies both the displayed and resolved paths are contained by that root with segment-aware containment; a missing/failed realpath lookup, symlink escape, or mixed unavailable multi-selection disables the whole relative-copy action rather than copying misleading repository-relative text. Differ's corresponding action remains explicitly Git-repository-relative. `tests/tabber.test.js` covers the pure/multi-select matrix, and `tests/test_browser_finder.py::test_finder_context_relative_copy_uses_visible_root_and_fails_closed` exercises the real context-menu and clipboard route.
- Finder Sync root opens caused by an explicit session/tab click are user-owned filesystem reads, not background refreshes. They must bypass client-push background fetch suppression, immediately update the path field to the planned target root, and show the shared `searching...` row while `/api/fs/list` is pending. Otherwise a slow uncached session root looks like Sync did nothing.
- Finder Sync roots a home-contained active worktree at the user's home directory and displays that root in home-compacted form (`~`). It expands only the full ancestor chain needed to reveal the active repo/working directory, such as `~/Documents` -> `~/Documents/bin` -> `~/Documents/bin/yolomux.dev8881`, and leaves outside-home worktrees rooted at their focused repo/root path. Every affected repository/directory from the selected session shows a bright localized `★` marker, including the visible root and touched ancestors inside a repository; unrelated home-chain pass-through directories stay unstarred. Hovering a starred directory uses the shared Finder detail popover to show `★ Modified by <selected session>`; ordinary rows have no such line. The marker set moves with the selected session and is independent of ancestor repo-root bolding. Per-tab expansion, manual-collapse, keyboard cursor/selection, and anchor memory are keyed by `(session, root)`, so two tabs sharing the home root keep independent disclosure and keyboard-traversal state. Returning to a target restores its visible cursor as the tree active-descendant and scrolls it into view without stealing terminal/browser focus; a collapsed or absent cursor falls back to its nearest visible ancestor. A subsequent sync tick or `Expand all` must not reopen an intermediate directory the user manually collapsed; a manually collapsed synced row keeps its `★` marker while staying collapsed.
- `tests/test_browser_finder.py::test_sync_mode_opens_common_repo_parent_and_expands_affected_dirs` covers Finder Sync root/non-root stars, multiple touched paths, per-session movement, pass-through exclusion, and collapse behavior.
- Finder Sync session changes are cache-first. When every visible listing for the selected session is already in the shared bounded filesystem-resource LRU, the remembered expansion and manual-collapse state is reconciled synchronously with zero blocking directory requests and existing row nodes are reused. On the next animation frame, stale-while-revalidate refreshes only the visible directories, compares their existing signatures, and reconciles in place only when content changed; an unchanged signature preserves the mounted row identity. A cold target uses at most eight concurrent listing workers rather than a sequential folder loop. `tests/layout_async.test.js` covers synchronous warm switching, next-frame changed-listing replacement without disclosure flicker, unchanged identity preservation, bounded `[8, 4]` cold batches, and LRU eviction.
- Finder Sync has one explicit target at a time: the last clicked/typed session or explicitly opened file. A normal Finder file click also pins that file's current root, highlight, expansion, and scroll until another explicit session/file click or typed path changes the target. Hover auto-focus, passive terminal focus, metadata/transcript refresh, and SSE updates must never clear that pin or re-scope Finder. When the target changes, any in-flight sync for a different target is stale and must be invalidated or dropped before it can set the visible root, remembered sync target, highlight state, or applied-plan key. A same-root short-circuit is not enough: if session 5 reactivates while a session 6 root fetch is still in flight and the path field already shows session 5's root, the session 5 sync must still cancel/guard the session 6 transaction so a late session 6 response cannot retarget Finder back to the wrong worktree.
- `tests/test_browser_finder.py::test_sync_mode_does_not_follow_hovered_tmux_session` covers a plain Finder file click followed by another pane hover and a session-metadata refresh, then verifies that the selected row, root, and expansion remain pinned.
- Finder Sync transaction identity, applied-plan identity, and expansion generation live in one record. Manual root selection clears that complete record before its directory read, and an older root completion must not replace the manual root or restore an applied-plan key.
- Differ shows both current git changes and recent transcript-touched repo paths for the selected session. A repo must remain visible when the session modified files there and then committed or cleaned the working tree; those rows use transcript attribution instead of pretending there is a current dirty diff.
- Differ and full-editor commit-reference pickers share one responsive width owner: they may use up to two-thirds of the browser viewport (subject to available viewport padding) so full commit descriptions remain readable, while leaving browser context visible. They must not use a fixed compact-width cap. Their reusable popover node, owning input, rendered items, active keyboard cursor, and global-listener installation live in one `diffRefPopoverState` lifecycle; hide clears input/items/cursor together, and a scroll against a detached input closes the popover.
- Finder Sync mode `Expand all` is bounded to the sync plan: expand affected repositories/directories such as the active worktree path and do not recursively crawl every directory under a broad home root. Fixed-root Finder mode may still recursively expand the current root.
- Differ's target session is committed only by explicit user intent: clicking/selecting a pane or typing/pasting/inserting input into a terminal. Hover auto-focus, passive xterm focus, xterm mouse-tracking bytes emitted by hover, pointerenter pane selection, background refresh, SSE updates, and tab-hover popovers must not switch the Differ session or fetch another session's changed-files payload.
- When Auto-focus hovers Finder, Differ, or Tabber, that active file-surface panel owns its tree keyboard navigation even if browser focus remains in the previously hovered xterm: arrow, selection, and activation keys route to the visible tree. This ownership is one shared gate for all three fixed views; it is off when Auto-focus is off and never overrides an input, CodeMirror, menu, popover, modal, or other editable control.
- Hover popovers and passive Auto-focus are cursor-capability behavior, not a device-name rule. They require a hover-capable mouse, trackpad, or pen (`any-hover: hover`, or its active pointer event); touch-only phones/tablets must neither open nor retain hover popovers and must not passively move pane/keyboard focus. An iPad with a mouse or trackpad retains normal hover and Auto-focus. Explicit taps, clicks, keyboard focus, and terminal input remain available on every device.
- A notification addressed to the exact visible active tab is already acknowledged: it must not appear in YOLOmux or as a browser notification, including when a status render briefly blurs xterm. The shared logical active-pane record, rather than transient DOM focus alone, decides this; selecting/clicking the target also removes any existing notification for it.
- The Finder path field shows its invalid (red) state only on a real directory-open failure, scoped to that exact path. A deferred or suppressed background refresh — the SSE push channel will supply the listing, a push refresh is mid-flight, or a fresh cache hit — is not a failure and must never flag the field, and a recorded error for one path must not taint another. (Otherwise the path flashed red for ~one push cycle whenever an idle background refresh was skipped.)
- Finder caches each observed directory's listing signature and prior entry-name set in one bounded record. Refresh comparison and new-entry highlighting must consume the same record generation: broad filesystem invalidation clears the signature but preserves prior names so the next listing can still mark additions, while invalidating a renamed/deleted root removes the complete record and its child highlight deadlines.
- Typing a path into Finder and pressing Enter is an explicit manual-root action. It must immediately leave Sync mode, clear any pending Sync root open, update the visible path to the typed target, clear stale tree rows, and show a passive `searching...` row using the shared moving-ellipsis helper until the requested directory entries are fetched and rendered. A slow or large directory request must not bounce the path back to the previously synced session root. Any already-started session-sync root fetch that resolves after the manual open is stale and must be dropped; it must not flash that session path before the manual request resolves. Broad directories are guarded by a bounded directory-entry listing so paths like `/tmp` can open without forcing YOLOmux to stat and render an unbounded number of children.

## Reconnect And Wake Rules

- Client-events reconnect must backfill live auto-approve status so YO markers reflect the current working/idle state immediately after the stream is ready again. Page visibility returning to visible and browser `online` events must schedule the shared refresh path, and a narrow auto-status fallback poll may run only while the client-events stream is disconnected.
- Client-event source/connectivity, the coalesced delivery queue/frame, and wake/online resync debounce live in one `clientEventTransportState` lifecycle. Hidden tabs cancel a pending animation frame and flush status events immediately. Background-owner and activity-summary HTTP requests are generation guarded: a newer forced request or SSE payload wins, an older success/failure cannot overwrite it, and only the current request may clear its handle. YO!info's cached render signature and HTML commit together; hidden renders simply skip, and no write-only pending-render flag is supported.
- Transcript metadata, Search & Runs results, and YO!agent conversation/jobs hydration follow the same generation rule. A newer query, forced refresh, streamed/direct payload, empty-query reset, or SSE payload invalidates older HTTP work. The input text, visible result rows, transcript path, pending waits, job list, pane/Tabber metadata, loading state, and request handle must come from the same record generation; old success/failure/finally paths cannot resurrect them or clear newer work.
- The browser title, YO favicon badge, and topbar activity line consume one `globalActivityCounts()` snapshot and count active Claude/Codex windows, not only tmux sessions. Two working agent sub-windows in one session therefore count as two on every surface. They must not count tmux's selected pane/window flags, open terminal websockets, idle Claude/Codex processes, or attention-only states as active work; every tmux sub-window has a selected pane even when idle.
- For sessions with multiple detected Claude/Codex panes, the backend keeps per-agent-window status rows keyed by tmux sub-window/pane target. Any scalar session summary follows the selected/preferred agent pane first, then a deterministic agent-pane fallback, but YO markers, working counts, Tabber, popovers, and Recent Agents must consume the per-window rows so they do not get stuck on an older idle agent when a later window in the same tmux session is working.
- A session's YO marker is GREEN/active ("on here") when THIS server is auto-approving the session, YELLOW/locked ("on elsewhere") when ANOTHER YOLOmux server on the same machine is auto-approving it, and inactive otherwise. With multiple servers sharing one tmux, clicking YO green on one server must make the same session read yellow on every other server. Cross-server detection: a server with no local worker for the session reports `enabled_elsewhere`/`locked` by probing the auto-approve lock for that session's agent-pane targets (`auto_approve_agent_targets`) — i.e. the exact lock a YO worker holds — falling back to the bare session-name lock for the no-agent case. Probing only the session-name lock silently dropped the yellow marker for every agent-backed session, because YO workers lock per agent pane (e.g. `%1`), not the bare session.
- Local YO workers are target-keyed records containing both the owning session and worker. Fanout, prune, takeover, persistence, status aggregation, and shutdown consume that one record owner; parallel worker/session maps are not supported because a partial mutation can attach a live pane worker to the wrong session.
- Session tabs reserve their fixed-width numeric prefix only when the shared leading status ball is present. A tab with no ball compacts that prefix so its branch and PR metadata does not begin after an empty status column.

## Tabber Pane Rules

- Tabber is an independent Vertical-Side-required file-surface tab, not a mode that replaces Finder or Differ. Fresh wide layouts place it in `side:left` after Finder and Differ; File selection activates an existing Tabber in its Vertical Side Pane or recreates it in `side:left`. In constrained one-pane layouts it is the one selected file surface in the Generic Pane. Tabber active styling follows the shared visible-pane predicate, not focus, and its current tree row follows the shared focused-pane item.
- Tabber shows a live tree of everything open and defaults to fully expanded. Level 0 = tmux sessions (sorted first), then the other open tabs (Preferences, YO!info/YO!agent, file editors) as leaf rows at the end. Session rows are collapsible like other directory rows: clicking only the disclosure arrow toggles collapse/expand, while clicking anywhere else on the green `N <description>` session line opens/activates that tmux tab. The Tabber session label must render through `tmuxPaneTabTokenHtml` and use the shared `.tmux-pane-tab-token` shell for visible tab styling, including inactive border, hover, active colors, and top-only rounded corners; Tabber CSS may only add row layout/truncation overrides and the selected-row focus ring. Level 1 = the session's tmux sub-windows (`index:process (pid=N)`, e.g. `0:claude (pid=12345)`) rendered through the same compact `.tmux-window-button` shell used by YO!info tmux sub-window values and the pane Info Bar. Claude/Codex tmux sub-windows show the shared agent icon and smaller sub-window status ball inside that button shell. The session's current tmux sub-window is shown by row emphasis such as bold text plus `aria-current`, not by adding another circle/dot beside the agent activity icons. Under a `claude`/`codex` tmux sub-window, Level 2 = absolute touched repo roots attributed to that same tmux agent sub-window from the backend `agent_windows.path_entries` record in `/api/activity`, with a folder icon and the git branch on the matching repo row. A repo touched by `0:codex` must not render under `1:codex` or `1:bash` just because it belongs to the same tmux session. The backend builds those path entries from the cached `/api/session-files` attribution and git metadata; the browser must not refetch or reparse `/api/session-files` to decide per-sub-window path/git/active state. These touched-path rows are file-kind leaves and use normal text color; Tabber must not list every individual file, nested repo subdirectory, `/tmp` artifact, or `.claude/projects/...` transcript path under them. Non-agent tmux sub-windows are leaf rows.
- Startup behavior: immediately after launch transcript metadata is available, Tabber should already be able to render the Level 0/Level 1 session, process, PID, active-window, and Claude/Codex icon rows. PID text is visible inline inside compact Tabber sub-window labels. The browser starts one `/api/activity?hours=<lookback>` request for the server-cached activity snapshot; when that snapshot contains `agent_windows`, path rows render from it. This first useful paint must not wait for a browser-side `/api/session-files` fetch; touched repo/path folding is backend-owned and may update on the normal cached activity cadence.
- Toolbar behavior: Tabber owns its toolbar and pane frame controls. Finder/Differ session controls, Finder-only controls (`.*`, Sync/path controls, create file, create folder, Finder sort/date/reload controls), and Differ-only controls (Differ expand/collapse) are absent because Tabber is an inventory/navigation view, not a filesystem editing or changed-files view.
- Rows render through the SHARED Finder/Differ row pipeline (`renderTreeChildren` -> `updateFileTreeRow` -> `updateFileTreeRowContents`) via a `mode: 'tabber'` render option whose per-row display values are precomputed as data — never a bespoke per-pane row builder. Selection, the icon column, and the shared date modes (`None` / `Date` / `Ago`) all apply unchanged. Tabber uses restrained level colors: level 0 session/other-pane rows share one accent, level 1 window/process rows share one accent, and level 2 paths stay normal foreground text. Purple remains reserved for merged PR status. Tabber rows store hover text in `data-tabber-title`, not the native `title` attribute, so the browser cannot show a second tooltip on top of YOLOmux's full Tabber hover. The collapse state is a persisted COLLAPSED set (default expanded) keyed by a stable id-based node path (`s_<id>/w_<i>/r_<n>`), so it survives reloads and a session add/remove never shifts another node's open/closed state. Clicking a row's disclosure icon toggles that node; the toolbar `Expand all` / `Collapse all` apply to the whole Tabber tree.
- Active-tab, focus, and layout-only changes must patch the mounted Tabber rows through the shared lightweight state synchronizer; they must not rebuild the Tabber tree or reload the Dockview layout. With a visible 41-row Tabber fixture over 12 sessions and 24 tmux windows, 30 tab clicks must produce zero full Tabber renders, zero Dockview `fromJSON()` calls, at most one lightweight sync per click, preserve mounted panel identity/view state, and meet p95 under 50 ms plus max under 100 ms in the focused Chrome run. The shared xdist gate treats those operation counts as the primary invariant because unrelated workers can delay animation frames.
- Time column + sort: every Tabber tmux sub-window row has one semantic recency timestamp used for visible date/age text, row `mtime`, and parent bubbling. Claude/Codex tmux sub-window rows use the backend agent list from `/api/activity`, whose recency comes from meaningful transcript event timestamps; raw transcript file mtime is not activity unless the transcript has no parseable event timestamp. Live working/attention state stays visible through the shared agent icon/status ball, but it must not replace the date/age column or make the row look newer than its transcript event recency. Agent sub-window user-input heartbeats and touched repo/path child mtimes must not affect that sub-window's time or sort order. Non-agent tmux sub-window rows use `activity[session:window].active_recency_ts` from the activity ledger. A parent session row inherits the max final semantic `mtime` of its child tmux sub-window rows, so it must not show a newer age than every visible child sub-window. Touched repo-root rows use the newest actual file mtime among files folded into that repo root, falling back to transcript time only when the touched file no longer exists, and those touched-path mtimes never bubble into the Claude/Codex window row or parent session row. The date/ago column is always reserved at every pane width; row label/tree text truncates first instead of hiding recency. The tree honors the active Finder sort control for top-level sessions and touched-path rows — `A-Z`/`Z-A` sort by the human label, `new`/`old` by the recency time — and re-sorts on the normal poll without collapsing the tree (collapse state is keyed by the stable path, not display order). The immediate sub-window/process rows under each session are the exception: they always stay in tmux sub-window index order (`0`, `1`, `2`, ...), regardless of recency or the active tree sort. The server owns the `/api/activity` cache: it builds the first snapshot at startup, runs a dedicated background warmer on the `performance.tabber_activity_refresh_ms` cadence, and serves the latest cached snapshot immediately while any stale refresh runs; the default cadence is 15 seconds and Preferences shows it in Performance. Opening Tabber may read `/api/activity?hours=<lookback>`, but that request must read the server snapshot rather than rebuild Tabber activity on click. An empty follower response marked `cache.refreshing_elsewhere` is a refresh placeholder, not a new snapshot: Tabber must preserve the last useful activity payload so visible Date/Ago values cannot disappear and return on alternating polls. Older overlapping activity requests must not overwrite a newer accepted snapshot, while a complete non-placeholder empty snapshot remains authoritative when records legitimately expire from the lookback. YO!agent Recent Agents uses the same server-cached agent list instead of building a separate browser-side list. The server folds `/api/session-files` touched-path data into `agent_windows.path_entries`; known repo roots are the display unit, so descendant touched directories fold into a top-level repo such as `/home/keivenc/ai-config` instead of rendering redundant child rows like `/home/keivenc/ai-config/claude/skills/...`.
- Tabber session descriptions have no character-count truncation. They pass the full PR/Linear/branch-subject text into the shared tmux tab renderer; CSS may show an ellipsis only when the rendered row actually runs out of inline space.

## Session Work Graph And Repo Detection

How a tmux-session tab header and YO!info decide which git repo(s) a session is working in, and when they show `no git checkout detected`. Implemented by `work_graph_path_observations` -> `resolve_path_observation` -> `session_work_graph` in `yolomux_lib/metadata.py`, with the transcript edited-dir signal from `session_touched_dirs` in `yolomux_lib/session_files.py`.

- The repo is derived from MULTIPLE signals, not just the live pane cwd. Candidate directories, strongest first: (1) the directories the agents have actually EDITED files in, read from each agent transcript's edit-tool scan (Claude `Edit`/`Write`/`MultiEdit`/`NotebookEdit`, Codex apply-patch — edits only, NOT files merely read), (2) the focused pane's current path, (3) each agent's launch cwd, (4) other panes' current paths, (5) the session-number default workdir when configured, (6) configured companion dirs. The edited-dir signal (1) is strongest because it is real work evidence, not a guess.
- Each candidate directory is resolved to its git top-level (`git rev-parse --show-toplevel`) and deduped by repo root. The PRIMARY repo shown in the tab header is selected by signal strength first; inside the same signal tier, a repo whose current branch has a PR ranks first, then repos rank by the most recent git activity (the later of dirty-file mtime and last-commit time), with stable discovery order as the final tiebreaker. The live pane cwd, agent launch cwd, and other pane cwd signals share one tier; transcript edited dirs sit above them. A tmux-session tab renders exactly one work description from that primary repo; it does not concatenate PRs, Linear issues, commit subjects, or branch names from the session's other repos. The Info Bar repo carousel is presentation-local and does not change the primary repo or retarget the tab description. YO!info lists every distinct touched repo and is the complete view when a session works across multiple repos.
- CONDITION — agent launched from a non-repo cwd: a Claude/Codex started in `$HOME` (or any directory with no git checkout) that is editing files inside a real repo MUST still surface that repo and branch via signal (1). The header must not show `no git checkout detected` while the transcript shows edits in a real repo.
- REQUIREMENT — `$HOME` is never a project repo. `focus_root_for_session` explicitly excludes `Path.home()`, so a home-directory checkout (even a valid one) is never treated as the session's project; home-based sessions surface their repo only through the other signals.
- REQUIREMENT — a directory counts as a "checkout" only if `git rev-parse --show-toplevel` succeeds. A `.git` directory existing on disk is NOT sufficient: a stub/incomplete `.git` that lacks `HEAD`/`objects`/`refs` (for example a placeholder with empty `config`/`hooks`) makes git return rc=128, and the directory must be treated as repo-less. This is why `$HOME` with a placeholder `.git` reads as `no git checkout detected` rather than as a repo.
- CONDITION — no signal yields a checkout: when none of the candidate directories resolve to a git repo (fresh shell in a non-repo directory, no edits yet), the tab subtitle is `no git checkout detected` (`t('git.noCheckout')`). That is the correct terminal state, not a failure.
- PERFORMANCE REQUIREMENT — detection runs on the hot metadata-refresh path (every session, every refresh) and now scans agent transcripts for edited dirs. The Claude transcript scan MUST be memoized by `(path, mtime, size)` (matching the Codex scanner's cache) so an unchanged transcript reads from disk once; the cache key MUST include mtime+size so an appended transcript re-scans and a newly-touched repo appears without a stale-cache miss.

## YO!share (Live Session Sharing)

YO!share lets the host share the live YOLOmux layout to other browsers via a short `/share/<id>#t=<secret>` URL. Read-only viewers observe; write viewers may also type through validated input intents. The guiding rule: the client mirrors the host's visible state and never re-derives geometry or invents transient UI state. The detailed DOM replay architecture spec lives in [`SHARE_MIRRORING.md`](SHARE_MIRRORING.md). See also the File menu entry in Search, Menus, And Popovers and the `share.*` Preferences.

Architecture: read-only and write viewers use host-rendered DOM replay by default. The host is the only renderer for mirrored app state, streams sanitized `#appRoot` keyframes/deltas, and terminal panes stay on the host-sized byte stream because xterm internals do not serialize through DOM mutation. Viewer DOM is inert. Write clients send allowed input intents to the host/server and receive host replay frames for visual feedback. Normal viewer chrome shows replay health (`mirrored`, `resyncing`, `host disconnected`, `viewer behind`) rather than semantic bucket names.

Replay repair is keyframe-based. Any host topology or global-UI change that cannot be trusted as a small delta may publish or trigger a complete epoch-keyed DOM keyframe. Clients apply frames only in sequence for the current epoch; gaps, digest mismatch, slow-viewer backpressure, and manual debug requests ask the host/server for a new keyframe. Digestless mutations that target an unknown replay node are treated as stale deltas from a replaced subtree: the viewer advances the replay cursor without entering `viewer behind` and waits for the next scheduled keyframe. Terminal resize remains separate: `terminal-host-resize` resets viewer xterms to host rows/cols before fresh tmux repaint bytes arrive.

The default fit is `cover` from `share.view_fit`, cropping the long axis symmetrically; `contain` letterboxes instead. The viewer must not reflow the app against its own viewport, and raw `window.innerWidth`/`window.innerHeight` reads outside the shared viewport owner are forbidden. Host terminal cols/rows stay pinned and viewer resize only changes the mirror transform, never the shared tmux size. Read-only viewers are passive: Dockview may apply host-authored layout frames, but its local measured geometry must never be adopted back into `layoutSlots`, including in the semantic diagnostic escape hatch and across cover/contain changes. Write viewers may send validated input intents over HTTPS, but they do not publish semantic layout, Finder, editor, Preferences, popup, or scroll state; host replay frames provide visual feedback.
- YO!share URLs must stay short (`/share/<id>#t=<secret>`); sessions, layout, tabs, and UI state live server-side behind the id. Host-owned popups, menus, context menus, language pickers, modals, and transient messages mirror as normal sanitized DOM inside `#appRoot` through replay. Share-modal URLs and token-bearing fields are marked as share secrets and redacted before DOM keyframes, deltas, debug copy, or server UI frames reach clients. Mirrored modals use the host DOM and host content sizing, with no client-only reflow, forced child height, or extra scrollbar. Active share rows are compact: `New share` and `Active share URLs` sections, max time/viewers/read-only/protocol grouped, URL plus copy in one row, and Stop inline with mode/protocol. The active URL block is the primary share affordance in the modal; it must be visually stronger than metadata rows while keeping exactly one copy target beside the URL input. Only the host can stop or extend active shares; viewers only receive refreshed countdown and viewer-count status. Read-only viewers must preserve browser text selection, copy, context-copy, and harmless navigation defaults while blocking activation and mutation. Editor/diff parity comes from host replay plus share-scoped file reads for every host-visible file editor identity (`file:`, `filediff:`, `filecopy:`, and `image:` tabs). Saves still publish file-version invalidations for open shared files. Pre-auth login/setup language pickers use in-page buttons and hidden inputs rather than native selects so OS popups cannot leak as broken share-view boxes.
- When a share viewer's host-provided expiry reaches zero, the viewer must navigate to the login screen immediately through the shared login redirect path; it must not wait for a websocket close, terminal reconnect loop, or the low-frequency `/api/share` backup poll. The login `next` target includes only the pathname and query, never the `#t=` fragment token.
- The YO!share manage view must show a `Users` section inside each active share row. The section lists one row per distinct connected browser, with connected duration, IP address, and summarized browser type. It updates from pushed share status on join/leave without requiring the host to close and reopen the modal, and it must not expose full user-agent strings or share secrets in the mirrored popup layer. The Users section is an inline subsection of the active share row: row separators are allowed, but it must not draw a second card border, framed table box, or card background inside the already-bordered share row.
- YO!share clients mirror the host's visible UI state, not the client's inferred state. When the host opens, closes, hovers, unhovers, scrolls, changes language, changes theme, changes layout, changes active panes/tabs, toggles tab metadata, switches YO!info/YO!agent, or expands/collapses editor diff context, the client must converge to that exact host state and clear any stale mirrored surface immediately. This includes the YO!share modal, all app menus, submenu hover state, language popovers, tab hover popovers, context menus, transient messages, editor scroll, Finder/Differ scroll, pane split geometry, active/focused pane chrome, tab metadata visibility, YO!info/YO!agent active sub-tab, and popup font sizing. Client-side hover or focus must not be required to clear stale host UI, and read-only replay clients must not synthesize local tab popovers from their own pointer position. Desktop session tab popovers prefer the owning pane width and may exceed it only by a bounded allowance of 15% up to 64px when the browser has room; they remain viewport-clamped, while file-preview popovers keep their existing content sizing. Host tab popovers must carry their computed app-space `left`, `top`, measured inline `width`, and measured `height` on the replayed popup node, not only in document-level CSS variables or an `inset` shorthand, so Safari/Chrome viewers under a mirror transform render the same host position and wrapping. Client-side scroll must not persist or snap mirrored panes back to a local default; read-only client scroll input is blocked for mirrored app surfaces except where browser text selection requires a harmless default. Host menu and popover mirroring is an open/close contract as well as an HTML snapshot contract: a closed host popup means no mirrored popup remains on any client. Language changes must update host and clients atomically enough that no pane blanks or loses its current content while theme/locale text is re-rendered. Theme changes must update replay-shell viewers through live `appearance` frames as well as DOM replay, preserving the host `theme`, `resolvedTheme`, and `terminalTheme` values so mirrored terminals repaint across light, dark, and system. User-facing language choices show human-readable language names/endonyms only; raw locale codes such as `en`, `es`, `ja`, or `zh-Hans` are not visible UI labels except in explicit developer/debug output.
- In the semantic YO!share diagnostic path, each scroll target has one record for its host payload, snapshot generation, restore generation, and remaining restore frames. A full `ui-state.scroll` snapshot replaces target membership and deletes absent records; queued restore callbacks must verify current record identity/generation so a closed and later recreated Finder/editor target cannot inherit stale scroll.
- YO!share replay keyframes must include the host's current scroll targets, compact YO/auto-approval state, theme/language/font readiness, popup/menu/modal DOM, terminal placeholders, and share-visible file/editor identities, not only the pane tree. This is required for late-joining viewers, keyframe repair, Preferences scroll, tmux/Tabs badge parity, and host menu rows that depend on YO state. Live editor/file refresh for share viewers is push-driven through share UI/file-version frames; `/api/share` polling on read-only viewers is only a low-frequency status backup for banner metadata, not a high-frequency editor refresh mechanism. Same-epoch incremental DOM replay updates should keep applying even when a viewer detects a sequence gap, while viewer repair keyframe requests and host repair keyframes are throttled to a five-second floor. Topology and join keyframes keep host mutation deltas quiet for the keyframe floor plus slack so Finder/Differ/minimize loops do not interleave node-id deltas from an old DOM base between full replay frames. Semantic geometry resync stays on a ten-second floor.
- YO!share visual parity is a replay-health contract. Host and viewer must load the bundled `YOLOmux UI` and `YOLOmux Mono` fonts before the first mirrored layout so serialized DOM dimensions stay stable, but viewers do not compute their own wrapped-text or pane geometry to decide whether they are synced. Read-only replay health asserts `domDigest`, epoch/sequence continuity, dropped-frame/keyframe-request counts, redaction policy version, node count, and terminal placeholder health. Legacy semantic geometry/text-wrap digest buckets such as `slots`, `tabStrips`, `editors`, and `textWraps` may remain only for the explicit semantic escape hatch or targeted diagnostics; normal read-only replay must not show bucket names or repair by applying another semantic widget snapshot.
- YO!share debug diagnostics are opt-in with `?shareDebug=1` on a share-view URL or `localStorage.yolomux.shareDebug=1`. Replay debug records one sanitized `share-replay-health` report on `window.yolomuxShareDebug` with DOM digest, epoch, sequence, dropped-frame count, keyframe request count, redaction policy version, node count, terminal placeholder health, fit mode, viewport, `visualViewport`, DPR, user agent, root/stage rects, and mirror transform. The debug banner copy control appears only in debug mode and copies the latest report as JSON. Share tokens and `#t=...` fragments must be redacted before console logging, window exposure, or clipboard copy. Legacy semantic drift reports are allowed only when the semantic escape hatch is active.
- YO!info is part of the same host-owned share state as Finder, Differ, editors, and Preferences. Branch-table row content, sort column/direction, resizable Branch and desc column widths, vertical scroll, and horizontal scroll must publish from the host and apply on read-only clients. A viewer must not rebuild YO!info rows from its own scoped transcript metadata after a host snapshot arrives; it renders the host's rows until the host sends a newer row snapshot. Column widths are part of the mirrored UI state, not browser-local storage on the viewer; applying them on a client must not persist them or echo-publish a client-authored width. A read-only client wheel, scrollbar drag, hover, focus, or other local interaction over YO!info must not publish or persist a client-authored position; it restores to the last host-authored `#info-content` scroll frame.
- Finder, Differ, and Tabber state is host-owned at the model layer, not only at the DOM event layer. The host snapshot owns each item's independent identity, pane location, active state, Finder root mode/root path/selected session/expanded paths, Differ target session/changed-file collapsed sets/per-repo refs, Tabber collapsed set, and shared selected-path/date/sort state where applicable. Finder and Differ session selectors are separate owners: changing Finder may synchronize its root to that Finder session but must not retarget Differ, and changing Differ must not change Finder's selected session. A read-only client's local transcript refresh, `/api/session-files` result, hover/focus state, active-tab sync, or manual open attempt must not retarget a surface or collapse/expand its tree. Repeated host frames with the same Finder root must refresh in place when needed and must not reopen the root in a way that briefly clears expansion; only a changed host root should reopen the tree.
- HTTP read-only share parity covers the same visible surfaces as HTTPS share parity. A viewer loaded from an HTTP share URL must render the host's Finder rows without a red path field, pane tab metadata without `no path` or `finding branch...` fallbacks, file editor contents without `File could not be opened` / `TypeError: Load failed`, and Differ/Tabber data without local fetch-error placeholders. These checks must run in Selenium against a real `http://.../share/<id>#t=...` page so token propagation, plaintext redirect exemptions, and browser fetch/WebSocket behavior are exercised together.
- YO!share popups, menus, context menus, language pickers, modals, and transient UI mirror as ordinary sanitized DOM inside `#appRoot`. Popup roots must mount under the app overlay owner so the replay serializer sees them; stale popup resurrection is prevented by replay epoch/sequence handling and keyframe replacement, not a separate popup-layer shell.
- Share-view terminal grids are host-PTY authoritative. A viewer terminal must size from the host's rows/cols only; it must not fall back to `estimateTerminalSize()` from the viewer pane, native viewport, local font metrics, or transient container visibility. A host resize, `terminal-host-resize` frame, or terminal placeholder update that changes rows/cols must resize the viewer xterm, call `reset()` to discard any smeared/refowed buffer rows, and rely on the server-side `tmux refresh-client` repaint to refill a clean grid at the host size. The server must deliver `terminal-host-resize` before requesting the tmux repaint so clients reset before fresh terminal bytes arrive. A viewer that reattaches to an already-running shared upstream must request a bounded `tmux refresh-client` repaint; otherwise layout rebuilds can create a blank xterm that does not receive a full curses screen until manual refresh. If host dimensions are unavailable, the viewer skips resizing rather than inventing local dimensions.
- Dockview panel show/hide is part of the terminal lifecycle. When a share-view host frame removes or minimizes Finder and Dockview remounts tmux panels or activates tmux tabs that were hidden during the topology change, the viewer must ensure the xterm is bound to the currently connected terminal container, reconnect/restart stale bindings, and keep using the host rows/cols.
- Read-only share viewers are visually host-identical inside the mirrored app. They must not add client-only disabled styling, `readonly access` labels, native scrollbars, draggable scrollbar tracks/thumbs, or other read-only chrome inside the mirrored Preferences/Finder/Differ/editor/terminal surfaces. Read-only protection is enforced by the inert replay shell and host-authored scroll frames; host scroll is applied programmatically and retried after keyframe/delta apply so a late DOM rebuild cannot snap a viewer back to a local default. Viewer-only controls such as the share banner, fit toggle, and replay health stay outside the mirrored app root.

### Transport and server architecture

- HTTP read-only share works even when YOLOmux is launched with HTTPS, on the SAME port (no second listener). The server muxes protocols by peeking the connection's first byte (`TLS_FIRST_BYTES`); the peek timeout must stay generous (`tls_peek_timeout_seconds`, currently 2.0s) and must NOT run so short that a remote client's first byte (one RTT away) is misread as a TLS preconnect — a 50ms peek silently broke the plaintext->https redirect for remote browsers. Plaintext requests on a TLS host are SERVED only for share-scoped paths (`/share/<id>`, `/`, `/static/*`, `/ws/share-view`, `/ws/share-ui`, `/api/ping`, `/api/share`, `/api/share-stream`, readonly share data APIs such as `/api/session-metadata`, `/api/transcripts` compatibility metadata, `/api/activity`, `/api/session-files`, `/api/fs/list`, `/api/fs/info`, `/api/fs/index-status`, scoped file APIs such as `/api/fs/read`, `/api/fs/diff`, `/api/fs/raw`, and readonly batch `/api/fs/batch`) AND only when the matching share record has `http_allowed` (`plaintext_share_scope_allowed`); every other plaintext request gets a 308 redirect to https (`redirect_plaintext_to_https_if_needed`). Plaintext share connections are read-only regardless of token mode, and no cookies are set on them. Write mode requires https.
- Terminal mirroring fans out from ONE upstream pty+`tmux attach` per shared session (`share_terminal_upstream`), never one attach per viewer (viewers x panes attaches would melt tmux). The upstream pty is pinned to the host's rows/cols; viewers never resize it. Each output chunk is serialized into one frame once and enqueued to every viewer (`broadcast`). Backpressure is per-viewer: a bounded outbound queue (`SHARE_VIEWER_QUEUE_HIGH_WATER_BYTES`) — on overflow the viewer's backlog is dropped and ONE rate-limited `tmux refresh-client` (`SHARE_REFRESH_CLIENT_MIN_SECONDS`) repaints all lagging viewers from a single screenful; a viewer that overflows repeatedly (`SHARE_VIEWER_OVERFLOW_LIMIT` within `SHARE_VIEWER_OVERFLOW_WINDOW_SECONDS`) is disconnected `too-slow`. One slow consumer must never buffer the server into OOM or stall the fan-out loop. Upstreams with no viewers are reaped (`close_inactive_share_upstreams`).
- Host resize propagation is a terminal repaint contract, not only a layout contract. `record_host_pty_dimensions` updates each matching share upstream, broadcasts `terminal-host-resize`, and only then requests `tmux refresh-client` after `set_pty_size`/SIGWINCH so terminal bytes following the resize redraw a full clean screen after the client has reset to the host rows/cols. Layout commits publish replay keyframes or deltas so terminal placeholders, tab metadata, YO!agent sub-tab state, Finder/Differ state, editor diff expansion, and Preferences state converge after pane rebuilds; semantic `layout`/`ui-state` snapshots are legacy escape-hatch behavior, not the normal share path.
- Viewer accounting is per distinct browser (`viewer_ids` on the record), not per websocket; status (viewer count, expiry) is PUSHED to host pill/manage-modal and viewer banner via `broadcast_share_status` on join/leave/extend, not only polled. The read-only viewer `/api/share` status poll is a backup and must stay low-frequency because normal UI, editor, file-version, scroll, popup, and viewer-count changes arrive through the share WebSocket. The viewer cap refuses the WS upgrade past `max_viewers`. Records are in-memory: a server restart drops all shares (URLs become invalid -> login), which is acceptable and intended. The data model is keyed by token to allow N concurrent shares later (mixed modes/TTLs); v1 may gate to one active share at mint time.
- Host-local surfaces must NOT be mirrored to viewers: the latency meter is the host browser's own RTT and is meaningless on a client; viewer-only chrome (the share banner, fit toggle, replay health status) belongs to the viewer, not the host; the real OS cursor is replaced by the ghost cursor; share secrets/URLs stay redacted. Each remote sender reuses one ghost DOM node and one replaceable hide timer in the same sender record; different senders remain isolated, self-echoed frames are ignored, and a stale timer cannot hide a newer frame. Decision rule for any new mirrored surface: host-owned visible state inside `#appRoot` mirrors through sanitized DOM replay with the client inert; client-local or host-private state is redacted or excluded. When unsure whether a surface belongs in the mirror, register it with the replay/redaction policy and add a replay health or sanitizer test rather than adding another semantic snapshot/apply pair.

## Preferences Contract

- Preferences is a normal tab type and may be moved, activated, minimized, and closed like other non-Finder tabs.
- Preferences must expose a search field, collapsible sections, per-row reset controls, a global reset flow, the settings file path, and a settings age/status indicator.
- Preference changes save through the shared settings API and live-apply when the setting affects visible UI. Appearance, terminal/editor, Finder/Differ, popover timing, notification, and YOLO changes must not require a page reload unless the setting explicitly says so.
- General settings must include language, auto-focus, and startup tips. The legacy default-sessions setting is accepted from old config files but is hidden because running servers default to all discovered tmux sessions unless launched with `--sessions`.
- Appearance settings must include global theme, default layout, UI and Finder font sizes, tab width, max tabs per pane, pane spacing, pane ring opacity, inactive pane opacity, active color, separator color, editor cursor color, and 12-hour/24-hour clock mode. Clock mode is global: changing it live-repaints every mounted timestamp surface, including YO!stats axes/hover readouts, YO!agent, YO!chat, Finder/Differ, event logs, and editor blame. Notification settings include the green/red/yellow workflow transition glow duration. Changing Active color must live-apply the matching tmux status/window/pane styles to every existing tmux session/window, and newly created tmux sessions must receive that saved Active color immediately after creation.
- Terminal / Editor settings must include terminal theme, terminal/editor/preview font sizes, terminal scrollback, editor dark/light schemes, editor cursor style, word wrap, line numbers, autosave, autosave delay, and all-lines blame.
- Notifications must include both update controls: the server/client reload prompt (`general.reload_on_update` / `general.reload_on_update_auto`) and the origin/main update checker (`updates.check_enabled`). The reload prompt reacts when the running server reports an app version that differs from the open page's boot client version at the enabled semantic threshold, or when the served client bundle revision differs from the page's boot bundle revision; it does not check GitHub. The prompt must ask whether to reload the browser and must reuse the existing safe-reload gate so dirty editors or active typing are not interrupted. The origin/main checker is the control that shows the admin `Update & restart` card, and it is off by default. `updates.notify_level` controls the minimum semantic version change that produces the origin/main update notification: `major`, `minor`, `patch`, or `none`, default `patch`.
- Notification titles are surface-aware through one shared formatter. In-page toasts omit the redundant `YOLOmux[host/session]` external identity and show only the actionable title such as `Needs input`, `terminal`, or `notifications enabled`; browser-to-OS notifications keep the full `YOLOmux[host/session] ...` title so the source remains identifiable outside the app. Browser-to-OS notifications use a badge-free 192px YOLOmux icon from the same theme/accent-aware canvas renderer as the browser favicon; individual notification call sites do not duplicate icon assets or rendering options.
- YO!chat history hydration is never a new-message notification. Only messages received after a person has established their chat read cursor may create one in-app/browser notification. The server keys that cursor only by room and authenticated username, never by browser, tab, IP, or client-supplied identifier: acknowledging as Keiven synchronizes all of Keiven's browsers, while Calvin and Jeremy retain independent unread state. Dismissing a YO!chat toast, opening it, or explicitly focusing/clicking/typing in YO!chat advances that same durable person cursor; a reload therefore cannot re-create an acknowledged historical toast. The generic notification target acknowledgement hook owns navigation-triggered acknowledgement so producers do not add parallel focus listeners.
- Finder/Differ settings must include root mode, image open mode, image preview size, indexed directories, explicit Quick Open exclusions, index refresh, companion directories, directory cache timing, and new-entry highlight timing. Finder/File Explorer right-click on a directory owns the matching indexed-root action: **Allow index** adds that directory to Indexed directories and **Disallow index** removes it. Quick Open exclusions are shared multiline preferences: directory-name exclusions and one-rule-per-line path exclusions (a plain absolute/home-relative subtree path, `glob:<root-relative glob>`, or `regex:<regular expression>` against root-relative POSIX paths). Multiline controls preserve their own caret and internal scroll position across passive Preferences refreshes; the directory-name list grows and shrinks with its content through 20 lines, then scrolls internally. The path rule list must be searchable by exclusion, ignore, glob, regex, and performance terms; invalid regex entries do not break indexing and are ignored.
- An in-page agent attention toast names its exact target as `[session] window: Needs approval` or `[session] window: Needs input`, for example `[8002] 2:codex: Needs input`, then shows the same non-interactive session Tab and tmux sub-window status control used elsewhere: the shared `[session]` Tab plus the red stop or yellow pause glyph, stable Claude/Codex identity icon, and window label such as `2:codex`. Those controls form one floating group, so the remaining message text wraps around it instead of being held in a permanent side-by-side column; it must not repeat the same window label. Clicking anywhere on the alert, including either control, activates the terminal tab and switches to the named tmux sub-window through the shared direct-window route. A toast pauses both its countdown bar and removal timer while hovered or keyboard-focused, and resumes when the pointer/focus leaves unless it is kept. Upload-result toast titles are local UI context, so they show the session and upload action without a redundant `YOLOmux` or hostname prefix; browser-to-OS notifications retain their source identity. Each session's upload-result state is one `{entries, cleanupTimer}` record, retains at most the newest eight lines, clears its timer when kept or hidden, does not schedule an infinite timer for persistent lines, moves intact across session rename, and is deleted on expiry or teardown.
- Uploads/Downloads must include file transfer size cap, filename template, central temporary-upload retention, terminal file-action suggestions, read-only shell autorun, and custom file-drop actions. Every terminal and Markdown-editor upload uses `/tmp/yolomux.<authenticated-user>/uploads/<session>/`; the authenticated YOLOmux username, not the Unix account, owns the namespace. The private tree rejects symlinks, non-directories, and foreign Unix ownership, uses `0700` directories and exclusively-created `0600` files, and prunes only expired regular files inside that verified tree. Existing worktree `.upload`/`.uploads` references remain readable but the server never creates new ones. Editor paste inserts the returned absolute path.
- Performance must include server SSE file-change polling, server SSE background file-change polling, server SSE directory-change polling, cached Tabber activity refresh, latency refresh, event-log refresh, popover show/hide timing, menu hover timing, tab popover timing, and remote resize debounce. The visible-editor file-change poll checks open editor files in visible panes for fast `files_changed` pushes. The background-editor file-change poll checks hidden/background editor tabs less often so inactive tabs still notice disk edits without making every server tick expensive.
- GitHub must include watched PRs.
- YOLO must include auto-approve interval, rule file path, dry-run mode, and prompt source.
- YO!agent must include backend, invocation mode, Claude/Codex model and effort controls, system prompt, intro, and format. It must not expose a background transcript-summary interval; rolling transcript summaries run once on the first visible YO!agent launch in a server run, then stop until the server restarts. The invocation row currently exposes only CLI; reserved transports such as API-key mode must stay hidden until they work end to end. Model selects must be wider than compact selects, but only enough to show the longest model label plus padding; they must not stretch across the whole Preferences row.
- YO!agent has Preferences parity. It must answer settings questions from the backend settings catalog, not from scraped Preferences DOM, and include current value, default, valid choices/range, GUI location, and config path when useful. Admin setting writes from chat must route through the same server settings save path as Preferences, publish the normal `settings_changed` event, preserve concurrent unrelated settings edits, and render as a normal YO!agent answer with a compact before/after block. Read-only users may ask explanatory settings/product-state questions but cannot write. Background/white/black writes map to the global Appearance theme (`change background to white` -> `appearance.theme=light`, `change background to black` -> `appearance.theme=dark`). Bare UI color writes such as `change color from green to orange` mean Appearance -> Active color (`appearance.active_color`) unless the user names cursor color, separator color, theme, or another specific color setting. Ambiguous settings, broad resets, YO!agent prompt/format edits, YOLO rule paths, share access, and path-sensitive list changes require clarification or confirmation; credential-heavy paths and secret-looking values are rejected. Successful live settings changes must update the visible GUI without manual refresh unless the catalog marks the setting as next-use/restart. Opening the standalone YO!agent tab must render the latest transcript at the bottom; background SSE/result refreshes may preserve manual scrollback, but explicit open/attach paths must bottom-scroll. YO!agent has one vertical scroll owner (`.yoagent-chat-history`), and wheel gestures over the composer must forward to that same history owner so the bottom of the panel is not a dead scroll zone.
- YO!agent starts the persistent Codex app-server as part of server startup when the configured YO!agent backend resolves to Codex CLI. Opening the YO!agent tab should not spend the first visible turn only warming the process; if the conversation is empty, the tab requests one real LLM startup answer, streams safe answer text as it arrives, persists the assistant message, and replaces the transient stream with the saved conversation. This startup response is shown once per empty conversation and must not disappear when the prewarm request finishes. Clear conversation invalidates an in-flight startup before deleting persisted state and blocks another startup until clearing finishes. An invalidated response must end its stream as stopped/aborted, return the current conversation without saving its answer, and never clear a newer replacement startup's active state.
- YO!agent message timestamps include seconds. Assistant replies keep all safe diagnostics collapsed in an expandable Details block: backend, response time in seconds and milliseconds, model CLI elapsed seconds, prompt size, seed/resume state, context-change note, stream phase, and fallback reason. This metadata must never appear as ordinary reply text, including beside a collapsed Usage/stream block. Raw hidden chain-of-thought, reasoning deltas, or `<think>` blocks must not render in the visible answer or Details block; show only a short note that raw model thinking was hidden and keep a safe thinking/progress indicator visible while waiting.
- YO!agent messages that are direct results from another tmux/agent session (`agent_result`) must read like quoted output from that session: keep the stronger result bubble, leave the `Result from tmux session ...` heading unindented, and put the returned output below it in an indented block with a full-height left bar so users can distinguish relayed target-session output from YO!agent's own narration.
- YO!agent prompt latency should be dominated by the work the selected path actually needs. Settings reads/writes, skill-file operations, job creation, and explicit send routing must not build the full activity summary before answering; they may use settings payloads and direct target/session discovery only. Product-state questions such as `what changed`, `what did I last work on`, `what PR was that`, session-state explanations, and model-backed fallback may load the server activity payload because they need that evidence. The response Details `response time` is end-to-end server response time, not pure model first-token time.
- YO!agent persisted-job rows use the shared queue renderer and show localized job status/type, watched roster, separate send destination, stable quiet duration, current blockers, redacted command preview, and Confirm/Cancel controls where applicable. `firing` uses the shared sending label; protocol values such as `wait_roster_then_send` and `all_calm` never appear in browser-visible text. A roster job is cancellable while queued or pending confirmation, and the row converges from `yoagent_jobs_changed` SSE updates without duplicate cards. Deterministically parsed jobs follow their normal confirmation/risk policy; a job proposed from flexible model interpretation always starts pending confirmation and never exposes the provider's control object in the chat transcript.
- YO!agent product-state answers should prefer server-owned caches and registries: activity summary, Recent Agents/Tabber cache, session-file metadata, watched PR metadata, skill/context locations, and the product capability registry. It should answer questions such as `what did I last work on`, `what changed`, `what PR was that`, `which sessions are idle/running/blocked`, `where are YO!skills/context stored`, `what can I do from here`, and `why is this session behaving this way` from cached evidence, and say when a fact is unavailable instead of guessing.
- YO!agent is the central command surface for YOLOmux. Common-intent examples and cross-agent communication methods are specified in [`../YOAGENT.md`](../YOAGENT.md); that guide is the durable place for natural-language Preference aliases, explicit-send result defaults, wait-before-send jobs, original/excerpt/summary/modified handoff forms, artifact handoff rules, and the transport reliability ladder.
- YO!agent chat uses the terminal font-size setting for transcript messages, action cards, recent agents, and the composer. Transcript path stays compact above the scrollable chat history. Current info and Recent agents render inside the chat history as assistant-style YO!agent responses, not as detached chrome; their activity-snapshot timestamp participates in the same chronological timeline as persisted and streaming messages, so a newer question or answer always renders below an older activity snapshot. The list is a bullet list, not chips: each row shows the shared agent icon, tmux session label, tmux sub-window name, recent touched path roots when available, and compact activity recency. When the user scrolls upward in the YO!agent history, background summary/result refreshes preserve that scrollback until the user scrolls back near the bottom or explicitly sends/clears a message.
- YO!agent transcript messages that contain a result fetched from another target agent must be visually distinct from normal YO!agent answers. They keep the assistant-side placement but get a stronger colored left rule so the user can quickly see where the relayed agent result starts.
- YO!skills are loaded from built-in YAML first, then user-local YAML under `~/.config/yolomux/skills.d/`; user-local context Markdown lives under `~/.config/yolomux/context.d/`. Built-ins are read-only. User-local skills/context can be inspected and edited only through the narrow admin skill-file API or direct filesystem edits to those two directories.
- YO!agent composer `Up` recalls prior user messages, newest first, and places the caret at the end so the text can be edited and resubmitted. Repeated `Up` clamps at the oldest message. `Down` walks newer and then returns to the latest composer slot, restoring any unsent draft; when that slot is empty the normal `Ask anything...` placeholder is visible.
- YO!agent active request/action work, queued asks, composer history/draft, visible error, and fallback notice live in one chat record. Every busy-work completion uses the same finish path and drains queued asks, including action-send completion; cancel, retry, and Clear must not leave `busy` without an active operation or strand queued text. Prewarm flags and the frozen/visible startup activity snapshot live in a separate startup record: the snapshot stays fixed across ordinary activity pushes, explicit refresh replaces it, and one empty-conversation prewarm remains one-shot until Clear resets it.
- YO!agent background transcript summaries are first-launch-only. The first visible YO!agent launch in a server run may start one background summary pass across live tmux sessions; later opens, watch ticks, transcript changes, and settings saves must not start a recurring summary loop. The old `yoagent.auto_refresh` boolean and `yoagent.refresh_interval_seconds` value are ignored compatibility inputs and must not appear as live Preferences rows.
- Tabber rows, pane tabs, YO marker/ring state, activity APIs, and YO!agent target status consume the shared screen state. They may show working from a static visible working row, recent transcript activity, or an advancing visible status counter; stale repeated counters must not keep a target busy forever. Visible-counter temporal history is pane-lifetime state with one bounded owner: a non-counter capture clears that pane, records unseen for 30 seconds are pruned, at most 512 pane records remain, and regularly observed static counters become stale after 75 seconds.
- Typing into one terminal may acknowledge that terminal's current red/yellow attention event once, but repeated xterm data for the same event must not restart its acknowledgement visual, rebuild unrelated pane/Tabber/status surfaces, restart another session's ball animation, or send duplicate acknowledgement requests. Clicking a session Tab is also explicit engagement: it opens that pane and immediately acknowledges the current highest-priority red/yellow child for that session, so its ball enters the normal gray fade even when terminal focus is deferred. The click must capture that child before the pane/window switch; deferred focus must not subsequently acknowledge a different child that became current. Attention acknowledgement is host-global across YOLOmux servers that share a state directory. Cross-server acknowledgement fanout carries the changed keys as a scoped patch, while every explicit auto-status read merges the shared acknowledgement revision before using its local cache so a missed fanout or disconnected event stream cannot leave another port stale. Already-acknowledged keys are a server-side no-op that preserves the original acknowledgement timestamp and does not increment the shared revision or trigger another client refresh.
- YO!agent is the central command surface for YOLOmux. It must reason over all visible tmux sessions, not only the focused pane, and explicit send requests run without an extra confirmation unless the user asks for preview or confirmation. Explicit target-session sends return results by default: the first YO!agent response must confirm the send immediately, state that the target session was verified as accepting an AI prompt, show the exact prompt YO!agent is sending after perspective rewriting/redaction, and say it is awaiting the response; a background watcher then reads the target agent transcript delta, falls back to visible pane text on timeout, and appends a result message through the persisted conversation plus `yoagent_conversation_changed` client event. Phrases such as `do not wait for the result`, `just send it`, or `no output needed` opt out of the watcher and keep the action send-only. YO!agent must keep the composer usable while background result waits are pending, and multiple waits must render and finish independently. Wait-before-send jobs must re-resolve the exact pane, re-check that the target is still a Claude/Codex prompt, reject busy/approval/stale targets, clear any current input draft before paste, send through the same server-owned transport, and start the default result watcher after the send unless opted out. Target-session prompts preserve perspectives: YO!agent keeps the user's routing perspective local, then writes the target prompt from the target agent's point of view instead of blindly forwarding the user's wording. `ask agent 1 to <do ...>` sends `<do ...>` to agent `1`, not `ask agent 1 to <do ...>`, and `ask session 1 what it has done today` sends `what have you done today?` to session `1`, while YO!agent's local explanation may still say it is asking tmux session `1`. Generated handoff prompts should be concise clean tasks/questions, not multi-line routing/context wrappers, unless the user explicitly provided raw quoted multi-line text. YO!agent may pass original, modified, or summarized information to another target session, but it should do that centrally: capture the source output, bound or validate it, transform it when requested, and send a clean target prompt. When a current target input draft exists and the user explicitly asks YO!agent to send, YO!agent clears that draft first, verifies the composer is empty, then pastes and submits the new prompt; if the draft cannot be cleared, it refuses and does not paste. After a paste-plus-Return it must not claim success if the sent text is still sitting in the target composer. Old submitted `❯ ...` prompt lines visible above a current blank composer must not be treated as current unsent text. For Claude/Codex transcript deltas that show a tool/turn lifecycle, the watcher must not append the first readable text as final; it waits for a terminal assistant/task-complete marker such as Claude `stop_reason: end_turn` or Codex `task_complete`, so an intermediate pre-tool sentence is not shown as the completed target result.
- YO!agent action cards appear under assistant messages only for confirmed/preview paths. A ready card shows session, agent/model, transport, pane id, path, risk state, and redacted text to send, plus a single Confirm/send button for admin users. Waiting, expired, sent, and error cards show a state label instead of an active send button. Public action/job payloads and persisted chat history must redact known `token=`, `secret=`, `password=`, and `api_key=` values, while the server keeps the raw pending preview privately until it expires or sends.
- YO!agent jobs are one-shot background watches. Notify-session-idle, notify-all-idle, and wait-then-send jobs live in server state, survive reload/restart, and publish `yoagent_jobs_changed` client events when created, confirmed, cancelled, fired, failed, or timed out. Firing notify jobs use the existing toast/browser notification path. Wait-then-send jobs do not paste into a busy, non-agent, approval-prompt, missing, or stale target; they re-preview and revalidate the exact pane/agent before sending. High-risk prompt text forces confirmation even when the user did not explicitly ask for preview.
- YO!agent `all sessions` means all tmux sessions visible to the current user, not only the configured local YOLOmux session roster or currently open tabs. The job state tracks the per-session states and blockers so a future job list can show which session is still preventing the watch from firing. Batch fanout actions should remain per-target confirm/cancel instead of one opaque batch button.
- YO!agent direct-send transport is selected through the backend transport registry and must be named in action cards, send confirmations, result records, jobs, and diagnostics. Existing user-owned visible Claude/Codex panes use `tmux-legacy`: server-resolved visible pane paste plus Return, prompt-state preflight, stale-target checks, and post-send composer-cleared verification. `tmux-legacy` is not a native agent API and must never be described as more reliable than terminal automation. Managed transports may be preferred only when YO!agent owns or can safely resume that managed conversation and can surface the effective auth, sandbox, approval, and result-source semantics. The planned order is Claude SDK for YOLOmux-owned Claude sessions, Claude Channels for opt-in visible Claude panes, Codex SDK before direct Codex app-server calls for YOLOmux-owned Codex sessions, Codex MCP server for central orchestration, `codex-exec` for one-shot jobs, Claude stream-json as a CLI fallback, and `tmux-legacy` last. Blind `tmux send-keys` is a fallback implementation detail.
- YO!agent-managed structured sessions are background conversation/job targets first, not fake terminal tabs. A managed Claude/Codex SDK/app-server/MCP/exec session may appear in YO!agent action cards, waits, job rows, Recent Agents, and result transcript views with its provider, thread/session id, cwd, model, sandbox/approval settings, started time, result source, and latest final response. It must not appear as a normal tmux tab unless YOLOmux actually started or attached a visible TUI/PTY for that session. If a future visible TUI is attached to a managed session, the UI must label both identities: the visible tmux pane and the structured provider/thread that YO!agent controls.
- YO!agent multi-session handoffs are always orchestrated by YO!agent, not delegated from one target session to another. In a normal request, session `1` and session `2` do not know about each other; YO!agent coordinates and passes clean messages between them unless the user explicitly asks for disclosure. Direct relay/chaining between agents is rare and allowed only when the user explicitly requests relay/chaining; when allowed, YO!agent must pass concrete instructions that say how to relay, and must not leave the agent to infer the route. For a request such as "ask session 1..., get info from session 1, pass that to session 2", YO!agent asks tmux session `1`, waits for and records session `1`'s response, derives a bounded handoff prompt itself, verifies tmux session `2` is accepting an AI prompt, sends the handoff to session `2`, waits for session `2`, and appends session `2`'s result back into the YO!agent conversation. The prompt sent to session `2` must be the clean user-facing question or task, not a meta wrapper such as "tmux session 1 replied..." plus instructions. If the user asks YO!agent to pass exact original text, an excerpt, a summary, or a modified/derived prompt, YO!agent performs that transformation centrally before sending and labels uncertainty; for example, adding 35 minutes to a reported time produces only the derived question `Is 6:10 PM the correct time now?`. While any background result watcher is active, the chat history shows a compact waiting queue with one row per active wait; a handoff wait row says ``Waiting for tmux session `1` to respond (regarding ...), before handing off the next request to tmux session `2` (regarding ...)``. Multiple simultaneous waits must render as multiple rows and clear independently as each wait finishes or times out.
- YO!agent orchestration regressions are pinned in `tests/test_yoagent_orchestration.py`: send-by-number, wait-then-send, sequential dependent asks, prompt/selector answering, and no-refusal direct sends must be repeatable without a live manual tmux run.
- Any new Preference row must be represented in the server defaults, sanitization/clamping rules, frontend `preferenceSections()`, locale catalogs, settings tests, and live-apply runtime if it affects visible UI.

## Keyboard Shortcuts Contract

- The app modifier is platform-specific: Mac uses `Cmd`; PC/Linux uses `Ctrl`. Shortcut labels must say the concrete platform modifier and must not display combined `Ctrl/Cmd` text.
- Mac app shortcuts use `Cmd` so they can be captured even when a terminal is focused; Mac `Ctrl` stays available for terminal/tmux input. PC/Linux app shortcuts use `Ctrl`.
- Find is concrete rather than a combined label: macOS uses `Cmd-F`; PC/Linux uses `Ctrl-F`. Focused tabs with registered search UI claim it, reveal/focus the existing field, and prevent native browser Find; all other tabs leave native Find unchanged.
- Global shortcuts are captured before focused controls swallow them, but text inputs, rename fields, editors, and terminal input must keep their normal typing shortcuts unless a shortcut is explicitly platform-owned by the app.
- Implemented app shortcuts: `Mod+P` opens unified search with file-first ranking, `Shift+Mod+P` opens the same search with pane/command-first ranking, `Mod+K` opens YO!share, `Mod+B` hides Finder/Differ/Tabber or restores `side:left` on wide layouts, `Mod+Alt+B` opens YO!agent in a right Generic Pane, `Mod+,` opens Preferences, `?` opens Keyboard Shortcuts and Legends, `Esc` closes overlays, and `Mod+W` closes the active closable tab.
- Help -> Keyboard Shortcuts and Legends must document implemented global/context shortcuts plus the color, status, icon, and YO button meanings used by tabs, Info Bars, YO!info, and menus. On wide viewports it should use compact three- or four-column packing with balanced section stacks so users can scan it without scrolling. YO buttons are square rounded-control badges, not circular pills. Every title, section, action, color/icon description, and YO sample glyph in an already-open Help overlay must update together on a runtime language change through the shared locale catalog; the overlay shell title and its rebuilt body cannot retain different locales.
- Close-tab fallback: outside text-editing contexts, `Mod+Backspace` / `Mod+Delete` may close the active closable tab. This fallback must not fire inside inputs, rename fields, editors, or terminal text entry.
- Editor shortcuts: `Mod+S` saves, `Mod+F` finds, `Mod+H` replaces, `Mod+G` goes to line, `Mod+/` toggles comments, `Tab` / `Shift+Tab` indent and outdent, `Mod+Z` / `Shift+Mod+Z` undo and redo, and `Mod+Alt+[` / `Mod+Alt+]` move editor navigation back and forward.
- Diff editor chunk shortcuts reuse editor undo/redo: `Mod+Z` undoes the selected chunk action and `Shift+Mod+Z` redoes it when the diff editor owns the focus.
- Finder/Differ keyboard behavior follows Finder-style list navigation: Arrow Up/Down moves selection, Shift+Arrow extends selection, Home/End jump, Shift+Home/End extend to edge, Arrow Right expands or steps into children, Arrow Left collapses or steps to parent, Space toggles quick preview, type-ahead selects by prefix, `Mod+A` selects all, `Mod+O` and `Mod+Down` open, `Mod+Up` opens the enclosing folder, and Return starts rename.
- Finder/Differ delete is platform-specific: Mac uses `Cmd+Delete` or `Cmd+Backspace`; PC/Linux uses plain `Delete`. It must be scoped to Finder/Differ and must not run from text inputs or rename controls.
- Terminal copy handling must preserve normal terminal input: `Cmd+C` on Mac or `Ctrl+C` on PC/Linux copies xterm selection when present; without a browser/xterm selection, the tmux copy bridge may request the tmux copy-mode selection. Claude and other TUIs can own the visible highlighted block and expose it only through OSC 52; terminal right-click `Copy` must prefer the selection captured before xterm clears it, treat an explicitly captured empty selection as authoritative, and then fall back to the recent OSC 52 app clipboard payload instead of re-reading a tiny under-cursor browser/xterm fragment. After a terminal menu action consumes selected text, stale browser/xterm selection must be cleared. Explicit `Copy tmux selection` must not leave tmux copy-mode rows visibly selected after the clipboard payload is captured.
- Terminal URL links must stitch hard-wrapped TUI URLs into one clickable link even when the TUI emits each visual row as a separate non-`isWrapped` line. Hanging-indent continuations and zero-indent width-1 continuations are allowed only when the previous row ends in an unterminated URL token at the terminal edge; a following fresh `https://`, `file://`, or `www.` row stays a separate link.
- Terminal file path references that resolve to an existing file through the same context as the right-click/open path action show a subtle persistent underline in the terminal. Missing/unresolved path-looking text stays unmarked. Hovering a resolved file reference keeps the normal xterm link handling but uses only a slight underline contrast increase so the hovered target remains visible without a bright bar. Output, redraw, resize, and scroll events coalesce one cached visible-row repaint so underlines disappear as soon as their text leaves the viewport. Passive path resolution runs at most once every 1.7 seconds and only for an active pane tab whose xterm and browser document are visible; switching away cancels pending work and clears its underline overlay. Resolved targets, misses, and in-flight resolutions share the bounded 512-entry frontend LRU, while explicit context-menu/open actions bypass passive cached misses.
- Tab drag, pane split, and root-edge layout changes are currently pointer-driven. Keyboard equivalents for moving tabs/panes are not implemented and must not be advertised as available until they exist and are tested.
