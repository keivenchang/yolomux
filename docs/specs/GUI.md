# YOLOmux GUI Behavior Spec

This document is the working GUI contract for pane, tab, Finder/Differ, editor, preview, Preferences, keyboard shortcuts, drag/drop, and YO!share live-mirror behavior. It is meant to drive future implementation and tests. If a UI change conflicts with this file, update the spec and the tests in the same change.

Section order follows the reading path: a capabilities overview and glossary, then layout mechanics (panes, tabs, chrome, drop previews), then the special panes (Finder/Differ, Tabber), then editor, search/menus, and YO!share, then the Preferences and Keyboard contracts, and finally test coverage and the audit backlog.

## Popular IDE Parity Checklist

- [x] Activity/workbench panes can be arranged as tabs inside split panes.
- [x] Tabs can be reordered within the same pane by pointer drag.
- [x] Tabs can be moved from one pane to another pane's tab strip.
- [x] Tabs can be dropped on pane edges to split the target pane.
- [x] Tabs can be dropped on root edges to create full-span panes beside the current layout.
- [x] Tab insertion preview appears between tabs, not over the target tab.
- [x] Active tabs have rounded top corners and a distinct active background.
- [x] Inactive tabs inherit the tab-container background instead of using a disconnected color.
- [x] Pinned tabs stay at the front of a pane, show a pin icon, are protected from LRU tab eviction, and can be toggled from the tab context menu or the `Mod+K` then `Enter` shortcut.
- [x] Tab hover shows a detail popover with session/file state.
- [x] Tab context menu exposes tab-level actions before destructive session actions.
- [x] Open editors support edit, preview, split, and diff-capable modes based on file capability.
- [x] Markdown task checkboxes in Preview are interactive and update the source document.
- [x] Editor back/forward buttons navigate the visited-file stack.
- [x] Search results can render image/file references as `[Image #N] '/absolute/path'`.
- [x] File quick-open ranks basename matches ahead of path-only matches.
- [x] Typed Quick Search results keep the first visible screen mixed across relevant result types. Cmd-P can still rank files first and Shift-Cmd-P can still rank panes/commands first, but a broad query such as `agent` must not fill the whole first screen with only indexed file hits when matching tabs, commands, or settings exist.
- [x] Newly opened tabs (files, Preferences, YO!info/YO!agent, and other non-tmux tab types) open in the pane that owns the currently active tab; Finder/Differ stays reserved and tmux sessions keep their placement rules. DONE 2026-06-11: generalized from the former quick-open-only targeting rule.
- [x] Finder/Differ rows support selection, multi-select, rename/delete, context menu, date/sort controls, and image preview.
- [x] Preferences exposes searchable, collapsible sections with reset controls and live-applied settings.
- [x] Keyboard shortcut labels use the concrete platform modifier: Mac uses `Cmd`; PC/Linux uses `Ctrl`.
- [x] Global app shortcuts are captured before focused panes swallow them when the shortcut is platform-owned.
- [x] Code editor shortcuts include save, find, replace, go-to-line, comment toggle, indent/outdent, undo/redo, and editor back/forward.
- [x] Inline git blame can annotate the current caret line, with an option to annotate all lines.
- [x] Editor themes include dark/light mode switching, separate dark/light scheme defaults, caret style, and caret color.
- [ ] Partial: pinned-tab behavior is per browser profile through local storage, not yet a shareable URL or server-side preference.
- [ ] Partial: file tabs and tmux tabs share pinning, movement, and pane placement; Finder/Differ remains a reserved special pane.
- [ ] Partial: preview/edit interactions are implemented for Markdown task checkboxes, not for every possible rendered widget.
- [ ] Keyboard-only tab reordering and pane splitting are not implemented.
- [ ] Multi-root project workspaces are represented through Finder roots and companion directories, not a full workspace model.
- [ ] Extension/plugin APIs for editor behavior are not implemented.
- [ ] Integrated source-control side panels are partial: Differ and inline blame exist, but a full branch/commit workbench is not implemented.
- [ ] Problems/debug/test runner panels are not implemented as first-class workbench panels.
- [ ] Terminal/profile management is tmux-session based, not a full shell-profile workbench.

## Terms

- Pane: a visible browser layout region that contains one tab strip and one active tab body.
- Tab: a layout item inside a pane. Current tab types include tmux sessions, Finder/Differ, file editors and image viewers, Preferences, and YO!info/YO!agent.
- Finder/Differ: one special file pane. Finder mode shows the file tree. Differ mode shows changed files and diff controls. Tabber mode shows a live tree of open tabs, tmux windows, and touched repo roots, sorted by recent activity. Legacy `changes` / `__changes__` URLs resolve to Finder in Differ mode.
- Root edge: the outside edge of the pane layout. Dropping there creates a full-span pane beside the existing layout.
- Pane edge: the edge of one target pane. Dropping there splits only that target pane.
- Cross-gutter: the separator between two sibling panes. Dropping there creates a full-span split at that sibling boundary.

## Layout Model

- Pane layout is percentage-based and reload-safe. `layout` and `tabs` URL parameters are the shareable source of truth.
- A pane split must not advertise a preview if the target rect cannot fit both resulting panes. Left/right splits require enough width for the current target pane plus the incoming tab. Top/bottom splits require enough height for both panes.
- A middle drop must not advertise a preview if the target pane cannot display the incoming tab at its minimum width and height.
- Pane minimum sizes come from shared CSS/JS sizing tokens and tab-type minimums. Do not hard-code a one-window pixel capacity in a local drag handler.
- A local pane-edge drop splits only the target pane. It must not flatten same-axis siblings into equal thirds. Example: dragging into the right half of `1/2 | 1/2` should be able to produce `1/2 | 1/4 | 1/4`.
- A root-edge drop creates a full-span pane beside the entire current content area. If panes are stacked top-to-bottom, dragging to the left or right root edge creates a full-height leftmost or rightmost pane.
- Root top/bottom drops with a docked Finder/Differ column split only the non-Finder content area, so the preview and resulting pane do not cover the reserved Finder/Differ column.
- Root drops over the Finder/Differ column are reserved and must not show a preview.
- Cross-gutter drops preserve the sibling boundary and create a full-span pane at that boundary.
- Closing or moving tabs must preserve user-chosen split percentages where possible. Empty placeholder panes are allowed when they preserve a meaningful user split.
- The app layout menu exposes Single pane, Split, and Grid. The separate `tmux_wall.py` companion is not an app layout mode.
- Resizing the Finder/Differ sash changes the root Finder/Differ percentage and preserves the nested content split percentages. Example: in `Finder | Pane1 | Pane2`, dragging the Finder sash changes the available content width while Pane1 and Pane2 keep their relative ratio.
- The pane-header detail-toggle button controls the pane detail strip only. It must be labeled as `show details` or `hide details` based on state and must not reuse the YO!info pane label.

## Tab Strip Behavior

- Clicking a tab activates it in its current pane.
- Dragging a tab within the same tab strip reorders it. The preview must appear between tabs, not on top of the target tab.
- A same-strip reorder must commit on a slow, deliberate drag, not only on a quick flick. Drag-and-drop is pointer-based (`dndStrategy: 'pointer'`), so the dragged tab smooth-reorders under the cursor; the drop-target tab must be resolved from the pointer x-coordinate with the dragged tab excluded (nearest remaining tab when the pointer sits over the dragged tab's own gap), never by hit-testing whatever element is under the pointer. Hit-testing the element under the pointer resolves to the dragged tab itself, makes the drop look like a `source == target` no-op, and silently swallows the reorder — only a flick that outran the smooth-reorder transform would land, which reads to the user as "drag sometimes does nothing".
- In a two-tab strip the dragged tab fully covers the drop point. The reorder must still commit, and a secondary pinned pointer-reorder fallback must stand down once the primary drop pipeline has handled the gesture, so it cannot re-run and mirror-swap the order back (which makes the completed drag look undone).
- Dragging a tab to another tab strip moves it to that pane at the indicated insertion position.
- Pinned tabs stay in their current pane. A pinned tab can reorder within the pinned partition of its own pane, but it cannot be moved to another pane or split into a new pane.
- Invalid pinned-tab affordances must not take ownership of drag state. They must be derived from the current Dockview preview/target geometry and must clear immediately when the target changes or the drag ends; do not cache an invalid target and replay or repair the drop after release. Invalid pinned/non-pinned partition targets, including pinned tab-strip drops into another pane, show the normal between-tabs dashed preview in the danger color.
- Dragging a tab to another pane edge splits that target pane only.
- Dragging a tab to a root edge creates a full-span pane beside the current layout.
- The tab insertion preview is a visible dashed box between tabs and uses the same configurable color as the pane separator.
- The active tab keeps rounded top corners. Inactive tabs use the same background as the tab container.
- The active pane's tab container is slightly brighter than inactive panes. Inactive tabs inside the active pane use that same active-pane tab-container background.
- Tab hover details must show the session/file information popover. Dockview tab popovers must not be clipped by the Dockview tab scroller.
- Header actions such as back/forward, minimize, close, and add must stay on the first header row when there is room.
- The pane info (detail) row reads `path · dirty-count | tmux window buttons | ×` with NO separate agent badge — terminal agent identity is carried by the window buttons (an agent window's name falls back to its process label, so it reads `N:claude` / `N:codex`). The agent marker must never appear in the first tab-title row.
- Tmux session detail rows show a window bar with one button per tmux window, immediately before the detail-row close control. Buttons use compact `index:name` labels, keep duplicate-name suffixes when needed, never show process IDs in the visible button text, switch directly to that tmux window on click, hide with collapsed details, and fall back to number-only labels when the name bar would be too crowded. The ACTIVE window's button reads as a pressed toggle styled through the shared `--active-control-*` tokens (so it follows the user's Active color preference and flips with the theme); inactive buttons keep the plain button look.
- Many Dockview tabs must keep the configured Preferences tab width when the pane has room for one tab at that width. When there are too many tabs for the pane width, the tab strip wraps to additional rows, the Dockview header grows to contain every row, and content/toolbars start below the complete header. Only the first row reserves horizontal space for the right-side action cluster; wrapped rows use the full pane width, including the space to the right of the first-row actions. Tabs must not be squeezed by tab count, scroll horizontally under hidden scrollbars, paint under the right-side action cluster on row 1, or overlap the editor toolbar/details row.

## Pane Chrome And Resize

- Pane separators are skinny at rest and use the shared separator token. Hover and active hit targets may be wider, but the visible rest line stays skinny.
- Separator hover thickness is at least 5px.
- Dashed tab/file/root previews use the same configurable separator color.
- Pane spacing and active pane rings must not cover terminal/editor content. Content surfaces resize inside the ring/spacing.
- Dockview sash hit targets are transparent at rest so only the shared separator line is visible.
- Active/typing-ready panes feed the same active-ring color into Dockview and legacy pane chrome.
- The active ring is a state indicator, not layout capacity. Changing pane spacing must not shift terminal text under the ring.

## Visual Preview Contract

- Valid drops show exactly one preview owner: root preview on the grid, pane preview on the target group/pane, or tab insertion preview between tabs.
- Invalid drops show no dashed preview and must suppress any native Dockview overlay for that invalid target, except pinned/non-pinned tab partition violations which reuse the native tab insertion preview geometry in the danger color.
- Finder/Differ reserved areas must not show a preview except the allowed bottom split case.
- Root top/bottom previews with Finder/Differ docked must start at the non-Finder content column and never cover the Finder/Differ column.
- Preview geometry must be measured from DOM rects, not guessed from a fixed browser size.
- Preview color follows the pane separator preference. Tests should compare computed preview border color with the separator token.

## Finder/Differ Pane Rules

- Finder/Differ is a real pane in the layout, not an overlay.
- Finder/Differ/Tabber are three modes of the same reserved pane. Dockview adoption, Tabber refresh, Differ refresh, and Finder refresh must preserve that pane and its current tree position unless the user explicitly hides it with the close button, File -> Finder toggle, or the Finder shortcut.
- Finder/Differ must self-heal when a URL parse, Dockview adoption, reconnect, wake, hidden-tab measurement, or resize path produces a layout that lost the reserved pane without an explicit user close. Explicit close intent is per browser tab and suppresses self-heal until the user restores Finder/Differ.
- Dockview must not adopt a layout snapshot while its host is hidden or measured at zero area. A visible small host still lays out at its real size; only serialized snapshots use functional fallback dimensions so a sleep/hidden-tab measurement cannot become the persisted layout.
- Finder/Differ is reserved as a target. Nothing can be dropped into its center, left edge, right edge, or top edge.
- The only allowed drop onto Finder/Differ is the bottom edge, and only when Finder/Differ is large enough for the existing Finder/Differ pane and the incoming tab after the split.
- Finder/Differ itself is not draggable as a layout tab. Dragging Finder/Differ must not advertise a pane split, root split, or gutter split.
- A file dragged from Finder or Differ into a normal pane opens in that pane. Dropping near a normal pane edge opens it in a new split and shows the same dashed preview used for tab drops.
- A file dragged from Finder or Differ onto Finder/Differ follows the reserved-target rule: no preview except the allowed bottom split when there is enough room.
- Directory drags into terminal content keep the terminal path-insertion affordance. They may show a normal-pane split preview when the target is not Finder/Differ and the pane has enough room.
- File or directory drops into terminal content route through one shared `DROP_ACTIONS` registry. Agent panes may insert the path first and then show a transient non-modal suggestion overlay on `1` through `9` (top-row or numpad); plain shell panes show explicit read-only shell commands (`file`, `wc`, `tail -F`, `jq`, `column -t`, directory size scans) instead of inserting a stray bare path; server actions can show bounded file info, preview, log scan, data-stats/chart, or OCR results in a result panel. External/upload drops and pasted image uploads use the same action registry after the upload path is known; pasted image uploads must still insert the `[Image #N] '/path'` reference immediately. Escape, click-away, normal typing, or timeout dismisses the overlay without running anything.
- File-drop actions support multi-file prompt actions, remember the last chosen action for categories without an explicit order, and include user-defined custom actions from Upload Preferences. Image paste/drop ordering is controlled by `uploads.image_action_order`; that Preference is authoritative, stores prompt actions as `Popup label: ; prompt text inserted after the image path`, accepts non-AI rows as the popup label or a short special action such as `info`, and is capped at 9 rows because the popup assigns shortcut keys `1` through `n`. Image popups must only show configured image-order entries that apply to the current pane, and their visible text must match the configured Preference row verbatim except `info`, which resolves to the runnable `Server: file info` label. The pasted image path/reference is inserted before the popup, so `Insert path` is not part of the image action order. The default image order is agent OCR, error diagnosis, image description, then `info`; stale image-order rows for server OCR and file type are dropped. Custom action lines use `Label | prompt text or shell:command | optional comma-separated categories` and support `{path}`, `{qpath}`, `{paths}`, `{qpaths}`, `{name}`, `{count}`, and `{category}` placeholders. Read-only shell autorun is allowed only when the `uploads.suggestion_autorun` Preference is on; agent prompts and server result-panel actions never auto-enter terminal text.
- Finder and Differ rows share selection, multi-select, context menu, date columns, status columns, image preview, rename/delete, and keyboard behavior through shared helpers.
- Finder and Differ changed directory/path rows render metadata in one shared order: path/name, inherited agent marker, diff counts (`+N -N`), changed-file count (`N file changed` / `N files changed`), git status, then date. The diff column is the first right-side metadata column when it exists; the changed-file count follows it and must not appear before the diff counts.
- Differ file rows open the diff on a normal single left-click in the active non-Finder editor pane; modifier clicks remain selection-only for multi-select. Normal row opens reuse the Differ preview editor tab and start that editor item with collapsed unchanged context without changing the global/default diff expansion setting. Transcript-touched `T` rows still request the Diff editor first when the file is tracked and has useful git history; only a proven unavailable/non-renderable diff falls back to Edit, and that fallback must show a `diff unavailable: ...` status. The row context menu lists Open actions first in this exact order: `Open in a Differ`, `Open in a new Differ`, `Open in a new Editor`. When the same physical file is already open, those actions focus that editor and switch modes instead of creating a duplicate editable tab.
- Background file reloads must preserve each same-file editor tab's CodeMirror cursor and scroll. If the server push/watch path replaces shared file content for `/path`, every visible editor item for `/path` must capture its own selection/scroll before the shared state swap and restore it after the render so duplicate editor tabs do not jump to the top. Background reload must also defer replacement while any editor for that path has focus, or immediately after the file was saved/returned to clean, and surface the normal changed-on-disk status instead of replacing the live document under the cursor.
- File refresh and inspection failures must preserve the backend/API reason and HTTP status before falling back to a generic `Cannot inspect <path>` message. A valid inspectable local file must not be marked `file state unknown`; if the UI shows that state for an existing file, the next diagnosis starts with the actual `/api/fs/list` or `/api/fs/info` error shape, allowed-root validation, and stale absolute-path/worktree state.
- Editor scroll and selection must survive a tab switch away and back (for example `2026.md` -> Preferences -> `2026.md`); switching back must not snap the editor to the top of file. The CodeMirror scroller captures its own view state on scroll through one shared scheduler, so "scroll, then switch away" persists state regardless of tab-event order. A manual Dockview tab activation captures the outgoing pane's editor view state on `pointerdown`/`mousedown` before Dockview switches panels, because manual Dockview tab clicks do not route through `activatePaneTab()`. A pooled, disconnected, or zero-height editor pane must never overwrite an existing saved scroll with `scrollTop: 0`, and restore must reapply the saved scroll after CodeMirror focus and across layout/animation frames. This tab-switch contract is distinct from the background-reload contract above.
- Uploaded files in Differ stay under their actual repo section and repo-relative path, usually as a `.uploads` directory. Differ must not pull them into a synthetic top-level `Uploaded files` group. `.uploads` directories are collapsed by default so screenshots and pasted images do not dominate the first view.
- Image file rows in Finder, Explorer, Differ, and Tabber path contexts expose `Copy image` through the shared file context menu. When browser image clipboard writes are available, it copies the image bytes; when unavailable, it falls back to copying the absolute path with a clear status message.
- Finder, Differ, and Tabber share one top toolbar row so they read as modes of the same pane. The `Finder / Differ / Tabber` mode switcher must look like pane tabs, but it is not a draggable/reorderable tab strip; each mode tab uses exactly `3px` inline label padding and no artificial minimum width so the words sit tightly together. The `Session ...` selector appears immediately to the right of the `Tabber` mode tab in Finder and Differ, while Tabber hides session-specific controls.
- The Finder/Differ/Tabber top row must not include a duplicate Differ collapse/expand-all button next to the close button; tree expand/collapse controls live in the normal Finder/Differ toolbar row below.
- Finder's path row is exactly `Sync button | absolute path input | copy-path button`. The path may be synced or manually typed, but the Sync control must not be duplicated on another line.
- Finder's file-action row starts with create-file/create-folder controls, then puts the `.*` hidden-file toggle before the `A-Z` sort selector. The `~`, `/*`, and `/tmp` quick-root buttons are not visible in the Finder pane chrome because they consume too much space.
- Finder mode and Differ mode must keep the same toolbar alignment rules, date-mode vocabulary (`None`, `Date`, `Ago`), sort vocabulary (`A-Z`, `Z-A`, `new`, `old`), and readable row metadata. The trailing tree controls are ordered `Date | Expand all | Collapse all | Reload` on both Finder and Differ, and Expand all/Collapse all use compact skinny in/out arrow toolbar icons rather than square text glyphs.
- When Finder or Differ auto-switches to a session whose changed-file payload is being fetched and no cached payload is available, it must show an inline loading state with subtle movement, not `not loaded`. The current standard is the small spinning YO marker followed by loading text and the session label. In Finder Sync mode, an explicit session change must immediately plan the root from the cached changed-file payload or the latest tmux transcript metadata, then refresh changed-files in the background; it must not leave the path on the prior session while waiting for `/api/session-files`. If a loaded cached payload exists for that session and comparison, Finder/Differ must apply it immediately, use it for Sync root planning, and refresh in the background instead of replacing it with the loading state.
- Finder Sync root opens caused by an explicit session/tab click are user-owned filesystem reads, not background refreshes. They must bypass client-push background fetch suppression, immediately update the path field to the planned target root, and show the shared `searching...` row while `/api/fs/list` is pending. Otherwise a slow uncached session root such as `yolomux.dev2` looks like Sync did nothing.
- Finder Sync has one explicit target at a time: the last clicked/typed session or explicitly opened file. When that target changes, any in-flight sync for a different target is stale and must be invalidated or dropped before it can set the visible root, remembered sync target, highlight state, or applied-plan key. A same-root short-circuit is not enough: if session 5 reactivates while a session 6 root fetch is still in flight and the path field already shows session 5's root, the session 5 sync must still cancel/guard the session 6 transaction so a late session 6 response cannot retarget Finder back to the wrong worktree.
- Differ shows both current git changes and recent transcript-touched repo paths for the selected session. A repo must remain visible when the session modified files there and then committed or cleaned the working tree; those rows use transcript attribution instead of pretending there is a current dirty diff.
- Finder Sync mode `Expand all` is bounded to the sync plan: expand affected repo/directories such as the active `yolomux.dev2` path and do not recursively crawl every directory under a broad home root. Fixed-root Finder mode may still recursively expand the current root.
- Differ's target session is committed only by explicit user intent: clicking/selecting a pane or typing/pasting/inserting input into a terminal. Hover auto-focus, passive xterm focus, pointerenter pane selection, background refresh, SSE updates, and tab-hover popovers must not switch the Differ session or fetch another session's changed-files payload.
- The Finder path field shows its invalid (red) state only on a real directory-open failure, scoped to that exact path. A deferred or suppressed background refresh — the SSE push channel will supply the listing, a push refresh is mid-flight, or a fresh cache hit — is not a failure and must never flag the field, and a recorded error for one path must not taint another. (Otherwise the path flashed red for ~one push cycle whenever an idle background refresh was skipped.)
- Typing a path into Finder and pressing Enter is an explicit manual-root action. It must immediately leave Sync mode, clear any pending Sync root open, update the visible path to the typed target, clear stale tree rows, and show a passive `searching...` row using the shared moving-ellipsis helper until the requested directory entries are fetched and rendered. A slow or large directory request must not bounce the path back to the previously synced session root. Any already-started session-sync root fetch that resolves after the manual open is stale and must be dropped; it must not flash that session path before the manual request resolves. Broad directories are guarded by a bounded directory-entry listing so paths like `/tmp` can open without forcing YOLOmux to stat and render an unbounded number of children.

## Reconnect And Wake Rules

- Client-events reconnect must backfill live auto-approve status so YO markers reflect the current working/idle state immediately after the stream is ready again. Page visibility returning to visible and browser `online` events must schedule the shared refresh path, and a narrow auto-status fallback poll may run only while the client-events stream is disconnected.

## Tabber Pane Rules

- The Tabber is the third mode of the one shared file-pane mode control (`Finder / Differ / Tabber`); like Differ it takes over the pane (the file tree is hidden, the changes panel fills the pane). Legacy `files`/`diff` behavior is unchanged. A newly restored/default file pane opens in Finder mode; Tabber is an explicit mode selection, not the startup default for the left file pane.
- Tabber shows a live tree of everything open and defaults to fully expanded. Level 0 = tmux sessions (sorted first), then the other open tabs (Preferences, YO!info/YO!agent, file editors) as leaf rows at the end. Session rows are collapsible like other directory rows: clicking only the disclosure arrow toggles collapse/expand, while clicking anywhere else on the green `N <description>` session line opens/activates that tmux tab. Level 1 = the session's tmux windows (`index:process (pid=N)`, e.g. `0:claude (pid=12345)`, tinted by agent); Claude/Codex windows show the same shared agent icon used by the tmux menu after the process name and before the pid, scaled to the row font, and the session's current tmux window is marked with a trailing `●`. Under a `claude`/`codex` window, Level 2 = absolute touched repo roots from `/api/session-files`, lazily fetched and cached per session, with a folder icon and the git branch on the matching repo row. The `/api/session-files` cache is separate from the `/api/activity` Tabber activity cache: it keeps a short in-process entry, persists the same key under the YOLOmux state directory for reuse by parallel servers, and uses a per-key cross-process lock so two servers do not rebuild the same cold touched-path payload at once. These touched-path rows are file-kind leaves and use normal text color; Tabber must not list every individual file, nested repo subdirectory, `/tmp` artifact, or `.claude/projects/...` transcript path under them. While the first touched-path fetch is in flight and no usable memory/disk cache exists, the agent window shows a moving `Fetching paths...` row. Non-agent windows (e.g. `bash (pid=54321)`) are leaf rows.
- Startup behavior: immediately after launch transcript metadata is available, Tabber should already be able to render the Level 0/Level 1 session, process, PID, active-window, and Claude/Codex icon rows. This first useful paint must not wait for `/api/session-files`; touched repo/path hydration is background/lazy and may fill in later under the already-visible agent window.
- Toolbar behavior: Tabber keeps only the shared mode switcher and pane frame controls. Finder/Differ session controls, Finder-only controls (`.*`, Sync/path controls, create file, create folder, Finder sort/date/reload controls), and Differ-only controls (Differ expand/collapse) are hidden because Tabber is an inventory/navigation view, not a filesystem editing or changed-files view.
- Rows render through the SHARED Finder/Differ row pipeline (`renderTreeChildren` -> `updateFileTreeRow` -> `updateFileTreeRowContents`) via a `mode: 'tabber'` render option whose per-row display values are precomputed as data — never a bespoke per-pane row builder. Selection, the icon column, and the shared date modes (`None` / `Date` / `Ago`) all apply unchanged. Tabber uses restrained level colors: level 0 session/other-pane rows share one accent, level 1 window/process rows share one accent, and level 2 paths stay normal foreground text. Purple remains reserved for merged PR status. The collapse state is a persisted COLLAPSED set (default expanded) keyed by a stable id-based node path (`s_<id>/w_<i>/r_<n>`), so it survives reloads and a session add/remove never shifts another node's open/closed state. Clicking a row's disclosure icon toggles that node; the toolbar `Expand all` / `Collapse all` apply to the whole Tabber tree.
- Time column + sort: Claude/Codex window rows use the backend agent list from `/api/activity`, whose recency comes from the agent transcript mtime/newest timestamp; if the transcript shows an active turn, the row says `running`. Agent-window user-input heartbeats and touched repo/path child mtimes must not affect that window's time or sort order. Non-agent window rows use the activity-ledger recency (`max(last_user_input_ts, last_agent_active_ts, last_output_ts)`) and may fall back to the session ledger when the heartbeat cannot be attributed to a window. A parent row inherits the most-recent time among its children, except that an agent window contributes only its own transcript recency to the session parent. Touched repo-root rows use the newest actual file mtime among files folded into that repo root, falling back to transcript time only when the touched file no longer exists. The tree honors the active Finder sort control for top-level sessions and touched-path rows — `A-Z`/`Z-A` sort by the human label, `new`/`old` by the recency time — and re-sorts on the normal poll without collapsing the tree (collapse state is keyed by the stable path, not display order). The immediate window/process rows under each session are the exception: they always stay in tmux window-index order (`0`, `1`, `2`, ...), regardless of recency or the active tree sort. The server owns the `/api/activity` cache: it builds the first snapshot at startup, runs a dedicated background warmer on the `performance.tabber_activity_refresh_ms` cadence, and serves the latest cached snapshot immediately while any stale refresh runs; the default cadence is 15 seconds and Preferences shows it in Performance. Opening Tabber may read `/api/activity`, but that request must read the server snapshot rather than rebuild Tabber activity on click. YO!agent Recent Agents uses the same server-cached agent list instead of building a separate browser-side list. The `/api/session-files` touched-path data is fetched lazily once per agent session, stored in memory plus shared disk cache, and bypassed/replaced on explicit forced refresh; known repo roots are the display unit, so descendant touched directories fold into a top-level repo such as `/home/keivenc/ai-config` instead of rendering redundant child rows like `/home/keivenc/ai-config/claude/skills/...`.
- Row actions: clicking any non-arrow part of a session row opens/activates that tab and records normal tab navigation so Back returns to the previously active tab; clicking the session disclosure arrow toggles collapse/expand; pressing a bare numeric key `1` through `9` while Tabber owns focus opens the matching visible tmux session; clicking a non-session tab row opens/activates that tab; clicking the YO!info row opens the merged info pane on the YO!info sub-tab, not the remembered YO!agent sub-tab; clicking a window row opens the tab and switches to that tmux window (shared `tmuxWindow({windowIndex})` path); clicking an absolute path row points the Finder at that path (switching to Finder mode) and also opens the tab + switches the window; right-clicking an absolute path row reuses the shared file context menu for that directory. Hover/passive focus never act.
- YO!agent transcript Details controls are user-owned disclosure state. If the user expands a response Details block, background activity refreshes, conversation refreshes, busy-state rebuilds, locale rerenders, and Markdown rerenders must keep that same message's Details block expanded instead of collapsing it.

## Tab Strip Behavior

- Clicking a tab activates it in its current pane.
- Dragging a tab within the same tab strip reorders it. The preview must appear between tabs, not on top of the target tab.
- A same-strip reorder must commit on a slow, deliberate drag, not only on a quick flick. Drag-and-drop is pointer-based (`dndStrategy: 'pointer'`), so the dragged tab smooth-reorders under the cursor; the drop-target tab must be resolved from the pointer x-coordinate with the dragged tab excluded (nearest remaining tab when the pointer sits over the dragged tab's own gap), never by hit-testing whatever element is under the pointer. Hit-testing the element under the pointer resolves to the dragged tab itself, makes the drop look like a `source == target` no-op, and silently swallows the reorder — only a flick that outran the smooth-reorder transform would land, which reads to the user as "drag sometimes does nothing".
- In a two-tab strip the dragged tab fully covers the drop point. The reorder must still commit, and a secondary pinned pointer-reorder fallback must stand down once the primary drop pipeline has handled the gesture, so it cannot re-run and mirror-swap the order back (which makes the completed drag look undone).
- Dragging a tab to another tab strip moves it to that pane at the indicated insertion position.
- Re-opening an existing file or diff tab activates it in the pane where the user last put that tab. A normal Finder, Differ, Quick Search, or context-menu open must not silently move an existing file editor back to the active, first, largest, or leftmost pane; only an explicit drag/drop or split action may rehome an existing tab.
- Pinned tabs stay in their current pane. A pinned tab can reorder within the pinned partition of its own pane, but it cannot be moved to another pane or split into a new pane.
- Invalid pinned-tab affordances must not take ownership of drag state. They must be derived from the current Dockview preview/target geometry and must clear immediately when the target changes or the drag ends; do not cache an invalid target and replay or repair the drop after release. Invalid pinned/non-pinned partition targets, including pinned tab-strip drops into another pane, show the normal between-tabs dashed preview in the danger color.
- Dragging a tab to another pane edge splits that target pane only.
- Dragging a tab to a root edge creates a full-span pane beside the current layout.
- The tab insertion preview is a visible dashed box between tabs and uses the same configurable color as the pane separator.
- The active tab keeps rounded top corners. Inactive tabs use the same background as the tab container.
- The active pane's tab container is slightly brighter than inactive panes. Inactive tabs inside the active pane use that same active-pane tab-container background.
- Tab hover details must show the session/file information popover. Dockview tab popovers must not be clipped by the Dockview tab scroller.
- Header actions such as back/forward, minimize, close, and add must stay on the first header row when there is room.
- The pane info (detail) row reads `path · dirty-count | tmux window buttons | ×` with NO separate agent badge — terminal agent identity is carried by the window buttons (an agent window's name falls back to its process label, so it reads `N:claude` / `N:codex`). The agent marker must never appear in the first tab-title row.
- Tmux session detail rows show a window bar with one button per tmux window, immediately before the detail-row close control. Buttons use compact `index:name` labels, keep duplicate-name suffixes when needed, never show process IDs in the visible button text, switch directly to that tmux window on click, hide with collapsed details, and fall back to number-only labels when the name bar would be too crowded. The ACTIVE window's button reads as a pressed toggle styled through the shared `--active-control-*` tokens (so it follows the user's Active color preference and flips with the theme); inactive buttons keep the plain button look.
- Many Dockview tabs must keep the configured Preferences tab width when the pane has room for one tab at that width. When there are too many tabs for the pane width, the tab strip wraps to additional rows, the Dockview header grows to contain every row, and content/toolbars start below the complete header. Only the first row reserves horizontal space for the right-side action cluster; wrapped rows use the full pane width, including the space to the right of the first-row actions. Tabs must not be squeezed by tab count, scroll horizontally under hidden scrollbars, paint under the right-side action cluster on row 1, or overlap the editor toolbar/details row.

- Pane separators are skinny at rest and use the shared separator token. Hover and active hit targets may be wider, but the visible rest line stays skinny.
- Separator hover thickness is at least 5px.
- Dashed tab/file/root previews use the same configurable separator color.
- Pane spacing and active pane rings must not cover terminal/editor content. Content surfaces resize inside the ring/spacing.
- Dockview sash hit targets are transparent at rest so only the shared separator line is visible.
- Active/typing-ready panes feed the same active-ring color into Dockview and legacy pane chrome.

## File Editor And Preview Behavior

- Markdown opens in the right mode for the source context. README-style opens may default to Preview; code/source files default to edit/search/diff-capable editor views.
- Editable file editors are unique by backend physical-file identity, not by path string. If `/repo/file.txt`, a symlink path, or a hard-link path resolve to the same `(st_dev, st_ino)` file identity, opening any alias focuses the existing editor and applies the requested edit/diff mode. The dirty buffer, cursor/scroll state, and tab pane stay with the existing editor; same basename, size, or search-result similarity is never enough to fold two editors.
- Markdown Preview task checkboxes rendered from `- [ ]` and `- [x]` are clickable for admin users. Clicking one updates the Markdown source, refreshes preview panes/popouts, updates editor documents, and uses the normal dirty/autosave path.
- Markdown Preview is not read-only when the user interacts with task checkboxes, but unsafe HTML remains blocked.
- Preview capability and rendering kind must come from the shared `PREVIEW_RENDERERS` registry plus its helper layer, not ad hoc extension checks in toolbar, popout, open-file, reload, Finder/Differ, or split-view code. The registry owns extension, MIME, media kind, text-vs-raw, default mode, sandbox policy, recognized-fallback policy, and fallback behavior. Markdown and HTML are text-backed previews, known code/plain-text files use the code-preview fallback, structured data and delimited tables use bounded preview renderers, direct browser-native media files use raw-file preview, misleading-extension binaries can recover through sniffed MIME from `/api/fs/info`, and unsupported binaries show a clear fallback instead of opening a fake text editor.
- Markdown Preview resolves local image URLs relative to the Markdown file's directory and serves them through `/api/fs/raw`; it must not let relative images resolve against the YOLOmux app URL. Safe external `http:`/`https:` images may remain external, but protocol-relative URLs, unsafe protocols, SVG data URLs, and unsafe image data URLs stay blocked by the existing sanitizer policy. Markdown Preview must have a local fallback parser for normal headings, tables, lists, task checkboxes, images, links, code fences, and local URLs with spaces or parentheses when external Markdown/highlight assets fail to load.
- Direct SVG file Preview renders through an image/raw-file path such as `<img src="/api/fs/raw?...">`; direct SVG file contents must not be inserted as trusted app DOM. Mermaid-generated SVG is a separate generated-output path and must be sanitized before display.
- Mermaid Preview is local-vendored, lazy-loaded only when needed, configured with strict security and deterministic IDs, rendered after Markdown sanitization, and followed by a YOLOmux SVG sanitizer before display. A failed Mermaid render keeps the source visible and shows the error in a bounded preview error block.
- The Mermaid palette (node fills, line/edge color, text color, surface background) follows the PREVIEW display mode (Bright / Dark / Vanilla), which is independent of the app theme — not document `--text`. In Bright/Vanilla preview the surface is white, so lines, edge labels, and surface-resting node labels are dark; in Dark preview they are light. Reading the palette from the app `--text` (which tracks the app theme, not the preview mode) is the bug that painted light-gray lines on a white Bright surface. Changing the preview display mode re-renders the diagram with the new palette.
- Every Mermaid node label must be readable against its own node background, in both light and dark mode: the label color is chosen to contrast with the node's actual background SHAPE fill (light text on a dark/transparent default node, dark text on a light `classDef` node), not with an ancestor or label fill. A node with no parseable shape fill is treated as transparent over the preview surface and gets the surface foreground color (which follows the preview mode). Fill detection must tolerate Mermaid's `!important` declarations (e.g. `fill:#fef3c7 !important`). Subgraph/cluster/edge labels that sit on the preview background use the surface foreground color.
- HTML Preview and PDF Preview use sandboxed frames/objects where possible. The explicit same-origin HTML "open with JS" path stays separate from the safe in-pane HTML preview. Audio and video previews use native browser controls backed by `/api/fs/raw`, with no autoplay; browser decode failures fall back to explicit open/download actions.
- JSON, GeoJSON, Excalidraw JSON, JSONL/NDJSON, notebook `.ipynb`, YAML, TOML, XML/Draw.io, INI/properties/env-style files use bounded structured previews. Notebook preview never executes code and does not render stored outputs; it summarizes cells and hides outputs. CSV and TSV previews render bounded row/column tables with truncation notice.
- Direct browser-rendered images include PNG/APNG, JPG/JPEG, GIF, WebP, BMP, ICO, AVIF, and SVG-as-image. TIFF/TIF, HEIC/HEIF, Office files, SQLite, Parquet/Arrow, and archives are recognized fallback formats: the preview says why YOLOmux is not rendering them and offers Open/Download without unpacking, executing, or trusting embedded content.
- Unsupported binaries, recognized fallbacks, and failed media decodes show file path, guessed MIME or format, size when available, and explicit open/download actions; they must not silently render as empty Preview or corrupt editor text.
- Direct image and Mermaid Preview share one zoom/pan surface (`-` / `Fit` / `1:1` / `+`, wheel zoom, drag pan) across full Preview, split Preview, inline Markdown diagrams, and the pop-out. When the preview/image pane panel is itself the zoom surface, its box must come from the pane's absolute layout `inset` (the full pane in preview mode, exactly the right half in split-preview), never from a `width:100%`/`height:100%` rule: an explicit width on an absolutely-positioned pane overrides the split `inset: 0 0 0 50%` `right:0`, so the pane renders full-width offset to 50% and the diagram's right half runs off-screen and is chopped. In split view the diagram at `Fit` must fit inside the visible right half, `1:1` must center it with scroll/pan available for overflow, and the diagram must never extend past the content area's right edge. The zoom surface's resize observer must coalesce to one apply per animation frame so applying zoom (which resizes the observed content and can toggle a scrollbar) does not synchronously re-trigger itself.
- A fit-mode diagram must not visibly jump/resize after it first appears. A file editor pane opens at a transient height and Dockview re-lays-it-out shortly after (and a hover that triggers a relayout does the same), so the zoom surface keeps the diagram hidden (stage `visibility:hidden`, viewport still laid out and measurable) until its viewport size has settled — debounced after the last size change — then reveals it once at the settled fit, instead of fitting against the transient size and re-fitting.
- Useful rendered Preview paths must have concrete sample files under `docs/preview-samples/`. Do not add sample files that only show ordinary text rendering, broken-by-design audio/video decode fallbacks, archive fallbacks, or unsupported placeholders; DOT/PlantUML stay out of the sample set until YOLOmux actually renders those diagram formats.
- The file editor Differ button appears only when blame/diff capability exists for that file. Blame and Differ controls appear or hide as a pair.
- A clean git-tracked file with meaningful history can stay in diff mode so the FROM/TO picker remains usable.
- Preview pop-outs and preview panes must not drive Differ editor scrolling.

## Search, Menus, And Popovers

- Search results that reference images or files should use a Popular IDE-style display when possible: `[Image #N] '/absolute/path'`.
- Cmd-P file quick-open and other newly opened non-tmux tabs preserve active-pane context. A normal file, Preferences, or YO!info/YO!agent open lands in the pane that owns the currently active tab; explicit split-open keeps its split behavior; Finder/Differ remains a reserved pane target.
- In Cmd-P / Quick Search, typing an absolute file path pins that exact file as the Enter target, including image paths. The parent folder may still appear as a secondary row below it, but it must not steal Enter from the exact file path.
- Quick Search and Tabs search should collapse duplicate open-file rows by path and surface open view chips instead of showing duplicate file entries.
- Tab search indexes session labels, branch/PR metadata, Linear IDs, file paths, and tab details.
- Unified search ranking contract: `Mod+P` and `Shift+Mod+P` search the SAME universe (files, panes/tabs, commands, settings) and differ ONLY in ranking prior — `Mod+P` ranks files first, `Shift+Mod+P` ranks panes first. Dominance order within results: (1) per-surface domain prior, (2) anchored prefix on the primary name (typing `6` puts file `6.md` first in `Mod+P`, pane `6 …` first in `Shift+Mod+P`), (3) contiguity (`123` ranks `hello 123` above `1 a 2 b 3 c`), (4) recency (file index mtime; pane recency uses latest agent transcript mtime from the session payload, then summary timestamps as fallback), (5) focused-session repo affinity. Match quality dominates recency; recency dominates repo affinity. For typed blended results, the first visible screen keeps a bounded mix of file and pane matches so async file-index results cannot crowd a strong pane hit offscreen. The scoring and first-screen mix weights are ONE exported data table guarded by table-driven ranking tests, so behavior is tuned by editing weights + expected-order rows, not by rewriting ranking code.
- Unified search shows a compact loading status while backend file search is in flight, even when local pane/command/recent results are already displayed.
- Menus and popovers are global UI state. Hover must not auto-open menus when auto-focus is off unless the user explicitly opened a menu first.
- Background topbar refreshes must not detach a focused topbar control. If the language app-menu button or a future topbar input/select is active, passive topbar renders defer until the control blurs; explicit user-triggered forced renders may still rebuild immediately.
- Popovers use shared timing and ownership so tab, image, menu, and file-preview popovers do not fight each other.
- Custom popovers should replace duplicate native `title` text when the custom popover already provides the same information.

## YO!share (Live Session Sharing)

YO!share lets the host share the live YOLOmux layout to other browsers via a short `/share/<id>#t=<secret>` URL. Read-only viewers observe; write viewers may also type through validated input intents. The guiding rule: the client mirrors the host's visible state and never re-derives geometry or invents transient UI state. The detailed DOM replay architecture spec lives in [`SHARE_MIRRORING.md`](SHARE_MIRRORING.md). See also the File menu entry in Search, Menus, And Popovers and the `share.*` Preferences.

Architecture: read-only and write viewers use host-rendered DOM replay by default. The host is the only renderer for mirrored app state, streams sanitized `#appRoot` keyframes/deltas, and terminal panes stay on the host-sized byte stream because xterm internals do not serialize through DOM mutation. Viewer DOM is inert. Write clients send allowed input intents to the host/server and receive host replay frames for visual feedback. Normal viewer chrome shows replay health (`mirrored`, `resyncing`, `host disconnected`, `viewer behind`) rather than semantic bucket names.

Replay repair is keyframe-based. Any host topology or global-UI change that cannot be trusted as a small delta may publish or trigger a complete epoch-keyed DOM keyframe. Clients apply frames only in sequence for the current epoch; gaps, unknown node ids, digest mismatch, slow-viewer backpressure, and manual debug requests ask the host/server for a new keyframe. Terminal resize remains separate: `terminal-host-resize` resets viewer xterms to host rows/cols before fresh tmux repaint bytes arrive.

### Sharing UI, mirror frame, and viewer rules

- File -> Open File is the first command under File, and File -> YO!share sits immediately after the Finder/YO!info/YO!agent commands and before Preferences. YO!share opens the shared modal chrome, not a separate dialog, and shares the entire current YOLOmux layout: pane tree, split percentages, active/background tabs, Finder/Differ/Tabber state, editor tabs and view mode/theme/preview/split state, Preferences state, app-menu state, and every tmux session present in pane tabs. When the share action is available or sharing is already active, the File menu detail says only `sharing`. The create view field order is Max time to share, Max viewers, Read-only, Protocol; Max time to share is a typable minutes number field, not a dropdown. Defaults come from Preferences `share.ttl_seconds`, `share.max_viewers`, `share.read_only`, and `share.scheme`; `share.max_viewers` defaults to 2, read-only defaults on, protocol defaults to http, unchecking Read-only forces https and disables http with a short hint, and a non-TLS page forces Read-only and http. The manage view lists every active share with a readonly URL input whose copy icon sits immediately to the right of the URL text, then expiry, viewer count, mode, protocol, and per-share Stop control, while keeping the create form available for another share inside a bounded, scrollable modal body. Viewer caps count distinct viewer browsers, not the number of terminal websockets opened for that browser's shared pane layout. While shares are active, the host topbar shows a compact single-share pill (`YO!share: N viewers · M:SS left`) or an aggregate multi-share pill (`YO!share: S shares · N viewers · M:SS left`) that reopens the manage view; read shares use the green/good mode color and write shares use the red/bad mode color. A share-view browser shows a slim banner with host, mode, expiry, pushed status, replay health, and a cover/contain fit toggle; this banner lives outside the mirrored app root and must never shift host layout geometry. Share viewers render sanitized host DOM inside a host-sized `#appRoot` replay frame whose viewport, appearance settings, bundled UI/mono fonts, and terminal placeholders come from the host, then one root transform maps that frame into the viewer window. The default fit is `cover` from `share.view_fit`, cropping the long axis symmetrically; `contain` letterboxes instead. The viewer must not reflow the app against its own viewport, and raw `window.innerWidth`/`window.innerHeight` reads outside the shared viewport owner are forbidden. Host terminal cols/rows stay pinned and viewer resize only changes the mirror transform, never the shared tmux size. Read-only viewers are passive. Write viewers may send validated input intents over HTTPS, but they do not publish semantic layout, Finder, editor, Preferences, popup, or scroll state; host replay frames provide visual feedback.
- YO!share URLs must stay short (`/share/<id>#t=<secret>`); sessions, layout, tabs, and UI state live server-side behind the id. Host-owned popups, menus, context menus, language pickers, modals, and transient messages mirror as normal sanitized DOM inside `#appRoot` through replay. Share-modal URLs and token-bearing fields are marked as share secrets and redacted before DOM keyframes, deltas, debug copy, or server UI frames reach clients. Mirrored modals use the host DOM and host content sizing, with no client-only reflow, forced child height, or extra scrollbar. Active share rows are compact: `New share` and `Active share URLs` sections, max time/viewers/read-only/protocol grouped, URL plus copy in one row, and Stop inline with mode/protocol. The active URL block is the primary share affordance in the modal; it must be visually stronger than metadata rows while keeping exactly one copy target beside the URL input. Only the host can stop or extend active shares; viewers only receive refreshed countdown and viewer-count status. Read-only viewers must preserve browser text selection, copy, context-copy, and harmless navigation defaults while blocking activation and mutation. Editor/diff parity comes from host replay plus share-scoped file reads for every host-visible file editor identity (`file:`, `filediff:`, `filecopy:`, and `image:` tabs). Saves still publish file-version invalidations for open shared files. Pre-auth login/setup language pickers use in-page buttons and hidden inputs rather than native selects so OS popups cannot leak as broken share-view boxes.
- When a share viewer's host-provided expiry reaches zero, the viewer must navigate to the login screen immediately through the shared login redirect path; it must not wait for a websocket close, terminal reconnect loop, or the low-frequency `/api/share` backup poll. The login `next` target includes only the pathname and query, never the `#t=` fragment token.
- The YO!share manage view must show a `Users` section inside each active share row. The section lists one row per distinct connected browser, with connected duration, IP address, and summarized browser type. It updates from pushed share status on join/leave without requiring the host to close and reopen the modal, and it must not expose full user-agent strings or share secrets in the mirrored popup layer. The Users section is an inline subsection of the active share row: row separators are allowed, but it must not draw a second card border, framed table box, or card background inside the already-bordered share row.
- YO!share clients mirror the host's visible UI state, not the client's inferred state. When the host opens, closes, hovers, unhovers, scrolls, changes language, changes theme, changes layout, changes active panes/tabs, toggles tab metadata, switches YO!info/YO!agent, or expands/collapses editor diff context, the client must converge to that exact host state and clear any stale mirrored surface immediately. This includes the YO!share modal, all app menus, submenu hover state, language popovers, tab hover popovers, context menus, transient messages, editor scroll, Finder/Differ scroll, pane split geometry, active/focused pane chrome, tab metadata visibility, YO!info/YO!agent active sub-tab, and popup font sizing. Client-side hover or focus must not be required to clear stale host UI, and read-only replay clients must not synthesize local tab popovers from their own pointer position. Host tab popovers must carry their computed app-space `left`, `top`, measured inline `width`, and measured `height` on the replayed popup node, not only in document-level CSS variables or an `inset` shorthand, so Safari/Chrome viewers under a mirror transform render the same host position and wrapping. Client-side scroll must not persist or snap mirrored panes back to a local default; read-only client scroll input is blocked for mirrored app surfaces except where browser text selection requires a harmless default. Host menu and popover mirroring is an open/close contract as well as an HTML snapshot contract: a closed host popup means no mirrored popup remains on any client. Language changes must update host and clients atomically enough that no pane blanks or loses its current content while theme/locale text is re-rendered. User-facing language choices show human-readable language names/endonyms only; raw locale codes such as `en`, `es`, `ja`, or `zh-Hans` are not visible UI labels except in explicit developer/debug output.
- YO!share replay keyframes must include the host's current scroll targets, compact YO/auto-approval state, theme/language/font readiness, popup/menu/modal DOM, terminal placeholders, and share-visible file/editor identities, not only the pane tree. This is required for late-joining viewers, keyframe repair, Preferences scroll, tmux/Tabs badge parity, and host menu rows that depend on YO state. Live editor/file refresh for share viewers is push-driven through share UI/file-version frames; `/api/share` polling on read-only viewers is only a low-frequency status backup for banner metadata, not a high-frequency editor refresh mechanism. Same-epoch incremental DOM replay updates should keep applying even when a viewer detects a sequence gap, while viewer repair keyframe requests and host repair keyframes are throttled to a five-second floor. Semantic geometry resync stays on a ten-second floor.
- YO!share visual parity is a replay-health contract. Host and viewer must load the bundled `YOLOmux UI` and `YOLOmux Mono` fonts before the first mirrored layout so serialized DOM dimensions stay stable, but viewers do not compute their own wrapped-text or pane geometry to decide whether they are synced. Read-only replay health asserts `domDigest`, epoch/sequence continuity, dropped-frame/keyframe-request counts, redaction policy version, node count, and terminal placeholder health. Legacy semantic geometry/text-wrap digest buckets such as `slots`, `tabStrips`, `editors`, and `textWraps` may remain only for the explicit semantic escape hatch or targeted diagnostics; normal read-only replay must not show bucket names or repair by applying another semantic widget snapshot.
- YO!share debug diagnostics are opt-in with `?shareDebug=1` on a share-view URL or `localStorage.yolomux.shareDebug=1`. Replay debug records one sanitized `share-replay-health` report on `window.yolomuxShareDebug` with DOM digest, epoch, sequence, dropped-frame count, keyframe request count, redaction policy version, node count, terminal placeholder health, fit mode, viewport, `visualViewport`, DPR, user agent, root/stage rects, and mirror transform. The debug banner copy control appears only in debug mode and copies the latest report as JSON. Share tokens and `#t=...` fragments must be redacted before console logging, window exposure, or clipboard copy. Legacy semantic drift reports are allowed only when the semantic escape hatch is active.
- YO!info is part of the same host-owned share state as Finder, Differ, editors, and Preferences. Branch-table row content, sort column/direction, resizable Branch and desc column widths, vertical scroll, and horizontal scroll must publish from the host and apply on read-only clients. A viewer must not rebuild YO!info rows from its own scoped transcript metadata after a host snapshot arrives; it renders the host's rows until the host sends a newer row snapshot. Column widths are part of the mirrored UI state, not browser-local storage on the viewer; applying them on a client must not persist them or echo-publish a client-authored width. A read-only client wheel, scrollbar drag, hover, focus, or other local interaction over YO!info must not publish or persist a client-authored position; it restores to the last host-authored `#info-content` scroll frame.
- Finder/Differ/Tabber share state is host-owned at the model layer, not only at the DOM event layer. The host snapshot owns mode, root mode, root path, target session, expanded paths, selected paths, tree date/sort settings, changed-file collapsed sets, Tabber collapsed set, and per-repo Differ refs. A read-only client's local transcript refresh, `/api/session-files` result, hover/focus state, active-tab sync, or manual open attempt must not retarget the root, collapse/expand the tree, or switch Finder/Differ context. Repeated host frames with the same root must refresh in place when needed and must not reopen the root in a way that briefly clears expansion; only a changed host root should reopen the tree.
- HTTP read-only share parity covers the same visible surfaces as HTTPS share parity. A viewer loaded from an HTTP share URL must render the host's Finder rows without a red path field, pane tab metadata without `no path` or `finding branch...` fallbacks, file editor contents without `File could not be opened` / `TypeError: Load failed`, and Differ/Tabber data without local fetch-error placeholders. These checks must run in Selenium against a real `http://.../share/<id>#t=...` page so token propagation, plaintext redirect exemptions, and browser fetch/WebSocket behavior are exercised together.
- YO!share popups, menus, context menus, language pickers, modals, and transient UI mirror as ordinary sanitized DOM inside `#appRoot`. Popup roots must mount under the app overlay owner so the replay serializer sees them; stale popup resurrection is prevented by replay epoch/sequence handling and keyframe replacement, not a separate popup-layer shell.
- Share-view terminal grids are host-PTY authoritative. A viewer terminal must size from the host's rows/cols only; it must not fall back to `estimateTerminalSize()` from the viewer pane, native viewport, local font metrics, or transient container visibility. A host resize, `terminal-host-resize` frame, or terminal placeholder update that changes rows/cols must resize the viewer xterm, call `reset()` to discard any smeared/refowed buffer rows, and rely on the server-side `tmux refresh-client` repaint to refill a clean grid at the host size. The server must deliver `terminal-host-resize` before requesting the tmux repaint so clients reset before fresh terminal bytes arrive. A viewer that reattaches to an already-running shared upstream must request a bounded `tmux refresh-client` repaint; otherwise layout rebuilds can create a blank xterm that does not receive a full curses screen until manual refresh. If host dimensions are unavailable, the viewer skips resizing rather than inventing local dimensions.
- Dockview panel show/hide is part of the terminal lifecycle. When a share-view host frame removes or minimizes Finder and Dockview remounts tmux panels or activates tmux tabs that were hidden during the topology change, the viewer must ensure the xterm is bound to the currently connected terminal container, reconnect/restart stale bindings, and keep using the host rows/cols.
- Read-only share viewers are visually host-identical inside the mirrored app. They must not add client-only disabled styling, `readonly access` labels, native scrollbars, draggable scrollbar tracks/thumbs, or other read-only chrome inside the mirrored Preferences/Finder/Differ/editor/terminal surfaces. Read-only protection is enforced by the inert replay shell and host-authored scroll frames; host scroll is applied programmatically and retried after keyframe/delta apply so a late DOM rebuild cannot snap a viewer back to a local default. Viewer-only controls such as the share banner, fit toggle, and replay health stay outside the mirrored app root.

### Transport and server architecture

- HTTP read-only share works even when YOLOmux is launched with HTTPS, on the SAME port (no second listener). The server muxes protocols by peeking the connection's first byte (`TLS_FIRST_BYTES`); the peek timeout must stay generous (`tls_peek_timeout_seconds`, currently 2.0s) and must NOT run so short that a remote client's first byte (one RTT away) is misread as a TLS preconnect — a 50ms peek silently broke the plaintext->https redirect for remote browsers. Plaintext requests on a TLS host are SERVED only for share-scoped paths (`/share/<id>`, `/`, `/static/*`, `/ws/share-view`, `/ws/share-ui`, `/api/ping`, `/api/share`, `/api/share-stream`, readonly share data APIs such as `/api/transcripts`, `/api/activity`, `/api/session-files`, `/api/fs/list`, `/api/fs/info`, `/api/fs/index-status`, scoped file APIs such as `/api/fs/read`, `/api/fs/diff`, `/api/fs/raw`, and readonly batch `/api/fs/batch`) AND only when the matching share record has `http_allowed` (`plaintext_share_scope_allowed`); every other plaintext request gets a 308 redirect to https (`redirect_plaintext_to_https_if_needed`). Plaintext share connections are read-only regardless of token mode, and no cookies are set on them. Write mode requires https.
- Terminal mirroring fans out from ONE upstream pty+`tmux attach` per shared session (`share_terminal_upstream`), never one attach per viewer (viewers x panes attaches would melt tmux). The upstream pty is pinned to the host's rows/cols; viewers never resize it. Each output chunk is serialized into one frame once and enqueued to every viewer (`broadcast`). Backpressure is per-viewer: a bounded outbound queue (`SHARE_VIEWER_QUEUE_HIGH_WATER_BYTES`) — on overflow the viewer's backlog is dropped and ONE rate-limited `tmux refresh-client` (`SHARE_REFRESH_CLIENT_MIN_SECONDS`) repaints all lagging viewers from a single screenful; a viewer that overflows repeatedly (`SHARE_VIEWER_OVERFLOW_LIMIT` within `SHARE_VIEWER_OVERFLOW_WINDOW_SECONDS`) is disconnected `too-slow`. One slow consumer must never buffer the server into OOM or stall the fan-out loop. Upstreams with no viewers are reaped (`close_inactive_share_upstreams`).
- Host resize propagation is a terminal repaint contract, not only a layout contract. `record_host_pty_dimensions` updates each matching share upstream, broadcasts `terminal-host-resize`, and only then requests `tmux refresh-client` after `set_pty_size`/SIGWINCH so terminal bytes following the resize redraw a full clean screen after the client has reset to the host rows/cols. Layout commits publish replay keyframes or deltas so terminal placeholders, tab metadata, YO!agent sub-tab state, Finder/Differ state, editor diff expansion, and Preferences state converge after pane rebuilds; semantic `layout`/`ui-state` snapshots are legacy escape-hatch behavior, not the normal share path.
- Viewer accounting is per distinct browser (`viewer_ids` on the record), not per websocket; status (viewer count, expiry) is PUSHED to host pill/manage-modal and viewer banner via `broadcast_share_status` on join/leave/extend, not only polled. The read-only viewer `/api/share` status poll is a backup and must stay low-frequency because normal UI, editor, file-version, scroll, popup, and viewer-count changes arrive through the share WebSocket. The viewer cap refuses the WS upgrade past `max_viewers`. Records are in-memory: a server restart drops all shares (URLs become invalid -> login), which is acceptable and intended. The data model is keyed by token to allow N concurrent shares later (mixed modes/TTLs); v1 may gate to one active share at mint time.
- Host-local surfaces must NOT be mirrored to viewers: the latency meter is the host browser's own RTT and is meaningless on a client; viewer-only chrome (the share banner, fit toggle, replay health status) belongs to the viewer, not the host; the real OS cursor is replaced by the ghost cursor; share secrets/URLs stay redacted. Decision rule for any new mirrored surface: host-owned visible state inside `#appRoot` mirrors through sanitized DOM replay with the client inert; client-local or host-private state is redacted or excluded. When unsure whether a surface belongs in the mirror, register it with the replay/redaction policy and add a replay health or sanitizer test rather than adding another semantic snapshot/apply pair.

## Preferences Contract

- Preferences is a normal tab type and may be moved, activated, minimized, and closed like other non-Finder tabs.
- Preferences must expose a search field, collapsible sections, per-row reset controls, a global reset flow, the settings file path, and a settings age/status indicator.
- Preference changes save through the shared settings API and live-apply when the setting affects visible UI. Appearance, terminal/editor, Finder/Differ, popover timing, notification, and YOLO changes must not require a page reload unless the setting explicitly says so.
- General settings must include language, auto-focus, and startup tips. The legacy default-sessions setting is accepted from old config files but is hidden because running servers default to all discovered tmux sessions unless launched with `--sessions`.
- Appearance settings must include global theme, default layout, UI and Finder font sizes, tab width, max tabs per pane, pane spacing, pane ring opacity, inactive pane opacity, active color, separator color, editor cursor color, YOLO rotation timing, and 12-hour/24-hour clock mode.
- Terminal / Editor settings must include terminal theme, terminal/editor/preview font sizes, terminal scrollback, editor dark/light schemes, editor cursor style, word wrap, line numbers, autosave, autosave delay, and all-lines blame.
- Notifications must include both update controls: the server-version reload prompt (`general.reload_on_update` / `general.reload_on_update_auto`) and the origin/main update checker (`updates.check_enabled`). The reload prompt only reacts when the running server reports a newer app version than the open page booted with; it does not check GitHub. The origin/main checker is the control that shows the admin `Update & restart` card, and it is off by default. `updates.notify_level` controls the minimum semantic version change that produces the origin/main update notification: `major`, `minor`, `patch`, or `none`, default `patch`.
- Finder/Differ settings must include root mode, image open mode, image preview size, quick-access paths, indexed directories, index refresh, companion directories, directory cache timing, and new-entry highlight timing.
- Uploads must include filename template, upload size cap, upload subdirectory, terminal file-action suggestions, read-only shell autorun, and custom file-drop actions.
- Performance must include server SSE file-change polling, server SSE background file-change polling, server SSE directory-change polling, cached Tabber activity refresh, latency refresh, event-log refresh, popover show/hide timing, menu hover timing, tab popover timing, and remote resize debounce. The visible-editor file-change poll checks open editor files in visible panes for fast `files_changed` pushes. The background-editor file-change poll checks hidden/background editor tabs less often so inactive tabs still notice disk edits without making every server tick expensive.
- GitHub must include watched PRs.
- YOLO must include auto-approve interval, rule file path, dry-run mode, and prompt source.
- YO!agent must include backend, invocation mode, Claude/Codex model and effort controls, the background transcript summary interval, system prompt, intro, and format. The invocation row currently exposes only CLI; reserved transports such as API-key mode must stay hidden until they work end to end. Model selects must be wider than compact selects, but only enough to show the longest model label plus padding; they must not stretch across the whole Preferences row.
- YO!agent has Preferences parity. It must answer settings questions from the backend settings catalog, not from scraped Preferences DOM, and include current value, default, valid choices/range, GUI location, and config path when useful. Admin setting writes from chat must route through the same server settings save path as Preferences, publish the normal `settings_changed` event, preserve concurrent unrelated settings edits, and render as a normal YO!agent answer with a compact before/after block. Read-only users may ask explanatory settings/product-state questions but cannot write. Background/white/black writes map to the global Appearance theme (`change background to white` -> `appearance.theme=light`, `change background to black` -> `appearance.theme=dark`). Bare UI color writes such as `change color from green to orange` mean Appearance -> Active color (`appearance.active_color`) unless the user names cursor color, separator color, theme, or another specific color setting. Ambiguous settings, broad resets, YO!agent prompt/format edits, YOLO rule paths, share access, and path-sensitive list changes require clarification or confirmation; credential-heavy paths and secret-looking values are rejected. Successful live settings changes must update the visible GUI without manual refresh unless the catalog marks the setting as next-use/restart.
- YO!agent starts the persistent Codex app-server as part of server startup when the configured YO!agent backend resolves to Codex CLI. Opening the YO!agent tab should not spend the first visible turn only warming the process; if the conversation is empty, the tab requests one real LLM startup answer, streams safe answer text as it arrives, persists the assistant message, and replaces the transient stream with the saved conversation. This startup response is shown once per empty conversation and must not disappear when the prewarm request finishes.
- YO!agent message timestamps include seconds. Assistant replies may include an expandable Details block for safe diagnostics: backend, response time in seconds and milliseconds, model CLI elapsed seconds, prompt size, seed/resume state, context-change note, stream phase, and fallback reason. Raw hidden chain-of-thought, reasoning deltas, or `<think>` blocks must not render in the visible answer or Details block; show only a short note that raw model thinking was hidden and keep a safe thinking/progress indicator visible while waiting.
- YO!agent messages that are direct results from another tmux/agent session (`agent_result`) must read like quoted output from that session: keep the stronger result bubble, leave the `Result from tmux session ...` heading unindented, and put the returned output below it in an indented block with a full-height left bar so users can distinguish relayed target-session output from YO!agent's own narration.
- YO!agent prompt latency should be dominated by the work the selected path actually needs. Settings reads/writes, skill-file operations, job creation, and explicit send routing must not build the full activity summary before answering; they may use settings payloads and direct target/session discovery only. Product-state questions such as `what changed`, `what did I last work on`, `what PR was that`, session-state explanations, and model-backed fallback may load the server activity payload because they need that evidence. The response Details `response time` is end-to-end server response time, not pure model first-token time.
- YO!agent product-state answers should prefer server-owned caches and registries: activity summary, Recent Agents/Tabber cache, session-file metadata, watched PR metadata, skill/context locations, and the product capability registry. It should answer questions such as `what did I last work on`, `what changed`, `what PR was that`, `which sessions are idle/running/blocked`, `where are YO!skills/context stored`, `what can I do from here`, and `why is this session behaving this way` from cached evidence, and say when a fact is unavailable instead of guessing.
- YO!agent common-intent examples and cross-agent communication methods are specified in [`docs/specs/YOAGENT_COMMON_INTENTS_AND_AGENT_COMMUNICATION.md`](specs/YOAGENT_COMMON_INTENTS_AND_AGENT_COMMUNICATION.md). That spec is the durable place for natural-language Preference aliases, multi-session handoff patterns, artifact handoff rules, and the transport reliability ladder.
- YO!agent chat uses the terminal font-size setting for transcript messages, action cards, recent agents, and the composer. Transcript path stays compact above the scrollable chat history. Current info and Recent agents render inside the chat history as assistant-style YO!agent responses at the bottom after persisted transcript messages, not as detached chrome and not pinned above older chat. The list is a bullet list, not chips: each row shows the shared agent icon, tmux session label, tmux window name, recent touched path roots when available, and compact activity recency. When the user scrolls upward in the YO!agent history, background summary/result refreshes preserve that scrollback until the user scrolls back near the bottom or explicitly sends/clears a message.
- YO!agent transcript messages that contain a result fetched from another target agent must be visually distinct from normal YO!agent answers. They keep the assistant-side placement but get a stronger colored left rule so the user can quickly see where the relayed agent result starts.
- YO!skills are loaded from built-in YAML first, then user-local YAML under `~/.config/yolomux/skills.d/`; user-local context Markdown lives under `~/.config/yolomux/context.d/`. Built-ins are read-only. User-local skills/context can be inspected and edited only through the narrow admin skill-file API or direct filesystem edits to those two directories.
- YO!agent composer `Up` recalls prior user messages, newest first, and places the caret at the end so the text can be edited and resubmitted. Repeated `Up` clamps at the oldest message. `Down` walks newer and then returns to the latest composer slot, restoring any unsent draft; when that slot is empty the normal `Ask anything...` placeholder is visible.
- YO!agent background transcript summaries use one Preference: `yoagent.refresh_interval_seconds`. `0s` disables background transcript summaries entirely; positive values below `30s` clamp to `30s`, and the worker exits on the next pass when the interval returns to `0s`. The old `yoagent.auto_refresh` boolean is input-only migration compatibility and must not appear as a live Preferences row.
- YO!agent explicit send requests run without an extra confirmation unless the user asks for preview or confirmation. If the user asks to show, print, return, or tell them the result here, the first YO!agent response must confirm the send immediately, state that the target session was verified as accepting an AI prompt, show the exact prompt YO!agent is sending after perspective rewriting/redaction, and say it is awaiting the response; a background watcher then reads the target agent transcript delta, falls back to visible pane text on timeout, and appends a result message through the persisted conversation plus `yoagent_conversation_changed` client event. Target-session prompts preserve perspectives: YO!agent keeps the user's routing perspective local, then writes the target prompt from the target agent's point of view instead of blindly forwarding the user's wording. `ask agent 1 to <do ...>` sends `<do ...>` to agent `1`, not `ask agent 1 to <do ...>`, and `ask session 1 what it has done today` sends `what have you done today?` to session `1`, while YO!agent's local explanation may still say it is asking tmux session `1`. Generated handoff prompts should be concise clean tasks/questions, not multi-line routing/context wrappers, unless the user explicitly provided raw quoted multi-line text. When a current target input draft exists and the user explicitly asks YO!agent to send, YO!agent clears that draft first, verifies the composer is empty, then pastes and submits the new prompt; if the draft cannot be cleared, it refuses and does not paste. After a paste-plus-Return it must not claim success if the sent text is still sitting in the target composer. Old submitted `❯ ...` prompt lines visible above a current blank composer must not be treated as current unsent text. For Claude/Codex transcript deltas that show a tool/turn lifecycle, the watcher must not append the first readable text as final; it waits for a terminal assistant/task-complete marker such as Claude `stop_reason: end_turn` or Codex `task_complete`, so an intermediate pre-tool sentence is not shown as the completed target result.
- YO!agent action cards appear under assistant messages only for confirmed/preview paths. A ready card shows session, agent/model, transport, pane id, path, risk state, and redacted text to send, plus a single Confirm/send button for admin users. Waiting, expired, sent, and error cards show a state label instead of an active send button. Public action/job payloads and persisted chat history must redact known `token=`, `secret=`, `password=`, and `api_key=` values, while the server keeps the raw pending preview privately until it expires or sends.
- YO!agent jobs are one-shot background watches. Notify-session-idle, notify-all-idle, and wait-then-send jobs live in server state, survive reload/restart, and publish `yoagent_jobs_changed` client events when created, confirmed, cancelled, fired, failed, or timed out. Firing notify jobs use the existing toast/browser notification path. Wait-then-send jobs do not paste into a busy, non-agent, approval-prompt, missing, or stale target; they re-preview and revalidate the exact pane/agent before sending. High-risk prompt text forces confirmation even when the user did not explicitly ask for preview.
- YO!agent `all sessions` means the configured local YOLOmux session roster. The job state tracks the per-session states and blockers so a future job list can show which session is still preventing the watch from firing. Batch fanout actions should remain per-target confirm/cancel instead of one opaque batch button.
- YO!agent direct-send transport is selected through the backend transport registry and must be named in action cards, send confirmations, result records, jobs, and diagnostics. Existing user-owned visible Claude/Codex panes use `tmux-legacy`: server-resolved visible pane paste plus Return, prompt-state preflight, stale-target checks, and post-send composer-cleared verification. `tmux-legacy` is not a native agent API and must never be described as more reliable than terminal automation. Managed transports may be preferred only when YO!agent owns or can safely resume that managed conversation and can surface the effective auth, sandbox, approval, and result-source semantics. The planned order is Claude SDK for YOLOmux-owned Claude sessions, Claude Channels for opt-in visible Claude panes, Codex SDK before direct Codex app-server calls for YOLOmux-owned Codex sessions, Codex MCP server for central orchestration, `codex-exec` for one-shot jobs, Claude stream-json as a CLI fallback, and `tmux-legacy` last. Blind `tmux send-keys` is a fallback implementation detail.
- YO!agent-managed structured sessions are background conversation/job targets first, not fake terminal tabs. A managed Claude/Codex SDK/app-server/MCP/exec session may appear in YO!agent action cards, waits, job rows, Recent Agents, and result transcript views with its provider, thread/session id, cwd, model, sandbox/approval settings, started time, result source, and latest final response. It must not appear as a normal tmux tab unless YOLOmux actually started or attached a visible TUI/PTY for that session. If a future visible TUI is attached to a managed session, the UI must label both identities: the visible tmux pane and the structured provider/thread that YO!agent controls.
- YO!agent multi-session handoffs are always orchestrated by YO!agent, not delegated from one target session to another. In a normal request, session `1` and session `2` do not know about each other; YO!agent coordinates and passes clean messages between them unless the user explicitly asks for disclosure. Direct relay/chaining between agents is rare and allowed only when the user explicitly requests relay/chaining; when allowed, YO!agent must pass concrete instructions that say how to relay, and must not leave the agent to infer the route. For a request such as "ask session 1..., get info from session 1, pass that to session 2", YO!agent asks tmux session `1`, waits for and records session `1`'s response, derives a bounded handoff prompt itself, verifies tmux session `2` is accepting an AI prompt, sends the handoff to session `2`, waits for session `2`, and appends session `2`'s result back into the YO!agent conversation. The prompt sent to session `2` must be the clean user-facing question or task, not a meta wrapper such as "tmux session 1 replied..." plus instructions. If the user asks YO!agent to transform session `1`'s answer first, such as adding 35 minutes to a reported time and asking session `2` whether it is correct, YO!agent performs the transformation before sending and asks only the derived question, for example `Is 6:10 PM the correct time now?`. While any background result watcher is active, the chat history shows a compact waiting queue with one row per active wait; a handoff wait row says ``Waiting for tmux session `1` to respond (regarding ...), before handing off the next request to tmux session `2` (regarding ...)``. Multiple simultaneous waits must render as multiple rows and clear independently as each wait finishes or times out.
- Any new Preference row must be represented in the server defaults, sanitization/clamping rules, frontend `preferenceSections()`, locale catalogs, settings tests, and live-apply runtime if it affects visible UI.

## Keyboard Shortcuts Contract

- The app modifier is platform-specific: Mac uses `Cmd`; PC/Linux uses `Ctrl`. Shortcut labels must say the concrete platform modifier and must not display combined `Ctrl/Cmd` text.
- Mac app shortcuts use `Cmd` so they can be captured even when a terminal is focused; Mac `Ctrl` stays available for terminal/tmux input. PC/Linux app shortcuts use `Ctrl`.
- Global shortcuts are captured before focused controls swallow them, but text inputs, rename fields, editors, and terminal input must keep their normal typing shortcuts unless a shortcut is explicitly platform-owned by the app.
- Implemented app shortcuts: `Mod+P` opens unified search with file-first ranking, `Shift+Mod+P` opens the same unified search with pane/command-first ranking, `Mod+B` toggles Finder/Differ, `Mod+,` opens Preferences, `?` opens Keyboard shortcuts, `Esc` closes shortcut/menu overlays, and `Mod+W` closes the active closable tab.
- Close-tab fallback: outside text-editing contexts, `Mod+Backspace` / `Mod+Delete` may close the active closable tab. This fallback must not fire inside inputs, rename fields, editors, or terminal text entry.
- Editor shortcuts: `Mod+S` saves, `Mod+F` finds, `Mod+H` replaces, `Mod+G` goes to line, `Mod+/` toggles comments, `Tab` / `Shift+Tab` indent and outdent, `Mod+Z` / `Shift+Mod+Z` undo and redo, and `Mod+Alt+[` / `Mod+Alt+]` move editor navigation back and forward.
- Diff editor chunk shortcuts reuse editor undo/redo: `Mod+Z` undoes the selected chunk action and `Shift+Mod+Z` redoes it when the diff editor owns the focus.
- Finder/Differ keyboard behavior follows Finder-style list navigation: Arrow Up/Down moves selection, Shift+Arrow extends selection, Home/End jump, Shift+Home/End extend to edge, Arrow Right expands or steps into children, Arrow Left collapses or steps to parent, Space toggles quick preview, type-ahead selects by prefix, `Mod+A` selects all, `Mod+O` and `Mod+Down` open, `Mod+Up` opens the enclosing folder, and Return starts rename.
- Finder/Differ delete is platform-specific: Mac uses `Cmd+Delete` or `Cmd+Backspace`; PC/Linux uses plain `Delete`. It must be scoped to Finder/Differ and must not run from text inputs or rename controls.
- Terminal copy handling must preserve normal terminal input: `Cmd+C` on Mac or `Ctrl+C` on PC/Linux copies xterm selection when present; without a browser/xterm selection, the tmux copy bridge may request the tmux copy-mode selection. Claude and other TUIs can own the visible highlighted block and expose it only through OSC 52; terminal right-click `Copy` must prefer the selection captured before xterm clears it, treat an explicitly captured empty selection as authoritative, and then fall back to the recent OSC 52 app clipboard payload instead of re-reading a tiny under-cursor browser/xterm fragment.
- Terminal URL links must stitch hard-wrapped TUI URLs into one clickable link even when the TUI emits each visual row as a separate non-`isWrapped` line. Hanging-indent continuations and zero-indent width-1 continuations are allowed only when the previous row ends in an unterminated URL token at the terminal edge; a following fresh `https://`, `file://`, or `www.` row stays a separate link.
- Tab drag, pane split, and root-edge layout changes are currently pointer-driven. Keyboard equivalents for moving tabs/panes are not implemented and must not be advertised as available until they exist and are tested.
