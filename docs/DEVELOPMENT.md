# YOLOmux — Development

Conventions, architecture, build/test workflow, restart workflow, and API notes for contributors live here. End-user/operator docs live in [`README.md`](../README.md). AI-agent behavior rules live in [`AGENTS.md`](../AGENTS.md).

## Setup

Standard install via the Makefile (wraps `pip install -e .` + xterm.js assets + the static bundle build):

```bash
make setup        # pip install -e ".[yoagent]"  +  npm install (xterm.js)  +  python3 tools/static_build.py
make dev          # same, plus dev/test deps (pytest-xdist) — see `make help`
```

`pyproject.toml` is the source of truth for Python deps and the floor (`requires-python = ">=3.10"`), so pip checks the Python version and resolves the runtime dependencies plus the `yoagent` / `dev` extras. `make setup` and `make dev` run the same Python-floor preflight before invoking pip. The `requirements*.txt` files are kept for `pip install -r` workflows and are referenced by `boot.sh`/provisioning, but mirror the same packages, including `watchfiles`. Plain pip equivalents:

```bash
pip install -e ".[yoagent,dev]"      # deps + the `yolomux` command + agent SDKs + pytest-xdist
# or the legacy split: pip install -r requirements.txt -r requirements-dev.txt -r requirements-yoagent-managed.txt
```

On externally managed system Python installs (PEP 668), create and activate a virtual environment first (`python3 -m venv .venv && . .venv/bin/activate`) instead of forcing `--break-system-packages`.

## Project Conventions

### Version bump policy

`YOLOMUX_VERSION` in `yolomux_lib/common.py` is bumped only when publishing to `origin/main`. LOCAL `cps` / `yolo-cps` lands work on local `main` with no version bump and no push; ORIGIN / REMOTE `cps` rebases onto `origin/main`, increments the patch segment, folds the bump into the work commit, and includes `Version: 0.4.N` in the commit message body. The self-updater compares `YOLOMUX_VERSION` on `origin/main`, so SHA-only published commits do not cue update notifications, but local integration commits must not create fake versions.

To find the latest committed version before ORIGIN publish: `git show origin/main:yolomux_lib/common.py | grep YOLOMUX_VERSION` and add 1 to the patch segment.

### Timing constants

JavaScript and CSS timing constants in `static_src/`, `static/yolomux.js`, and `static/yolomux.css` are split by purpose:

- UI / popup / display / animation timings use round whole numbers, for example `300`, `1000`, `1550`, `10000`, `20000`.
- Backend polling / refresh intervals SHOULD prefer slightly-staggered (often odd) values, for example `1257`, `3001`, `5003`, to spread client requests across ticks instead of piling up on the same one. This is a preference, not an invariant — several shipped defaults in `yolomux_lib/settings.py` are still round (`event_log_refresh_ms: 5000`, `server_event_poll_ms: 850`, `server_directory_event_poll_ms: 3000`); nudge them toward staggered values when you touch that area, but nothing enforces it today.

UI durations are perceived by users and read as deliberate at round values. When a request says "make it 1000ms", a UI duration keeps `1000`; a backend poll may use a nearby staggered value like `1003`.

Recurring browser work must register through `resetRuntimeInterval()`/`clearRuntimeInterval()`. The named recursive scheduler waits for an async callback to settle before starting the next full delay, so slow YO!stats, share-status, or share-geometry work cannot overlap or degrade into back-to-back requests. Do not add direct application `setInterval()` loops or a parallel timer registry.

### Responsive UI sizing

Do not hard-code layout capacity around one browser window, OS, zoom level, or font rendering. For menus, panes, dropdowns, tab lists, and editor/viewer surfaces, prefer intrinsic sizing, flex/grid allocation, percentages, viewport units, and shared CSS variables over fixed pixel width buckets. Fixed pixels are acceptable for hairlines, icon glyphs, and the base spacing-token definitions, but component gaps, padding, and margins from 1–12 px must consume `--space-1` through `--space-12`; negative token spacing uses `calc(-1 * var(--space-N))`, never invalid `-var(...)`. If JavaScript must set a size, derive it from measured DOM content and clamp it to the current viewport/container.

## Source Layout

### Runtime And Layout Names

Use explicit names whenever runtime tmux metadata and browser layout appear together: `TmuxSession`, `TmuxWindow`, and `TmuxPane` describe tmux objects; `YOPane` and `YOTab` describe the browser layout. `TmuxPaneInfo` is the backend record for a `TmuxPane`; do not introduce a new unqualified `PaneInfo` use, because it is ambiguous with `YOPane`.

The main server entry point is `yolomux.py`, which delegates to `yolomux_lib/cli.py`. Request routing lives in `yolomux_lib/server.py`, application coordination lives in `yolomux_lib/app.py`, local Unix RPC framing/lifecycle lives in `yolomux_lib/local_services/`, visible Claude/Codex tmux pane control lives in `yolomux_lib/agent_tui.py`, shared Claude/Codex structured communication lives in `yolomux_lib/agent_comms/` (Codex app-server, Claude stream-json, and normalized stream events), and shared helpers live in smaller modules such as `metadata.py`, `sessions.py`, `session_files.py`, `transcripts.py`, `uploads.py`, `events.py`, `websocket.py`, `approvals.py` (the approval-prompt detection pipeline), `runtime_env.py` (one dependency-light PATH healer used by both the server and Codex child environment), `atomic_file.py` (cross-process file lock + atomic write), `cache.py` (the shared `TtlCache`), and `activity.py` (the per-session/window user+agent activity ledger). Local service entry points are `statsd.py` for the sole YO!stats SQLite writer plus its independent read-only `stats-reader` WAL peer, `search_indexer.py` for `indexd` Quick Open SQLite ownership, `jobd.py` for bounded stateless CPU tasks, and `approvald.py` for target-keyed YO auto-approval workers; the elected `app.py` process owns independent YO!stats collection workers and talks to services through `StatsClient`, `SearchIndexerClient`, `JobClient`, and `ApprovalClient` rather than importing service internals. Recursive tree discovery is never request-path/API-process work: indexed-repository discovery is a maintenance-priority `jobd` task, and `app.py` serves the last completed snapshot while a refresh is pending.

Frontend source for the interactive UI lives in ordered partials under `static_src/js/yolomux/` and `static_src/css/yolomux/`. Generated served assets are `static/yolomux.js` and `static/yolomux.css`; do not edit those directly except as generated outputs. Python keeps only the small HTML shell in `yolomux_lib/web.py`, plus bootstrap JSON and versioned static asset URLs. The main app's non-tmux tab types are centralized in the `TAB_TYPES` registry in the JS source partials. The read-only wall has its own frontend files, `static/tmux-wall.js` and `static/tmux-wall.css`, so `tmux_wall.py` stays focused on tmux capture, JSON endpoints, and Server-Sent Events.

Detached pane pop-outs are owned by `static_src/js/yolomux/96_pane_popout.js` and the `/pane-popout` placeholder route. Add support through `TAB_TYPES.canPopout`, `openPopout`, or `popoutRenderer`; do not create per-pane popout registries or document writers. Phase-1 popouts are rendered snapshots, so live terminal/transcript and interactive command panes must remain disabled with an explicit `popoutDisabledReason` until they have a live bridge design.

### Central State Containers

`agentWindowActivityRecords` owns per-agent-window transition state; `sessionStatusRecords` owns per-session status and notification state; `toastRecords` owns toast nodes/timers; `dragState` owns one active drag; `paneViewState` owns durable pane scroll; `layoutUrlState` owns URL application; `clientEventTransportState` owns the browser SSE lifecycle; `fileExplorerDirectoryRecords` owns directory-observation signatures; `fileExplorerFsResourceRecords` owns filesystem resource cache records; `shareHostConnectionRecords` owns share host sockets/queues; `StatsHistoryService` owns the elected app's per-family scheduler threads, locks, wake events, diagnostics, and sample/token caches, `statsd` alone mutates durable history, and `stats-reader` alone serves potentially long history aggregation/encoding from a read-only WAL connection; `SessionFilesService`, `ActivityTranscriptService`, and `ClientWatchService` own their server locks/cache/work/watcher records; and `approvald` owns target-keyed auto-approve workers. Readers use their named selectors/helpers and lifecycle owners; consumer-local mirrors of the same keyed fact are forbidden.

For every refactor or new UI behavior, record `Existing parent checked: <symbol>` or `New shared parent: <symbol and reason>` plus the exact searches used. A DONE note or handoff also records the negative search proving retired copies are gone and the net non-generated line change.

The approval-prompt detection pipeline lives in `yolomux_lib/approvals.py` (one shared owner: `app.py`'s read-path, the `AutoApproveWorker` act-path in `yolomux_lib/auto_approve_worker.py`, and the standalone `auto_approve_tmux.py` CLI all call it; the CLI re-exports it). `agent_tui.py` is the public owner for combining that pure detector state with tmux captures, cursor facts, composer draft/ghost state, transcript activity upgrades, clear, paste-submit, and submit verification. An enabled session may have one worker per detected agent-pane target; `approvald` is the single target-keyed owner of each worker, its target flock, and its session association.

## Normalized Session Work Metadata

`yolomux_lib.metadata.session_work_graph()` is the only backend owner of repository/work metadata. It serializes `work_graph.version`, a monotonic `generation`, and canonical maps for `tmux_sessions`, `tmux_windows`, `tmux_panes`, `runtime_actors`, `path_observations`, `git_worktrees`, `local_repositories`, `hosted_repositories`, `local_branches`, `pull_requests`, `linear_issues`, and `worktree_branch_activity`. The graph uses snake_case API fields; browser selectors use camelCase rendering projections only after reading the graph.

`PathObservation` links tmux/actor evidence to at most one nearest `GitWorktree`. A `GitWorktree` belongs to exactly one `LocalGitRepository`, identified by Git common-dir, so primary and linked worktrees share one local branch inventory without duplicating it. Separate clones retain separate LocalGitRepository records even when they map to one HostedRepository; PullRequests belong to the HostedRepository and associate to zero or more LocalBranches. Graph validation runs before serialization and rejects dangling IDs.

`session_to_json()` emits one authoritative `work_graph` plus non-Git `window_metadata` cwd rows. New consumers must use the graph-aware shared frontend selectors (`sessionWorkSummary`, `focusedRepositoryIdsForTmuxTarget`, `branchIdsForGitWorktree`, and `pullRequestIdsForTmuxTarget`); neither `session.project` nor `window_metadata[].git` exists in the API. A schema-valid graph, including an empty one, is authoritative. Metadata refreshes use the graph generation to reject stale results rather than wall-clock ordering.

## Static Build

After changing any frontend source file, rebuild the served single-file assets:

```bash
python3 tools/static_build.py
```

Run with `--check` before committing to verify generated assets match source:

```bash
python3 tools/static_build.py --check
```

Transient Claude/Codex window state has one browser owner: `agentWindowActivityRecords`, keyed by the stable window transition key. Each record owns the activity generation, stopped refresh, transition-pulse refresh, stopped acknowledgement timestamp, and temporary gray acknowledgement visual. Do not add another map keyed by the same transition key; extend the record so timer replacement, acknowledgement, and rendering cannot advance different generations.

YO!stats captured API-event lifecycle state has one browser owner: `jsDebugGraphEventRecords`, keyed by event id. Each record owns the graph bucket, response-byte count, and last-seen timestamp used by response completion, retention, clear, and debug counts; do not split these fields into parallel maps.

Other paired lifecycle state follows the same record rule. `sharePointerRecords` owns each remote sender's ghost node and hide timer; `uploadResultRecords` owns each session's upload entries and cleanup timer; `sessionStatusRecords` owns each browser session's state snapshot, notification throttle map, working-agent tone map, and metadata badge pulse map; `toastRecords` owns each toast node and numeric-ID removal timer; backend `metadata_badge_records` owns each session's signature and pulse deadlines. Watched-PR notification throttles are global and must stay outside session records. Extend these records and their shared lifecycle helpers instead of adding another map keyed by the same sender/session or putting a composite/numeric key through direct session rekeying. Durable session status and upload records migrate on rename, but transient toast records are discarded on detach or rename because their callbacks capture the original session identity. Pending pane-view captures are also transient: `detachSessionUi()` is their only session-lifecycle cleanup owner, while durable `paneViewState` remains in the clear/migrate path. Metadata badge persistence serializes signatures only and writes the legacy pulse key as empty because live pulse deadlines must not survive restart.

Non-keyed frontend transactions follow the same rule. `selfUpdateReloadState` owns one post-update reload attempt, `serverWatchRootsState` owns the debounced watch registration pipeline, `diffRefPopoverState` owns the reusable ref-picker node/input/items/cursor/listeners, `layoutUrlState` owns pending deep-link application and URL refresh scheduling, `fileExplorerSyncState` owns Finder Sync target/application generations, `fileExplorerSessionFilesState` owns Finder/Differ payload/signature/loading/request generations, `tabberActivityState` owns Tabber activity request/application readiness, `commandPaletteState` owns the palette node/query/cursor/rendered items, `fileQuickOpenState` owns Quick Open scope/results/debounce/abort identity, `backgroundOwnerStatusState` owns owner-status request generations, `activitySummaryState` owns activity payload/request generations, `transcriptMetadataState` owns session metadata payload/load/request generations, `searchHistoryState` and `runHistoryState` own their query/result generations, `yoagentConversationState` and `yoagentJobsState` own YO!agent hydration generations, `yoagentChatState` owns active chat/action work plus queue/composer/error state, `yoagentStartupState` owns prewarm and frozen startup-info state, `infoPanelRenderCache` owns signature/HTML together, and `clientEventTransportState` owns the SSE source/connectivity/dispatch/reconnect lifecycle. Reset or replacement paths update the whole record rather than leaving timer, pending-option, input, cursor, promise, generation, frame, or cached-HTML scalars behind. External SSE/direct payloads invalidate older HTTP generations before applying; stale success, failure, and `finally` paths may not replace data or clear a newer request handle. Semantic CSS aliases that intentionally share paint reference one stable palette token with `var(--owner)`; they do not repeat the owner's raw hex under a second token name or point a scoped component override back at itself.

Backend YO!agent prewarm and visible-startup state follows the same generation rule through one `YoagentPrewarmRecord`. Clear conversation increments the startup generation and marks reset in progress before deleting CLI/conversation state; a visible startup cannot begin during that clearing window. Startup persistence checks and writes under the same lock, stale work publishes a stopped/aborted completion and returns the current conversation, and cleanup clears the active generation only when it still owns that generation.

Backend auto-approve status caching follows the same generation rule through one `AutoApproveCacheRecord`: payload, worker identity, and generation advance together, and a retired refresh cannot publish or clear a replacement worker after invalidation. Finder filesystem list/info caching likewise uses one bounded `fileExplorerFsResourceRecords` map keyed by normalized resource type and path; payload, timestamp, request, and generation share a record, so rename/reindex/push invalidation and newer fresh reads retire older completions instead of letting them repopulate stale state.

The client-event watcher uses one `ClientEventWatcherRecord` per generation. The record owns the main worker, directory-poll worker, wake/stop events, and every poll deadline. Stop retires the whole record; restart creates fresh events, and main/directory cleanup may mutate app state only when both the record and worker identity still match. Never split a new watcher phase into a parallel app attribute or reuse a retired generation's events.

### Demand-aware browser transport

`clientEventTransportState` is the one browser owner for client-event demand. Its descriptor is recomputed from document visibility, active pane tabs, active subtabs, and system-notification delivery, then opens `/api/client-events` with only the required channels: `core`, `status`, `files`, `activity`, `transcripts`, `yoagent`, or hidden-page `attention`. A hidden page with system notifications disabled closes the stream and unregisters server watches; a hidden page with system notifications enabled retains only compact attention delivery. Pane, layout, Finder-mode, and notification-setting changes must resync this descriptor instead of opening feature-local streams or timers.

The backend `ClientEventBroker` keeps per-subscriber channels and aggregates producer demand across every connected browser. Filesystem/transcript/activity polling, auto-approve/tmux status polling, watched-PR polling, and YO!agent job polling run only while at least one subscriber needs their channel. Runtime diagnostics expose bounded sanitized client records and published/delivered/filtered SSE event and byte counters by type. HTTP response counts/bytes remain under the existing endpoint performance metrics; do not create a second accounting path.

For a traffic trace, clear JS Debug, hold one pane/document state for a fixed interval, copy the JS Debug report, and compare endpoint counts/bytes and SSE type counts against the runtime report's `client_events` snapshot. Test at least visible terminal-only, visible Finder/Tabber/YO!agent, hidden with system notifications off/on, no active share, and two browsers with disjoint panes. A hidden surface must catch up from one current snapshot when made visible; it must not receive a backlog.

Browser drag state also has one owner: `dragState` contains the tab/pane item and source slots, file payload, custom/native/transparent preview nodes, preview offset, and tab-geometry cache. Tab, pane, Finder, and Differ drag starts replace incompatible fields; drop, dragend, pointer/mouse release, blur, and visibility cancellation clear the complete operation through `cancelDragOperationState()`. The transparent native drag-image node is a reusable singleton and remains cached between operations; no other operation field may survive cleanup.

YO!share browser transport state follows the same rule: `shareHostConnectionRecords` owns one `{socket, queue}` record per share token, and `sendOrQueueShareHostMessage()` is the only open-or-enqueue policy. A matching socket close clears only `record.socket` so queued frames survive an intended reconnect; active-token reconciliation closes and deletes the complete record when the share stops or expires. Do not add parallel token-keyed socket and queue maps or another direct queue mutation path.

Finder directory observation state has one bounded browser owner: `fileExplorerDirectoryRecords`, keyed by normalized directory and holding the last listing signature plus prior entry-name set. Broad filesystem invalidation clears only signatures so a later listing can still identify newly added names; root invalidation removes the complete record and any child-path highlight deadlines. Do not split signature and prior-name fields into independently evicted maps.

## Tests

`python3 tools/check.py` runs the local gate as parallel lanes and exits non-zero if any lane fails — use it as the one pre-commit/CPS command. Default/full runs and focused expensive lanes acquire the per-user `~/.cache/yolomux/expensive-tools.lock`, shared by every worktree and launch context; a contender fails immediately instead of queuing another full gate to run later. `YOLOMUX_TOOL_LOCK_PATH` may override the path. When live YOLOmux servers are present in the shared background-owner state the runner lowers its process priority before launching child lanes. Use `python3 tools/check.py --serial` when debugging order, process load, or interleaved output: it runs lanes one at a time and omits xdist from pytest, browser, and E2E commands entirely, even if `YOLOMUX_PYTEST_WORKERS` is set. Use `python3 tools/check.py --list-lanes` to see lane names and repeat `--lane <name>` for focused runs. Use `--no-tool-guard` only when you intentionally want overlapping high-load local tools.

The non-browser gate includes AST invariants that all Python imports under runtime, tools, and tests remain module-scoped and that runtime/tool functions with two or more statements do not repeat an identical body. Optional dependencies may be detected at module scope before conditional module-scope imports, but imports must not be hidden inside functions or import-time `try`/`except` blocks. Shared process checks such as PID liveness live in `yolomux_lib/background_owner.py`; tools import that owner rather than carrying a second implementation.

The default lanes are `py-compile`, `static`, `node-syntax`, `node-layout`, `pytest`, `pytest-browser`, `pytest-e2e`, and `whitespace`. The runner derives one concurrent pytest budget from the host's logical core count, capped at 8 CPUs on macOS. Linux makes 75% of that capacity available to pytest; macOS uses 50% because every browser worker owns a Chrome process tree and host security/indexing services scan its temporary profile. The budget is split one-half to non-browser, one-third to browser, and the remainder to E2E, with at least one worker per pool. Thus a Mac reports at most an 8-CPU scheduling budget (2/1/1 workers), 10 Linux cores produce 3/2/2, and 32 Linux cores produce 12/8/4. This replaces independent CPU-sized `auto` pools, which oversubscribed development Macs and caused control-socket handoff and cold-daemon timing failures that passed in isolation. `YOLOMUX_PYTEST_WORKERS=N` gives all three pools the same explicit size; `YOLOMUX_PYTEST_WORKERS=nonbrowser,browser,e2e` tunes them independently outside `--serial` mode.

The `pytest-browser` lane first runs the fatal boot-smoke suite, then behavioral browser tests with xdist work stealing, and finally the local `visual_golden` matrix serially. Serial goldens are intentional: the comparison uses reviewed, machine-local pixels, and parallel Chrome compositor/font pressure can cause small raster deltas without a product-layout change. ### Restart workflow

After every YOLOmux change in a dev worktree, restart that active dev server and verify it before reporting the change ready for testing. Restart prod only when explicitly requested.

Use `boot.sh` from the checkout you want to serve. `boot.sh` and `yolo-dev-start` share one process-lifetime startup lock from `tools/startup_common.sh`, so a second launcher fails before it stops or starts any server. They also share a slow-ramp load gate: before each agent or server mutation, one- and five-minute load must be below the CPU budget, capped at 8 CPUs on macOS even when the OS reports more logical CPUs. The launcher waits up to `YOLOMUX_START_LOAD_WAIT_SECONDS` (300 seconds by default) and stops the ramp rather than adding another process under sustained load. `boot.sh` also serializes each port, restarts only that listener, waits for the old listener to leave the port before launching the replacement, sets `PATH` explicitly before launch so agent CLIs under `~/.local/bin` are visible, defaults `MALLOC_ARENA_MAX` to 2 so threaded JSON/transcript work cannot retain one glibc arena per request thread, and appends logs under `/tmp` by default. The server holds a process-lifetime per-port lease, so direct launch and self-update races fail before app/background initialization. On Linux the script finds listeners with `ss`; on macOS it falls back to `lsof`, and it uses plain `nohup` when `setsid -f` is unavailable.

```bash
./boot.sh
```

The script accepts any valid TCP port. It defaults to HTTPS `8880` on macOS and `7770` on Linux and still passes the compatible `--self-signed` spelling plus `--dang`; the CLI itself now defaults to the same self-signed HTTPS behavior. `YOLOMUX_PORT` selects an arbitrary primary port and keeps the normal non-dev launch behavior. The macOS dev worktrees conventionally use `8001` through `8003`; the Linux fleet conventionally uses `7771` through `7773`. For a dev worktree, run the script from that checkout and pass the port you chose; non-primary ports use `--dev` automatically:

```bash
(cd ~/path/to/dev-checkout && ./boot.sh <dev-port>)
```

Override with `YOLOMUX_PORT`, `YOLOMUX_HOST`, `YOLOMUX_LOG_DIR`, `YOLOMUX_START_LOAD_WAIT_SECONDS`, `--port`, `--dev`, or `--no-dev` when testing a non-standard instance. `YOLOMUX_BACKGROUND_OWNER_PRIMARY_PORT` pins background ownership to one preferred server while retaining automatic follower failover; `boot.sh` defaults it to its selected primary port. Use `./boot.sh --print-command <dev-port>` to inspect the launch command without restarting anything.

For the standard multi-worktree stack, `yolo-dev-start` defaults the preferred server to `8882` on macOS when that worktree exists (`8880` otherwise), and to `7770` on Linux. Override it with `YOLOMUX_PRIMARY_PORT`. The preferred server starts first; each port must then pass readiness, exactly-one-process, three-sample ownership, empty-refresh-queue, and system-load gates before the next port starts. Discovery checks use the read-only `/api/tmux-session-exists` roster endpoint and must never force a full transcript/metadata rebuild. Every checkout must advertise the same `BACKGROUND_OWNER_PROTOCOL_VERSION`; a stale worktree causes a preflight refusal before tmux, launchd, listeners, or ownership are changed.

To restart a dev worktree in no-auth test mode:

```bash
YOLOMUX_TEST_AUTH_BYPASS=1 ./boot.sh <dev-port>
```

Four footguns: (1) never `pkill -f` broad YOLOmux patterns; kill only the listener for the selected port; (2) run `boot.sh` from the checkout you intend to serve so a dev port does not accidentally serve another worktree; (3) never bypass the shared startup lock or replace the read-only roster check with `/api/transcripts?force=1`; (4) a backend `.py` change only takes effect after the Python process restarts, because `static_build.py` does not touch backend code. If a restarted server appears ready and then disappears, check whether readiness was served by the old listener; `boot.sh` must fail instead of launching over a still-bound old PID. The in-app self-update path likewise treats a failed, timed-out, or unlaunchable `tools/static_build.py` run as blocked and must not restart into stale generated assets; periodic update-check exceptions are logged rather than hidden.

Verify after restart:

```bash
ps -ef | grep "yolomux\.py.*<port>" | grep -v grep
ss -tlnp 2>/dev/null | grep ":<port> "
curl -sk -o /dev/null -w "ping: %{http_code} %{time_total}s\n" https://localhost:<port>/api/ping
curl -skL -u <user>:<pass> https://localhost:<port>/ | grep -oE 'YOLOmux [0-9.]+' | head -1
```

Expected: one `yolomux.py` process, `LISTEN` on the intended port, `ping: 401` in under roughly 100ms, and the rendered version matches `YOLOMUX_VERSION`. If verification fails, inspect the appended `/tmp/yolomux-<port>.log` log for the previous launch error and the latest `boot.sh launching port ...` marker.

If you intentionally started with `YOLOMUX_TEST_AUTH_BYPASS=1`, `/api/ping` and `/api/settings` should return `200` without cookies. If a normal login-gated server returns `200` for those unauthenticated routes, auth is accidentally bypassed.

## Production Sync (`cps`)

`cps` (the `yolo-cps` skill) has two modes, chosen by the trigger word:

- **LOCAL (default — "cps" / "yolo-cps" with no qualifier):** rebase the dev branch onto LOCAL `main` and land it on local `main`. NO version bump, NO push.
- **ORIGIN ("cps origin" / "cps remote"):** rebase onto `origin/main`, bump the version, land on local `main`, then push to `origin/main`.

Edits happen in a dev checkout; the integration/production checkout (which holds `main`) is read-only during sync and shares the same `origin`. Because `main` is checked out in the production checkout, you cannot check it out in a dev worktree — land work by committing on the dev branch, rebasing it onto `main` / `origin/main`, then fast-forwarding `main` from the production checkout.

ORIGIN-mode sequence from the dev checkout:

```bash
python3 tools/check.py                       # the full gate (see Tests above)
git fetch origin && git rebase origin/main   # rebase onto the published tip
# bump YOLOMUX_VERSION in yolomux_lib/common.py, folded into the work commit
git add -- <explicit-files>
git commit -m "<message including Version: 0.4.N>"
git -C <production-checkout> merge --ff-only <branch>    # land on local main — its OWN exit-checked command
git -C <production-checkout> push origin main            # only after the ff-merge succeeds
```

LOCAL mode is the same minus the version bump and the final `push` (stop after the ff-merge).

Rules:

- ORIGIN mode MUST bump `YOLOMUX_VERSION` in `yolomux_lib/common.py` in the same commit; the auto-updater checks this value on `origin/main`, not the commit SHA, so SHA-only commits do not cue the update. LOCAL mode does NOT bump.
- Never use `git add -A`; screenshots and scratch files must not get swept in.
- The `merge --ff-only` into the production checkout MUST be its own exit-checked command — NOT piped through `tail`/`grep` and NOT `&&`-chained straight into the push. A pipe's exit status is the last stage's, so a DIVERGED ff-merge fails silently and a chained `push origin main` then publishes whatever local `main` already points at, not your commit. Confirm the merge succeeded before pushing.
- This is a shared multi-worktree: local `main` can advance from another worktree mid-`cps`, so your ff-merge can suddenly refuse (diverged). Recovery: `git fetch origin`, `git rebase origin/main` your dev branch (disjoint files rebase clean), re-run the gate, then ff-merge + push. No force-push, nothing lost — the other commit is usually a sibling off the same base.
- Production pull/merge is `--ff-only`. Never edit, stage, or commit inside the production checkout.
- Restart is NOT part of `cps` itself. After code changes in a dev worktree, restart the active dev server before reporting ready; restart prod only when explicitly asked (see Restart workflow above), then verify `/api/ping`, the process cwd, and `YOLOMUX_VERSION`. The login page may not expose the version to unauthenticated curl; do not rely on a blank version grep alone.

## How The Webterm Works

The server is a dependency-light Python `ThreadingHTTPServer`. It serves one HTML page, local xterm.js assets, JSON APIs, Server-Sent Events streams, and a WebSocket endpoint.

For each terminal connection, the browser opens `/ws?session=<tmux-session>`. The server creates a PTY, runs `tmux attach-session -t <tmux-session>` for admin users or `tmux attach-session -r -t <tmux-session>` for readonly users on the PTY slave, reads terminal bytes from the PTY master, and sends those bytes to xterm.js as WebSocket binary frames.

Closing the WebSocket before the initial resize payload is an expected disconnect. `bridge_tmux()` returns before PTY allocation for that case and uses the same `OSError` family boundary as later terminal disconnects, so reloads cannot emit request-thread tracebacks or allocate an orphan PTY.

Browser input is sent as JSON messages over the same WebSocket. Normal keyboard data becomes `{"type": "input", "data": "..."}`. Resize messages become `{"type": "resize", "cols": ..., "rows": ...}`. Scroll messages become `{"type": "tmux-scroll", "direction": "up|down", "lines": ...}`.

Terminal image paste is latency-sensitive. The browser upload path inserts the `[Image #N] <path>` reference from the `/api/upload` response, repaints the terminal, and may refresh upload/event UI, but it must not force a transcript/session-metadata refresh. Upload-directory resolution is static and authenticated-user scoped: `/tmp/yolomux.<login-user>/uploads/<session>/`; it must not inspect the session cwd, discover agents, resolve transcripts, or trigger macOS `lsof` process scans. The per-user root is verified as a non-symlink directory owned by the server's Unix uid before it is secured to `0700`; files are reserved with exclusive creation for multi-server safety. On macOS, general agent cwd discovery uses `libproc.proc_pidinfo(PROC_PIDVNODEPATHINFO)`, while Codex transcript attribution enumerates open vnode paths with `PROC_PIDLISTFDS`/`PROC_PIDFDVNODEPATHINFO`. A native lookup failure emits a deduplicated server warning and falls back to a bounded, TTL-cached `lsof` query; do not restore an unconditional all-process or all-file-descriptor scan.

Resizing is handled on the PTY slave file descriptor, then the server sends `SIGWINCH` to the tmux attach process group. The browser sends resize updates only after a debounce, except for an initial fast resize during WebSocket startup.

The browser creates panel DOM nodes for visible sessions at boot, checks that the backing tmux sessions exist, and starts terminal connections for them. Visible layout changes move existing panel nodes into slot containers. Hidden panels move back to `#panelPool`. This keeps xterm instances and WebSocket connections alive while changing layout.

The full layout is encoded in the page URL through readable `sessions`, `layout`, and `tabs` query parameters; split positions are stored as percentages in `layout`. This makes a layout reload-safe and shareable/bookmarkable without browser storage.

Transcript metadata comes from tmux pane discovery plus local process-tree inspection. YOLOmux looks for Claude or Codex processes across the session's tmux sub-windows, finds their transcript/session metadata, and exposes both per-session summaries and per-sub-window rows in the pane header, transcript tab, popovers, Tabber, and API responses. `work_graph` is the only repository/branch/PR owner. The backend-owned `agent_windows` rows are the source of truth for Claude/Codex tmux sub-window UI state, pid, and active sub-window; graph `PathObservation` edges supply touched paths and Git associations. `window_metadata[]` carries only raw per-sub-window cwd metadata, so popover, Tabber, Info Bar, and tmux sub-window bar UI must not derive Git state from it.

The transcript tab uses Server-Sent Events from `/api/context-stream`. The AI summary tab uses `/api/summary-stream`, which builds a scoped prompt from the selected session's recent transcript and streams a Codex-generated summary back to the browser. Summary provider defaults live in `settings.yaml` under `summary.*` and in the backend settings catalog; `YOLOMUX_SUMMARY_MODEL`, `YOLOMUX_SUMMARY_EFFORT`, and `YOLOMUX_SUMMARY_SERVICE_TIER` only seed defaults when they name valid catalog values.

YOLO uses `auto_approve_tmux.py` workers behind `/api/auto-approve`. The browser polls YOLO status every `paneStateRefreshMs` and reflects the active state in each pane tab.

## YO!share Transport

YO!share visual sharing uses host-rendered DOM replay. The host opens `/ws/share-host` and publishes sanitized `#appRoot` keyframes, DOM deltas, replay control replies, status, and terminal placeholder metadata. Viewers open `/ws/share-ui` to receive replay frames and share status; HTTPS write viewers may also send validated `input-intent` frames there. `/ws/share-view` remains the terminal byte stream for shared tmux sessions, and terminal content is represented in replay DOM only by placeholders that bind the existing xterm stream to host rows/cols.

Replay deltas are ordered by epoch, sequence, and base sequence. Old or already-applied deltas are stale and should be ignored without repair; missing future deltas are gaps and should request a keyframe. Replay debug/profiling output should expose both paths separately (`staleFrames` versus dropped frames/keyframe requests) plus the frame numbers needed to explain a `viewer behind` report.

The replayed DOM is inert serialized markup. Host app event listeners do not survive keyframe rebuilds or child-list delta rebuilds, so viewer-local read-only affordances such as tab hover details must be rebound by the replay shell after replay applies, or represented as host-owned replay DOM.

Write access requires HTTPS. A write viewer does not publish `layout`, `ui-state`, Finder, editor, popup, or scroll semantic state; it sends allowed input intents and waits for host replay frames for visible feedback. `shareReplay=0`, `shareSemantic=1`, bootstrap `shareReplay: false`, and `localStorage.yolomux.shareReplaySemantic=1` are temporary semantic escape hatches for diagnostics, not the normal product path.

## Webterm API

All API routes require auth unless the process was intentionally started with the local-only `YOLOMUX_TEST_AUTH_BYPASS=1` test bypass. Read endpoints accept `readonly` or `admin`, except `/api/summary-stream` because it launches Codex and requires `admin`. Mutating POST routes require `admin` except `/api/event`, which accepts readonly client telemetry. `/ws` accepts readonly users but attaches tmux with `-r` and ignores keyboard input, tmux-scroll, and resize messages. The `Route` registry in `yolomux_lib/http_routes.py` is the single owner of method/path, role, handler, body limit, group, and `share_access` (`none`, `readonly`, or `scoped-file`). Request admission, readonly filesystem scoping, and plaintext HTTP-share admission all resolve that registry policy; do not add manual API-path allowlists in server/auth code. Share-token guests are narrower than `auth.yaml` readonly users: tokens are scoped to one share and admitted only through registry-marked share page, static, replay, terminal, status, and readonly file/data routes. The test bypass never escalates a share-token request; share-token scoping still wins.

Pricing catalog policy is server-owned: `GET /api/pricing-catalog` is available to authenticated readonly/admin users and returns local catalog status plus safe source evidence; it may initialize the offline seed cache but never fetches a provider. `POST /api/pricing-catalog/refresh` is admin-only, returns promptly, and delegates the crawl to the single-flight refresh coordinator. Refresh accepts only reviewed canonical HTTPS allowlist sources, bounds response size/time, records ETag/Last-Modified/digests, and fails closed per source: a provider parser or network failure retains that provider's seed/fallback rates while independently validated official provider catalogs may activate, and the response reports every rejected source. OpenAI, Anthropic, and Google provider pages are the activation sources; OpenRouter and LiteLLM inputs are corroboration/discovery only and cannot activate a catalog by themselves. Exact provider/model aliases and effective dates are required; precedence is explicit override, accepted official rate, then packaged seed. The UI labels its result API list price because subscription, credits, enterprise terms, service tier, regional, long-context, and tool charges may differ from a user's bill.

The public filesystem package facade is the single OS-error normalization boundary. Raw `OSError` values become localizable `FilesystemError` responses before they reach single-route or batch handlers; `PermissionError` maps to 403 and other OS failures map to 500 unless a caller provides a more specific status.

Linear fallback links use the slug-less `https://linear.app/issue` base by default. Set `YOLOMUX_LINEAR_ISSUE_BASE_URL=https://linear.app/<workspace>/issue` when you want local fallback links to point at a specific Linear workspace; Linear API results keep their server-provided URL.

- `GET /api/session-metadata` returns pane, process, transcript-path, `work_graph`, and non-Git `window_metadata[]` cwd metadata. `/api/transcripts` is a compatibility alias for old clients and share links; it does not return transcript text. Claude/Codex per-window state and active status use the backend-owned `agent_windows` rows from `/api/activity` or `/api/auto-approve`; path/branch/PR association uses the canonical graph, never `window_metadata[]` or a singular session-project field.

### Background Ownership

Expensive background coordination is single-owner per `YOLOMUX_STATE_DIR`, while durable stateful work is owned by local services in separate PIDs. `STATE_DIR` defaults to `~/.local/state/yolomux`; setting `YOLOMUX_STATE_DIR=/some/path` makes every YOLOmux process that uses that path share the same coordination files, caches, locks, service sockets, and service-discovery records. The background-owner election files live under `STATE_DIR/background-owner/`: `owner.lock` is the global flock, `owner.json` is the current owner heartbeat/status record, and `generations/*.json` are live process generation records. The latest live generation owns Tabber activity refresh, session-files refresh demand, watch-root polling, and independent YO!stats metric workers; older servers become followers even if a browser is actively connected to them. Followers serve shared ready/stale snapshots and request refresh from the owner over the control socket. Token-consumer demand is a family-specific owner refresh that wakes only the token worker; there is no statsd-to-web sampling RPC. Followers must not start warmers, metric workers, or derived watch-root directory polling/listing. They still update the shared watch-root intent index at `STATE_DIR/watch-index.json` so the owner can observe connected browser interest.

### Offline token/cost history rebuild

`tools/rebuild_stats_tokens.py` is the one offline repair path for rebuilding retained YO!stats input/cache/output/cost components from Claude and Codex JSONL transcript families. It reuses `session_files.transcript_usage_atoms()` and statsd's component projector, so live collection and repair have the same event identities, cache semantics, model evidence, pricing inputs, and retention tiers. It does not recreate CPU, memory, GPU, API/SSE, latency, bandwidth, or agent-status history because transcripts do not contain those measurements.

The command fails closed. A dry run or apply refuses while a registered YOLOmux generation, `statsd`, fresh background owner, DB/WAL/SHM opener, or statsd socket remains. `--apply --stop-services` first stops every recorded server sharing the target `YOLOMUX_STATE_DIR`, then stops statsd, holds the service-registry and daemon ownership locks, rechecks that the database is exclusive, creates a timestamped SQLite backup, and performs the wipe+rebuild in one transaction. It preserves live metrics and the generated-output scalar by default; transcriptless retained buckets remain visible as explicit output-only lower bounds. `--include-live` permanently erases live-only history and cannot reconstruct it.

```bash
# Inspect the complete contract and options. A live-state dry run intentionally refuses.
python3 tools/rebuild_stats_tokens.py --help

# Stop all servers sharing the default state directory, back up, and rebuild retained token/cost data.
python3 tools/rebuild_stats_tokens.py --apply --stop-services

# Restart only the intended server after checking the JSON report and backup path.
./boot.sh 8881
```

After restart, verify one listener on the intended HTTP port, exactly one `yolomux_lib.statsd` process, a healthy auth-gated `/api/ping`, and `billable_lt_output_buckets: 0` in the rebuild report. Keep the reported `stats-history.sqlite3.backup-<UTC timestamp>` until the repaired graphs have been reviewed. Re-running the same command is idempotent at the component/event level and creates another backup before replacing the projection.

Local services share one lifecycle and RPC parent. `local_services/rpc.py` owns the versioned length-prefixed JSON envelope with optional bounded binary payload and rolling legacy newline reads. `local_services/runtime.py` owns service-side mode-`0600` Unix socket binding, peer UID checks where available, singleton locks, response correlation, and idle cleanup. `local_services/registry.py` owns client-side discovery, stale PID/socket cleanup, `sys.executable -m <module> --serve` spawning, service records under `STATE_DIR/services/*.service.json` or next to the socket, exponential crash backoff from 0.25 seconds to 8 seconds, and lease-protected idle exit. A service failure may produce stale/pending/unavailable responses, but the web process must not run a retired expensive local fallback.

Stats history requests are bounded by both caller and service. The browser scales its timeout from 8 seconds for short ranges to 30 seconds for 24 hours and keeps the explicit timeout reason for retry/backoff UI; automatic retries start at 10 seconds, double, and cap at five minutes. A timed-out current-protocol RPC is never replayed through the legacy fallback, because replay would duplicate queued work during overload. The elected app owns one non-overlapping deadline worker per metric family: CPU at 1 second, Agent Status and GPU at 10 seconds, system memory/hardware at 60 seconds, and transcript tokens at 10 seconds while watched or 60 seconds while idle. Each worker collects and durably merges only its partial record, then publishes SSE; a slow or failed family advances/skips only its own deadline and cannot back off CPU. API handlers read the latest cached CPU and durable history without invoking collectors. Family writes take the live statsd socket directly, skip synchronous compaction/rollup rebuilding, and start a daemon only for a structured absent/refused-socket error; a busy timeout is not process-death evidence. CPU's durable write deadline is 0.9 seconds. The single SQLite/RPC owner services queued clients before maintenance, round-robins one token/cost/rollup maintenance family after a real accept-idle turn, and persists historical token atoms one at a time so maintenance cannot monopolize the listener. Token demand wakes only the token worker, including through a follower-to-owner refresh, and a wake during an active scan is coalesced rather than starting back-to-back work. `statsd.sampler_families` exposes cadence, alive/running state, attempts, successes, failures, runtime, last success/failure, and late/missed counts per family; the legacy aggregate sampler fields are derived from those records. Token recovery/migration remains isolated from CPU while the statsd listener retains sole SQLite ownership. Browser coverage is interval- and resolution-aware: a partial live-tail response never grants infinite coverage, so durable earlier CPU buckets stay requested and displayed while the shared in-chart loading state covers an older-prefix fetch. Browser history uploads are chunked under both 1,000 records and 96 KiB, below the route's 128 KiB body limit. An `ack_only` upload merges the batch and returns only the durable sequence; it must not build a retained-history response that the caller will discard or that can overflow the 256 KiB metadata frame. `tests/test_statsd.py::test_statsd_cpu_write_stays_responsive_during_atom_spool_maintenance`, `tests/test_app.py::test_statsd_fast_status_samples_continue_while_one_token_worker_is_blocked`, the per-family cadence/wake/stale-generation/API-cache tests beside it, sampler-family diagnostic coverage, `tests/test_browser_layout.py::test_debug_graph_wider_range_fetches_and_paints_older_history_after_inflight_poll`, and the YO!stats Node suite cover those boundaries.

Protocol 20/schema 3 records bounded coverage intervals atomically with metric buckets. Adjacent sampler owner generations remain distinct, while a schema-2 read-only database derives intervals from retained gaps without mutation. Prefix, suffix, middle, multiple, and empty shapes are valid data: each chart uses its own family's intervals, Agent Status carry-forward ends at an epoch boundary, and an absent token sample stays uncovered rather than becoming a zero. The compatibility min/max envelope is descriptive only and cannot prove that a middle interval is covered.

The web process CPU budget is below 10% when idle and below 25% with YO!stats visible on the standard macOS development host. The existing one-second CPU sampler owns enforcement: more than 30% process CPU sustained for five minutes emits one deduplicated `stats-cpu` warning with the three largest endpoint/background compute owners, and the System tab exposes current/budget/sustained values. This adds no monitoring timer. Metadata refreshes are process-wide single-flight: concurrent readers share one real status/signature check, and unchanged repositories reuse the full Git metadata snapshot instead of repeating branch, remote, worktree, and inventory subprocesses. A dirty worktree or Git HEAD/index/config/ref signature change rebuilds immediately. Concurrent Finder refresh owners may bypass the completed listing TTL with `fresh`, but identical in-flight directory listings still coalesce into one `/api/fs/batch`; explicit user freshness is preserved without multiplying enumeration.

Repeated visible-client token demand may shorten a stale 60-second idle deadline, but demand inside the active 10-second cadence is coalesced even after the previous scan has completed; UI polling therefore cannot create a token-scan wake storm.

Browser/client metric uploads use the same direct-first write path and enqueue both rollups and retention instead of running full-store compaction in the request. Retention freezes a pass epoch, finds only expired or mis-tiered candidates, and atomically processes one row per idle maintenance turn; concurrent live writes schedule the next epoch without restarting the current pass. This preserves the graduated 24-hour tiers and exact browser/server totals without letting a once-per-minute upload block the one-second CPU writer.

### YO!stats contract budgets

The deterministic backend contract subset enforces production-facing history budgets against the real retained-tier schema (2,700 source slots) with clients, server processes, host metrics, eight Agent/Model series, and billable component dimensions. A 1-hour request must finish the `statsd` SQLite/aggregation/encoding path within 1 second; a 24-hour request must finish within 2 seconds. Both return at most 360 main points, preserve exact Agent/Model/cost totals and complete coverage, stay below 4 MiB uncompressed JSON, and stay below 768 KiB with the same gzip representation used by HTTP. History response aggregation groups cost dimensions incrementally and projects each output bucket once; do not restore per-source rebuilding of the growing component list. These are request budgets, not permission to retry tightly: callers must preserve the response `coverage` envelope and use bounded backoff when the owner is unavailable. The live service deadline remains short enough that a slow or unavailable `statsd` cannot block the HTTP handler indefinitely; browser retry cadence and timeout policy are documented immediately above.

Focused service tests are split by ownership. `tests/test_local_services_rpc.py` covers framing, rolling legacy reads, peer UID handling, and request correlation. `tests/test_search_indexer.py` covers `indexd` coalescing, socket protocol, leases/idle exit, stale record recovery, common status, and read-only server snapshots. `tests/test_statsd.py` covers SQLite WAL transactions, rollback, retention/cursor history, pre-encoded binary responses, registry spawning, and legacy imports. `tests/test_jobd.py` covers bounded spawn workers, registered task payload/result bounds, priorities, stale-generation supersession, cancellation, queue saturation, timeouts, crash recovery, and request-path no-spawn behavior. `tests/test_approvald.py` covers target worker start/status/stop, duplicate lock ownership, event callbacks, profile, and drain. `tests/test_app.py::test_runtime_report_exposes_shared_local_service_lifecycle_clients` and `tests/test_app.py::test_runtime_report_payload_reports_owner_cache_endpoints_events_and_transcripts` pin the runtime-report service schema.

Sampled background-refresh event logging keeps `count` and `last_emit_count` in one typed record keyed by `(event_type, role)` under one lock. First/every-N emission and `suppressed_since_last` are one transaction; do not split those counters into parallel maps.

In-process caches follow the same record rule. `StatsSampleRecord` owns the process/system CPU baselines and the short-lived sampled payload; `TranscriptsPayloadCacheRecord` owns transcript payload age plus its stale-while-revalidate flag; and `TabberActivityCacheRecord` owns activity payload age, the source signature that classifies that payload, per-session source signatures/rendered rows, and per-key refresh futures. Cache get/set/invalidate/refresh paths must update these records under their existing locks rather than adding parallel timestamp, payload, signature, row, future, or refreshing attributes. The Tabber warmer thread is a separate long-lived scheduler and stays outside the cached-value record. A warmer tick reuses unchanged session rows, rebuilds only invalidated rows, removes vanished sessions, and returns the existing payload without an all-session rebuild when the complete source signature is unchanged.

### Background CPU measurement

`python3 yolomux.py --print-runtime-report` reports the elected owner, accepted/coalesced refresh counters, local service status, and a rolling 60-second performance summary. The JSON top level includes `state_dir`, `owner`, `refresh`, `caches`, `search_index`, `local_services`, `top_endpoints`, `top_background_work`, `top_event_types`, `client_events`, `chat`, `largest_active_transcripts`, and `transcripts_cache`. `local_services.services[]` is ordered as `indexd`, `statsd`, `jobd`, `approvald`; rows report safe fields such as `service`, `pid`, `started_at`, `version`, `socket`, `healthy`, `clients`, `queues`, `active_task`, `cache`, `last_success`, `last_failure`, `restart_backoff_seconds`, `generation`, `target_count` where applicable, bounded registry `record`, and `resources` (`cpu_percent`, `rss_bytes`). `local_services.totals` reports counted worker processes plus combined CPU/RSS. It must not include prompts, transcript bodies, secrets, or unbounded paths. Session-files phase rows have these meanings: `phase:transcript-attribution` maps transcript events to touched files; `phase:repository-discovery` resolves session paths to repositories; `phase:git-signature` reads the complete invalidation identity; `phase:git-snapshot` builds or reuses repository-wide diff/ref facts; `phase:session-merge-render` combines shared Git facts with session attribution; and `phase:cache-serialization` signs and writes the result. A `phase:git-snapshot` cache hit must report zero Git compute. Phase details are bounded and omit composite values so runtime reports do not become another payload or secret store.

For an accepted before/after comparison, restart only the dev port being measured, confirm that port is `owner.current_owner.port`, keep the same visible Tabber/YO!info workload, save `--print-runtime-report` output under `/tmp`, and run `pidstat -u -p <owner-pid>,<peer-pid>,<peer-pid>,<peer-pid> 1 60` for both windows. Compare the owner's CPU against the mean of the three non-owners as well as absolute CPU, because live agent output changes every process's baseline. For each role/surface, compute seconds per minute as `count * compute_ms_avg / 1000`; also compare `compute_ms_max`, accepted/coalesced counts, and cache status. A valid freshness check must change a real worktree input, observe the complete session-files payload within the documented refresh interval, remove it, and confirm stale-while-refresh never substitutes an empty Differ/Tabber payload.

The 2026-07-03 equal-duration live sample used one dev owner. Before the change, owner CPU was 33.66% versus 27.66%, 26.31%, and 25.05% for the non-owners; after the change, overall workload was higher at 41.43% versus 41.17%, 36.23%, and 34.12%, but the owner premium over peer mean narrowed from 7.32 to 4.26 CPU points (42%). In the comparable rolling report, session-files `payload` work fell from 3.61 to 2.54 CPU-seconds/minute and `cache-entry` work fell from 5.31 to 3.38 CPU-seconds/minute; maximum payload time fell from 2367ms to 752ms. A visible unchanged Tabber refresh reused the existing snapshot in 0.904ms. An untracked-file probe appeared in the live owner payload in 0.79 seconds (621 to 622 rows), and its removal served the complete 621-row last-known-good payload while the replacement refresh ran. This workload realized the expected session-files reduction and a smaller owner-premium reduction than the older 105% leader sample because the accepted pre-change owner was already only 33.66% and live agent CPU rose during the post-change window.

The required final cold-restart stress window deliberately kept Tabber visible every 10 seconds. The owner was at 50.72% CPU versus a 35.81% non-owner mean, and the phase rows explained why this window did not retain the warm-cache improvement: first-use transcript attribution for active transcripts up to 140.9 MB consumed 70.8 CPU-seconds/minute, while shared Git snapshots consumed 2.3 seconds/minute. Tabber consumed 4.3 seconds/minute including its one cold build; 7 of 8 refreshes were unchanged-cache hits, all 11 session rows appeared by the first 10-second follow-up, and no later response lost a row or reported an error. Treat this as a separate cold transcript-attribution target; it does not justify undoing the shared Git, per-row reuse, or single-flight paths.

Transcript attribution now resumes large Claude and Codex JSONL scans from schema-versioned records under `STATE_DIR/transcript-scan-cache-v1/`. One provider-neutral owner validates parser version, device/inode, resolved path, options, a bounded first-line digest, parsed-tail bytes, truncation, and replacement before reusing an offset; invalid records reparse from byte zero. Records contain only derived path markers and bounded usage counters, never raw transcript lines, command bodies, or patch text. Files below 64 KiB remain memory-only because reparsing them is cheaper than durable fsync; large initial scans persist immediately, then active append state checkpoints at most every 30 seconds or 256 KiB. Per-record atomic writes, newer-offset protection, 0600 files in a 0700 directory, and count/64 MiB eviction keep rolling multi-server use bounded.

Background worker generations also have one typed owner. `SessionFilesDiskPruneRecord` owns the prune deadline, in-flight state, and completion result. `TabberActivityWarmerRecord` owns the warmer thread, running state, and visible-consumer deadline. `MetadataWarmRecord` owns the network metadata thread plus its demotion stop event. The elected app runs independent YO!stats family collectors while `statsd` alone persists their partial records; auto-approval target loops run in `approvald`; CPU transforms run in `jobd` executors. Metadata warming checks its demotion event between session lookups, so a demoted process cannot continue owner-only GitHub/Linear work. Worker loops may clear their record only if that exact thread/record remains current.

Global durable state must use one shared storage/reader path per state family, not parallel per-server JSON files. Current durable globals are Preferences in `~/.config/yolomux/settings.yaml`, YO!agent jobs in `~/.config/yolomux/state.json`, cross-server red/yellow tmux AI acknowledgement state in `STATE_DIR/tmux-AI-status.json` under `attention_acks`, YO!stats history and token/process/host rows in `STATE_DIR/stats-history.sqlite3`, watch-root intent in `STATE_DIR/watch-index.json`, Quick Open indexes under `STATE_DIR/search_index/`, session files under `STATE_DIR/session-files-cache/`, activity under `STATE_DIR/activity-cache/`, YO!chat in `STATE_DIR/yochat.sqlite3` plus `STATE_DIR/yochat-history/YYYY-MM-DD.jsonl`, service discovery under `STATE_DIR/services/`, and reconstructible model-price metadata in `YOLOMUX_CACHE_DIR/model-pricing/pricing.sqlite3` (default `~/.cache/yolomux/model-pricing/pricing.sqlite3`). The pricing catalog imports its packaged `yolomux_lib/data/model_pricing.json` seed without network access; source caches and its cross-process lock live beside that database. Legacy `stats-client-history-v4.json`, `stats-client-history-v3.json`, `stats-client-history.json`, and `tmux-AI-status.json` `stats_history` data are import sources for `statsd`; normal durable stats writes go to SQLite. Browser latency, API/SSE counts, bandwidth/sec, disconnected time, selection/hover zoom state, and other per-client graph details are scoped to the browser `client_id` and merged with shared server fields only in the response payload.

YO!stats retains 24 hours with graduated compaction: 0–30 minutes uses one-second buckets, 30 minutes–2 hours uses ten-second buckets, 2–4 hours uses one-minute buckets, 4–8 hours uses two-minute buckets, 8–12 hours uses five-minute buckets, and 12–24 hours uses ten-minute buckets. A fully populated history family therefore has at most 2,700 time slots. `statsd` is the single writer for row-based SQLite WAL storage in `STATE_DIR/stats-history.sqlite3`; it stores durable bucket rows plus normalized client, process, agent-rate, host-metric, and metadata tables, then returns cursor/range results as bounded metadata or validated pre-encoded JSON bytes. During a rolling multi-worktree update, registry health requests identify the caller's expected service protocol. A newer stats daemon reports the pinned compatibility version to legacy callers and its current version to current callers, so a current port can replace an old owner once without legacy ports reclaiming the socket and rewriting newer fields. Migration combines every available compatible legacy journal from newest to oldest: current-version rows win in overlapping time ranges, older versions fill only uncovered ranges, legacy bucket durations compact into the current tiers, and duration-scaled sanity checks discard counters corrupted by the old row-duplication bug. The SQLite `migrated_versions` marker prevents a restart from importing the same rows twice. Disk and heap use scale with the number of populated client/process records inside those bounded time slots, not only with elapsed time. A browser retains the ranges it has fetched until the same 24-hour tier compactor removes them. Use the `statsd` database file size plus bucket/client/process/agent/host row counts for storage audits, and measure `statsd` RSS separately from the web process.

Usage atoms add provider/model, input/output direction, modality, cache role, unit, effort, execution-source/thread identity, event identity, timestamp, and completeness to the same retained stats buckets. `statsd` projects effective-dated catalog rates into integer micro-USD component totals; it keeps the existing generated/output token projection unchanged. Recoverable transcript usage can be backfilled only inside the retained 24-hour window. Missing or legacy output-only transcript evidence remains partial, so a Cost summary must use a lower-bound estimate rather than fabricate input/cache prices; data already compacted out of the retention window cannot be reconstructed.

`GET /api/stats-sample` accepts `history_start` (inclusive epoch seconds), `history_end` (exclusive epoch seconds; zero means the live tail), `history_resolution`, and `history_max_points`. Widening a graph requests only `[new_start, previous_loaded_start)`, while a bounded older response reports its global latest sequence separately and never advances the browser's steady live cursor. `history.coverage` reports requested, available, and covered bounds; completeness/older continuation; effective resolution and point budget; source/returned record counts; and safe plus latest cursors. One encoder owns client, global-owner, per-process CPU, status, and token aggregation. It preserves every source record when the mixed-resolution input fits the requested budget and coarsens only when the encoded record count exceeds that cap; token data appears in either the main records or the dedicated token response, never both.

Dynamic JSON uses the same response-level gzip negotiation owner as static text assets. Bodies at or above the shared threshold honor explicit `gzip`/wildcard q-values, emit `Vary: Accept-Encoding`, and keep HEAD/error semantics. HTTP endpoint samples record uncompressed bytes, representation/wire bytes, JSON encode time, compression time, and write time; the stats endpoint also records client-history merge, token-consumer, sample, compact, history-encode, and total app-build phases. Browser debug counters separately record history fetch, parse, apply, render, paint, no-data sweep, and total loading duration; the stats fetch remains excluded from its own API/SSE chart.

The stats readiness state is one browser record: `idle`, `loading-initial`, `loading-older`, `retrying`, `ready`, or `error`. Before the first accepted sample, the graph renders exactly one localized `Waiting for server stats...` message; `No events yet` is valid only after a successful empty response. Boot, SSE ready/reconnect, visibility recovery, and periodic refresh call the shared snapshot coordinator. A concurrent call joins the current request; a stale/reconnected resource starts exactly one refresh after its freshness/revision check. Do not add endpoint-local startup fetches or duplicate loading/error state.

The per-request HTTP profile is initialized at dispatch and is never stored on a reusable handler object. It records accept/queue wait, handler compute, JSON encoding, compression, write time, uncompressed/representation/wire bytes, plus bounded owner phase counters. Shared owners increment candidate discovery, transcript bytes/JSON decode/cache outcomes/cwd resolution, Git, disk-cache maintenance, activity, stats encode/compress, filesystem policy/list/identity/Git, generation/PID, and watch-root signature counters. Routine topbar responses do not carry these diagnostics; fetch them only through the bounded diagnostics/report route.

Search-index files live under `STATE_DIR/search_index/`. Each indexed root gets digest-named files: `<digest>.sqlite3` for entries, `<digest>.manifest.json` for cheap status/count metadata, `<digest>.lock` for the per-root build lock, `<digest>.tomb` for removals, plus SQLite WAL/SHM sidecars when SQLite needs them. Servers read the small manifest and committed SQLite snapshots; `indexd` is the only process that applies row deltas, full rebuilds, and unindex writes. Quick Open queries only consume ready/stale data and never schedule a TTL rebuild. The watcher coalesces changed paths per root, `indexd` incrementally replaces only those subtrees, and the 30-minute full safety scan is independent of query traffic. In-memory entries, persisted entry count, and estimated/actual cache bytes are bounded; truncated or over-budget roots remain usable as explicit partial in-memory indexes and are not persisted. Recoverable SQLite writes are optional and debounced, while locks, tombstones, settings, and user data keep their existing durability. The default skip set rejects generic generated/dependency/build trees; `file_explorer.index_exclude_paths` adds machine/project-specific rules, one per line: plain absolute/home-relative subtree paths, `glob:<root-relative glob>`, or `regex:<expression>` against root-relative POSIX paths. Invalid regex rules are ignored, and the complete effective rule set participates in the index signature so policy changes rebuild rather than reuse incompatible rows. `/api/fs/index-status` and the runtime report expose build/full/incremental counts, last duration, scanned/ignored entries, dirty subtrees, truncation, cache bytes, and write bytes. The browser keeps `/api/background/status` fresh so the top-right ownership chip can show whether this connected process is the background owner/follower, while `--print-runtime-report` shows the actual `indexd`, `statsd`, `jobd`, and `approvald` service rows. `background_owner_changed` pushes update that status after election changes, and the status is refreshed on EventSource reconnect and manual refresh.

If a background owner dies or stops heartbeating and its process lock is gone, a follower heartbeat observes that the old generation is no longer live, acquires `owner.lock`, writes a new `owner.json`, adopts warm session-files, Tabber activity, and search-index disk snapshots into memory, and publishes `background_owner_changed` so connected clients update without waiting for a poll. If the owner heartbeat/control socket is stale but the flock is still held by a live process, a follower must not steal ownership because that would split coordination; it may run only the existing bounded immediate-response fallback and then return to follower mode. Local service ownership remains guarded separately by service locks and SQLite single-writer rules, so owner failover must not create a second `statsd` or `indexd` writer.

YO auto-approve is not owned by `background-owner`. `approvald` uses per-target flock files under `STATE_DIR/locks/auto-approve-<target>-<digest>.lock`, where the target is usually the agent pane target. That means the background owner can be one YOLOmux process while YO-on for a specific tmux pane is held by `approvald`; the auto-approve API probes and takes over those per-target locks separately from background ownership and indexing.

- `GET /api/auto-approve` returns YOLO status for all sessions.
- `GET /api/auto-approve?session=project1` returns YOLO status for one session.
- `POST /api/create-session?agent=claude`, `POST /api/create-session?agent=codex`, or `POST /api/create-session?agent=term` creates the next numbered tmux session with the selected agent, capped at `MAX_YOLOMUX_SESSION_TABS` (99) visible sessions.
- `POST /api/ensure-session?session=project1` checks that a tmux session still exists.
- `POST /api/auto-approve?session=project1&enabled=1` enables or disables YOLO for a session.
- `POST /api/tmux-next?session=project1` moves the session to the next tmux sub-window.
- `GET /ws?session=project1` attaches a browser terminal to tmux.

## Wall API

The read-only wall (`tmux_wall.py`) exposes:

- `GET /api/snapshot` returns the current six-pane dashboard payload.
- `GET /api/transcript?target=project1:0.0&lines=2000` returns one tmux pane transcript.
