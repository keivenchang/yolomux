# YOLOmux — Development

Conventions, architecture, build/test workflow, restart workflow, and API notes for contributors live here. End-user/operator docs live in [`README.md`](../README.md). AI-agent behavior rules live in [`AGENTS.md`](../AGENTS.md).

## Setup

Standard install via the Makefile (wraps `pip install -e .` + xterm.js assets + the static bundle build):

```bash
make setup        # pip install -e ".[yoagent]"  +  npm install (xterm.js)  +  python3 tools/static_build.py
make dev          # same, plus dev/test deps (pytest-xdist) — see `make help`
```

`pyproject.toml` is the source of truth for Python deps and the floor (`requires-python = ">=3.10"`), so pip checks the Python version and resolves the runtime dependencies plus the `yoagent` / `dev` extras. `make setup` and `make dev` run the same Python-floor preflight before invoking pip. Use editable installs; this repository deliberately has no mirrored `requirements*.txt` files.

```bash
pip install -e ".[yoagent,dev]"      # deps + the `yolomux` command + agent SDKs + pytest-xdist
```

On externally managed system Python installs (PEP 668), create and activate a virtual environment first (`python3 -m venv .venv && . .venv/bin/activate`) instead of forcing `--break-system-packages`.

## Project Conventions

### Version bump policy

Runtime `YOLOMUX_VERSION` lives in `yolomux_lib/infra/common.py`; the static package-version literal in `yolomux_lib/common.py` must always match it. Both are bumped only when publishing to `origin/main`. LOCAL `cps` / `yolo-cps` lands work on local `main` with no version bump and no push; ORIGIN / REMOTE `cps` rebases onto `origin/main`, increments the patch segment, folds both matching literals into the work commit, and includes `Version: 0.4.N` in the commit message body. The self-updater compares `YOLOMUX_VERSION` on `origin/main`, so SHA-only published commits do not cue update notifications, but local integration commits must not create fake versions. `tests/test_common.py::test_package_and_runtime_versions_match` rejects drift between the two required literals.

To find the latest committed version before ORIGIN publish: `git show origin/main:yolomux_lib/infra/common.py | grep YOLOMUX_VERSION` and add 1 to the patch segment, then apply that same version to `yolomux_lib/common.py`.

### Timing constants

JavaScript and CSS timing constants in `static_src/`, `static/yolomux.js`, and `static/yolomux.css` are split by purpose:

- UI / popup / display / animation timings use round whole numbers, for example `300`, `1000`, `1550`, `10000`, `20000`.
- Backend polling / refresh intervals SHOULD prefer slightly-staggered (often odd) values, for example `1257`, `3001`, `5003`, to spread client requests across ticks instead of piling up on the same one. This is a preference, not an invariant — several shipped defaults in `yolomux_lib/workspace/settings.py` are still round (`event_log_refresh_ms: 5000`, `server_event_poll_ms: 850`, `server_directory_event_poll_ms: 3000`); nudge them toward staggered values when you touch that area, but nothing enforces it today.

UI durations are perceived by users and read as deliberate at round values. When a request says "make it 1000ms", a UI duration keeps `1000`; a backend poll may use a nearby staggered value like `1003`.

Recurring browser work must register through `resetRuntimeInterval()`/`clearRuntimeInterval()`. The named recursive scheduler waits for an async callback to settle before starting the next full delay, so slow YO!stats, share-status, or share-geometry work cannot overlap or degrade into back-to-back requests. Each live scheduler record exposes its source-of-truth class, last run, and whether the callback produced useful work or `no-change`; do not add direct application `setInterval()` loops or a parallel timer registry.

The source-level `runtimeIntervalCatalog` is the authoritative classification for every production `resetRuntimeInterval()` owner. The shared taxonomy distinguishes `poll:no-change`, `sample`, `lease`, `external-reconcile`, `repair`, `fallback`, and `local-display`. `latency` is the sole browser health sample; it also supplies the bounded YO!stats client-health observation, so `debug-client-health` is retired. `chat-relative-times` is local display only; `share-geometry-digest` is repair work; `events-fallback`, `auto-approve`, `debug-stats`, `file-index-building`, Tabber fallback, and socket-unhealthy `share-status` are fallbacks; `file-index-refresh`, `debug-system`, and `debug-logs` remain visible-surface `poll:no-change` reconciliation until their owners publish revisions. A healthy EventSource suppresses normal discovery reads; disconnect, visibility recovery, a missed revision, or a terminal stream failure permits one bounded, keyed repair rather than `refreshAll()`. Browser-to-server stats-observations upload, background/typing leases, the one-second host sampler, SSE heartbeats, native-watch safety scans, and external PR/update checks are not browser runtime-interval entries; they retain their respective source owners. Do not remove a catalogued interval merely because SSE exists: remove it only when the named owner publishes an equivalent revision and a focused regression proves the fallback/repair behavior remains correct.

Three server cadences are deliberately always on: background-owner's 1 Hz heartbeat makes failover prompt despite two small state writes per second; the CPU family samples and publishes at 1 Hz to preserve continuous history; and the stats lease renews at 1 Hz to keep sole-writer ownership intact. Change any of those cadences only with an operator decision, not as a browser-idle optimization.

### Responsive UI sizing

Do not hard-code layout capacity around one browser window, OS, zoom level, or font rendering. For menus, panes, dropdowns, tab lists, and editor/viewer surfaces, prefer intrinsic sizing, flex/grid allocation, percentages, viewport units, and shared CSS variables over fixed pixel width buckets. Fixed pixels are acceptable for hairlines, icon glyphs, and the base spacing-token definitions, but component gaps, padding, and margins from 1–12 px must consume `--space-1` through `--space-12`; negative token spacing uses `calc(-1 * var(--space-N))`, never invalid `-var(...)`. If JavaScript must set a size, derive it from measured DOM content and clamp it to the current viewport/container.

## Source Layout

### Runtime And Layout Names

Use explicit names whenever runtime tmux metadata and browser layout appear together: `TmuxSession`, `TmuxWindow`, and `TmuxPane` describe tmux objects; `YOPane` and `YOTab` describe the browser layout. `TmuxPaneInfo` is the backend record for a `TmuxPane`; do not introduce a new unqualified `PaneInfo` use, because it is ambiguous with `YOPane`.

The main server entry point is `yolomux.py`, which delegates to `yolomux_lib/cli.py`. Request routing lives in `yolomux_lib/server.py`, application coordination lives in `yolomux_lib/app.py`, local Unix RPC framing/lifecycle lives in `yolomux_lib/local_services/`, visible Claude/Codex tmux pane control lives in `yolomux_lib/tmux/agent_tui.py`, shared Claude/Codex structured communication lives in `yolomux_lib/agent_comms/` (Codex app-server, Claude stream-json, and normalized stream events), and shared helpers live in smaller modules such as `metadata.py`, `sessions.py`, `session_files.py`, `transcripts.py`, `uploads.py`, `events.py`, `websocket.py`, `approvals.py` (the approval-prompt detection pipeline), `runtime_env.py` (one dependency-light PATH healer used by both the server and Codex child environment), `atomic_file.py` (cross-process file lock + atomic write), `cache.py` (the shared `TtlCache`), and `activity.py` (the per-session/window user+agent activity ledger). The current statsd service entry point is `stats_current/service.py`; it owns the sole YO!stats SQLite writer, asynchronous four-layer materializer, and exact snapshot/delta cache. Other local service entry points are `search_indexer.py` for `indexd` Quick Open SQLite ownership, `jobd.py` for bounded stateless CPU tasks, and `approvald.py` for target-keyed YO auto-approval workers. The elected `app.py` process owns independent YO!stats collection workers and talks to services through `StatsCurrentClient`, `SearchIndexerClient`, `JobClient`, and `ApprovalClient` rather than importing service internals. In the current-only architecture, web processes never open or aggregate the stats database; they forward current binary RPC snapshots produced by statsd. Recursive tree discovery is never request-path/API-process work: indexed-repository discovery is a maintenance-priority `jobd` task, and `app.py` serves the last completed snapshot while a refresh is pending.

### `TmuxWebtermApp` extraction direction

`TmuxWebtermApp` remains the HTTP/tmux coordination boundary. Do not split it by moving arbitrary method prefixes or by making a second app-shaped service. Extract one domain facade only when a change already owns that domain's behavior and can give the facade explicit dependencies, focused tests, and a narrow public contract.

The first candidate is chat, not share: the `chat_*` methods already delegate storage and paging to `ChatService`; their remaining coordination is the named `chat_messages_changed`/`chat_typing_changed` client events and the one YO!agent reply request. A future `ChatCoordinator` may receive `ChatService`, `YoagentController`, and an injected event publisher, while routes keep calling the same app-level methods during that change. Prove the extraction with `tests/test_app.py` and the chat route/event tests, including one created message, one typing transition, and one YO!agent reply. Do not move share in that change: its token lock, active session checks, replay sockets, terminal upstreams, and HTTP scope enforcement must remain one owner until a share-specific contract can name and test each dependency.

After chat, consider one facade per separately tested domain (`share`, `stats`, `session`/tmux, YO!agent, or filesystem watching). Each extraction must move its state records with the methods that own them, inject only named collaborators, preserve route payloads and lifecycle order, and delete the old app-owned path in the same change. Never run a broad mechanical split, retain compatibility aliases, or extract more than one domain in one change.

Frontend source for the interactive UI lives in ordered partials under `static_src/js/yolomux/` and `static_src/css/yolomux/`. Generated served assets are `static/yolomux.js` and `static/yolomux.css`; do not edit those directly except as generated outputs. Python keeps only the small HTML shell in `yolomux_lib/web.py`, plus bootstrap JSON and versioned static asset URLs. The main app's non-tmux tab types are centralized in the `TAB_TYPES` registry in the JS source partials. The read-only wall has its own frontend files, `static/tmux-wall.js` and `static/tmux-wall.css`, so `tools/tmux_wall.py` stays focused on tmux capture, JSON endpoints, and Server-Sent Events.

Detached pane pop-outs are owned by `static_src/js/yolomux/90_pane_popout.js` and the `/pane-popout` placeholder route. Add support through `TAB_TYPES.canPopout`, `openPopout`, or `popoutRenderer`; do not create per-pane popout registries or document writers. Phase-1 popouts are rendered snapshots, so live terminal/transcript and interactive command panes must remain disabled with an explicit `popoutDisabledReason` until they have a live bridge design.

### Central State Containers

`agentWindowActivityRecords` owns per-agent-window transition state; `sessionStatusRecords` owns per-session status and notification state; `toastRecords` owns toast nodes/timers; `dragState` owns one active drag; `paneViewState` owns durable pane scroll; `layoutUrlState` owns URL application; `clientEventTransportState` owns the browser SSE lifecycle; `fileExplorerDirectoryRecords` owns directory-observation signatures; `fileExplorerFsResourceRecords` owns filesystem resource cache records; `shareHostConnectionRecords` owns share host sockets/queues; `StatsCurrentRuntime` owns the elected app's independent per-family collection scheduler threads, locks, wake events, diagnostics, and source caches, while statsd alone owns durable original observations, usage atoms, family folds, immutable materialization generations, pre-encoded snapshots, and exact deltas; `SessionFilesService`, `ActivityTranscriptService`, and `ClientWatchService` own their server locks/cache/work/watcher records; and `approvald` owns target-keyed auto-approve workers. Readers use their named selectors/helpers and lifecycle owners; consumer-local mirrors of the same keyed fact are forbidden.

For every refactor or new UI behavior, record `Existing parent checked: <symbol>` or `New shared parent: <symbol and reason>` plus the exact searches used. A DONE note or handoff also records the negative search proving retired copies are gone and the net non-generated line change.

The approval-prompt detection pipeline lives in `yolomux_lib/approval/approvals.py` (one shared owner: `app.py`'s read-path, the `AutoApproveWorker` act-path in `yolomux_lib/approval/auto_approve_worker.py`, and the standalone `tools/auto_approve_tmux.py` CLI all call it; the CLI re-exports it). `agent_tui.py` is the public owner for combining that pure detector state with tmux captures, cursor facts, composer draft/ghost state, transcript activity upgrades, clear, paste-submit, and submit verification. An enabled session may have one worker per detected agent-pane target; `approvald` is the single target-keyed owner of each worker, its target flock, and its session association.

## Normalized Session Work Metadata

`yolomux_lib.metadata.session_work_graph()` is the only backend owner of repository/work metadata. It serializes `work_graph.version`, a monotonic `generation`, and canonical maps for `tmux_sessions`, `tmux_windows`, `tmux_panes`, `runtime_actors`, `path_observations`, `git_worktrees`, `local_repositories`, `hosted_repositories`, `local_branches`, `pull_requests`, `linear_issues`, and `worktree_branch_activity`. The graph uses snake_case API fields; browser selectors use camelCase rendering projections only after reading the graph.

`PathObservation` links tmux/actor evidence to at most one nearest `GitWorktree`. A `GitWorktree` belongs to exactly one `LocalGitRepository`, identified by Git common-dir, so primary and linked worktrees share one local branch inventory without duplicating it. Separate clones retain separate LocalGitRepository records even when they map to one HostedRepository; PullRequests belong to the HostedRepository and associate to zero or more LocalBranches. Graph validation runs before serialization and rejects dangling IDs.

`session_to_json()` emits one authoritative `work_graph` plus non-Git `window_metadata` cwd rows. New consumers must use the graph-aware shared frontend selectors (`sessionWorkSummary`, `focusedRepositoryIdsForTmuxTarget`, `branchIdsForGitWorktree`, and `pullRequestIdsForTmuxTarget`); neither `session.project` nor `window_metadata[].git` exists in the API. A schema-valid graph, including an empty one, is authoritative. Metadata refreshes use the graph generation to reject stale results rather than wall-clock ordering.

## Static Build

After changing any frontend source file, rebuild the served single-file assets:

```bash
python3 tools/static_build.py
```

Run with `--check` before committing to verify generated assets match source:

```bash
python3 tools/static_build.py --check
```

The static build is also the CSS/source-ownership gate. Transient Claude/Codex window state has one browser owner: `agentWindowActivityRecords`, keyed by the stable window transition key. Each record owns the activity generation, stopped refresh, transition-pulse refresh, stopped acknowledgement timestamp, and temporary gray acknowledgement visual. Do not add another map keyed by the same transition key; extend the record so timer replacement, acknowledgement, and rendering cannot advance different generations.

YO!stats browser observations originate in the existing bounded `jsDebugEvents` lifecycle owner and are uploaded in privacy-safe batches through `/api/stats-observations`. The browser never converts those events into graph buckets; statsd appends original observations and materializes shared per-client averages without exposing browser identities.

The observation writer fence has one server owner: `stats_current.storage.MIN_WRITER_PROTOCOL` and `SCHEMA_VERSION` are embedded as immutable `statsWriterFence` page-bootstrap data alongside the cache-busted bundle. The uploader validates that boot-time value before sending. A missing or invalid fence stops the page-local queue without a request; HTTP 426 remains terminal for an already-loaded stale page, while an actual reload creates a fresh epoch with the new bundle and fence.

Other paired lifecycle state follows the same record rule. `sharePointerRecords` owns each remote sender's ghost node and hide timer; `uploadResultRecords` owns each session's upload entries and cleanup timer; `sessionStatusRecords` owns each browser session's state snapshot, notification throttle map, working-agent tone map, and metadata badge pulse map; `toastRecords` owns each toast node and numeric-ID removal timer; backend `metadata_badge_records` owns each session's signature and pulse deadlines. Watched-PR notification throttles are global and must stay outside session records. Extend these records and their shared lifecycle helpers instead of adding another map keyed by the same sender/session or putting a composite/numeric key through direct session rekeying. Durable session status and upload records migrate on rename, but transient toast records are discarded on detach or rename because their callbacks capture the original session identity. Pending pane-view captures are also transient: `detachSessionUi()` is their only session-lifecycle cleanup owner, while durable `paneViewState` remains in the clear/migrate path. Metadata badge persistence serializes signatures only and writes the legacy pulse key as empty because live pulse deadlines must not survive restart.

Non-keyed frontend transactions follow the same rule. `selfUpdateReloadState` owns one post-update reload attempt, `serverWatchRootsState` owns the debounced watch registration pipeline, `diffRefPopoverState` owns the reusable ref-picker node/input/items/cursor/listeners, `layoutUrlState` owns pending deep-link application and URL refresh scheduling, `fileExplorerSyncState` owns Finder Sync target/application generations, `fileExplorerSessionFilesState` owns Finder/Differ payload/signature/loading/request generations, `tabberActivityState` owns Tabber activity request/application readiness, `commandPaletteState` owns the palette node/query/cursor/rendered items, `fileQuickOpenState` owns Quick Open scope/results/debounce/abort identity, `backgroundOwnerStatusState` owns owner-status request generations, `activitySummaryState` owns activity payload/request generations, `transcriptMetadataState` owns session metadata payload/load/request generations, `searchHistoryState` and `runHistoryState` own their query/result generations, `yoagentConversationState` and `yoagentJobsState` own YO!agent hydration generations, `yoagentChatState` owns active chat/action work plus queue/composer/error state, `yoagentStartupState` owns prewarm and frozen startup-info state, `infoPanelRenderCache` owns signature/HTML together, and `clientEventTransportState` owns the SSE source/connectivity/dispatch/reconnect lifecycle. Reset or replacement paths update the whole record rather than leaving timer, pending-option, input, cursor, promise, generation, frame, or cached-HTML scalars behind. External SSE/direct payloads invalidate older HTTP generations before applying; stale success, failure, and `finally` paths may not replace data or clear a newer request handle. Semantic CSS aliases that intentionally share paint reference one stable palette token with `var(--owner)`; they do not repeat the owner's raw hex under a second token name or point a scoped component override back at itself.

Backend YO!agent prewarm and visible-startup state follows the same generation rule through one `YoagentPrewarmRecord`. Clear conversation increments the startup generation and marks reset in progress before deleting CLI/conversation state; a visible startup cannot begin during that clearing window. Startup persistence checks and writes under the same lock, stale work publishes a stopped/aborted completion and returns the current conversation, and cleanup clears the active generation only when it still owns that generation.

Auto-approve status no longer caches in the web process at all: `statusd`'s `PersistentStatusService` is the sole owner of the same generation rule (snapshot bytes, generation, and `invalidation_reason` advance together under one lock; a build that fails after invalidation retains and serves the prior complete snapshot as stale rather than publishing a partial one). Web routes only forward those bytes (`auto_approve_status_bytes`); the retired in-web `AutoApproveCacheRecord`/`run_auto_approve_cache_refresh` TTL-refresh thread was deleted once every consumer moved to the daemon-owned snapshot. Finder filesystem list/info caching likewise uses one bounded `fileExplorerFsResourceRecords` map keyed by normalized resource type and path; payload, timestamp, request, and generation share a record, so rename/reindex/push invalidation and newer fresh reads retire older completions instead of letting them repopulate stale state.

The client-event watcher uses one `ClientEventWatcherRecord` per generation. The record owns the main worker, directory-poll worker, wake/stop events, and every poll deadline. Stop retires the whole record; restart creates fresh events, and main/directory cleanup may mutate app state only when both the record and worker identity still match. Never split a new watcher phase into a parallel app attribute or reuse a retired generation's events.

### Demand-aware browser transport

`clientEventTransportState` is the one browser owner for client-event demand. Its descriptor is recomputed from document visibility, active pane tabs, active subtabs, and system-notification delivery, then opens `/api/client-events` with only the required channels: `core`, `status`, `files`, `activity`, `transcripts`, `yoagent`, or hidden-page `attention`. A hidden page with system notifications disabled closes the stream and unregisters server watches; a hidden page with system notifications enabled retains only compact attention delivery. Pane, layout, Finder-mode, and notification-setting changes must resync this descriptor instead of opening feature-local streams or timers.

The backend `ClientEventBroker` keeps per-subscriber channels and aggregates producer demand across every connected browser. Filesystem/transcript/activity polling, auto-approve/tmux status polling, watched-PR polling, and YO!agent job polling run only while at least one subscriber needs their channel. Runtime diagnostics expose bounded sanitized client records and published/delivered/filtered SSE event and byte counters by type. HTTP response counts/bytes remain under the existing endpoint performance metrics; do not create a second accounting path.

For a traffic trace, clear JS Debug, hold one pane/document state for a fixed interval, copy the JS Debug report, and compare endpoint counts/bytes and SSE type counts against the runtime report's `client_events` snapshot. Test at least visible terminal-only, visible Finder/Tabber/YO!agent, hidden with system notifications off/on, no active share, and two browsers with disjoint panes. A hidden surface must catch up from one current snapshot when made visible; it must not receive a backlog.

Browser drag state also has one owner: `dragState` contains the tab/pane item and source slots, file payload, custom/native/transparent preview nodes, preview offset, and tab-geometry cache. Tab, pane, Finder, and Differ drag starts replace incompatible fields; drop, dragend, pointer/mouse release, blur, and visibility cancellation clear the complete operation through `cancelDragOperationState()`. The transparent native drag-image node is a reusable singleton and remains cached between operations; no other operation field may survive cleanup.

YO!share browser transport state follows the same rule: `shareHostConnectionRecords` owns one `{socket, queue}` record per share token, and `sendOrQueueShareHostMessage()` is the only open-or-enqueue policy. A matching socket close clears only `record.socket` so queued frames survive an intended reconnect; active-token reconciliation closes and deletes the complete record when the share stops or expires. Do not add parallel token-keyed socket and queue maps or another direct queue mutation path.

Finder directory observation state has one bounded browser owner: `fileExplorerDirectoryRecords`, keyed by normalized directory and holding the last listing signature plus prior entry-name set. Broad filesystem invalidation clears only signatures so a later listing can still identify newly added names; root invalidation removes the complete record and any child-path highlight deadlines. Do not split signature and prior-name fields into independently evicted maps.

## Tests

`python3 tools/check.py` runs the local gate as parallel lanes and exits non-zero if any lane fails — use it as the one pre-commit/CPS command. Default/full runs and focused expensive lanes acquire the per-user `~/.cache/yolomux/expensive-tools.lock`, shared by every worktree and launch context; a contender fails immediately instead of queuing another full gate to run later. `YOLOMUX_TOOL_LOCK_PATH` may override the path. When live YOLOmux servers are present in the shared background-owner state the runner lowers its process priority before launching child lanes. Use `python3 tools/check.py --serial` when debugging order, process load, or interleaved output: it runs lanes one at a time and omits xdist from pytest, browser, and E2E commands entirely, even if `YOLOMUX_PYTEST_WORKERS` is set. Use `python3 tools/check.py --list-lanes` to see lane names and repeat `--lane <name>` for focused runs. Use `--performance-report` (or `--performance-report /tmp/name.json`) for opt-in per-step/lane timing, pytest-worker, child CPU, and RSS evidence; reports are deliberately restricted to `/tmp`. Use `--no-tool-guard` only when you intentionally want overlapping high-load local tools.

The canonical pytest lanes use `tools/test_catalog.py`, which owns explicit phase files and is regression-checked against full collection so a node cannot be omitted or owned twice. `pytest-browser` remains boot -> behavior -> serial golden: `pytest-browser-behavior` exists only to measure an independent boot/behavior A/B and is not the default because boot must fail before an expensive browser run. Browser reuse defaults to one driver per test module; `YOLOMUX_SESSION_SCOPED_BROWSER=1` is an opt-in experiment with full profile reset, currently rejected as the default pending a material cross-platform win. Private tmux tests use the shared fast-then-backoff capture owner; do not restore a fixed 400ms loop or share private tmux/state across xdist workers.

The non-browser gate includes AST invariants that all Python imports under runtime, tools, and tests remain module-scoped and that runtime/tool functions with two or more statements do not repeat an identical body. Optional dependencies may be detected at module scope before conditional module-scope imports, but imports must not be hidden inside functions or import-time `try`/`except` blocks. Shared process checks such as PID liveness live in `yolomux_lib/infra/background_owner.py`; tools import that owner rather than carrying a second implementation.

The default lanes are `py-compile`, `static`, `node-syntax`, `node-layout`, `pytest`, `pytest-browser`, `pytest-e2e`, and `whitespace`. The runner derives one concurrent pytest budget from the host's logical core count, capped at 8 CPUs on macOS. Linux makes 75% of that capacity available to pytest; macOS uses 50% because every browser worker owns a Chrome process tree and host security/indexing services scan its temporary profile. The budget is split one-half to non-browser, one-third to browser, and the remainder to E2E, with at least one worker per pool. Thus a Mac reports at most an 8-CPU scheduling budget (2/1/1 workers), 10 Linux cores produce 3/2/2, and 32 Linux cores produce 12/8/4. This replaces independent CPU-sized `auto` pools, which oversubscribed development Macs and caused control-socket handoff and cold-daemon timing failures that passed in isolation. `YOLOMUX_PYTEST_WORKERS=N` gives all three pools the same explicit size; `YOLOMUX_PYTEST_WORKERS=nonbrowser,browser,e2e` tunes them independently outside `--serial` mode.

The `pytest-browser` lane first runs the fatal boot-smoke suite, then behavioral browser tests with xdist work stealing, and finally the local `visual_golden` matrix serially. Serial goldens are intentional: the comparison uses reviewed, machine-local pixels, and parallel Chrome compositor/font pressure can cause small raster deltas without a product-layout change. Boot smoke loads the real generated browser bundle from an isolated ephemeral server through representative fresh, saved-layout, malformed-state, Finder, and Preferences startup states; it fails on any JavaScript error, unhandled rejection, empty grid, zero-sized app root, or missing visible pane. All shipped locales are separately exercised by the live language-switch browser matrix. Browser hover/click/context-menu tests use the shared 50ms `fast_pointer_actions()` helper so they still emit real W3C pointer events and allow hover paint/popover geometry to settle without Selenium's unrelated 250ms human-speed travel animation; genuine drag tests construct `ActionChains` directly and retain deliberate motion duration. The `pytest` lane excludes the `node_bridge` marker because `test_node_suite.py` shells out to the same `node tests/layout_url.test.js` the always-on `node-layout` lane already runs, so the gate would otherwise run the node suite twice concurrently; e2e and browser tests have their own default lanes to keep expensive failures named by subsystem. The `node-layout` lane keeps node coverage in the gate; a bare `python3 -m pytest tests` (outside check.py) still runs the bridge. Focused opt-in lanes are `pytest-unit`, `pytest-socket`, `pytest-boot`, and `pytest-browser`; use them with `--lane` while iterating. Slow known tests are collected first through the one `SLOWEST_FIRST_TESTS` ranking in `tests/conftest.py`; a collection regression resolves every configured base node ID so renamed/moved tests cannot silently leave dead scheduling hints. Tests that monkeypatch a process-global function while daemon work may still run must tag spy calls by the current thread or another operation owner before asserting counts; do not use sleeps or count unrelated concurrent callers. The browser distribution change is backed by two same-code default runs (43.91 s, 45.46 s wall) and two work-stealing runs (34.89 s, 42.39 s), reducing median wall time from 44.69 s to 38.64 s, median user CPU from 102.02 s to 97.16 s, and median system CPU from 28.79 s to 23.99 s; the bounded macOS gate passed all eight lanes in 67.17 seconds with four live YOLOmux servers, including browser in 27.74 seconds, E2E in 46.29 seconds, and non-browser in 67.15 seconds. ## YO!stats stability contracts

Three standing rules exist because each was violated by a real incident (2026-07-14/15):

1. **One exact request and policy owner.** `yolomux_lib/stats_current/resolution.py` owns the `{1,10,60,300}` universe, Range matrix, AUTO choice, 600-bucket bound, and live cadence. The current browser path builds one `GET /api/stats-snapshot` request from server capabilities, and statsd returns that exact key or a structured error; no hand-rolled range table, hidden point limit, alternate reader dialect, or silent resolution substitution is allowed.

2. **Family manifest, no cross-family flags.** Per-family facts, native cadences, storage fields, fold rules, wire groups, and chart mapping live in one generated/shared family manifest. One family's inclusion or optimization flag cannot strip another family's data. Agent/Model token dimensions and YO!cost derive from the same identity-deduplicated usage atoms and must reconcile.

3. **Stats schema/protocol skew fails closed before mutation and coexists by scoped identity.** Every writer reads database application/schema/minimum-writer markers without mutation. Each compatible build uses `stats-v<schema>.sqlite3` and `statsd.p<protocol>s<schema>.sock`, so mixed worktrees never share a writer daemon or database. A stale page writer treats 426 as terminal; a current read client clears its local fence and retries only its own scoped service, never another build's daemon.

Contract suites exercise each current pipeline layer: real original-observation store, asynchronous four-layer materializer, statsd cache, HTTP forwarder, and render-only browser matrix, while separate RPC and route tests pin the binary transport and HTTP dispatch boundaries. Coverage includes every allowed Range x AUTO/explicit resolution, all families, exact deltas and cadence, migration/rollback/deletion, old-writer immutability, no-data, fair all-client browser aggregation, and cache-hit zero-DB/zero-aggregation behavior. The final cutover proof still runs the complete stack through a real socket and HTTP listener; old request/wire goldens are deleted rather than pinned behind a flag.

### Current-only storage and migration

The current schema-versioned stats database stores only original observations, identity-deduplicated usage atoms, covered epochs, explicit unavailable spans, migration reconciliation, and schema metadata. Explicit unavailable spans are coverage facts for known outages or migrated aggregates that cannot be reconstructed; they cannot overlap covered epochs. Writer fences preserve compatible migration repairs and startup diagnostics; older writers stop before mutation. Exact coverage overlapping an unavailable span is not repairable without guessing, so startup fails closed and preserves the database. Derived `1s/10s/60s/300s` display layers are immutable statsd memory, rebuilt asynchronously from a consistent snapshot and updated through the same deterministic dirty-cell fold; web requests never open SQLite or run a fold. [`docs/specs/STATS_API.md`](specs/STATS_API.md) owns the current schema, writer protocol/build, filename, and socket details.

Every process capable of writing first reads `application_id`, `user_version`, schema generation, minimum writer protocol, and minimum writer build through a read-only preflight. A newer schema or minimum writer produces terminal `SchemaTooNewError`/`upgrade_required` before mutation and stops spawn, replacement, and retry behavior while preserving the database/WAL/SHM bytes and metadata. Writer build 3 fences the fail-open legacy migration and startup diagnostics. Background-owner records advertise their stats writer build; when an older preferred owner cannot pass the active fence, the compatible newer generation inherits its priority through the existing release/takeover path and keeps the sole sampler alive. The versioned filename is not exposed through an alias or symlink to retired `stats-history.sqlite3`, so an old runner can recreate only its retired-path database; current startup never imports that file into serving and removes it through idempotent retirement cleanup. A genuinely corrupt database is preserved as a forensic sidecar; a merely newer database is never treated as corrupt. Transcript usage collection is incremental and reset-safe: one steady-state global pass reads at most 4 MiB or 4096 complete records across deduplicated Claude/Codex transcript families, persists offsets only through complete lines, rereads zero bytes for unchanged files, and resumes cold backfill on later native-cadence passes without blocking other collectors. Claude discovery uses the shared per-directory catalog and the same bounded reverse-name plus newest-mtime candidate selector as Codex, but only top-level project JSONL files seed independent session families; each root then expands through its own `subagents/` tree. A pane-owned root keeps its tmux key, while an unowned sibling gets a bounded stable project/session agent key and newest-mtime cold priority. The protocol-24 orphan-fork repair is a one-time asynchronous exception capped at 8 MiB or 8192 records per pass so tombstone receipts acknowledge and persist promptly on the target Mac; it records exact copied-prefix tombstones only through the explicit handoff, commits a versioned completion marker, and then uses the steady-state ceiling permanently.

The atomic all-source migration quiesces collectors and the sole writer, consumes every supported old table/journal/token/usage source, preserves the finest recoverable originals, converts unreconstructable coarse-only spans to explicit current no-data, recalculates all four layers, reconciles family counts/coverage/epochs/tokens/costs, then activates the new marker/fence. Success deletes old tables, files, sidecars, import markers, request dialects, response shapes, and runtime fallbacks; failure leaves the current database unactivated and the migration retryable. There is no dual-format aging period.

The atomic migration vacuums its shadow database, verifies zero freelist pages plus `integrity_check`, and activates that compact file only after reconciliation succeeds. Runtime retention pruning deletes expired originals without a second hidden VACUUM scheduler or undocumented schema field; any future periodic file compaction requires one explicit owner, nonblocking scheduling, diagnostics, and regression coverage before this contract changes.

## Login rate limiting

`yolomux_lib/login_rate_limit.py` is the one owner of login throttling; both password paths (`AuthMixin.handle_login_submit` and the Basic-auth branch of `require_auth`) route through `AuthMixin.verify_login_credentials`, which checks admission BEFORE the PBKDF2 verifier and records the outcome after. Design rules to preserve:

1. **Gate before the hash, one shared owner.** Admission is an atomic `BEGIN IMMEDIATE` check-and-reserve in a single WAL SQLite file under `STATE_DIR`, shared by every port. It runs before `auth_identity_for_credentials`, so a blocked attempt costs no password hash and cannot be used to burn hash CPU. Never add a second ad-hoc limiter map in `server_auth.py`; `ChatRateLimiter` is in-process/fixed-window and intentionally NOT reused here.

2. **Two O(1) primitives, chosen deliberately.** Token buckets for four hierarchical network scopes (exact, /24·/64, /16·/48, global) and a staged-backoff counter for the username scope. Token bucket was picked over fixed-window (boundary burst), sliding-window-log (O(N) state under a distributed attack — measured 2000 distinct-IP attempts collapse to ~400 rows via the coarse buckets), GCRA (equivalent, less legible), and an ASN/reputation DB (external-dependency cost). Network buckets charge every admitted attempt including successes; the username counter advances only on a completed failed verification and resets on success (NIST). A request blocked at admission must never advance the username counter or extend its cooldown.

3. **No information leak, fail closed.** Rows are keyed by HMAC(scope‖key) under a 0600 secret, so raw usernames/IPs are never stored. The 429 body/status/timing is identical for known and unknown usernames and reveals neither the firing scope, attempt counts, nor a precise `Retry-After` — only a coarse minutes/hours band. A store failure fails closed (generic bounded response), surfaced as `login_throttle.healthy=false` in the admin System report. Setup mode and `YOLOMUX_TEST_AUTH_BYPASS` short-circuit before the limiter; cookie and YO!share-token checks are disjoint from the password verifier and never throttled.

4. **Client IP = socket peer only.** `X-Forwarded-For`/`Forwarded` are never trusted. Escalation (`yolomux_lib/login_escalation.py`: decoy challenge, expiring firewall/proxy DROP) is defense-in-depth, OFF by default: edge rules build argv (never a shell string), carry a TTL and cap, exclude trusted CIDRs, and fire only for volumetric network scopes — never a username lock. Policy overrides load from a validated `CONFIG_DIR/login-rate-limit.json` (not `auth.yaml`, whose users-only parser would drop the section on its normalization rewrite); malformed/incoherent overrides fall back to the safe defaults.

The mechanical WAL open/migrate primitives (`open_wal_database`, `enable_wal_with_retry`, `begin_wal_migration`, `load_or_create_secret_key`) live in `atomic_file.py` and are shared with `ChatStore`/`ChatCursorCodec` rather than copy-pasted. Coverage: `tests/test_login_rate_limit.py` (fake-clock bucket/backoff/threat/persistence/concurrency), `tests/test_login_throttle_integration.py` (both HTTP paths, bypass/setup preservation, no-hash-on-block, edge trigger), and `tests/test_login_escalation.py` (decoy + edge controller, all mock-runner).

Browser fixtures are served over a throwaway localhost HTTP origin, not `file://`. Chrome 150+ blocks a `file://` document from loading `file://` subresources in an unrelated directory, so a fixture in a temp dir can no longer pull the `REPO_ROOT/static/*` bundle/vendor/font assets. `browser_layout.py` lazily starts one `ThreadingTCPServer` rooted at `REPO_ROOT`; reference repo assets with `fixture_asset_url("static", ...)` and write/serve a fixture page with `serve_repo_fixture_page()` + `fixture_page_url()` (never `page.as_uri()` for a page that loads the bundle or any repo asset). The single shared origin means: the handler sends `Cache-Control: no-store` (http caches by URL — a same-name overwrite would otherwise serve the stale page); each page load gets a unique URL so Chrome's back/forward cache cannot restore a prior test's live JS state; and `_reset_browser_state()` clears all origin storage between tests (per-file `file://` pages used to get opaque-origin isolation for free). Self-contained inline fixtures with no external assets may still use `file://`. If Chrome auto-updates and the lane breaks again, check the console for `net::ERR_ACCESS_DENIED` / mixed-origin `NetworkError` before assuming a product regression.

Every full-bundle browser fixture goes through `load_live_runtime_boot_fixture()`, which owns bundle readiness; the raw `_live_runtime_boot_fixture_html()` builder is private to that helper module. Readiness requires the post-bundle sentinel, a late-defined app function, and either the normal `#grid` anchor or replay-shell `#appRoot`, and it fails immediately on a captured synchronous boot error or unhandled rejection. Tests that intentionally redirect before bundle completion, such as an expired share going to `/login`, must opt into that exact expected pathname; normal fixtures reject arbitrary navigation and Chrome error pages. Keep asynchronous product waits for sockets, rendered panels, replay convergence, and layout state at the individual test, but do not add another pure `typeof <bundle global>` readiness wait.

All browser-document polling uses the `window.__yolomuxTestWaitFor(predicate, {timeoutMs, intervalMs, description})` helper injected by `new_chrome_driver()`. It returns the successful predicate value, uses animation-frame polling by default, permits bounded interval polling for network state, performs a final deadline-boundary check, and names the awaited condition in timeout errors. Do not add a local `waitFor` loop or a fixed delay when an observable state can be awaited.

Browser tests that only validate shipped CSS against static DOM state use `page_html()` and toggle the authoritative class/attribute directly; they do not boot the full app, filesystem stubs, and async render path. Their wrappers must use `load_static_html_fixture()` as the one UTF-8 write/file-navigation owner; an AST invariant rejects a local `page.write_text()` immediately followed by `browser.get(page.as_uri())`. Standalone writes that feed an HTTP fixture and repeated navigation of one already-written page at different viewport sizes remain distinct. Reserve live-runtime fixtures for behavior that actually calls app JavaScript. The disclosure-chevron computed-style test follows this rule and reduced its isolated total from 2.04 seconds to 0.34 seconds; the regular/compact tab-metadata visibility test follows this rule and reduced its focused invocation from 1.33 seconds to 0.63 seconds; the Info/Preferences shared-scrollbar test now follows it too and improved from 1.78 seconds / 2.085 seconds wall to 1.54 seconds / 1.819 seconds wall while removing runtime/tmux contention from xdist. Topbar owner/activity/latency rest and interaction paint also lives in the static fixture; the live owner fixture retains payload/order/role behavior only. Standalone and Finder-embedded comparison/Reload paint share the existing static light-surface fixture; adding both cascade assertions changed its focused wall sample from 0.60 seconds to 0.58 seconds and added no browser startup. The focused two-test wall samples (2.65 seconds before; 2.91, 3.06, and 3.19 seconds after) are Chrome-startup dominated, so that earlier assertion deletion claims ownership clarity rather than a measured speedup. The sub-window acknowledgement matrix still performs real W3C pointer clicks for red/yellow and pulsing/static variants, but seeds all four sub-windows in one live runtime and samples them over one shared 1.4-second window; its focused wall time fell from 9.11 seconds to repeat runs of 3.50 and 3.52 seconds. The complete xdist browser lane remained effectively flat at 40.35 seconds versus the 40.38-second baseline because the 13-14 second share matrix still sets the lane tail; the measured gain is isolated duplicate work and CPU, not a claimed full-lane wall reduction. Cadence loops wait only between real samples: a three-toggle 2000 ms Safari-shaped share test has exactly two waits and no trailing delay after its final assertions.

### Restart workflow

After every YOLOmux change in a dev worktree, restart that active dev server and verify it before reporting the change ready for testing. Restart prod only when explicitly requested.

Use `boot.sh` from the checkout you want to serve. `boot.sh` and `yolo-dev-start` share one process-lifetime startup lock from `tools/startup_common.sh`, so a second launcher fails before it stops or starts any server. They also share a slow-ramp load gate: before each agent or server mutation, one- and five-minute load must be below the CPU budget, capped at 8 CPUs on macOS even when the OS reports more logical CPUs. The launcher waits up to `YOLOMUX_START_LOAD_WAIT_SECONDS` (300 seconds by default) and stops the ramp rather than adding another process under sustained load. When an operator has measured a known persistent host service separately, `YOLOMUX_START_LOAD_DISCOUNT_CORES` may discount that many busy cores from both load checks; the value must be finite and non-negative, is capped at the scheduling CPU budget, and the launcher reports raw load, effective load, and the applied discount. This explicit adjustment does not bypass the shared lock, per-port lease, listener stability, or ownership checks. `boot.sh` also serializes each port, restarts only that listener, waits for the old listener to leave the port before launching the replacement, sets `PATH` explicitly before launch so agent CLIs under `~/.local/bin` are visible, defaults `MALLOC_ARENA_MAX` to 2 so threaded JSON/transcript work cannot retain one glibc arena per request thread, and appends logs under `/tmp` by default. The server holds a process-lifetime per-port lease, so direct launch and self-update races fail before app/background initialization. On Linux the script finds listeners with `ss` and uses the supported `setsid`/nohup detacher. On macOS it finds listeners with `lsof` and supervises web processes in per-port sessions on the private `yolomux-services` tmux socket; this keeps them out of the user's tmux roster while preserving the caller's Files and Folders access to checkouts under `~/Documents`. A legacy transient `local.yolomux.<port>` launchd job is removed during restart because launchd's daemon context can receive `EPERM` when serving those files.

```bash
./boot.sh
```

The script accepts any valid TCP port. It defaults to HTTPS `8880` on macOS and `7770` on Linux and still passes the compatible `--self-signed` spelling plus `--dang`; the CLI itself now defaults to the same self-signed HTTPS behavior. `YOLOMUX_PORT` selects an arbitrary primary port and keeps the normal non-dev launch behavior. The standard Linux fleet uses `7771` through `7773` for dev worktrees, and Linux autossh exposes `ereview:7770-7773` directly. The standard macOS fleet uses `8881` through `8883` for dev worktrees, and Mac autossh exposes `ereview:8880-8883` directly; there is no Linux-to-Mac forwarding hop and no standard `8000-8003` fleet. For a dev worktree, run the script from that checkout and pass the port you chose; non-primary ports use `--dev` automatically:

```bash
(cd ~/path/to/dev-checkout && ./boot.sh <dev-port>)
```

Override with `YOLOMUX_PORT`, `YOLOMUX_HOST`, `YOLOMUX_LOG_DIR`, `YOLOMUX_START_LOAD_WAIT_SECONDS`, `--port`, `--dev`, or `--no-dev` when testing a non-standard instance. `YOLOMUX_BACKGROUND_OWNER_PRIMARY_PORT` pins background ownership to one preferred server while retaining automatic follower failover; `boot.sh` defaults it to its selected primary port. Use `./boot.sh --print-command <dev-port>` to inspect the launch command without restarting anything.

For the standard multi-worktree stack, `yolo-dev-start` defaults the preferred server to `8882` on macOS when the second dev worktree exists (`8880` otherwise), and to `7770` on Linux. Override it with `YOLOMUX_PRIMARY_PORT`. The preferred server starts first; each port must then pass readiness, exactly-one-process, three-sample ownership, empty-refresh-queue, and system-load gates before the next port starts. Discovery checks use the read-only `/api/tmux-session-exists` roster endpoint and must never force a full transcript/metadata rebuild. Every checkout must advertise the same `BACKGROUND_OWNER_PROTOCOL_VERSION`; a stale worktree causes a preflight refusal before tmux, the private macOS service supervisor, listeners, or ownership are changed.

To restart a dev worktree in no-auth test mode:

```bash
YOLOMUX_TEST_AUTH_BYPASS=1 ./boot.sh <dev-port>
```

Four footguns: (1) never `pkill -f` broad YOLOmux patterns; kill only the listener for the selected port; (2) run `boot.sh` from the checkout you intend to serve so a dev port does not accidentally serve another worktree; (3) never bypass the shared startup lock or replace the read-only roster check with `/api/transcripts?force=1`; (4) a backend `.py` change only takes effect after the Python process restarts, because `static_build.py` does not touch backend code. If a restarted server appears ready and then disappears, check whether readiness was served by the old listener; `boot.sh` must fail instead of launching over a still-bound old PID. The in-app self-update path likewise treats a failed, timed-out, or unlaunchable `tools/static_build.py` run as blocked and must not restart into stale generated assets; periodic update-check exceptions are logged rather than hidden.

Verify after restart:

```bash
ps -ef | grep "yolomux\.py.*<port>" | grep -v grep
ss -tlnp 2>/dev/null | grep ":<port> "
curl -sk -o /dev/null -w "ping: %{http_code} %{time_total}s\n" https://localhost:<port>/api/ping
curl -skL -u <user>:<pass> https://localhost:<port>/ | grep -oE 'YOLOmux [0-9.]+' | head -1
```

Expected: one `yolomux.py` process, `LISTEN` on the intended port, `ping: 401` in under roughly 100ms, and the rendered version matches `YOLOMUX_VERSION`. If verification fails, inspect the appended `/tmp/yolomux-<port>.log` log for the previous launch error and the latest `boot.sh launching port ...` marker.

If you intentionally started with `YOLOMUX_TEST_AUTH_BYPASS=1`, `/api/ping` and `/api/settings` should return `200` without cookies. If a normal login-gated server returns `200` for those unauthenticated routes, auth is accidentally bypassed.

## Production Sync (`cps`)

`cps` (the `yolo-cps` skill) has two modes, chosen by the trigger word:

- **LOCAL (default — "cps" / "yolo-cps" with no qualifier):** rebase the dev branch onto LOCAL `main` and land it on local `main`. NO version bump, NO push.
- **ORIGIN ("cps origin" / "cps remote"):** rebase onto `origin/main`, bump the version, land on local `main`, then push to `origin/main`.

Edits happen in a dev checkout; the integration/production checkout (which holds `main`) is read-only during sync and shares the same `origin`. Because `main` is checked out in the production checkout, you cannot check it out in a dev worktree — land work by committing on the dev branch, rebasing it onto `main` / `origin/main`, then fast-forwarding `main` from the production checkout.

ORIGIN-mode sequence from the dev checkout:

```bash
python3 tools/check.py                       # the full gate (see Tests above)
git fetch origin && git rebase origin/main   # rebase onto the published tip
# bump matching YOLOMUX_VERSION literals in yolomux_lib/infra/common.py and yolomux_lib/common.py, folded into the work commit
git add -- <explicit-files>
git commit -m "<message including Version: 0.4.N>"
git -C <production-checkout> merge --ff-only <branch>    # land on local main — its OWN exit-checked command
git -C <production-checkout> push origin main            # only after the ff-merge succeeds
```

LOCAL mode is the same minus the version bump and the final `push` (stop after the ff-merge).

Rules:

- ORIGIN mode MUST bump matching `YOLOMUX_VERSION` literals in `yolomux_lib/infra/common.py` and `yolomux_lib/common.py` in the same commit; the auto-updater checks the runtime value on `origin/main`, not the commit SHA, so SHA-only commits do not cue the update. LOCAL mode does NOT bump.
- Never use `git add -A`; screenshots and scratch files must not get swept in.
- The `merge --ff-only` into the production checkout MUST be its own exit-checked command — NOT piped through `tail`/`grep` and NOT `&&`-chained straight into the push. A pipe's exit status is the last stage's, so a DIVERGED ff-merge fails silently and a chained `push origin main` then publishes whatever local `main` already points at, not your commit. Confirm the merge succeeded before pushing.
- This is a shared multi-worktree: local `main` can advance from another worktree mid-`cps`, so your ff-merge can suddenly refuse (diverged). Recovery: `git fetch origin`, `git rebase origin/main` your dev branch (disjoint files rebase clean), re-run the gate, then ff-merge + push. No force-push, nothing lost — the other commit is usually a sibling off the same base.
- Production pull/merge is `--ff-only`. Never edit, stage, or commit inside the production checkout.
- Restart is NOT part of `cps` itself. After code changes in a dev worktree, restart the active dev server before reporting ready; restart prod only when explicitly asked (see Restart workflow above), then verify `/api/ping`, the process cwd, and `YOLOMUX_VERSION`. The login page may not expose the version to unauthenticated curl; do not rely on a blank version grep alone.

## How The Webterm Works

The server is a dependency-light Python `ThreadingHTTPServer`. It serves one HTML page, local xterm.js assets, JSON APIs, Server-Sent Events streams, and a WebSocket endpoint.

For each terminal connection, the browser opens `/ws?session=<tmux-session>`. The server creates a PTY, runs `tmux attach-session -t <tmux-session>` for admin users or `tmux attach-session -r -t <tmux-session>` for readonly users on the PTY slave, reads terminal bytes from the PTY master, and sends those bytes to xterm.js as WebSocket binary frames.

Closing the WebSocket before the initial resize payload is an expected disconnect. `bridge_tmux()` returns before PTY allocation for that case and uses the same `OSError` family boundary as later terminal disconnects, so reloads cannot emit request-thread tracebacks or allocate an orphan PTY.

Browser input is sent as JSON messages over the same WebSocket. Normal keyboard data becomes `{"type": "input", "data": "..."}`. Resize messages become `{"type": "resize", "cols": ..., "rows": ...}`. Scroll messages become `{"type": "tmux-scroll", "direction": "up|down", "lines": ...}`.

Terminal image paste is latency-sensitive. The browser upload path inserts the `[Image #N] <path>` reference from the `/api/upload` response, repaints the terminal, and may refresh upload/event UI, but it must not force a transcript/session-metadata refresh. Upload-directory resolution is static and authenticated-user scoped: `/tmp/yolomux.<login-user>/uploads/<session>/`; it must not inspect the session cwd, discover agents, resolve transcripts, or trigger macOS `lsof` process scans. The per-user root is verified as a non-symlink directory owned by the server's Unix uid before it is secured to `0700`; files are reserved with exclusive creation for multi-server safety. On macOS, general agent cwd discovery uses `libproc.proc_pidinfo(PROC_PIDVNODEPATHINFO)`, while Codex transcript attribution enumerates open vnode paths with `PROC_PIDLISTFDS`/`PROC_PIDFDVNODEPATHINFO`. A native lookup failure emits a deduplicated server warning and falls back to a bounded, TTL-cached `lsof` query; do not restore an unconditional all-process or all-file-descriptor scan.

Resizing is handled on the PTY slave file descriptor, then the server sends `SIGWINCH` to the tmux attach process group. The browser sends resize updates only after a debounce, except for an initial fast resize during WebSocket startup.

The browser creates panel DOM nodes for visible sessions at boot, checks that the backing tmux sessions exist, and starts terminal connections for them. Visible layout changes move existing panel nodes into slot containers. Hidden panels move back to `#panelPool`. This keeps xterm instances and WebSocket connections alive while changing layout.

The full layout is encoded in the page URL through readable `sessions`, `layout`, and `tabs` query parameters; split positions are stored as percentages in `layout`. This makes a layout reload-safe and shareable/bookmarkable without browser storage.

Transcript metadata comes from tmux pane discovery plus local process-tree inspection. YOLOmux looks for Claude or Codex processes across the session's tmux sub-windows, finds their transcript/session metadata, and exposes both per-session summaries and per-sub-window rows in the pane header, transcript tab, popovers, Tabber, and API responses. `work_graph` is the only repository/branch/PR owner. The backend-owned `agent_windows` rows are the source of truth for Claude/Codex tmux sub-window UI state, pid, and active sub-window; graph `PathObservation` edges supply touched paths and Git associations. `window_metadata[]` carries only raw per-sub-window cwd metadata, so popover, Tabber, Info Bar, and tmux sub-window bar UI must not derive Git state from it.

The transcript tab uses Server-Sent Events from `/api/context-stream`. The AI summary tab uses `/api/summary-stream`, which builds a scoped prompt from the selected session's recent transcript and streams a Codex-generated summary back to the browser. Summary provider defaults live in `settings.yaml` under `summary.*` and in the backend settings catalog; `YOLOMUX_SUMMARY_MODEL`, `YOLOMUX_SUMMARY_EFFORT`, and `YOLOMUX_SUMMARY_SERVICE_TIER` only seed defaults when they name valid catalog values.

YOLO uses `tools/auto_approve_tmux.py` workers behind `/api/auto-approve`. The browser polls YOLO status every `paneStateRefreshMs` and reflects the active state in each pane tab.

## YO!share Transport

YO!share visual sharing uses host-rendered DOM replay. The host opens `/ws/share-host` and publishes sanitized `#appRoot` keyframes, DOM deltas, replay control replies, status, and terminal placeholder metadata. Viewers open `/ws/share-ui` to receive replay frames and share status; HTTPS write viewers may also send validated `input-intent` frames there. `/ws/share-view` remains the terminal byte stream for shared tmux sessions, and terminal content is represented in replay DOM only by placeholders that bind the existing xterm stream to host rows/cols.

Replay deltas are ordered by epoch, sequence, and base sequence. Old or already-applied deltas are stale and should be ignored without repair; missing future deltas are gaps and should request a keyframe. Replay debug/profiling output should expose both paths separately (`staleFrames` versus dropped frames/keyframe requests) plus the frame numbers needed to explain a `viewer behind` report.

The replayed DOM is inert serialized markup. Host app event listeners do not survive keyframe rebuilds or child-list delta rebuilds, so viewer-local read-only affordances such as tab hover details must be rebound by the replay shell after replay applies, or represented as host-owned replay DOM.

Write access requires HTTPS. A write viewer does not publish `layout`, `ui-state`, Finder, editor, popup, or scroll semantic state; it sends allowed input intents and waits for host replay frames for visible feedback. `shareReplay=0`, `shareSemantic=1`, bootstrap `shareReplay: false`, and `localStorage.yolomux.shareReplaySemantic=1` are temporary semantic escape hatches for diagnostics, not the normal product path.

## Webterm API

All API routes require auth unless the process was intentionally started with the local-only `YOLOMUX_TEST_AUTH_BYPASS=1` test bypass. Read endpoints accept `readonly` or `admin`, except `/api/summary-stream` because it launches Codex and requires `admin`. Mutating POST routes require `admin` except `/api/event`, which accepts readonly client telemetry. `/ws` accepts readonly users but attaches tmux with `-r` and ignores keyboard input, tmux-scroll, and resize messages. The `Route` registry in `yolomux_lib/http_routes.py` is the single owner of method/path, role, handler, body limit, group, and `share_access` (`none`, `readonly`, or `scoped-file`). Request admission, readonly filesystem scoping, and plaintext HTTP-share admission all resolve that registry policy; do not add manual API-path allowlists in server/auth code. Share-token guests are narrower than `auth.yaml` readonly users: tokens are scoped to one share and admitted only through registry-marked share page, static, replay, terminal, status, and readonly file/data routes. The test bypass never escalates a share-token request; share-token scoping still wins.

Pricing catalog policy is server-owned: `GET /api/pricing-catalog` is available to authenticated readonly/admin users and returns local catalog status plus safe source evidence; it may initialize the offline seed cache but never fetches a provider. `POST /api/pricing-catalog/refresh` is admin-only, returns promptly, and delegates the crawl to the single-flight refresh coordinator. The elected background owner also schedules a jittered refresh for seed-only/stale state and daily thereafter; failures use bounded exponential backoff. Refresh accepts only reviewed canonical HTTPS allowlist sources, bounds response size/time, records ETag/Last-Modified/digests, and fails closed per source. Provider tables take priority, structured LiteLLM JSON fills provider-page gaps, and OpenRouter is corroboration only. A partial refresh carries prior official aliases forward with their original evidence and emits a warning instead of reducing coverage. Resolution priority is explicit override, exact official rate, visibly labeled conservative family inference, then packaged seed. Update `yolomux_lib/data/model_pricing.json` only from reviewed provider sources; bump `catalog_revision`, preserve source `checked_at`, and use the actual effective date when known. Every projection keeps selected-profile marginal cost and the default-profile API-list counterfactual as separate integer micro-USD fields. A `subscription` profile produces `$0` marginal cost only for settings-authorized CLI usage while preserving the default rate, token volume, and source evidence; direct API usage remains `default`. The UI labels both values because subscriptions, credits, enterprise terms, service tier, regional, long-context, and tool charges may differ from a user's bill.

The public filesystem package facade is the single OS-error normalization boundary. Raw `OSError` values become localizable `FilesystemError` responses before they reach single-route or batch handlers; `PermissionError` maps to 403 and other OS failures map to 500 unless a caller provides a more specific status.

Each job records `id`, `type`, `target`, `predicate`, `action`, timestamps, status, confirmation state, timeout, idempotency hash, last observed state, and audit event ids. Jobs are one-shot in the first implementation; duplicate queued/pending jobs are rejected by idempotency hash. The server polls queued jobs from the existing client-event watch loop, debounces idle/done predicates with `quiet_seconds`, times out expired jobs once, and emits `yoagent_jobs_changed` events for created, confirmed, cancelled, fired, failed, and timed-out jobs.
- `wait_roster_then_send` is the reusable roster-condition-to-action job composition: `target.roster` is the watched set, `predicate.type=all_calm` requires every real Claude/Codex pane to be idle or done, accepting, and draft-free for its quiet window, and `action.session` is the independently validated destination. The local parser owns known wording. For flexible candidate wording, the shared model-intent registry selects exactly one enabled built-in skill, lowers its schema to the provider-compatible JSON Schema subset, and invokes an isolated read-only/no-tools Codex or Claude planner with only the request, known sessions, and that contract. A generic validator checks the returned object against both the provider-lowered and product schema; a named handler maps it to the existing job creator. User skill overrides cannot register a model intent, the server fixes the ten-second quiet default, and every model-derived job requires confirmation before it can fire. Its exact prompt is preserved, result capture is opt-in, the shared state-file lock atomically claims `firing` before any pane mutation, and a restart after a claimed send records a failed/unknown terminal state rather than risking a duplicate. The existing client-event watcher remains the only poller; job creation wakes it, and an active SSE subscriber keeps its `yoagent` demand while the page is hidden.
- Notify jobs cover one session becoming idle/done, one session entering `needs-input`, one session becoming blocked by approval/error/disconnect state, one session finishing after this job observed it working, and all visible tmux sessions becoming idle when all-session discovery is enabled. `POST /api/yoagent/jobs/cancel-session` cancels queued and pending-confirmation jobs for one session without touching fired/failed/timed-out jobs or other sessions. Wait-then-send jobs wait for the resolved target to accept a Claude/Codex prompt, then create a fresh send preview, revalidate the target, paste text plus Return, and start the same result watcher used by direct sends unless the user explicitly opted out. If the session disappears, the job fails and notifies rather than retargeting by name.
- YO!agent risk policy is intentionally narrow and boring. Normal explicit sends do not ask for extra confirmation unless the user asks for preview/confirmation, but high-risk prompt text forces confirmation: secret-like assignments, credential paths, `rm -rf`, `git reset --hard`, broad `pkill -f`, recursive `chmod`/`chown`, and SSH commands. Job idempotency keys are hashes, and stored/displayed YO!agent conversation text plus public job/action payloads redact known `token=`, `secret=`, `password=`, and `api_key=` style values.
- `GET /api/session-files?session=project1&hours=24` returns repo-aware AI file changes for one session. `GET /api/session-files-batch?session=project1&session=project2&hours=24` returns the same payload shape keyed by session after discovering tmux metadata once, then fills missing per-session payloads with bounded concurrency. These endpoints power Modified-files/Differ and feed the server-side `agent_windows` fold; Tabber no longer fetches them directly for per-window touched paths. They use a 30 second stale-while-revalidate in-memory cache, a shared disk cache under `~/.local/state/yolomux/session-files-cache/` (or `YOLOMUX_STATE_DIR/session-files-cache/`), and a per-key cross-process lock so parallel YOLOmux servers can reuse one touched-path payload instead of rebuilding it in every process. Durable cache identity is split: the filename derives ONLY from the stable logical view key (kind, key version, session, bounded hours, refs, per-repo ref overrides), while the volatile tmux/transcript/repository signatures are a replaceable `source_generation` stored inside the record — agent-status changes and transcript appends REPLACE one file per view instead of minting key-per-generation files, an older generation serves only as a stale last-known-good (never a fresh hit), and per-write index updates append one JSONL journal line that merge-on-read folds into the base (never an O(historical-entries) rewrite). Repository Git state has one owner per canonical root (`shared_git_identity`): the native filesystem watcher marks a repo's dirty generation from worktree events AND the narrow .git metadata set (HEAD, index, refs/, packed-refs, MERGE_HEAD), so a warm unchanged request reuses the recorded identity with ZERO Git commands, concurrent callers share one in-flight signature future, a 60 s safety bound recomputes even after a missed event, and an unhealthy watcher fails open to always-compute. Transcript discovery shares one per-directory mtime-keyed catalog in `sessions.py` (`_cataloged_jsonl_files`) for Codex rollouts, Claude session-id lookup, and subagent families — no request-path recursive glob remains, resumed old-name rollouts still surface through uncached file mtimes, and immutable rollout `session_meta` (thread id/parent) is identity-cached so appends read zero header bytes. Equivalent local requests also join one per-target future. Repository-wide Git facts use one bounded immutable snapshot per canonical worktree, git-dir, HEAD, index, untracked, and selected-ref identity; session-specific transcript attribution is merged afterward, and separate Git worktrees never share a snapshot merely because they use the same object database. Owner refresh completion fans out `background_refresh_done` so followers refetch instead of waiting for disk polling. The `/api/activity` cache reuses those payloads when building `agent_windows`. `force=1` bypasses existing cache data and stores the replacement result.

Filesystem watch pushes are MPEG-style frames. The first frame and periodic keyframes are full filesystem payloads; normal watch ticks publish a compact `fs_changed` invalidation with a shared snapshot token and no directory listings. Each browser client keeps its own last token and asks `GET /api/fs/watch-diff?since=<token>` for the changed roots; the server only keeps a bounded shared snapshot history and does not track per-client offsets. Stale or missing `since` falls back to a full frame, removed roots are sent as `removed_roots`, and the top-right refresh button forces `GET /api/fs/watch-diff?full=1` so users can always request a whole frame.

Each process keeps browser-requested file watches in one `ClientWatchFileRecord` per normalized path. The record owns both the TTL expiry and the mutually exclusive active/background mode. `POST /api/watch/roots` replaces that record set atomically, and active/background signature consumers derive their sorted path lists through one expiry-pruning helper. Do not reintroduce separate active and background maps: overlap repair and duplicate expiry paths previously let the same path occupy two modes.

### Background Ownership

Expensive background coordination is single-owner per `YOLOMUX_STATE_DIR`, while durable stateful work is owned by local services in separate PIDs. `STATE_DIR` defaults to `~/.local/state/yolomux`; setting `YOLOMUX_STATE_DIR=/some/path` makes every YOLOmux process that uses that path share the same coordination files, caches, locks, service sockets, and service-discovery records. The background-owner election files live under `STATE_DIR/background-owner/`: `owner.lock` is the global flock, `owner.json` is the current owner heartbeat/status record, and `generations/*.json` are live process generation records. The latest live generation owns Tabber activity refresh, session-files refresh demand, watch-root polling, and independent YO!stats metric workers; older servers become followers even if a browser is actively connected to them. Followers serve shared ready/stale snapshots and request refresh from the owner over the control socket. Token-consumer demand is a family-specific owner refresh that wakes only the token worker; there is no statsd-to-web sampling RPC. Followers must not start warmers, metric workers, or derived watch-root directory polling/listing. They still update the shared watch-root intent index at `STATE_DIR/watch-index.json` so the owner can observe connected browser interest.

### Current token/cost recovery

Transcript discovery is an input to the atomic all-source migration, not a separate retained-bucket repair mode. The migration reuses the one transcript-to-usage-atom identity owner, deduplicates provider events, reconciles Agent/Model/Cost totals, preserves incomplete evidence as an explicit lower bound, and converts unreconstructable spans to current no-data. After the current schema activates, normal repair appends missing current usage atoms through statsd and invalidates the affected materialization generation; no tool may rewrite derived display buckets because the current database does not contain them.

Migration and current repair fail closed while another writer or incompatible daemon owns the state directory. They acquire the same service/writer fences, preserve a forensic backup, validate reconciliation before activation, and leave the active database unchanged on failure. The retired offline bucket wipe/rebuild path is not part of current serving or repair.

Local services share one lifecycle and RPC parent. `local_services/rpc.py` owns the versioned length-prefixed JSON envelope with optional bounded binary payload and rolling legacy newline reads. `local_services/runtime.py` owns `LocalRpcServiceState`, service-side mode-`0600` Unix socket binding, peer UID checks where available, singleton locks, response correlation, and idle cleanup. `local_services/registry.py` owns client-side discovery, stale PID/socket cleanup, `sys.executable -m <module> --serve` spawning, service records under `STATE_DIR/services/*.service.json` or next to the socket, exponential crash backoff from 0.25 seconds to 8 seconds, and lease-protected idle exit. A service failure may produce stale/pending/unavailable responses, but the web process must not run a retired expensive local fallback.

`statusd` owns the shared public session/agent-status snapshot: lightweight discovery, pane capture/classification, assembly, and encoded auto-approve bytes. Web routes only authenticate, validate, and forward that result; the watcher publishes an `auto_approve_changed` transition from its generation/status rather than rebuilding a roster. `approvald` remains the separate action-worker owner. Statusd accepts version-stamped public requests, serves a bounded stale complete snapshot or `503`/`upgrade_required` without a web fallback, and exposes `invalidate` plus bounded `wait_generation(after_generation, timeout)` for status consumers. Statusd also owns a separate, private `inventory` action: a versioned session-inventory contract carrying only bounded per-session identifiers and a `source_signature` (never git/transcript/repo/PR enrichment), with the DAEMON discovering the canonical tmux roster itself (`list_tmux_session_names`) rather than trusting a web-supplied session list as authority; a web-supplied roster is only a fallback when tmux enumeration fails. `inventory_generation` advances only when the discovered roster's aggregate source signature changes, so refresh products (jobd) can key work on a stable upstream generation instead of re-deriving topology themselves.

Web-owned refresh compute is being migrated to `jobd` as typed materialized products, not just stateless one-shot tasks. `JobClient.product(coalesce_key)` returns the newest COMPLETED bytes for a product identity (`ready`/`stale`/`pending`/`none`) independent of whether a newer generation is still building, which is what makes stale-while-revalidate a broker property: a web route can serve a prior complete product while a newer submission runs, without ever falling back to in-process compute. The broker's existing `_submit` generation guard (an older-or-equal generation submission is coalesced into the existing job) and `_store_product` (a slower older-generation completion, or any failed/superseded completion, can never overwrite a newer complete product) together guarantee that a delayed result cannot regress a fresher one. `session_files_view` (registered 2026-07) was the first product built on this contract: the web owner submits the session's `SessionInfo`/options as JSON (never pickled — jobd workers run in a spawn `ProcessPoolExecutor` with no access to app state), the worker reconstructs the dataclasses, builds a per-task memoizing git-snapshot provider, and calls `session_files.session_files_payload(_for_info)` entirely inside the worker process. The web-side cache KEY for session-files uses the fs-watcher's per-repo dirty generation (`repo_dirty_generation`) instead of spawning `git status` to build a git-identity hash, so a warm request over an unchanged, watcher-covered repo performs zero git-identity spawns in the web process; an uncovered repo falls back to the spawn-based identity so freshness is never silently lost.

`tabber_activity_view` and `metadata_warm_view` (registered 2026-07-18, `JOBD_PROTOCOL_VERSION` 4→6) follow the same impure-gather/pure-assemble split for the two remaining refreshers. For Tabber, `agent_window_gathered_agents` (app.py) does the impure per-agent gathering (live tmux screen capture, attention/cooldown keys, path/git) that cannot run in a spawn worker; `build_activity_payload` batches every session whose per-session signature changed into one `tabber_activity_view` submission, and the worker (`activity_summary.tabber_activity_view_result`) does pure dict assembly/sort only, reusing unchanged sessions' prior rows. The per-session signature folds in the statusd-owned roster row (not just tmux screen text) so a roster-only change still busts reuse. For metadata warming, `warm_metadata_cache` submits one `metadata_warm_view` job per session (not batched, to preserve per-session demotion-cancellation granularity); the worker runs `session_work_graph(info, fresh_cache, allow_network=True)` against a worker-local `MetadataCache`, snapshots every GitHub/Linear/git cache entry it populated as `{key: {value, ttl_remaining}}`, and the web owner replays those entries into its own long-lived `metadata_cache` via `.set(key, value, ttl=...)` — reproducing the same warm effect without the web process ever making a network call or git spawn itself. Both raise a dedicated `*JobdUnavailable` exception (never an inline fallback compute) so an unreachable broker serves last-known-good/stale data or simply skips a warm cycle instead of rebuilding in web.

Exact stats requests are bounded by the canonical matrix and service protocol. `GET /api/stats-snapshot` asks for one preset Range and `AUTO|1|10|60|300`; statsd returns one already-published generation with at most 600 uniform buckets or a structured `pending`, `unsupported`, `unavailable`, or `upgrade_required` result. A request never opens SQLite, runs a family fold, encodes a new body, or retries through another protocol. The browser keeps one generation-tagged loading/error owner, rejects stale or mismatched responses, and uses bounded retry/backoff without paging older ranges.

The elected app owns one non-overlapping deadline worker per metric family: CPU at 1 second, Agent Status/GPU/service load at 10 seconds, system memory/hardware at 60 seconds, and transcript tokens at 10 seconds while watched or 60 seconds while idle. Each worker appends only its original observation through the current statsd writer, and one slow or failed family advances/skips only its own deadline. CPU's durable write deadline remains 0.9 seconds. The materializer consumes committed generations asynchronously, recomputes deterministic dirty cells through the same fold as a full build, and publishes one immutable generation without blocking the writer/listener. `statsd.sampler_families` and materializer diagnostics expose cadence, attempts/successes/failures, late/missed, runtime, queue depth, warm/dirty state, generations, bytes, and last failure.

The current schema stores original per-family observations, usage atoms, coverage/epoch facts, unavailable spans, migration reconciliation, and schema/minimum-writer metadata. Adjacent sampler owner generations remain distinct; prefix, suffix, middle, multiple, and empty coverage shapes are ordinary data; absent samples remain no-data rather than zero. Derived `1s/10s/60s/300s` layers exist only in statsd memory. Counts, tokens, costs, and event rates are never held or repeated.

The web process CPU budget is below 10% when idle and below 25% with YO!stats visible on the standard macOS development host. The existing one-second CPU sampler owns enforcement: more than 30% process CPU sustained for five minutes emits one deduplicated `stats-cpu` warning with the three largest endpoint/background compute owners, and the System tab exposes current/budget/sustained values. This adds no monitoring timer. Metadata refreshes are process-wide single-flight: concurrent readers share one real status/signature check, and unchanged repositories reuse the full Git metadata snapshot instead of repeating branch, remote, worktree, and inventory subprocesses. A dirty worktree or a real ref change (`.git/HEAD`, `refs/`, `packed-refs`, `MERGE_HEAD`) invalidates and rebuilds; a repo's own `.git/index`/`*.lock` churn is NOT treated as an external invalidation, because `git status` itself rewrites `.git/index` on every read, and feeding that write back in made a compute bump the generation that then discarded its own result — a self-perpetuating refresh livelock — while the 15 s cache TTL backstops any staging-only change skipped. On a mid-compute invalidation the owner caches and returns the value it just computed and refreshes on the next call, rather than discarding it and recomputing inline. The single-flight aggregate (session-metadata) refresh worker carries a bounded deadline so a stalled build can be superseded instead of pinning the guard and freezing the header indefinitely. Concurrent Finder refresh owners may bypass the completed listing TTL with `fresh`, but identical in-flight directory listings still coalesce into one `/api/fs/batch`; explicit user freshness is preserved without multiplying enumeration.

Repeated visible-client token demand may shorten a stale 60-second idle deadline, but demand inside the active 10-second cadence is coalesced even after the previous scan has completed; UI polling therefore cannot create a token-scan wake storm.

Browser/client metric upload uses the bounded `POST /api/stats-observations` acknowledgement path with stable event identities, a server-bound client ID, and the current writer protocol/schema generation. statsd deduplicates and appends originals, then marks deterministic materialization cells dirty; the request never compacts history or returns a snapshot. Retention deletes only expired originals/atoms/coverage, and incremental/full materialization stays outside the request and writer/listener threads.

Tabber consumed 4.3 seconds/minute including its one cold build; 7 of 8 refreshes were unchanged-cache hits, all 11 session rows appeared by the first 10-second follow-up, and no later response lost a row or reported an error. Treat this as a separate cold transcript-attribution target; it does not justify undoing the shared Git, per-row reuse, or single-flight paths.

Transcript attribution now resumes large Claude and Codex JSONL scans from schema-versioned records under `STATE_DIR/transcript-scan-cache-v1/`. One provider-neutral owner validates parser version, device/inode, resolved path, options, a bounded first-line digest, parsed-tail bytes, truncation, and replacement before reusing an offset; invalid records reparse from byte zero. Records contain only derived path markers and bounded usage counters, never raw transcript lines, command bodies, or patch text. Files below 64 KiB remain memory-only because reparsing them is cheaper than durable fsync; large initial scans persist immediately, then active append state checkpoints at most every 30 seconds or 256 KiB. Per-record atomic writes, newer-offset protection, 0600 files in a 0700 directory, and count/64 MiB eviction keep rolling multi-server use bounded.

Background worker generations also have one typed owner. `SessionFilesDiskPruneRecord` owns the prune deadline, in-flight state, and completion result. `TabberActivityWarmerRecord` owns the warmer thread, running state, and visible-consumer deadline. `MetadataWarmRecord` owns the network metadata thread plus its demotion stop event. The elected app runs independent YO!stats family collectors while statsd alone appends their original observations and owns derived materializations; auto-approval target loops run in `approvald`; CPU transforms run in `jobd` executors. Metadata warming checks its demotion event between session lookups (one `metadata_warm_view` jobd job per session, not batched), so a demoted process cannot continue owner-only GitHub/Linear work. Worker loops may clear their record only if that exact thread/record remains current.

Global durable state must use one shared storage/reader path per state family, not parallel per-server JSON files. Durable globals include Preferences in `~/.config/yolomux/settings.yaml`, YO!agent jobs in `~/.config/yolomux/state.json`, cross-server red/yellow tmux AI acknowledgement state in `STATE_DIR/tmux-AI-status.json` under `attention_acks`, watch-root intent in `STATE_DIR/watch-index.json`, Quick Open indexes under `STATE_DIR/search_index/`, session files under `STATE_DIR/session-files-cache/`, activity under `STATE_DIR/activity-cache/`, YO!chat in `STATE_DIR/yochat.sqlite3` plus `STATE_DIR/yochat-history/YYYY-MM-DD.jsonl`, service discovery under `STATE_DIR/services/`, and reconstructible model-price metadata in `YOLOMUX_CACHE_DIR/model-pricing/pricing.sqlite3` (default `~/.cache/yolomux/model-pricing/pricing.sqlite3`). The current YO!stats durable entry is original observations, usage atoms, covered epochs, explicit unavailable spans, migration reconciliation, and schema metadata in schema-versioned `STATE_DIR/stats-v<schema>.sqlite3`. It is never exposed through the retired `stats-history.sqlite3` pathname or a compatibility symlink, so even a runner that predates the writer fence cannot open or clobber the current database. The pricing catalog imports its packaged `yolomux_lib/data/model_pricing.json` seed without network access; source caches and its cross-process lock live beside that database. Old stats journals/files are migration inputs only and are deleted after successful reconciliation/activation; if an old runner later recreates retired-path files, current startup removes them without importing them into current serving. Browser latency, API/SSE counts, bandwidth/sec, and disconnected time are stored under server-bound browser identities and materialized as privacy-safe all-client averages; zoom/hover/render state remains browser-local.

The current YO!stats store retains 24 hours of original evidence and deletes it only after expiry; retention never replaces originals with derived coarse rows. statsd builds four immutable memory layers at `1s/10s/60s/300s`, and shorter Range snapshots slice the layer for their concrete resolution. Disk audits measure current original/atom/coverage rows and database/WAL size; heap audits measure statsd shared/private structured bytes and pre-encoded cache bytes separately from web-process RSS.

Usage atoms record provider/model, input/output direction, modality, cache role, unit, effort, execution-source/thread identity, stable event identity, timestamp, and completeness as original facts. statsd projects effective-dated catalog rates into integer micro-USD plot values inside the materializer. Missing or output-only evidence remains partial, so Cost uses a lower-bound estimate rather than fabricated input/cache prices; Agent, Model, and Cost projections reconcile to the same atoms.

The current routes are `GET /api/stats-capabilities`, `GET /api/stats-snapshot`, `GET /api/stats-delta`, `GET /api/stats-stream`, and `POST /api/stats-observations`. A snapshot accepts one preset `range_seconds`, `resolution=AUTO|1|10|60|300`, a server-bound `client_id`, and optional `since_generation`; it echoes the exact key, supplies one concrete resolution and at most 600 uniform plot-ready buckets plus per-family no-data, and identifies source/cache generations and open-tail state. `/api/stats-stream` emits exact `delta`, `ready`, `repair`, or `unavailable` SSE events over the same delta cursor contract; `/api/stats-delta` exposes the bounded one-shot equivalent. Observation upload accepts bounded identity-deduplicated browser originals and returns only a durable acknowledgement. Full request/response/error and migration contracts live in [`docs/specs/STATS_API.md`](specs/STATS_API.md).

Live delivery and repaint cadence derives only from the concrete server-returned resolution: `/api/stats-stream` checks for exact deltas and presentation advances every second for `1s`, every 10 seconds for `10s`, and once per minute for `60s` and `300s`. This is `min(resolution_seconds, 60 seconds)`, not a Range heuristic. Slower or sparse families do not fabricate fine observations; their existing geometry drifts left or remains empty while the live axis continues to move. Hidden and fixed historical views do no live work and recover from one exact snapshot.

Dynamic JSON uses the same response-level gzip negotiation owner as static text assets. Bodies at or above the shared threshold honor explicit `gzip`/wildcard q-values, emit `Vary: Accept-Encoding`, and keep HEAD/error semantics. statsd owns structured/encoded cache bytes; the HTTP forwarder returns them without decoding, temporal transformation, or re-encoding the JSON body.

Each mounted current chart has one readiness state: `loading`, `pending`, `ready`, or `error`. Before the first accepted generation and after a selection change it shows `Loading stats…`; a structured pending response shows `Stats are still being prepared. Retrying with backoff.`; validation or transport failure shows `Could not load stats.` A successful empty generation shows the view-specific empty state. Range/Resolution selection, reconnect, missed generation, server identity change, and hidden-to-visible recovery use the exact snapshot coordinator; selection generations reject stale work and retry is bounded. There is no older-page loading phase, full-retention prefetch, or legacy fallback.

The per-request HTTP profile is initialized at dispatch and is never stored on a reusable handler object. It records accept/queue wait, handler compute, JSON encoding, compression, write time, uncompressed/representation/wire bytes, plus bounded owner phase counters. Shared owners increment candidate discovery, transcript bytes/JSON decode/cache outcomes/cwd resolution, Git, disk-cache maintenance, activity, stats encode/compress, filesystem policy/list/identity/Git, generation/PID, and watch-root signature counters. Routine topbar responses do not carry these diagnostics; fetch them only through the bounded diagnostics/report route.

For live YO!stats performance investigations, use the bounded System diagnostics and `python3 yolomux.py --print-runtime-report`, save raw profiles under `/tmp`, and compare equivalent cold and settled workloads. Do not treat one developer-machine sample as a portable timing threshold.

Search-index files live under `STATE_DIR/search_index/`. Each indexed root gets digest-named files: `<digest>.sqlite3` for entries, `<digest>.manifest.json` for cheap status/count metadata, `<digest>.lock` for the per-root build lock, `<digest>.tomb` for removals, plus SQLite WAL/SHM sidecars when SQLite needs them. Servers read the small manifest and committed SQLite snapshots; `indexd` is the only process that applies row deltas, full rebuilds, and unindex writes. Quick Open queries only consume ready/stale data and never schedule a TTL rebuild. The watcher coalesces changed paths per root, `indexd` incrementally replaces only those subtrees, and the 30-minute full safety scan is independent of query traffic. In-memory entries, persisted entry count, and estimated/actual cache bytes are bounded; truncated or over-budget roots remain usable as explicit partial in-memory indexes and are not persisted. Recoverable SQLite writes are optional and debounced, while locks, tombstones, settings, and user data keep their existing durability. The default skip set rejects generic generated/dependency/build trees; `file_explorer.index_exclude_paths` adds machine/project-specific rules, one per line: plain absolute/home-relative subtree paths, `glob:<root-relative glob>`, or `regex:<expression>` against root-relative POSIX paths. Invalid regex rules are ignored, and the complete effective rule set participates in the index signature so policy changes rebuild rather than reuse incompatible rows. `/api/fs/index-status` and the runtime report expose build/full/incremental counts, last duration, scanned/ignored entries, dirty subtrees, truncation, cache bytes, and write bytes. The browser keeps `/api/background/status` fresh so the top-right ownership chip can show whether this connected process is the background owner/follower, while `--print-runtime-report` shows the actual `indexd`, `statsd`, `jobd`, and `approvald` service rows. `background_owner_changed` pushes update that status after election changes, and the status is refreshed on EventSource reconnect and manual refresh.

If a background owner dies or stops heartbeating and its process lock is gone, a follower heartbeat observes that the old generation is no longer live, acquires `owner.lock`, writes a new `owner.json`, adopts warm session-files, Tabber activity, and search-index disk snapshots into memory, and publishes `background_owner_changed` so connected clients update without waiting for a poll. If the owner heartbeat/control socket is stale but the flock is still held by a live process, a follower must not steal ownership because that would split coordination; it may run only the existing bounded immediate-response fallback and then return to follower mode. Local service ownership remains guarded separately by service locks and SQLite single-writer rules, so owner failover must not create a second `statsd` or `indexd` writer.

YO auto-approve is not owned by `background-owner`. `approvald` uses per-target flock files under `STATE_DIR/locks/auto-approve-<target>-<digest>.lock`, where the target is usually the agent pane target. That means the background owner can be one YOLOmux process while YO-on for a specific tmux pane is held by `approvald`; the auto-approve API probes and takes over those per-target locks separately from background ownership and indexing.

- `GET /api/auto-approve` returns YOLO status for all sessions.
- `GET /api/auto-approve?session=project1` returns YOLO status for one session.
- `POST /api/create-session?agent=claude`, `POST /api/create-session?agent=codex`, or `POST /api/create-session?agent=term` creates the next numbered tmux session with the selected agent, capped at `MAX_YOLOMUX_SESSION_TABS` (99) visible sessions.
- `POST /api/ensure-session?session=project1` checks that a tmux session still exists.
- `POST /api/auto-approve?session=project1&enabled=1` enables or disables YOLO for a session.
- `POST /api/tmux-next?session=project1` moves the session to the next tmux sub-window.
- `GET /ws?session=project1` attaches a browser terminal to tmux.

## Wall API

The read-only wall (`tools/tmux_wall.py`) exposes:

- `GET /api/snapshot` returns the current six-pane dashboard payload.
- `GET /api/transcript?target=project1:0.0&lines=2000` returns one tmux pane transcript.
- `GET /api/summary-input?lines=1200` returns the active dashboard panes and container metadata as one JSON payload.

These wall endpoints do not call an LLM. They are the stable input surface for a later summarizer.
